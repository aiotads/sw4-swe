# 第 10 章｜Documentation（文件）

> *"Documentation is hugely important over time and scale. … Write for your audience, not yourself."*
> 「文件在時間與規模兩個維度上都極其重要。……為你的讀者而寫，別只為自己寫。」

## 摘要

幾乎每位工程師都抱怨過「文件品質差、數量少、根本沒有」。本章的核心診斷是：**2010 年代末工程文件的處境，像極了 1980 年代末的軟體測試**——大家都知道該投入更多心力，但組織還沒真正認識到它的關鍵價值。Google 最成功的做法，是**把文件當成程式碼來對待（treat documentation like code）**，納入既有的工程工作流，讓寫與維護都更容易。

「文件」泛指工程師為了做好工作而寫的一切補充文字——不只獨立文件，**程式碼註解也算**（事實上 Google 工程師寫的文件，多數是程式碼註解）。文件的效益**都在下游**：你只寫一次，但會被讀數百、甚至數千次，初始成本被攤提到所有未來讀者身上——這就是它在時間與規模上的價值。但也因為效益不立即（尤其對作者），加上工具支援不足、被視為額外負擔，文件常被工程師輕視。書中最重要的心法是 C++ 風格指南的格律：**「為讀者最佳化（optimize for the reader）」**。

全章圍繞幾個可立即套用的主張：**文件要有 owner、進版本控制、走 code review、用 bug 系統追蹤、定期檢查（freshness date）**；寫之前先**認清受眾**（seekers 要一致性、stumblers 要清晰；customer 與 provider 的文件要分開）；一份文件只服務**單一目的**；五種主要文件類型（reference／design docs／tutorials／conceptual／landing pages）各有寫法；好文件在 **completeness（完整）、accuracy（準確）、clarity（清晰）** 三者間取捨，很難三者兼得。結論坦承：**文件在 Google 還不是「一等公民」**，但工程師必須認知「自己既是問題、也是解方」。

## 為什麼要讀這一篇

- **你也覺得「寫文件」是浪費時間、是寫程式以外的雜事。** 這章用「攤提」的角度翻轉它：文件**寫一次、被讀上千次**，成本分攤到所有讀者；它和測試一樣，是對未來的投資。看懂這點，你會願意付那筆前期成本。
- **你的團隊 wiki 早已腐爛——一堆過時、重複、沒人維護的頁面。** GooWiki 的崩壞（**約 90% 文件數月無人瀏覽或更新**、文件品質一度成為 Google **開發者年度調查的頭號抱怨**）就是你的前車之鑑。解方很具體：**把文件搬進版本控制、給它 owner、走 code review、用 bug 系統追蹤**。
- **你寫文件時其實只寫給「現在的自己」。** 本章最常被引用的錯誤就是這個。學會先問「**我的讀者是誰**」（seekers／stumblers、expert／novice、customer／provider），並用 WHO/WHAT/WHEN/WHERE/WHY 框住開頭兩段，你的文件會立刻好用很多。
- **你想知道什麼時候該找 technical writer。** 答案違反直覺：**團隊幾乎都能寫好「給自己」的文件**，真正需要協助的是**寫給「另一個受眾」**、跨 API 邊界的文件——technical writer 的關鍵價值，是「以局外人身分挑戰你團隊對自家專案有用性的假設」。

## 細節重點

### Why Is Documentation Needed?（為什麼需要文件）
核心主張：好文件讓程式與 API 更易懂、減少錯誤；讓團隊目標清楚、讓 onboarding 省力。但效益**都在下游**、不立即回饋作者——這正是它和測試的差別（測試很快回饋程式設計師）。然而**投資文件會隨時間自行回本**：你只寫一次，卻會被讀數百數千次，初始成本攤提到所有未來讀者。它回答的是「**為什麼**當初這樣設計／這樣實作」——包括兩年後回看自己程式的你。

工程意涵：即使對作者本人，寫文件也有四個即時好處——**①幫你把 API 想清楚**（「如果你解釋不了、定義不了，大概就是還沒設計好」）；②留下維護路線圖與歷史紀錄；③讓程式看起來更專業（「文件好」通常是「維護得好」的良好指標）；④**減少別人來問你問題**（同一件事要解釋超過一次，就該寫成文件）——這是長期最大的好處。但**絕大多數效益歸於讀者**，所以一切回到那句格律：**為讀者最佳化**。

### Documentation Is Like Code（把文件當程式碼對待）
核心主張：文件就是一種**工具**，只是用另一種語言（通常是英文）寫成，目的和程式碼裡的規範相同：**強制一致、提升清晰、避免（理解上的）錯誤**。所以文件該盡量**比照程式碼**處理：有內部規則、進版本控制、有清楚的 owner、變更走 review（並隨它所描述的程式一起改）、用 bug 系統追蹤問題、定期評估（某種意義上的「測試」）、可能的話衡量其準確度與新鮮度（此處工具仍未跟上）。

工程意涵：沒有 owner 的文件會腐爛、難維護。當多份文件衝突時，要指定 **canonical documentation（權威文件）**：確立主要來源，把其他併入或棄用重複者。Google 用 **`go/` 連結**（見 [[ch03]]）與「**把文件放進原始碼、與程式並列**」兩招來扶植權威文件。工程師越把寫文件當成軟體開發的「必要任務之一」，就越不會排斥前期成本、越能收割長期效益。

#### Case Study：The Google Wiki（GooWiki 的崩壞）
核心主張：早期 Google 小而精時，用內部 wiki（**GooWiki**）共享資訊看似合理。但隨規模成長問題浮現：**沒有真正的 owner →文件過時**；沒有新增文件的流程 →**重複文件激增**；扁平命名空間、缺乏階層（光是設定 Borg 生產運算環境就有 **7~10 份**文件、多半各自為政）。更致命的是：**能修文件的人不是用文件的人**——新使用者發現文件壞了卻無從確認或回報，而最有能力修的人寫完後根本不需再看。文件品質惡化到一度成為 **Google 開發者年度調查的頭號抱怨**。

工程意涵：解方是**把重要文件搬進和程式碼一樣的版本控制**——文件有了 owner、在原始碼樹裡有 canonical 位置、有回報與修復 bug 的流程，品質就**戲劇性改善**。引進 **Markdown**（讓人不必懂 HTML/CSS 就能編輯）與 **g3doc**（把文件嵌在程式碼旁、在開發環境裡並列）後更好——工程師能在**同一個變更裡**同時改程式與文件。關鍵不是建立新流程，而是**沿用既有的開發者工作流**（filed bugs、changelists、專家 review）。

### Know Your Audience（認清你的受眾）
核心主張：工程師寫文件**最重要的錯誤就是只寫給自己**。組織越大，你的假設越站不住，連「少數迷路的讀者」都是大成本。動筆前先（正式或非正式地）**辨識主要受眾**並對其書寫：設計文件要說服決策者、tutorial 要對全然陌生者給極明確的步驟、API reference 要對專家與新手都完整準確。

好文件**不必精雕細琢或「完美」**——以為「得是更好的作者才能寫」是個迷思（照這標準沒幾個工程師能寫了）。**「如果你能讀，你就能寫。」** 記住：你的讀者站在你曾站過的地方，只是還沒有你現在的領域知識；你只需要讓「一個像你的人」變得和你現在一樣熟悉即可。

工程意涵：受眾依**經驗（專家／新手）、領域知識（隊友／只懂 API endpoint 的外人）、目的（要快速找資訊的終端使用者／要維護艱澀實作的高手）** 分層。訣竅多半是**寫得盡量廣納各群**：對專家可以走捷徑、卻會搞糊塗新手；對新手鉅細靡遺、又會惹惱專家。最有用的單一技巧是**讓文件保持精短**——而寫出短文常需要先寫長（把資訊蒐齊），再做一次刪修（去掉重複）。Pascal：「**要是我有更多時間，這封信會寫得更短。**」此外用兩種「相遇方式」區分讀者：**seekers（搜尋者）** 知道自己要什麼，關鍵在**一致性**（reference 格式統一好掃讀）；**stumblers（誤撞者）** 還不確定要什麼，關鍵在**清晰**（在檔頭給概觀、甚至用 TL;DR 標明「不感興趣可停讀」）。最後，**customer（API 使用者）與 provider（專案成員）的文件要盡量分開**——實作細節對維護者重要，終端使用者不該需要讀到。

### Documentation Types（文件的類型）
核心主張：工程師會寫多種文件——**reference（含程式碼註解）、design docs、tutorials、conceptual、landing pages**。重點是**知道類型差異、不要混用**：一份文件通常該有**單一目的**並堅守它（就像 API 該「只做一件事、把它做好」）。早期 Google 常見的「巨型 wiki 頁」把連結、概念、API reference 全攪在一起，正因不服務單一目的而失敗（且長到沒人讀）。

- **Reference Documentation（參考文件，含程式碼註解）：** 最常見、幾乎每天都在寫。註解分兩派：**API 註解**（不必談實作細節或設計決策，不能假設讀者像作者一樣熟）vs **實作註解**（可假設較多領域知識，但別假設太多——人會離開專案）。reference 應盡量**單一來源（single-sourced）**、從程式碼裡的註解生成（Javadoc/PyDoc/GoDoc；C++ 因 API 介面在 `.h`、實作在 `.cc`，慣於在 header 寫 reference）。Google 因 Code Search（[[ch17]]）夠強，幾乎不另外產生獨立 reference。**好註解**：**file comment** 概述內容、用途、目標讀者（「第一兩段就描述不清的 API，通常代表沒設計好」）；**class comment** 名詞化描述物件；**function comment** 用**祈使動詞開頭**（Merges, Deletes, Creates…）帶來一致性，讓 seeker 掃過動詞就知道合不合用。Google **不採** `Returns:`／`Throws:` 之類樣板——一段自然散文把後置條件、參數、回傳、例外**一起**講，常更清楚。
- **Design Docs（設計文件）：** 多數團隊在啟動重大專案前**要求一份核准的設計文件**（常用團隊核可的模板、放在協作良好的 Google Docs、在會議上被專家討論——某種意義上是「**寫程式之前的 code review**」）。模板強制考慮 **security、internationalization、storage、privacy** 等面向（由各領域專家審）。好設計文件涵蓋**目標、實作策略、關鍵設計決策與其取捨、替代方案的優劣**；核准後既是**歷史紀錄**，也是日後**衡量專案是否達成目標**的標尺。
- **Tutorials（教學）：** 加入新團隊時最想快速上手——一份**「Hello World」**文件無價。寫 tutorial 的最佳時機就是**你剛加入團隊時**（也最容易抓出既有 tutorial 的 bug）：邊做邊記下每一步、**不假設任何 setup／權限／領域知識**，事後再精簡。**務必明確編號每個步驟**（「沒有比『第 4 步出錯，因為你忘了叫人先授權帳號』更惱人的事」）；把使用者的原子操作**合併成單一步驟**、別把「系統的回應」也編號；輸入輸出用等寬粗體另起一行。書中給了「壞 tutorial → 改良」的對照例（原本步驟 4、5 其實發生在 server 端、6 與 7 根本是同一件事）。
- **Conceptual Documentation（概念文件）：** 提供 API/系統的概觀，**augment（補充）而非取代** reference。**最難寫、也最常被忽略**。常無法嵌進原始碼（沒有 canonical 位置）。它要對專家與新手都有用，必須**犧牲完整性（留給 reference）、有時也犧牲嚴格準確性，以換取清晰**——重點是「**傳達理解**」。一句精妙的類比：**「如果註解是文件的 unit test，概念文件就是 integration test。」**（例：Abseil 的 `StrFormat` 另附一份 format concepts 文件。）
- **Landing Pages（著陸頁／團隊首頁）：** 常一團亂。修法其實簡單：**明確標示用途，只放通往其他頁的連結**——「如果著陸頁上的東西不只是在當交通警察（traffic cop），它就沒做好本分」。連結太多就按 taxonomy 分區。**別讓一頁服侍兩個主人**（既當 API 使用者的入口、又當團隊首頁）——把「team page」獨立出來，因為團隊要知道的和顧客要知道的往往很不同。

### Documentation Reviews（文件審查）
核心主張：程式都要 review，文件**一般也該 review**（只是這共識較不普及）。想「測試」文件有沒有用，就找別人讀讀看。一份技術文件受益於**三種不同的審查**：**technical review（求準確）**——由領域專家做，常就是 code review 的一部分；**audience review（求清晰）**——由不熟該領域者（新人或 API 顧客）做；**writing review（求一致）**——常由 technical writer 或志願者做。高曝光或將對外發布的文件，應收到更多種審查。重點：**只要有人 tied 進工程工作流，文件就會隨時間改善**——多數文件如今隱性地通過了「audience review」，因為它的受眾終會使用它、並（但願）在不管用時回報。

### Documentation Philosophy（文件哲學）
> 此節作者自陳偏「技術寫作最佳實務與個人意見」，非「Google 怎麼做」，對工程師為選讀。

- **WHO / WHAT / WHEN / WHERE / WHY：** 多數技術文件回答 **HOW**（怎麼運作／怎麼用這個 API／怎麼架這台 server），工程師常一頭栽進 HOW 而忽略其他。雖然這幾個一般都不如 HOW 重要（**設計文件例外——對應的 WHY 往往同等重要**），但少了框架文件會令人困惑。**在開頭兩段內交代**：WHO＝受眾（有時要明寫「本文給 Secret Wizard 專案的新工程師」）；WHAT＝本文目的（寫出 WHAT 常能幫你框住範圍）；WHEN＝建立／審查／更新日期；WHERE＝該放哪（最好進版本控制、與程式並列）；WHY＝讀完該帶走什麼（在引言立下、寫摘要時驗證是否達標）。
- **Beginning, Middle, and End（開頭、中段、結尾）：** 聽來蠢，但多數文件至少該有這三段。只有一段的文件只說了一件事，而**很少有文件只有一件事要說**。別怕加 section——它把內容切成有邏輯的塊、給讀者一張地圖。**在文件裡，冗餘（redundancy）常是有用的**：把重點先在引言介紹並摘要，再用其餘篇幅詳述，能幫讀者記住其重要性。
- **The Parameters of Good Documentation（好文件的三參數）：** 好文件有三面向——**completeness（完整）、accuracy（準確）、clarity（清晰）**，但**很難三者兼得**：想更完整，清晰常受損；對程式語言而言，把所有副作用都記全也會傷清晰。所以「好文件」的定義是**「正在做它該做的那件事」**——故很少需要一份文件做超過一件事。寫概念文件就不必涵蓋全部 API；寫 reference 就要完整、但或許得犧牲一點清晰；寫 landing page 就專注組織、把討論降到最低。快速提升品質的法門：**聚焦受眾需求，常常「少即是多」**（例：別把設計決策塞進 API 文件——使用者不需要）。

### Deprecating Documents（棄用文件）
核心主張：舊文件和舊程式一樣會出問題。當文件不再有用，**要嘛移除、要嘛標為過時**（並指向新資訊）——即使是無主文件，有人加一句「**這個已經不能用了！**」也勝過留著一份看似權威、實則失效的東西。Google 常用 **freshness date（新鮮度日期）**：記錄上次審查時間，metadata 會在文件**三個月**未動時寄 email 提醒；owner 有動機保持日期更新（若在版本控制下，更新就需一次 code review），是**低成本**確保文件定期被看的辦法。加上「**Last reviewed by…**」署名後，採用率也提高了。

### When Do You Need Technical Writers?（何時需要技術寫手）
核心主張：technical writer 是**稀缺資源**。早年「重要專案就配一個 writer」的假設被證明是錯的——它製造了**反向誘因**：成為重要專案，你的工程師就不必寫文件了，而「打消工程師寫文件的念頭」恰恰是你最不該做的事。實際學到的是：**團隊幾乎都能寫好給「自己」的文件**（回饋迴路即時、假設清楚），真正需要協助的是**寫給「另一個受眾」、跨 API 邊界**的文件。technical writer 的關鍵角色，是**以「不熟領域的局外人」身分，挑戰你團隊對自家專案有用性的假設**——支援單一團隊「不scale」，把他們的專長用在跨邊界文件才對。

工程意涵（結論）：相較於測試（工程師已普遍接受「再小的變更也要測」、工具完備且嵌入工作流），**文件在 Google 還不是一等公民**。部分原因是文件**本質主觀**——品質不由作者、而由讀者衡量，且常是非同步的；測試可原子化、可自動化，文件難。但組織已認知其重要、流程在改善。要改變文件品質，工程師與整個組織都得接受：**自己既是問題、也是解方**——對任何你預期能活超過幾個月的程式，投入文件的額外週期不只幫別人，也幫你自己維護。

> **EN:** "If comments are the unit tests of documentation, conceptual documents are the integration tests."

> *語境/註解：Conceptual Documentation——以測試金字塔類比文件層次：註解貼著單一程式（單元），概念文件描述跨 API/模組的整體行為（整合）。*

> **EN:** "If I had more time, I would have written you a shorter letter." (Blaise Pascal)
> **ZH:** 「要是我有更多時間，這封信會寫得更短。」（巴斯卡）
> *語境/註解：Know Your Audience——短而清晰的文件能同時滿足專家與新手；寫短常需先寫長再刪修，這成本攤提給所有讀者很划算。*

> **EN:** "Write for your audience, not yourself."
> **ZH:** 「為你的讀者而寫，別只為自己寫。」
> *語境/註解：TL;DRs——全章對「最常見錯誤」的一句總結；先辨識受眾（seekers/stumblers、customer/provider）再下筆。*

> **EN:** "They are both the problem and the solution."
> **ZH:** 「他們既是問題、也是解方。」
> *語境/註解：Conclusion——要改善文件品質，工程師與整個組織必須認知這一點；產出好文件是工作的一部分，長期替自己省時省力。*

## 關鍵字補充

- **Treat Documentation Like Code（把文件當程式碼對待）** — 本章核心方法論：文件要有 owner、進版本控制、走 code review、用 bug 系統追蹤、定期評估。沿用既有開發者工作流，而非另建新流程。
- **Optimize for the Reader（為讀者最佳化）** — 取自 C++ 風格指南的格律；文件絕大多數效益歸於讀者，寫一次被讀千次，故一切以讀者視角為準（呼應 [[ch08]]）。
- **Canonical Documentation（權威文件）** — 當多份文件衝突時指定的唯一主要來源；用 `go/` 連結與「放進原始碼並列」來扶植，其他重複者併入或棄用。
- **g3doc** — Google 把文件嵌在程式碼旁的框架（Markdown 撰寫、可在 Code Search 瀏覽）；讓工程師能在同一個變更裡同時改程式與文件。
- **Seekers vs Stumblers（搜尋者 vs 誤撞者）** — 兩種讀者：seeker 知道要什麼、關鍵是**一致性**（好掃讀）；stumbler 還不確定、關鍵是**清晰**（檔頭給概觀、TL;DR）。
- **Customer vs Provider（顧客 vs 提供者）** — API 使用者 vs 專案成員；兩者的文件要分開，實作細節與設計決策不該塞進 API 文件。
- **API Comments vs Implementation Comments（API 註解 vs 實作註解）** — 前者不談實作/設計、不假設讀者熟 API；後者可假設較多領域知識（但別假設太多，人會離開專案）。
- **Reference / Design / Tutorial / Conceptual / Landing（五種文件類型）** — 各有單一目的，不可混用；conceptual 最難寫、補充而非取代 reference，犧牲完整與準確換清晰。
- **Completeness / Accuracy / Clarity（完整／準確／清晰）** — 好文件的三參數，難三者兼得；「好文件」＝「正在做它該做的那件事」，故一份文件別做超過一件事。
- **Freshness Date（新鮮度日期）** — 文件記錄上次審查時間、逾期（如三個月）自動寄提醒；配「Last reviewed by…」署名，低成本確保文件被定期檢視。
- **WHO/WHAT/WHEN/WHERE/WHY** — 多數技術文件只答 HOW；在開頭兩段補足這五問可避免混淆（設計文件的 WHY 與 HOW 同等重要）。

## 延伸提問

1. 把你最近寫的一份文件攤開：它其實是**寫給「現在的你」還是寫給「讀者」**？它的主要受眾是 seeker 還是 stumbler、是 customer 還是 provider？如果你在開頭兩段補上 WHO/WHAT/WHY，它會不會立刻好用很多？
2. 盤點你團隊的文件現況：有多少像 GooWiki 那樣**沒有 owner、過時、重複**？哪幾份其實可以**搬進版本控制、給 owner、走 code review、加 freshness date**，讓它「像程式碼一樣」被維護？（對照 [[ch15]] 棄用：哪些該直接標為過時或移除？）
3. 你的 API 文件裡有沒有混進**設計決策或實作細節**？把它們挪到 design doc，API 文件會不會更清晰？你的文件有沒有試圖「一頁服侍兩個主人」（landing page 同時當團隊首頁與顧客入口）？
4. 用 **completeness／accuracy／clarity 三角**檢視你最不滿意的一份文件：它想三者兼得卻什麼都沒做好嗎？如果明確選定它的**單一目的**（這是 reference 還是 conceptual？），你會犧牲哪一項來成全另一項？
5. 你的團隊是不是把「找 technical writer」當成寫文件的萬靈丹？對照本章：你們真正需要外援的，是**跨 API 邊界、寫給另一個受眾**的文件嗎？還是其實「給自己的文件」團隊自己就能寫好，只是缺少把它當程式碼維護的紀律？

- [[ch09]] Code Review — 文件變更走 code review、文件的 technical review 常就是 code review 的一部分；「把文件當程式碼」最關鍵的一步就是讓它走同一套審查流程。

- [[ch15]] Deprecation — 「棄用文件」與「棄用程式碼」同理：當文件不再有用就移除或標為過時，freshness date 是文件版的生命週期管理。
