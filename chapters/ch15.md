# 第 15 章｜Deprecation（棄用）

> *"Removing things is often more difficult than building them to begin with, because existing users are often using the system beyond its original design."*
> 「移除一個東西，往往比當初建造它還難——因為既有使用者經常以超出它原始設計的方式在用它。」

## 摘要

**所有系統都會老。** 即使位元不會腐朽，新技術、函式庫、語言與生態演進也會讓舊系統過時——而舊系統需要持續維護、艱澀的專業知識、且隨生態偏離而愈來愈費工。**Deprecation（棄用）** 就是「有序地遷移離開、並最終移除過時系統」的流程。它屬於軟體工程（管理系統跨時間）而非單純的程式設計。本章的根本前提呼應 [[ch09]]：**程式碼是負債，不是資產**——帶來價值的是它提供的「功能」；故我們該最大化的不是程式碼的量，而是**「每單位程式碼能交付多少功能」**，而移除不再需要的程式與系統，正是最容易的做法之一。

**棄用為什麼這麼難？** ①**Hyrum's Law**：使用者愈多，愈可能有人以未預期的方式依賴它——移除等於「終極的變更」，會抖出一堆意外的依賴者；②新系統必然**不同**（否則沒有遷移誘因），故與舊系統一對一對應很罕見；③**情感依附**（「我喜歡這段程式！」）；④**政治與經費**（讓系統苟延殘喘的成本不易觀察，難說服 stakeholder——可用 [[ch07]] 的研究方法提供證據）。因此**在原地漸進演進（in-place refactoring）通常比完全替換便宜**——Google 發現整體遷移到全新系統極貴、且成本常被低估。

棄用最好**在設計時就規劃**（如核電廠設計時就把除役納入、甚至預撥經費）——問兩個問題：使用者多容易遷移到替代品？我能不能漸進替換各部分？棄用是一個**光譜**，兩端是 **advisory（建議性，無期限、無強制，「hope is not a strategy」）** 與 **compulsory（強制性，有期限與強制機制、且必須由專門團隊配人力執行到底）**。最後談管理：**要有明確的 process owner**（否則不會有進展）、設**漸進里程碑**（別只把「完全移除」當唯一里程碑）、用三類工具（**discovery 發現／migration 遷移／backsliding prevention 防回潮**）。棄用警告必須同時**actionable（可行動）與 relevant（相關，在對的時機出現）**，否則造成 **alert fatigue（警告疲勞）**。

## 為什麼要讀這一篇

- **你的程式碼庫塞滿「沒人敢關、也沒人維護」的舊系統。** 本章告訴你為何該關（**程式碼是負債**、多系統並存拖慢新系統演進）、以及為何那麼難關（Hyrum's Law、情感依附、政治經費）。看懂後，「移除」會變成你工具箱裡正當的一招。
- **你想推同事遷移到新系統，卻只在舊的上面貼個 deprecated 警告就走人。** 本章直接點破：「**hope is not a strategy（希望不是策略）**」——advisory deprecation 鮮少真的讓人遷移；舊系統會持續對新用法產生「概念/技術上的拉力」。真要完成，得靠 **compulsory deprecation＋專門團隊配人力**。
- **你被指派去做一個沒人想做、看不到進度的「清理」專案。** 本章給你管理它的方法：**設漸進里程碑**（刪掉一個關鍵子元件就該慶祝，別只等最後關燈）、**集中專業到一個團隊**、把它當「重要但不緊急」的 20% 時間好題材。
- **你在設計新系統。** 本章提醒你一個少見卻關鍵的視角：**為「未來的除役」而設計**（像核電廠）——現在就問「使用者多好遷移走？能不能漸進替換？」，並謹記「**別啟動一個組織不打算支撐其預期壽命的專案**」。

## 細節重點

### Why Deprecate?（為何棄用）
核心主張：起點是 **「程式碼是負債，不是資產」**。程式有成本——建造時的、以及更多在**維護期間**（運行的營運資源、隨生態演進持續更新的工夫）承擔的成本。**系統老不等於過時**（LaTeX 歷經數十年改良仍非過時）；棄用最適合「**可證明已過時、且存在功能相當的替代品**」的系統（新系統可能更省資源、更安全、更可維護或修了 bug）。兩個系統做同一件事看似不急，但**長期維護成本會大幅成長**：它們可能得互相介接（需複雜的轉換碼）、互相依賴而更難移除；**多系統並存還會拖慢新系統的演進**（因它仍被要求與舊系統相容）。

工程意涵：別執著於「寫了多少程式、程式碼庫多大」，而要最大化**「每單位程式碼交付的功能」**——最容易的辦法之一不是寫更多程式，而是**移除不再需要的程式與系統**。但要注意：組織能**同時進行的棄用工作有上限**（道路鋪設比喻：若所有路同時封閉施工，誰都動不了；集中火力反而更快）——故要審慎選棄用專案、並承諾把它做完。

### Why Is Deprecation So Hard?（為何這麼難）
核心主張：
- **Hyrum's Law（見 [[ch01]]）：** 使用者愈多，愈可能有人以未預期的方式依賴它——他們的用法是「碰巧能動」而非「保證能動」。移除是**終極的變更**（不只改行為，而是完全拿掉），會抖出一堆意外的依賴者。
- **新系統必然不同：** 棄用通常要等到有了功能相當（或更好）的新系統才可能；但「更好」也意味「不同」（否則沒遷移誘因）——故一對一對應罕見，舊系統的每個用法都得在新系統脈絡下重新評估。
- **情感依附：** 一種 change aversion——「我喜歡這段程式！」很難說服人拆掉自己花數年建的東西。Google 的對策之一：讓 source repo **不只能搜尋 trunk、也能搜尋歷史**（被移除的程式仍找得回來，見 [[ch17]]），減輕「丟掉就沒了」的焦慮。
- **政治與經費：** 配人力、花時間移除過時系統要花真金白銀，而「什麼都不做、讓系統苟延殘喘」的成本卻不易觀察——難說服 stakeholder，尤其當它衝擊新功能開發時。可用 [[ch07]] 的研究方法提供具體證據。

工程意涵：鑑於棄用之難，**在原地漸進演進（in-place refactoring）通常比完全替換便宜**——漸進不能免除棄用流程，但能把它拆成更小、可管理、能持續產出價值的區塊。Google 發現整體遷移到全新系統**極貴、且成本常被低估**。（書中老笑話：Google 有兩種做事方法——「已被棄用的」與「還沒準備好的」。）

### Deprecation During Design（在設計時規劃棄用）
核心主張：棄用可在系統初建時就規劃——語言、架構、團隊組成、甚至公司政策與文化都影響「日後移除有多容易」。**「為了能被棄用而設計」在軟體界或許激進，在其他工程領域卻常見**：核電廠在設計時就把「服役一生後的除役」納入考量、甚至預撥經費。可惜軟體少有這般深思（工程師多愛建新系統而非維護舊的；企業文化獎勵快速出貨；且為自己心血規劃「終局」在心理上也難）。鼓勵團隊問兩個問題：**①我的使用者要從我的產品遷移到潛在替代品有多容易？②我能不能漸進替換系統的各部分？**（多與「如何提供與消費依賴」有關，見 [[ch16]]。）

工程意涵：**是否長期支撐一個專案，這決定在「組織決定建它」的那一刻就做下了**。系統存在後只剩三選項：支撐它、小心地棄用它、或讓它在某外部事件弄壞它時停止運作。一句話：**別啟動一個你的組織不打算支撐其預期壽命的專案**——即使最終選擇棄用移除仍有成本，但可透過規劃與工具/政策投資來緩解。

### Types of Deprecation（棄用的兩種類型）
核心主張：棄用不是單一流程，而是一個**光譜**——從「我們希望某天關掉它」到「這系統明天就消失，客戶最好準備好」。大致分兩區：
- **Advisory（建議性）：** 無期限、非高優先（組織不願投入資源），亦稱 aspirational（願景式）。常**缺乏強制**——「我們希望客戶遷移，但無法逼他們」（如 SRE 所言：「**Hope is not a strategy.**」）。它擅長**宣傳新系統存在、鼓勵早期採用者試用**（但新系統不該被當 beta——它得已能承擔生產負載、且準備好無限期支撐新使用者）。但**好處必須是「變革性」而非「漸進性」**：使用者不會為邊際好處自行遷移。Advisory 能「**輕推**」使用者往期望方向，但**不能指望它完成大部分遷移工作**——只貼個警告就走人，鮮少讓人主動遷移；舊系統會持續對新用法產生「概念/技術上的拉力」。
- **Compulsory（強制性）：** 這種「主動鼓勵」通常帶**移除期限**（過期還依賴它，你的系統就會壞）。反直覺地，**強制棄用要能擴展，最好把遷移專業集中在「單一專家團隊」**（通常就是負責徹底移除舊系統的團隊）——他們有誘因幫人遷移、能發展出可全組織重用的經驗與工具（常用 [[ch22]] 的 LSC 工具）。要真正有效，**排程需要強制機制**：授權執行團隊在充分警告後可以「弄壞」不合規的使用者，否則客戶團隊會把棄用工作擱在功能之後。但**沒有配人力的強制棄用**會被客戶當成「無經費的強制令（unfunded mandate）」、顯得刻薄、像「原地跑步」——故**強烈主張強制棄用要由專門團隊配足人力做到完成**。即便有政策撐腰仍可能遇政治阻礙（若最後一個依賴者是全組織仰賴的關鍵基礎設施，你願意為了一個任意期限而弄壞它嗎？）。發現未知依賴的辦法：**安排逐次拉長的計畫性停機**（類似 DiRT）、或更改「僅供實作用」符號的名字看誰會壞。

#### Deprecation Warnings（棄用警告）
核心主張：用程式化的方式標記系統為 deprecated 很有用，但「標了就 hope 它消失」行不通。警告會隨時間累積，尤其在遞移情境（A→B→C，C 發警告卻冒在 build A 時），多到使用者乾脆全部忽略——醫療界稱此為 **alert fatigue（警告疲勞）**。任何棄用警告都需兩個性質：**①actionability（可行動）**——使用者能據以實際採取相關行動（如「把對某函式的呼叫換成更新版」、或 email 列出搬資料的步驟）；**②relevance（相關）**——在使用者「正要做該動作」時才浮現（在工程師寫用到該函式的程式時警告，而非 check-in 數週後；資料遷移 email 在移除前數月寄、而非前一個週末才補）。要克制「對所有東西都貼警告」的衝動——Google 雖很自由地標 deprecated，但用 ErrorProne／clang-tidy 把警告**限定在新改動的行**（見 [[ch20]]），更具侵入性的警告（如依賴圖裡的 deprecated 目標）只給強制棄用、且團隊正主動遷移使用者時才加。

### Managing the Deprecation Process（管理棄用流程）
核心主張：棄用專案的管理與其他軟體專案相似，但有幾處不同：
- **Process Owners（流程負責人）：** 沒有明確 owner，棄用幾乎不會有實質進展（不管系統發多少警告）。替代方案更糟：要嘛永不棄用任何東西（＝承諾永遠維護每個舊系統），要嘛把棄用委派給使用者（＝退化成 advisory，永遠不會有機地完成）。**集中棄用努力**才能讓專業真正降低成本。**被遺棄的專案**（原 owner 已轉去後繼專案、舊的還在地下室苟延、還是某關鍵專案的依賴）不會自己消失，仍需棄用專家來移除——這些團隊應**以「移除」為主要目標**而非副業（否則在優先序競爭中必敗）。這類「重要但不緊急」的清理是 **20% 時間**的好題材。
- **Milestones（里程碑）：** 蓋新系統的里程碑很清楚；但棄用常感覺「唯一的里程碑是完全移除舊系統」——團隊在關燈回家前都覺得毫無進展，而那一刻又因為早已無使用者而最不被外界注意。**別把「完全移除」當唯一可衡量里程碑**（它甚至未必發生）；要設**具體、可衡量、對使用者交付價值的漸進里程碑**，並像慶祝新產品成就一樣慶祝（如刪掉一個關鍵子元件）。
- **Deprecation Tooling（工具）：** 分三類——**Discovery（發現）：** 用 Code Search（[[ch17]]）與 **Kythe** 靜態判定誰用了某函式庫、抽樣看客戶意外依賴了什麼行為；用 logging/runtime sampling 找動態依賴；把**全域測試套件當 oracle**判定舊符號是否還有引用（[[ch11]]）。**Migration（遷移）：** 用 LSC 流程與工具（[[ch22]]）大規模更新程式碼指向新庫/服務。**Backsliding prevention（防回潮）：** 一個常被忽略卻關鍵的部分——若不防，棄用會變成「打地鼠」（使用者不斷加新用法、團隊不斷遷移），既無生產力又打擊士氣。微觀上用 **Tricorder** 在 review 時通知「你正加入對 deprecated 系統的呼叫」並給替代建議（owner 可加 `@deprecated` 等註解控制訊息、有時提供一鍵修正）；宏觀上用 build 系統的 **visibility whitelist** 確保不引入新依賴（並隨遷移自動修剪白名單）。

## 關鍵字補充

- **Deprecation（棄用）** — 有序地遷移離開、並最終移除過時系統的流程；屬軟體工程（管理跨時間）而非程式設計。
- **Code Is a Liability（程式碼是負債）** — 帶來價值的是功能而非程式本身；應最大化「每單位程式碼交付的功能」，移除過時系統是最容易的辦法之一（呼應 [[ch09]]）。
- **Obsolete ≠ Old（過時 ≠ 老）** — 棄用適合「可證明已過時且有功能相當替代品」者；LaTeX 般歷久彌新的系統雖老卻非過時。
- **Advisory Deprecation（建議性棄用）** — 無期限、無強制、非高優先；擅長宣傳新系統、輕推使用者，但不能指望它完成遷移（好處須變革性而非漸進性）。
- **Compulsory Deprecation（強制性棄用）** — 帶移除期限與強制機制；須由專門團隊配足人力做到完成，否則淪為「無經費的強制令」。
- **Hope Is Not a Strategy（希望不是策略）** — SRE 名言；只貼 deprecated 警告而不主動遷移，鮮少奏效。
- **In-place Refactoring vs Replacement（原地重構 vs 完全替換）** — 含 turndown 成本後，原地漸進演進通常比換新系統便宜；整體遷移極貴且常被低估。
- **Actionability & Relevance（可行動性與相關性）** — 好棄用警告的兩性質：能據以採取行動、且在對的時機浮現；否則造成 alert fatigue。
- **Alert Fatigue（警告疲勞）** — 警告（尤其遞移情境）累積到使用者全部忽略；故 Google 把警告限定在新改動的行（ErrorProne/clang-tidy）。
- **Backsliding Prevention（防回潮）** — 防止新增對「正在移除之物」的用法；微觀用 Tricorder＋`@deprecated`，宏觀用 build 系統的 visibility whitelist。
- **Process Owner（流程負責人）** — 棄用必須有明確 owner、以移除為主要目標；否則退化成 advisory 或永久維護。
- **Discovery / Migration / Backsliding（三類棄用工具）** — Code Search/Kythe＋全域測試（發現）、LSC（遷移）、Tricorder/whitelist（防回潮）。

## 延伸提問

1. 盤點你程式碼庫裡「**沒人敢關、也沒人明確擁有**」的舊系統：它們真的是「過時」（有功能相當的替代品）還是只是「老」？用「**每單位程式碼的功能**」這把尺，移除哪一個的投報率最高？對照 [[ch09]]「程式碼是負債」。
2. 你最近一次推遷移，是只貼了個 deprecated 警告就走人（advisory），還是有**期限＋專門團隊配人力**（compulsory）？回想結果——「hope is not a strategy」在你身上應驗了嗎？舊系統的「概念拉力」有多強？
3. 你的棄用警告**可行動且相關**嗎？它有告訴使用者「下一步該做什麼」、且在他「正要寫用到它的程式」時浮現嗎？還是淪為被全部忽略的 **alert fatigue**？能不能像 Google 那樣**限定在新改動的行**？
4. 你正在設計的新系統，有沒有**為未來的除役而設計**？問問核電廠的兩個問題：使用者要遷走有多容易？能不能漸進替換各部分？你的組織真的打算**支撐它的預期壽命**嗎？
5. 假設你被指派一個棄用專案：你會設哪些**漸進里程碑**（而非只等最後關燈）？你會用哪三類工具（discovery/migration/backsliding）？你能不能把遷移專業**集中到一個團隊**、並爭取到把使用者「弄壞」的強制權與人力？對照 [[ch22]] LSC。
