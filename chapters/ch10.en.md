# Chapter 10 — Documentation

> *"Documentation is hugely important over time and scale. … Write for your audience, not yourself."*
> Docs pay off across time and scale — and the reader, not you, is who they're for.

## Summary

Nearly every engineer has complained about documentation — its quality, quantity, or sheer absence. The chapter's core diagnosis: **the state of engineering documentation in the late 2010s resembles software testing in the late 1980s** — everyone knows more effort is needed, but the organization hasn't yet recognized its critical benefits. Google's most successful approach has been to **treat documentation like code** and fold it into the existing engineering workflow, making it easier to write and maintain.

"Documentation" means every supplemental text an engineer writes to do their job — not just standalone documents but **code comments too** (in fact, most documentation a Googler writes is comments). Documentation's benefits are all **downstream**: you write a doc once but it's read hundreds, even thousands of times — its initial cost is **amortized across all future readers**. That's its value over time and scale. But because the benefit isn't immediate (especially to the writer), plus weak tooling and being seen as an extra burden, engineers tend to undervalue it. The guiding maxim, from the C++ Style Guide: **"optimize for the reader."**

The chapter's directly applicable claims: documentation should **have owners, live in source control, go through code review, be bug-tracked, and be periodically checked (freshness dates)**; before writing, **know your audience** (seekers want consistency; stumblers want clarity; keep customer and provider docs apart); a document should serve a **single purpose**; the five main types (reference / design docs / tutorials / conceptual / landing pages) each have their craft; good docs trade off among **completeness, accuracy, and clarity** — you rarely get all three. The conclusion is candid: **documentation isn't yet a first-class citizen at Google**, but engineers must accept they are "both the problem and the solution."

## Why read this

- **You think writing docs is wasted time — a chore outside "real" programming.** This chapter flips that with amortization: a doc is **written once, read thousands of times**, its cost spread across all readers. Like testing, it's an investment in the future. See that and you'll pay the upfront cost.
- **Your team wiki has rotted into stale, duplicate, unmaintained pages.** GooWiki's collapse (**~90% of docs had no views or updates in months**; documentation quality once became Google's **#1 developer-survey complaint**) is your cautionary tale. The fix is concrete: **move docs into source control, give them owners, run code review, track bugs.**
- **You actually write only for present-you.** That's the chapter's most-cited mistake. Learn to ask "**who is my audience**" first (seekers/stumblers, expert/novice, customer/provider), and frame the first two paragraphs with WHO/WHAT/WHEN/WHERE/WHY — your docs get instantly more useful.
- **You want to know when to get a technical writer.** Counterintuitively: **teams can almost always write their own docs fine**; help is needed for docs aimed at **another audience**, crossing API boundaries — the writer's key value is "challenging your team's assumptions about your project's usefulness, as an outsider."

## Key details

### Why Is Documentation Needed?
The core claim: good docs make code and APIs more comprehensible (fewer mistakes), clarify team goals, and ease onboarding. But the benefits are all **downstream** and don't immediately reward the author — the opposite of testing, which pays the programmer back quickly. Still, **the investment pays for itself over time**: written once, read hundreds or thousands of times, with cost amortized across all future readers. Docs answer "**why** was this designed/implemented this way" — including for the you who reads your own code two years later.

Takeaway: even for the author, writing docs has four immediate wins — **① it helps you formulate the API** ("if you can't explain it and can't define it, you probably haven't designed it well enough"); ② it's a maintenance road map and historical record; ③ it makes code look more professional ("good docs" is a decent indicator of "well maintained"); ④ **fewer questions from others** (explain something more than once → document it) — the biggest long-term win. But the **lion's share of benefit accrues to readers**, so it all returns to the maxim: **optimize for the reader.**

### Documentation Is Like Code
The core claim: documentation is just a **tool**, written in another language (usually English) to accomplish the same goals as conventions within code: **enforce consistency, improve clarity, avoid (comprehension) errors.** So treat docs **like code** as much as possible: have internal rules, live under source control, have clear owners, go through reviews for changes (and change *with* the code they document), be bug-tracked, be periodically evaluated ("tested" in a sense), and where possible be measured for accuracy and freshness (tooling hasn't caught up here).

Takeaway: ownerless docs go stale and become hard to maintain. When docs conflict, designate **canonical documentation**: pick the primary source and consolidate (or deprecate) the rest. Google promotes canonical docs via **`go/` links** (see [[ch03]]) and by **placing docs in source control alongside the code**. The more engineers treat docs as "one of" the necessary tasks of development, the less they resent the upfront cost and the more they reap long-term benefits.

#### Case Study: The Google Wiki (GooWiki's collapse)
The core claim: when Google was small and lean, an internal wiki (**GooWiki**) seemed reasonable. As it scaled, problems surfaced: **no true owners → docs went obsolete**; no process for adding docs → **duplicates proliferated**; a flat namespace with no hierarchy (there were **7–10 docs** just on setting up Borg, the production compute environment, mostly unmaintained). Worst: **the people who could fix docs weren't the people who used them** — new users found broken docs but couldn't confirm or report the breakage, while those best able to fix them never needed to re-read them. Quality degraded until it became Google's **#1 developer-survey complaint.**

Takeaway: the fix was moving important docs into the **same source control used for code** — docs got owners, canonical locations in the source tree, and bug-and-fix processes, and quality **improved dramatically.** Introducing **Markdown** (edit without knowing HTML/CSS) and **g3doc** (docs embedded next to code, side by side in the dev environment) helped more — engineers could change code and its docs **in the same change.** The key wasn't a new process but **leveraging the existing developer workflow** (filed bugs, changelists, expert reviews).

### Know Your Audience
The core claim: the most important mistake engineers make is **writing only for themselves.** As the org grows, your assumptions stop holding, and even a few lost readers are a big cost. Before writing, (formally or informally) **identify your primary audience** and write to it: a design doc must persuade decision-makers; a tutorial must give explicit steps to a total newcomer; an API reference must be complete and accurate for experts and novices alike.

Good docs **need not be polished or "perfect"** — the myth that you must be a much better writer would silence nearly every engineer. "**If you can read, you can write.**" Your reader stands where you once stood, minus your new domain knowledge; you just need to get "someone like you" as familiar as you now are.

Takeaway: audiences split by **experience (expert/novice), domain knowledge (teammate / outsider who only knows API endpoints), and purpose (end user needing a quick answer / guru maintaining a hairy implementation).** The trick is usually to **write as broadly as possible**: cutting corners suits the expert but confuses the novice; spelling out everything annoys the expert. The single most useful tactic is to **keep docs short** — and writing short often means writing long first (get it all down), then editing down (cut duplication). Pascal: "**If I had more time, I would have written you a shorter letter.**" Also split readers by how they arrive: **seekers** know what they want — the key is **consistency** (uniform reference formats to scan); **stumblers** don't — the key is **clarity** (overviews at the top, even a TL;DR saying "not interested? stop reading"). Finally, **keep customer docs (API users) apart from provider docs (project members)** — implementation details matter to maintainers; end users shouldn't have to read them.

### Documentation Types
The core claim: engineers write **reference (incl. code comments), design docs, tutorials, conceptual docs, and landing pages.** Know the differences and **don't mix types**: a document should have a **single purpose** and stick to it (like an API that "does one thing well"). Early Google's monolithic wiki pages — links, concepts, API reference all jumbled — failed exactly because they served no single purpose (and grew too long to read).

- **Reference documentation (incl. code comments):** the most common, written almost daily. Comments split into **API comments** (no implementation details or design decisions; can't assume the reader is as versed as the author) vs **implementation comments** (can assume more domain knowledge — but not too much; people leave projects). Reference should be **single-sourced** and generated from in-code comments (Javadoc/PyDoc/GoDoc; C++ documents APIs in `.h` headers since it separates interface from `.cc` implementation). Google rarely generates separate reference because Code Search ([[ch17]]) is strong. Good comments: a **file comment** outlines contents, use cases, and audience ("an API you can't describe in the first paragraph or two usually isn't well thought out"); a **class comment** is "nouned"; a **function comment** starts with an **indicative verb** (Merges, Deletes, Creates…) for consistency, so a seeker can scan the verbs. Google **avoids** `Returns:`/`Throws:` boilerplate — a single prose sentence documenting postcondition, params, return, and exceptions together is often clearer.
- **Design docs:** most teams require an **approved design doc before any major project** (a team-approved template, often in Google Docs for collaboration, sometimes debated in meetings — "a form of code review before any code is written"). The template forces consideration of **security, internationalization, storage, privacy**, each reviewed by domain experts. A good design doc covers **goals, implementation strategy, and key decisions with their trade-offs, plus alternatives and their pros/cons.** Once approved, it's both a **historical record** and a yardstick for **whether the project met its goals.**
- **Tutorials:** new team members want to get up to speed fast — a **"Hello World"** doc is invaluable. The best time to write one is **when you first join** (also the best time to find bugs in existing tutorials): note every step as you go, **assuming no setup/permissions/domain knowledge**, then streamline. **Number every step explicitly** ("nothing is more annoying than an error on step 4 because you forgot to say to authorize their username"); combine atomic user operations into single steps; don't number the system's responses; put user-visible I/O on separate lines (monospaced bold). The chapter shows a "bad → better" tutorial (original steps 4–5 happen server-side; 6 and 7 are the same action).
- **Conceptual documentation:** overviews of APIs/systems that **augment, not replace,** reference. The **hardest to write and most neglected.** Often can't be embedded in code (no canonical home). Must serve experts and novices, **sacrificing completeness (leave that to reference) and sometimes strict accuracy, for clarity** — the point is to **impart understanding.** A neat analogy: "**If comments are the unit tests of documentation, conceptual documents are the integration tests.**" (e.g., Abseil's `StrFormat` ships a separate format-concepts doc.)
- **Landing pages:** often a mess. The fix is simple: **state the purpose, then include only links to other pages** — "if something on a landing page is doing more than being a traffic cop, it's not doing its job." Too many links → break up by taxonomy. **Don't serve two masters** (both an API user's entry point and a team home page) — split out a separate "team page," since what the team needs differs from what a customer needs.

### Documentation Reviews
The core claim: all code is reviewed, and docs **generally need review too** (less universally accepted). To "test" whether a doc works, have someone else review it. A technical doc benefits from **three kinds of review**: a **technical review (for accuracy)** by a subject-matter expert — often part of a code review; an **audience review (for clarity)** by someone unfamiliar with the domain (a newcomer or API customer); a **writing review (for consistency)**, often by a technical writer. High-profile or externally published docs warrant more review types. Key: **once docs are tied into the engineering workflow, they improve over time** — most docs now implicitly get an "audience review" because their audience eventually uses them and (hopefully) reports when they don't work.

### Documentation Philosophy
> The author flags this section as more of a "technical-writing best practices and personal opinion" treatise than "how Google does it" — optional for engineers.

- **WHO / WHAT / WHEN / WHERE / WHY:** most technical docs answer **HOW** (how does it work / how do I use this API / how do I set up this server), so engineers jump straight to HOW and ignore the rest. None is usually as important as HOW (**design docs are the exception — the equivalent WHY is often just as important**), but without framing, docs end up confusing. **In the first two paragraphs**, address: WHO = audience (sometimes call it out: "This document is for new engineers on the Secret Wizard project"); WHAT = purpose (writing the WHAT helps you frame scope); WHEN = created/reviewed/updated date; WHERE = where it lives (ideally source control, alongside the code); WHY = what the reader should take away (set in the intro, verify in the summary).
- **Beginning, middle, and end:** sounds silly, but most docs should have at least these three sections. A one-section doc says only one thing, and **few docs have only one thing to say.** Don't fear adding sections — they break flow into logical pieces and give readers a road map. **In docs, redundancy is often useful:** introduce and summarize a point in an intro paragraph, then make the case in detail — repetition helps the reader grasp importance.
- **The parameters of good documentation:** three aspects — **completeness, accuracy, clarity** — and you **rarely get all three.** More "complete" can hurt clarity; documenting every side effect can too. So a "good document" is defined as **one doing its intended job** — hence rarely should one doc do more than one job. Writing a conceptual doc? You needn't cover the whole API. A reference? Make it complete, perhaps sacrificing some clarity. A landing page? Focus on organization, minimize discussion. To improve quality fast: **focus on the audience's needs — often less is more** (e.g., don't stuff design decisions into an API doc; users don't need them).

### Deprecating Documents
The core claim: old docs cause problems like old code. When a doc no longer serves a purpose, **either remove it or mark it obsolete** (and point to new info) — even for unowned docs, someone noting "**This no longer works!**" beats leaving something that looks authoritative but is broken. Google attaches **freshness dates**: they note the last review and email a reminder when untouched for, say, **three months**; the owner is incentivized to keep the date current (under source control, that requires a code review) — a **low-cost** way to ensure periodic review. Adding a "**Last reviewed by…**" byline raised adoption.

### When Do You Need Technical Writers?
The core claim: technical writers are a **scarce resource.** The early assumption "important projects get a writer" proved bad — it created a **perverse incentive**: become an important project and your engineers won't have to write docs, which is exactly the opposite of what you want. The lesson: **teams can write their own docs fine** (immediate feedback loop, clear assumptions); help is needed for docs aimed at **another audience, crossing API boundaries.** A writer's key role is to **stand in as an outsider and challenge your team's assumptions about the project's usefulness** — supporting a single team doesn't scale; their specialty belongs on cross-boundary docs.

Takeaway (conclusion): compared with testing (engineers accept that even tiny changes need tests; tooling is robust and embedded in the workflow), **documentation isn't yet a first-class citizen at Google.** Partly because docs are **inherently subjective** — quality is judged by the reader, not the writer, and often asynchronously; tests can be atomic and automated, docs largely can't. But the importance is recognized and processes are improving. To change doc quality, engineers and the whole org must accept they are **both the problem and the solution** — for any code expected to live more than a few months, the extra cycles spent documenting help others *and* help you maintain it.

## Key terms

- **Treat documentation like code** — the chapter's core method: docs get owners, source control, code review, bug tracking, and periodic evaluation. Leverage the existing developer workflow rather than building a new one.
- **Optimize for the reader** — the C++ Style Guide maxim; docs' benefits accrue mostly to readers, written once and read thousands of times, so write from the reader's view (echoes [[ch08]]).
- **Canonical documentation** — the single primary source designated when docs conflict; promoted via `go/` links and "alongside the code," with duplicates consolidated or deprecated.
- **g3doc** — Google's framework for embedding docs next to code (Markdown, browsable in Code Search); lets engineers change code and docs in the same change.
- **Seekers vs stumblers** — two readers: seekers know what they want — the key is **consistency** (scannable); stumblers don't — the key is **clarity** (overviews, TL;DRs).
- **Customer vs provider** — API user vs project member; keep their docs apart, and don't put implementation details or design decisions in API docs.
- **API comments vs implementation comments** — the former skip implementation/design and don't assume API fluency; the latter assume more domain knowledge (but not too much — people leave projects).
- **Reference / design / tutorial / conceptual / landing (the five types)** — each has a single purpose, not to be mixed; conceptual is hardest, augmenting (not replacing) reference, trading completeness/accuracy for clarity.
- **Completeness / accuracy / clarity** — the three parameters of good docs, rarely all three; a "good document" is "one doing its intended job," so one doc shouldn't do more than one job.
- **Freshness date** — a doc records its last review; overdue (e.g., three months) triggers a reminder; paired with a "Last reviewed by…" byline, a low-cost way to keep docs reviewed.
- **WHO/WHAT/WHEN/WHERE/WHY** — most technical docs answer only HOW; covering these five in the first two paragraphs avoids confusion (a design doc's WHY matters as much as HOW).

## Questions to push your thinking

1. Open a doc you recently wrote: was it really written for **present-you or for the reader**? Is its primary audience a seeker or a stumbler, a customer or a provider? If you added WHO/WHAT/WHY to the first two paragraphs, would it get instantly more useful?
2. Audit your team's docs: how many are like GooWiki — **ownerless, stale, duplicated**? Which could be **moved into source control, given owners, code-reviewed, and given a freshness date** to be maintained "like code"? (Per [[ch15]] deprecation: which should just be marked obsolete or removed?)
3. Do your API docs leak **design decisions or implementation details**? Move them to a design doc — clearer? Does any doc try to "serve two masters" (a landing page that's both team home and customer entry point)?
4. Apply the **completeness/accuracy/clarity triangle** to your least-liked doc: is it failing all three by trying for all three? If you pick its **single purpose** (reference or conceptual?), which would you sacrifice to serve another?
5. Does your team treat "get a technical writer" as the cure-all for docs? Per this chapter: do you really need help only for **cross-API-boundary docs written for another audience**? Or could the team write "docs for itself" fine, lacking only the discipline to maintain them like code?
