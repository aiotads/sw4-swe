# Chapter 23 | Continuous Integration

> *"CI is the continuous assembling and testing of our entire complex and rapidly evolving ecosystem."*
> The dependency that breaks you increasingly lives outside your own code — so integrate the whole ecosystem, not just your binary.

## Summary

**Continuous Integration (CI)** is classically "a practice where team members integrate their work frequently, each integration verified by an automated build (including test) to detect integration errors as quickly as possible" — in one line: **automatically catch problematic changes as early as possible.** But in a world of microservices, cross-network RPCs, evolving OS/runtimes/cloud, and apps that periodically ingest data or update ML models, the change that breaks your app increasingly lives *not* in your own codebase but in an upstream service someone else owns and deploys on their own schedule. So the chapter offers a more realistic definition (CI v2): **the continuous assembling and testing of our entire complex and rapidly evolving ecosystem.**

CI is tightly coupled to testing and essentially answers two questions: **which tests to run *when* in the dev/release workflow**, and **how to compose the system under test (SUT) at each point, trading off fidelity against setup cost.** Four core concepts: **① fast feedback loops** — a bug's cost grows almost exponentially the later it's caught, so **shift left**; feedback must also be **accessible and actionable**. **② automation** — the Continuous Build (CB, integrates the latest at head and runs build + test, introducing *true head* and *green head*) and Continuous Delivery (CD, continuously assembles release candidates / RCs and promotes them through environments). **③ continuous testing (CT)** — as a change shifts right, subject it to progressively larger-scoped tests; presubmit runs only "fast, reliable" ones, leaving the rest for post-submit, the RC, and production (probers). **④ "CI is alerting"** — CI is the "left shift" of production monitoring/alerting; a flaky test is to CI what a spurious alert firing every few minutes is to the on-call — **if it isn't actionable, it shouldn't be alerting; if it doesn't violate an invariant, it shouldn't be a test failure**; chasing "100% green" is as expensive and misguided as chasing "100% uptime."

**Google's implementation:** the global continuous build **TAP** (>50,000 unique changes/day, >4 billion test cases/day, the gateway for almost all changes thanks to the monorepo) lets potentially-breaking changes land first (a change that passes presubmit has a 95%+ chance of passing the rest), with an average submit wait of ~**11 minutes**; it manages breakage with **batching + binary-search culprit finding**, a **Build Cop**, **fast/auto rollback** (two clicks, or automatic on high confidence), and running only the minimal test set via the **downstream dependency graph.** It closes with **Google Takeout's** four-act CI transformation (broken dev deploys → sandboxed presubmit; indecipherable logs → friendly UI + clickable links; debugging "all of Google" → run the same suite against prod for failure isolation; keeping it green → responsible tag-based disablement + auto-cleanup), and answers "I can't afford CI": CI isn't a new cost — it's **a cost shifted left to a cheaper stage.**

## Why read this

- **You still ask "why not just run all tests on presubmit?"** Because it's too expensive, you get blocked by unrelated instability/flakiness, and there are **mid-air collisions** (two changes touching different files combine to fail a test). The chapter gives a clear presubmit-vs-post-submit rule: **presubmit holds only fast, reliable tests** — accept some coverage loss and some rollbacks.
- **Your end-to-end suite is "almost always red" and everyone's stopped looking.** It tackles failure management head-on: use **hermetic testing** to crush instability (Assistant cut presubmit runtime 14× with virtually no flakiness after going fully hermetic), and **bug-hotlists / responsible disablement + auto-cleanup** to keep the suite green instead of a pile of commented-out dead tests.
- **You don't know what *useful* CI feedback looks like.** "Accessible & actionable" is the key: Takeout turned indecipherable logs into "red/green at a glance + error messages with clickable log links," cutting its team's debugging involvement by 35%. Feedback must be not just visible but usable enough to fix directly.
- **You treat "CI all green" as law and block commits when it's red.** The "CI is alerting" lens loosens that dogma: **not every failure equals a production problem**; if the root cause is well understood and won't affect production, blocking commits is unreasonable. This directly shapes your team policy.

## Key details

### CI's definition and two core questions
Core claim: classic CI = frequent integration + automated build verification; but a modern distributed app's dependencies long ago outgrew "the latest versioned code in the repo" — microservices, RPCs, evolving OS/runtime/cloud, data, and ML models should all be treated as dependencies and "continuously integrated." Hence **CI v2 = continuous assembling and testing of the whole complex, rapidly evolving ecosystem.** From a testing view, CI answers: **which tests to run when**, and **how to compose the SUT at each point (fidelity vs setup cost).** Example: letting presubmit code-under-review talk to real production backends is dangerous (security/quota vulnerabilities), but is often fine for staging. CI's value = **verifiable, timely proof** the app is good to progress — not just *hoping* everyone is careful, but *guaranteeing* the working state at each point from build through release.

### Fast Feedback Loops
Core claim: the later a bug is caught, the more it costs (triaged by an engineer unfamiliar with it, the author must re-investigate, it harms others or end users) — almost exponentially. So CI encourages **fast feedback loops** (a.k.a. "shifting left"). Fastest to slowest: the local edit-compile-debug loop → automated presubmit results → a post-submit integration error between two projects → a staging incompatibility with an upstream microservice caught by QA → bug reports from internal users → reports from external users or the press.
- **Canarying:** deploying to a small percentage of production first cushions issues that slip through; but multiple concurrent versions bring **version skew** — a distributed-system state holding multiple incompatible versions of code/data/config, another hard problem of developing software over time.
- **Experiments / feature flags:** extremely powerful feedback loops that isolate changes into modular components toggleable in production, reducing deployment risk; heavy flag-guarding is a common Continuous Delivery paradigm (see [[ch24]]).
- **Accessible & actionable:** feedback must be widely accessible — a unified test-reporting system where anyone can look up any build/test run's logs, when targets began failing, where/by whom they ran, plus flake classification — *and* actionable (usable enough to find and fix problems, e.g., by improving test-output readability).

### Automation: Continuous Build and Continuous Delivery
Core claim: automating by defining processes as code, plus peer review at check-in, reduces error probability; automated processes have bugs too, but done well they're faster and more reliable than manual effort.
- **Continuous Build (CB):** integrates the latest change at **head** and runs an automated build *and* test — so "breaking the build" covers breaking tests *and* breaking compilation. A change passing all tests is marked "green." This effectively creates two heads: **true head** (the latest commit) and **green head** (the latest CB-verified change). Engineers often sync to green head for a stable dev environment but must sync to true head before submitting.
- **Continuous Delivery (CD):** step one is **release automation** — continuously assembling the latest code and config from head into a **release candidate (RC)** (most Google teams cut at green, not true, head). **An RC = a cohesive, deployable unit produced by an automated process from code + config + dependencies that passed the CB.** Crucially, **include configuration in the RC and test it** — many production bugs come from "silly" config problems; config should be in version control and go through the same review. CD = continuously assembling RCs, then promoting and testing them through a series of environments (sometimes reaching production, sometimes not). As an RC is promoted it **should not be recompiled** (use containers like Docker, orchestrators like Kubernetes/Borg for consistency) → higher-fidelity earlier testing, fewer production surprises.

### Continuous Testing: presubmit, post-submit, RC, production
Core claim: as a change shifts right from local dev to production, subject it to **progressively larger-scoped** automated tests. A key CI objective is deciding what to test when along that progression.
- **Why presubmit isn't enough:** ① too expensive — waiting for every test before submit is severely disruptive; relaxing "presubmit must be exhaustive" yields big efficiency wins when tests pass far more than they fail. ② being blocked by unrelated instability/flakiness is too costly. ③ **mid-air collisions** — while presubmit runs, the underlying repo may change incompatibly; two changes touching completely different files can combine to fail a test, which happens most days at Google's scale (small repos can serialize submits to avoid it).
- **Presubmit vs post-submit rule of thumb:** presubmit holds **only fast, reliable** tests (usually just the change's own project, run concurrently), accepting coverage loss (caught later on post-submit) and some rollbacks; post-submit tolerates longer times and some instability given proper mechanisms. Most teams run small tests (unit) on presubmit; whether to run larger-scoped tests there is the interesting question — those who do can use **hermetic testing** to reduce instability, or allow them to be unreliable on presubmit but disable aggressively when they start failing.
- **Release candidate testing:** even if CD cuts at green and runs the same suite CB just ran, run a comprehensive automated suite against the whole RC — as a **sanity check** (nothing weird happened cutting/recompiling), for **auditability** (results attached to the RC, no digging through CB logs), to **allow cherry-picks** (your source has diverged from the CB-tested cut), and for **emergency pushes** (CD can cut from true head and run a minimal set).
- **Production testing:** continuous testing goes all the way to production — run the same suite against prod (sometimes called **probers**) to verify ① the working state of production per your tests, and ② the relevance of your tests per production. Each step's trade-offs are a reminder of **defense in depth** — quality and stability rely on many combined testing approaches, not one technology or policy.

### CI Is Alerting — Titus Winters
Core claim: sustainably maintaining software, like responsibly running production, needs continual automated monitoring. Production monitoring uses passive alerts and active probers on running systems; CI uses unit/integration tests to see how software responds to its environment *before* deployment. Both serve the same purpose — **identify problems automatically, as soon as possible** (CI at the early end of the workflow, alerting at the late end).
- **Shared fidelity principle:** a test should fail only when an important underlying invariant is violated, not because it's brittle or flaky; **a flaky test is to CI what a spurious alert firing every few minutes (paging the on-call) is to production.** If it isn't actionable, it shouldn't alert; if it doesn't violate the SUT's invariants, it shouldn't be a test failure.
- **Localized vs cross-dependency signals:** unit tests / isolated-statistic cause-based alerts are localized; integration/release tests and black-box probers are cross-dependency. End-to-end signals have the highest fidelity but you pay in flakiness, resource cost, and root-cause debugging difficulty.
- **Bring SRE's "error budget" to CI:** chasing 100% green is as expensive as 100% uptime (the biggest problem becomes a race condition between testing and submission); not every alert/failure deserves equal alarm — if production isn't impacted, silence the alert, and likewise CI should be more willing to accept a change disabling a test failing for irrelevant reasons; policies like "**nobody commits unless the latest CI is green**" are probably misguided — always investigate failures, but blocking commits when the root cause is understood and won't affect production is unreasonable. This "CI is alerting" insight is new — the coming years' task is reconceptualizing SRE monitoring practice in a CI context.

### CI Challenges & Hermetic Testing
Core claim: common CI challenges — **presubmit optimization** (which to run, how), **culprit finding / failure isolation** (which change, which system? — "integrating upstream microservices" isolates failures in distributed architectures but suffers version skew and false positives), **resource constraints** (large tests are expensive), and **failure management.**
- **Failure management:** large end-to-end suites are hard to keep consistently green (inherently break/flake, hard to debug); you need a mechanism to **temporarily disable and track** failures so releases proceed. Google often uses bug **hotlists** filed by an on-call/release engineer and triaged (GWS, Assistant auto-generate and file them); release-blockers fixed immediately, non-blockers prioritized too, or the suite becomes a growing pile of disabled old tests. **Often end-to-end failures are problems with the *tests*, not the code.** Flaky tests erode confidence and are hard to bisect (the failure isn't constant); some teams use a tool to temporarily pull flaky tests from presubmit while investigating. A common tactic for instability is allowing **retries**.
- **Hermetic testing:** for larger-scoped tests, use **hermetic backends** (entirely self-contained, no external deps like production backends), with two key properties: **determinism (stability)** and **isolation** — the same app + test code run twice gives the same result; a failure means *your* code or tests caused it (minor caveat: restructuring the hermetic environment can also fail it, but that rarely changes). Isolation means production problems don't affect these tests and vice versa; success shouldn't depend on who runs it, letting people reproduce CI results and run other teams' tests. **A fake is one kind of hermetic backend** (cheaper than a real backend, but needs maintenance and has limited fidelity). The cleanest presubmit-worthy integration test is **fully hermetic — the whole stack sandboxed** (Google ships out-of-the-box sandbox configs for popular components like databases); DisplayAds even starts ~400 servers from scratch on every presubmit. But **record/replay** has become a more popular, cheaper paradigm for large systems — record live backend responses, cache them, replay in a hermetic environment; the downside is brittleness, balancing **false positives** (too many cache hits, missing real problems) against **false negatives** (too few hits, responses need updating — slow, often submit-blocking).
- **The Hermetic Google Assistant:** going **fully hermetic on presubmit** was a major success — previously nonhermetic presubmit tests failed routinely (some days **>50 changes bypassed/ignored** the results); going hermetic **cut runtime 14× with virtually no flakiness.** Nonhermetic tests moved to post-submit (failures accumulate there; hard to debug, some teams just disable them, risking production failures). With decentralization (components becoming microservices), running the whole hermetic stack on presubmit is too costly; the team uses a clever **post-submit failure-isolation** scheme: for each of N microservices, run an environment with that service built at head + production versions of the other N−1 to isolate the problem — normally O(N²), but **hotswapping** (a request can tell a server to "swap" in a backend address) cuts it to **O(N)**, reusing the same prod backends across the N environments.

### CI at Google: TAP — Adam Bender
Core claim: **TAP (Test Automation Platform)** is the massive continuous build over the entire codebase, running the majority of automated tests. Thanks to the monorepo, **TAP is the gateway for almost all changes: >50,000 unique changes/day, >4 billion individual test cases/day** — the beating heart of Google's dev infrastructure. Conceptually simple: an engineer tries to submit → TAP runs the tests → passes are allowed in.
- **Presubmit optimization:** don't require presubmit to be exhaustive — each team creates a **fast subset** (often the project's unit tests) to run before submit. Empirically, **a change that passes presubmit has a 95%+ chance of passing the rest**, so it's optimistically integrated (and immediately visible company-wide); after submit, TAP asynchronously runs all potentially affected (including larger, slower) tests.
- **Culprit finding:** TAP evaluates so many changes (>1/second) it can't run every test on every change, so it **batches** related changes to cut total tests — but that obscures which change in the batch broke a test. Two ways to speed isolation: ① TAP automatically **splits a failing batch into individual changes and reruns** each in isolation; ② **culprit-finding tools** let a developer **binary-search** a batch to find the likely culprit.
- **Failure management — the Build Cop:** each team has a **Build Cop** who keeps all the project's tests passing (regardless of who broke them); on notification they drop everything and fix the build — usually by finding the offending change and deciding to **roll back (preferred)** or fix forward (riskier). The most effective tool is rollback — **any Google change can be rolled back in two clicks**, and TAP now auto-rolls-back when highly confident of the culprit. **Tests give confidence to change; rollbacks give confidence to undo** — without tests you can't safely roll back; without rollbacks you can't quickly fix broken tests. The "commit before verifying all tests" trade-off pays off: **~11-minute average submit wait** (often in the background) + Build Cop discipline = efficiently handling breakages from longer tests with minimal disruption.
- **Resource constraints:** most tests run in **Forge**, a distributed build-and-test system (maximizing datacenter parallelism). Even with vast compute, Forge/TAP are resource-constrained. The primary mechanism is **analyzing each change's downstream dependency graph** (Forge and Blaze maintain a near-real-time global graph for TAP) to run the minimal set proving the change safe. Another factor is test speed: TAP can often run changes triggering **fewer tests sooner**, biasing engineers toward **small, focused changes** (100 vs 1,000 triggered tests can mean tens of minutes' difference on a busy day).

### CI Case Study: Google Takeout
Core claim: Takeout (a 2011 data backup/download product championing "data liberation") grew from one product to a platform/service for ≥10, then >90 Google products — rapidly growing apps especially need effective CI. Four acts:
- **Act 1 | Continuously broken dev deploys:** deploying each new API as a customized instance (same binary, different config) led to "flag issues" — one instance's flags broke others, config incompatibilities stopped servers starting, nearly nightly breakages found only at the next day's deploy (unit tests weren't enough). **What they did:** temporary sandboxed mini-environments per instance, on presubmit, testing all servers start healthy — **prevented 95% of bad-config server failures and halved nightly deploy failures.** End-to-end tests couldn't move to presubmit (test accounts are subject to security/privacy safeguards), so they **reused the sandboxes as a post-submit environment run every two hours** (cut an RC from green head, run the same suite) — **cutting the "culprit set" 12×.**
- **Act 2 | Indecipherable test logs:** as a plugin platform integrating >90 products, dumping failures to a log didn't scale; failures piled up unseen and the suite was almost always red. **What they did:** refactored into a **dynamic, config-based suite (parameterized test runner)** with a friendly UI showing each result red/green; error messages **embed clickable log links** (e.g., a Gmail-fetch failure dynamically builds a link searching that file's ID in Takeout logs) — **cutting the team's involvement in debugging client failures by 35%.**
- **Act 3 | Debugging "all of Google":** by verifying the archive output of 90-odd user-facing products, they were effectively testing "all of Google" and catching unrelated issues — good, but needing better **failure isolation** to tell their build's problems from upstream microservices'. **What they did:** run the **same suite continuously against production** (same live backends, newly built binaries) — a cheap way to isolate which failures are new in their build vs in production. **Future:** try hermetic + record/replay in post-submit CI to remove backend-API noise.
- **Act 4 | Keeping it green:** plugin end-to-end tests failed constantly, many from plugin bugs Takeout couldn't control; commenting out tests made failures too easy to forget. **What they did:** **responsible disablement via bug tags** — tag a failing test with a bug, file it to the responsible team, and the framework suppresses that failure, keeping the suite **green** while still giving confidence everything else passes. For feature rollouts, plugin engineers specify a **feature-flag name / change ID + expected output with and without** the feature, and the test queries the environment to know which to check. Accumulating stale tags get **auto-cleaned** (query the bug API; a tagged-failing test that actually passes long enough prompts tag cleanup / marks the bug fixed; flaky tests excepted) — a **nearly self-maintaining suite** (DevOps would call the metric **MTTCU: mean time to clean up**).
- **But I Can't Afford CI:** first count the cost you **already pay for problems found and handled in production** — they harm users *and* the team (frequent firefighting is stressful and demoralizing). Building CI is expensive but **mostly not a new cost — it's a cost shifted left** to an earlier, more preferable stage, reducing the incidence (and cost) of problems too far right — buying a more stable product and a more confident, feature-focused dev culture.

## Key terms

- **Continuous Integration (CI)** — frequent integration + automated build verification to catch problems early; modern definition extends to "continuous assembling and testing of the whole evolving ecosystem."
- **Shift left / fast feedback loop** — a bug's cost grows almost exponentially the later it's caught, so move verification to the early end of the workflow and shorten feedback loops.
- **Continuous Build (CB) / true head vs green head** — build + test at head; true head = latest commit, green head = latest CB-verified change (dev often syncs green head).
- **Continuous Delivery (CD) / Release Candidate (RC)** — continuously assemble RCs and promote them through environments; an RC = a deployable unit of code + config + deps that passed CB (config must be tested with it).
- **Version skew** — a distributed system simultaneously holding multiple incompatible versions of code/data/config; the classic risk of canarying and multi-version coexistence.
- **Mid-air collision** — two changes touching different files each pass but combine to break a test; happens most days at Google's scale, a reason presubmit isn't enough.
- **Presubmit vs post-submit** — presubmit holds only fast, reliable tests (accepting coverage loss and rollbacks); post-submit tolerates longer times and instability, catching what slipped by.
- **CI is alerting** — shift SRE monitoring/alerting principles to CI; if not actionable don't alert, if no invariant violated it isn't a failure, and 100% green is as costly as 100% uptime.
- **Hermetic testing** — testing against an entirely self-contained environment with no external dependencies; deterministic and isolated, the key tool for reducing large-test instability.
- **Record/replay** — record live backend responses, cache, and replay in a hermetic environment; cheaper than a full sandboxed stack but brittle, balancing false positives against false negatives.
- **TAP (Test Automation Platform)** — Google's global continuous build; >50,000 changes and >4 billion test cases/day, the gateway for almost all monorepo changes.
- **Build Cop / culprit finding** — a dedicated person keeping a project's tests green and rolling back breakages; batching + binary search to find the offending change.
- **Probers / defense in depth** — run the same suite against production to verify working state and test relevance; quality comes from many combined testing approaches, not one mechanism.

## Questions to push your thinking

1. Which tests does your team run on presubmit? Are they all truly "fast and reliable," or are flaky/slow ones blocking everyone daily and forcing people to debug unrelated failures? Which should move to post-submit? Have you ever hit a **mid-air collision**? Per [[ch11]] on test size and stability.
2. Is your end-to-end suite green or "almost always red"? If the latter, do you crush instability with **hermetic testing**, keep it green with "tag-based responsible disablement + auto-cleanup" — or let it rot into a pile of commented-out dead tests? Per [[ch13]], [[ch14]].
3. Adopting the "**CI is alerting**" lens: is your "no commits unless the latest CI is green" policy reasonable? Can you distinguish "failures that will become production problems" from "failures with a clear root cause that won't affect production"? Is the "100% green" you chase worth its cost?
4. Is your CI feedback **actionable** enough? Can a failure message take you straight to the log, to the offending line? Recall Takeout cutting debugging involvement 35% just by turning logs into clickable links — is your feedback still "visible but not usable"?
5. Is your configuration in version control and tested *with* the code (included in the RC)? Recall "many production bugs come from silly config problems" — do you treat config as a second-class citizen or run it through the same review and testing? Per [[ch24]] on RC promotion.
