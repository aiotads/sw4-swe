# 第 23 章｜Continuous Integration（持續整合）

> *"CI is the continuous assembling and testing of our entire complex and rapidly evolving ecosystem."*
> 「持續整合，是對我們整個複雜且快速演化的生態系，做持續的組裝與測試。」

## 摘要

**Continuous Integration（CI，持續整合）** 的經典定義是「團隊成員頻繁整合工作，每次整合都由自動建置（含測試）驗證，以盡早偵測整合錯誤」——一句話：**自動、盡早地抓出有問題的變更**。但在微服務、跨網路 RPC、會演化的 OS/runtime/雲端、會定期吃資料或更新 ML 模型的現代系統裡，「會弄壞你應用的變更」越來越不在你自家程式碼裡，而在別人擁有、按自己時程部署的上游服務。所以本章給 CI 一個更貼近現實的定義（CI v2）：**對整個複雜且快速演化的生態系，做持續的組裝與測試**。

CI 與測試緊密耦合，本質上是回答兩個問題：**在開發/發布流程的「哪個時點」跑「哪些測試」**，以及**每個時點如何組裝待測系統（SUT），在保真度與建置成本間取捨**。核心概念有四：**①快速回饋迴圈（fast feedback loops）**——bug 抓得越晚成本近乎指數上升，故要「**左移（shift left）**」；回饋還要**可取得（accessible）且可行動（actionable）**。**②自動化**——Continuous Build（CB，在 head 整合最新變更、跑建置與測試，引入 true head 與 green head 兩個版本）與 Continuous Delivery（CD，持續組裝 release candidate／RC，並讓 RC 通過一連串環境）。**③持續測試（CT）**——隨變更右移，施以範圍逐漸放大的測試；presubmit 只跑「快又可靠」的，其餘留給 post-submit、RC、production（probers）。**④「CI 即告警（CI is alerting）」**——CI 是把生產監控/告警「左移」到部署前；flaky 測試之於 CI，就像每幾分鐘亂叫的假警報之於 on-call——**不可行動就不該告警，沒違反不變量就不該算測試失敗**；追求「100% 綠」就像追求「100% uptime」一樣昂貴且多半搞錯方向。

**Google 的實作**：全球持續建置 **TAP**（每天處理 >50,000 個變更、跑 >40 億個測試案例，是 monorepo 幾乎所有變更的閘門）——容許「可能弄壞的變更」先落地（通過 presubmit 的變更有 95%+ 機率通過其餘測試），提交平均只等約 **11 分鐘**；用**批次＋二分搜尋的 culprit finding（找元兇）**、**Build Cop（建置警長）**、**自動回退（兩下點擊／高信心自動回退）**、以及**靠下游依賴圖只跑必要測試**來管理。最後以 **Google Takeout** 的四幕 CI 轉型（壞掉的 dev deploy → 沙箱 presubmit；難讀的 log → 友善 UI＋可點連結；除錯「整個 Google」→ 對 prod 跑同一套測試做失敗隔離；保持綠燈 → 用 bug tag 負責任地停用＋自動清理）作結，並回應「我負擔不起 CI」：CI 不是新成本，而是**把成本左移到更便宜的階段**。

## 為什麼要讀這一篇

- **你還在問「為什麼不把所有測試都放 presubmit？」** 答案是太貴、會被無關的不穩定/flaky 卡住、還有 **mid-air collision（空中相撞）**（兩個碰不同檔的變更合起來弄壞測試）。本章給你 presubmit vs post-submit 的明確取捨原則：**presubmit 只放快又可靠的**，接受覆蓋率損失與一定的回退率。
- **你的端到端測試套件「幾乎總是紅的」，大家已經不看了。** 本章直球面對失敗管理：用 **hermetic testing（封閉測試）** 把不穩定打下來（Assistant 把 presubmit 改全封閉後 runtime 降 14 倍、近乎零 flaky），用 **bug hotlist／負責任地停用＋自動清理** 讓套件保持綠燈而非變成一堆被註解掉的死測試。
- **你不知道 CI 的回饋該長怎樣才有用。** 「accessible & actionable」是關鍵：Takeout 把難讀的 log 改成「紅/綠一目了然＋錯誤訊息直接帶可點的 log 連結」，就讓團隊介入除錯的比例降 35%。回饋不只要看得到，還要**好用到能直接修**。
- **你把「CI 全綠」當成鐵律、紅了就禁止提交。** 「CI 即告警」這個視角會鬆動你的教條：**不是每個失敗都等於生產問題**；若根因清楚且不影響 production，硬卡提交是不合理的。這直接影響你的團隊政策。

## 細節重點

### CI 的定義與兩個核心問題（CI Concepts 開場）
核心主張：傳統 CI＝頻繁整合＋自動建置驗證；但現代分散式應用的依賴早已超出「repo 裡最新的版本化程式碼」——微服務、RPC、演化的 OS/runtime/雲端、資料與 ML 模型，都該視為依賴並「持續整合」。故 **CI v2＝對整個複雜且快速演化的生態系做持續的組裝與測試**。從測試視角，CI 要回答：**何時跑哪些測試**、**每個時點如何組裝 SUT（保真度 vs 建置成本）**。例：presubmit 的 SUT 讓「待審程式」直連真實 production backend 很危險（安全/配額漏洞），但 staging 通常可接受。CI 的價值＝**可驗證且及時的保證**——不必只是「希望大家都很小心」，而是能保證應用在從建置到發布的各時點都處於可運作狀態。

### Fast Feedback Loops（快速回饋迴圈）
核心主張：bug 抓得越晚，成本近乎指數上升（要不熟該變更的人 triage、作者要重新回想調查、波及他人或終端用戶）。故 CI 鼓勵**快速回饋迴圈**（即「左移 shift left」）。由快到慢的常見回饋：本地 edit-compile-debug → presubmit 自動測試 → post-submit 抓到的兩專案整合錯誤 → staging 由 QA 抓到的上游微服務不相容 → 內部用戶回報 → 外部用戶/媒體回報。
- **Canary（金絲雀）：** 先部署到一小部分 production，可降低漏網問題的衝擊；但多版本並存會帶來 **version skew（版本偏移）**——分散式系統中同時存在多個不相容的程式/資料/設定版本，是「軟體隨時間演化」的又一難題。
- **Experiments / feature flags（實驗／功能旗標）：** 極強的回饋迴圈，把變更隔離進可在 production 動態開關的模組，降低部署風險；重度依賴 flag 是 Continuous Delivery 的常見範式（見 [[ch24]]）。
- **Accessible & actionable（可取得且可行動）：** 回饋要廣泛可取得——統一的測試報告系統（任何人可查任何 build/test run 的 log、失敗起點、何時何地由誰跑、flake 分類），也要 **actionable**（好用到能直接找到並修好問題，例如改善測試輸出可讀性）。

### Automation：Continuous Build 與 Continuous Delivery（自動化）
核心主張：把流程定義成程式碼來自動化，加上 check-in 時的同儕審查，能降低錯誤率；自動化流程雖也有 bug，但比人工更快更可靠。
- **Continuous Build（CB，持續建置）：** 在 **head** 整合最新變更、跑自動建置與測試——所以「break the build」同時指弄壞編譯與弄壞測試。通過全部測試的變更被標為「綠（green）」。這在 repo 裡有效製造兩個版本：**true head**（最新提交）與 **green head**（CB 已驗證的最新）。工程師常 sync 到 green head 取得穩定環境來開發，但提交前須 sync 到 true head。
- **Continuous Delivery（CD，持續交付）：** 第一步是**發布自動化**——持續把 head 的最新程式與設定組裝成 **release candidate（RC，發布候選）**（Google 多數團隊在 green head 而非 true head 切 RC）。**RC＝由自動流程產生、由通過 CB 的程式＋設定＋依賴組成的一個內聚、可部署單元**。關鍵：**設定（configuration）要納入 RC 一起測**——大量 production bug 來自「蠢」的設定問題，設定也要進版本控制、走同樣的 code review。CD＝持續組裝 RC，再讓 RC 通過一連串環境的晉級與測試（有時到 production、有時不到）。RC 跨環境晉級時**不該重新編譯**（用 Docker 等容器、Kubernetes/Borg 等編排工具確保一致性）→ 更高保真的早期測試、更少 production 驚喜。

### Continuous Testing：presubmit、post-submit、RC、production（持續測試）
核心主張：隨變更從本地開發右移到 production，施以**範圍逐漸放大**的自動測試。CI 的關鍵目標就是決定「在這個推進過程中何時測什麼」。
- **為何 presubmit 不夠：** ①太貴——等所有測試跑完才能提交會嚴重打斷生產力；放寬「presubmit 要窮盡」的限制，能在「測試通過遠多於失敗」時換得大量效率。②被無關的不穩定/flaky 卡住，代價太高。③**mid-air collision（空中相撞）**——跑 presubmit 的同時底層 repo 可能已被改成與待測變更不相容；兩個碰完全不同檔的變更可能合起來讓測試失敗，在 Google 規模幾乎天天發生（小 repo 可用「序列化提交」避免）。
- **presubmit vs post-submit 原則：** presubmit **只放快又可靠的**（通常只跑變更所在專案的測試、並行跑），接受覆蓋率損失但要在 post-submit 補抓、接受一定回退率；post-submit 可接受較長時間與一些不穩定，只要有對應機制。多數團隊把 small test（如單元測試）放 presubmit；larger-scoped 測試要不要放 presubmit 才是有趣的問題——想放的團隊可用 **hermetic testing** 降低不穩定，或允許它在 presubmit 不可靠但一開始失敗就積極停用。
- **Release candidate testing（RC 測試）：** 即使 CD 在 green 切 RC、跑的是 CB 剛跑過的同一套，仍要對整個 RC 跑全面自動測試——理由：**sanity check**（確認切 RC/重編沒出怪事）、**可稽核性**（結果直接綁在 RC 上、不必翻 CB log）、**允許 cherry-pick**（套了修補後源碼已偏離 CB 測過的）、**緊急推送**（CD 可從 true head 切、只跑最小必要測試）。
- **Production testing（生產測試）：** 持續測試一路到最終部署環境——對 production 跑與 RC 同一套測試（有時稱 **probers**），驗證 ①依測試看 production 的運作狀態、②依 production 看測試的相關性。每一步各有取捨，正是「**defense in depth（縱深防禦）**」——品質與穩定不靠單一技術或政策，而靠多種測試手段疊加。

### CI Is Alerting（CI 即告警）— Titus Winters
核心主張：永續維護軟體就像負責任地運行 production，都需要持續的自動監控。生產監控用「被動告警＋主動 prober」看運行中的系統如何回應變化；CI 用單元/整合測試在部署前看軟體如何回應其環境的變化。兩者目的相同——**盡快自動找出問題**（CI 在開發流程的早端，告警在晚端）。
- **共用的保真度原則：** 測試只該在「重要的底層不變量被違反」時失敗，而非因為它脆弱或 flaky；**flaky 測試之於 CI，等同每幾分鐘亂叫、害 on-call 被 page 的假警報**。不可行動就不該告警；沒違反 SUT 的不變量就不該算測試失敗。
- **localized vs cross-dependency 訊號：** 單元測試／隔離統計的 cause-based 告警是局部訊號；整合/發布測試、黑箱 prober 是跨依賴訊號。端到端訊號保真度最高，但代價是 flakiness、資源成本、除錯根因困難。
- **把 SRE 的「error budget」搬進 CI：** 追求 100% 綠就像追求 100% uptime 一樣昂貴（最大問題會是測試與提交間的 race condition）；不是每個告警/失敗都該一視同仁——若 production 沒被影響就該靜音告警，同理 CI 該更願意接受「停用一個因無關原因失敗的測試」的變更；「**最新 CI 不綠就誰都不准提交**」這類政策多半是搞錯方向——失敗一定要先調查，但若根因清楚且不影響 production，硬卡提交不合理。這個「CI 即告警」的洞見還很新，是未來幾年要把 SRE 監控實務重新概念化到 CI 的方向。

### CI Challenges & Hermetic Testing（CI 的挑戰與封閉測試）
核心主張：CI 的常見挑戰——**presubmit 最佳化**（該跑哪些、怎麼跑）、**culprit finding／failure isolation（找元兇／失敗隔離）**（哪個變更、哪個系統？分散式架構可用「整合上游微服務」來隔離，但受 version skew 與 false positive 之苦）、**資源限制**（大型測試很貴）、以及 **failure management（失敗管理）**。
- **失敗管理：** 大型端到端測試難以長保全綠（天生易壞/flaky、難除錯）；需要機制能**暫時停用並追蹤**，讓發布繼續。Google 常用 on-call/release 工程師建 bug **hotlist** 並 triage 給對應團隊（GWS、Assistant 等大產品會自動產生並歸檔）；release blocker 立刻修，non-blocker 也要排，否則套件淪為一堆被停用的舊測試。**很多端到端失敗其實是測試本身的問題，不是程式。** flaky 測試侵蝕信心、又因不是每次都失敗而難找回退對象；有團隊用工具在調查期間暫時把 flaky 測試移出 presubmit。對測試不穩定，常見手法是**允許多次嘗試（retry）**。
- **Hermetic testing（封閉測試）：** 對 larger-scoped 測試用**封閉後端**（完全自包含、無 production 等外部依賴），有兩大特性：**determinism（決定性/穩定）** 與 **isolation（隔離）**——同樣的應用與測試碼跑兩次該得同樣結果；失敗就知道是你的程式或測試造成的（小例外：重構封閉環境也可能害它失敗，但不常變）。隔離意味 production 的問題不影響測試、反之亦然；成功與否不該取決於誰跑，讓人能重現 CI 的結果、也能跑別隊的測試。**fake 是一種封閉後端**（比真後端便宜，但要維護且保真度有限）。最乾淨的 presubmit 整合測試是**完全封閉、把整個 stack 沙箱化啟動**（Google 為熱門元件如資料庫提供現成沙箱設定）；DisplayAds 甚至每次 presubmit 從零啟 ~400 台 server。但 **record/replay（錄製/重放）** 已成大型系統更流行、更便宜的範式——錄下真後端回應、快取、在封閉環境重放；缺點是測試脆弱：要在 **false positive**（命中快取太多、漏掉真問題）與 **false negative**（命中太少、回應要更新、常變成卡提交的失敗）間拿捏。
- **The Hermetic Google Assistant：** 把 presubmit 測試套件改成**完全封閉**是其最大成功之一——先前非封閉測試常失敗，某些日子有 **>50 個變更直接繞過/忽略測試結果**；改封閉後 **runtime 降 14 倍、近乎零 flaky**。非封閉測試被推到 post-submit（失敗在那累積，難除錯、有些團隊乾脆停用，可能導致 production 失敗）。去中心化（元件各自變微服務）下要在 presubmit 跑整個封閉 stack 成本太高；團隊用**聰明的 post-submit 失敗隔離**：對 N 個微服務，各跑「該微服務建在 head＋其餘 N−1 個用 production 版」的環境來隔離問題，本應 O(N²) 成本，但靠 **hotswapping（熱抽換）**（請求可指示 server 換掉要呼叫的後端位址）把成本降到 **O(N)**，且可重用同一組 prod 後端。

### CI at Google：TAP（Google 的全球持續建置）— Adam Bender
核心主張：**TAP（Test Automation Platform）** 是跑遍整個程式碼庫的大規模持續建置，負責跑多數自動測試。因 monorepo，**TAP 是 Google 幾乎所有變更的閘門：每天處理 >50,000 個獨特變更、跑 >40 億個測試案例**，是開發基礎設施「跳動的心臟」。概念很簡單：工程師要提交→TAP 跑相關測試→通過才准進程式碼庫。
- **Presubmit 最佳化：** 不要求 presubmit 窮盡——只要每隊建一份**快速子集**（常是專案單元測試）在提交前跑。經驗上**通過 presubmit 的變更有 95%+ 機率通過其餘測試**，故樂觀地讓它先整合進去（立刻對全公司可見）；提交後 TAP 非同步跑所有可能受影響的（含更大更慢的）測試。
- **Culprit finding（找元兇）：** TAP 每天評估的變更太多（每秒超過一個），無法對每個變更都跑每個測試，於是**批次**相關變更一起跑以減少總測試數，但這會模糊「批次裡哪個變更弄壞的」。兩招加速定位：①TAP 自動把失敗批次**拆成個別變更重跑**；②提供 **culprit finding 工具**讓開發者對一批變更做**二分搜尋**找出元兇。
- **Failure management（失敗管理）—— Build Cop：** 每隊有一位 **Build Cop（建置警長）**，負責讓該專案所有測試保持通過（不管是誰弄壞的）；被通知失敗就放下手邊事去修，通常是找出禍首變更並決定**回退（首選）**或往前修（較冒險）。最有效的工具是回退——**Google 任何變更兩下點擊就能回退**，TAP 近期更升級為「高信心是元兇時自動回退」。**測試給我們改變的信心，回退給我們撤銷的信心**；沒測試就不能安全回退，沒回退就不能快速修好壞掉的測試。容許「先提交再驗證全部測試」的取捨確實划算：**平均提交等待約 11 分鐘**（常在背景跑）＋Build Cop 紀律＝以最小干擾處理長測試抓到的破壞。
- **Resource constraints（資源限制）：** 多數測試在分散式建置測試系統 **Forge** 跑（在資料中心最大化並行）。即使算力龐大，Forge/TAP 仍受資源限制。主要機制是**分析每個變更的下游依賴圖**（Forge 與 Blaze 維護近即時的全域依賴圖供 TAP 用），只跑該變更下游的最小測試集。另一因素是測試速度：TAP 常能讓**觸發較少測試的變更更早跑完**——這偏好鼓勵工程師寫**小而聚焦的變更**（觸發 100 個 vs 1,000 個測試，忙碌日可差數十分鐘等待）。

### CI Case Study: Google Takeout（CI 案例：Google Takeout）
核心主張：Takeout（2011 起的資料備份下載產品，倡議「data liberation 資料解放」）從單一產品長成服務 ≥10、後達 >90 個 Google 產品的平台與服務——快速成長的應用尤其需要有效 CI。四幕轉型：
- **幕一｜持續壞掉的 dev deploy：** 為各 API 用同一份 binary 配不同設定部署成客製實例，導致「flag 問題」——一個實例的 flag 弄壞別的、設定不相容讓 server 起不來、近乎每晚壞一次，且要到隔天 deploy 才發現（單元測試不足以抓）。**做法：** 為每個實例建**臨時沙箱 mini 環境在 presubmit 測「所有 server 能否健康啟動」**——**擋下 95% 因壞設定起不來的 server、夜間部署失敗減半**。端到端測試難搬到 presubmit（用 test account 受安全隱私約束），就**重用沙箱環境擴成 post-submit 環境每兩小時跑一次**（從 green head 切 RC 跑同套端到端測試）——**把 culprit set 縮小 12 倍**。
- **幕二｜難以解讀的測試 log：** 平台支援 plug-in（各產品把抓資料的程式插進 Takeout binary），整合到 >90 個產品後，把失敗倒進 log 的做法不再 scale，失敗堆積易漏看、套件幾乎總是紅的。**做法：** 把測試重構成**動態、基於設定的套件（參數化 test runner）**，用友善 UI 紅/綠清楚呈現個別結果；錯誤訊息**直接內嵌可點的 log 連結**（如抓 Gmail 檔失敗，動態組一個搜該檔 ID 的 Takeout log 連結）——**把 Takeout 團隊介入除錯 client 失敗的比例降 35%**。
- **幕三｜為「整個 Google」除錯：** 因驗證 90 多個面向用戶產品的輸出（archive），等於在測「整個 Google」、抓到與 Takeout 無關的問題——好事，但需要更好的**失敗隔離**分清是自家 build 還是上游微服務的問題。**做法：** 把 post-submit 用的同一套測試**持續對 production 也跑一次**（同樣 live 後端、不同的新建 binary），便宜地隔離出哪些失敗是自家新進的、哪些在 production。**未來：** 試在 post-submit CI 用 hermetic＋record/replay 消除後端 API 失敗的雜訊。
- **幕四｜保持綠燈：** plug-in 各帶端到端測試、幾乎總有失敗，且很多是 Takeout 無法控制的 plug-in bug；註解掉太容易被遺忘。**做法：** 用 **bug tag 負責任地停用**——把失敗測試標上對應 bug 並歸檔給負責團隊，框架就抑制其失敗，讓套件**保持綠燈**同時仍對「已知問題以外的一切」給信心。對「功能漸進推出」造成的失敗，讓 plug-in 工程師指定 **feature flag 名/變更 ID＋開關兩種預期輸出**，測試查環境決定該驗哪個。bug tag 累積沒更新時**自動清理**（查 bug 系統 API；被標失敗的測試實際通過且持續夠久就提示清 tag／標記 bug 已修；flaky 測試例外）——做成**幾乎自我維護的測試套件**（此指標 DevOps 稱 **MTTCU：mean time to clean up**）。
- **But I Can't Afford CI（但我負擔不起 CI）：** 先想想你**已經在為「在 production 才發現並處理問題」付的成本**——傷用戶也傷團隊（頻繁救火令人壓力又士氣低落）。建 CI 雖貴，但**多半不是新成本，而是把成本左移到更早、更划算的階段**，降低問題發生在「太右邊」的機率與成本——換來更穩定的產品與更有信心、更專注做功能的開發文化。

## 關鍵字補充

- **Continuous Integration（CI，持續整合）** — 頻繁整合＋自動建置驗證以盡早抓問題；現代定義擴及「對整個複雜演化生態系的持續組裝與測試」。
- **Shift Left（左移）／Fast Feedback Loop** — bug 抓得越晚成本近乎指數上升，故把驗證盡量往開發流程的早端移、縮短回饋迴圈。
- **Continuous Build（CB）／true head vs green head** — 在 head 跑建置＋測試；true head＝最新提交，green head＝CB 已驗證的最新（開發常 sync green head）。
- **Continuous Delivery（CD）／Release Candidate（RC）** — 持續組裝 RC 並讓它通過一連串環境；RC＝程式＋設定＋依賴組成、通過 CB 的可部署單元（設定必須一起測）。
- **Version Skew（版本偏移）** — 分散式系統同時存在多個不相容的程式/資料/設定版本；canary 與多版本並存的典型風險。
- **Mid-air Collision（空中相撞）** — 兩個碰不同檔的變更各自通過卻合起來弄壞測試；Google 規模幾乎天天發生，是 presubmit 不夠的原因之一。
- **Presubmit vs Post-submit** — presubmit 只放快又可靠的（接受覆蓋率損失與回退率）；post-submit 容忍較長時間與不穩定，補抓漏網。
- **CI Is Alerting（CI 即告警）** — 把 SRE 監控/告警原則左移到 CI；不可行動不該告警、沒違反不變量不該算失敗、追求 100% 綠像追 100% uptime 一樣昂貴。
- **Hermetic Testing（封閉測試）** — 對完全自包含、無外部依賴的環境跑測試；具決定性與隔離性，是降低大型測試不穩定的利器。
- **Record/Replay（錄製/重放）** — 錄真後端回應、快取、在封閉環境重放；比啟整個沙箱 stack 便宜，但要在 false positive/negative 間拿捏，易脆弱。
- **TAP（Test Automation Platform）** — Google 全球持續建置；每天 >50,000 變更、>40 億測試案例，是 monorepo 幾乎所有變更的閘門。
- **Build Cop（建置警長）／Culprit Finding** — 專人讓專案測試保持綠、壞了就回退；批次＋二分搜尋找出弄壞測試的元兇變更。
- **Probers（生產探測）／Defense in Depth** — 對 production 跑與 RC 同套測試驗證運作狀態與測試相關性；品質靠多種測試手段疊加而非單一機制。

## 延伸提問

1. 你的團隊把哪些測試放在 presubmit？它們真的都「快又可靠」嗎，還是有 flaky/慢測試在天天卡大家、害人除無關的錯？哪些該下放到 post-submit？你有沒有承受過 **mid-air collision**（兩個無關變更合起來弄壞）？對照 [[ch11]] 的測試規模與穩定性。
2. 你的端到端測試套件現在是綠的還是「幾乎總是紅的」？如果是後者，你是用 **hermetic testing** 把不穩定打下來，還是用「bug tag 負責任地停用＋自動清理」保持綠——還是任由它爛成一堆被註解掉的死測試？對照 [[ch13]]、[[ch14]]。
3. 接受「**CI 即告警**」這個視角後：你的「最新 CI 不綠就禁止提交」政策合理嗎？你能不能區分「真的會變成 production 問題的失敗」與「根因清楚、不影響 production 的失敗」？你追求的「100% 綠」值那個代價嗎？
4. 你的 CI 回饋夠 **actionable** 嗎？一個失敗訊息能不能直接帶你到 log、到出錯的那一行？回想 Takeout 把難讀 log 改成可點連結就讓介入除錯降 35%——你的回饋還停在「看得到但不好用」嗎？
5. 你的設定（configuration）有沒有跟程式一起進版本控制、一起被測（納入 RC）？回想「大量 production bug 來自蠢設定問題」——你是把設定當二等公民，還是讓它走同樣的 review 與測試？對照 [[ch24]] 的 RC 晉級。
