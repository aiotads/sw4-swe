# 第 9 章｜Code Review（程式碼審查）

> *"Code review acts as the glue connecting engineers with one another, and the code review process is the primary developer workflow upon which almost all other processes must hang."*
> 「程式碼審查是把工程師彼此黏在一起的膠水；它是最核心的開發工作流，幾乎所有其他流程——從測試、靜態分析到 CI——都得掛在它上面。」

## 摘要

程式碼審查（code review）是「由作者以外的人來審視程式碼、通常在程式進入程式碼庫之前」的流程。業界做法天差地遠；Google 的版本極端徹底：**幾乎每一次變更、不論多小，都要在提交前被審（precommit review），而每位工程師既是發起審查的人、也是審查別人的人。** 本章談的是「流程」而非工具本身（工具 Critique 另闢 [[ch19]] 專章）。

審查的終點是拿到另一位工程師的 **LGTM（looks good to me，看起來沒問題）**。但 Google 把「批准」拆成**三個獨立的權限位元（bits）**：①另一位工程師給的 **LGTM**（程式正確且可理解）；②該目錄 **owner（擁有者）的核准**（這段程式適不適合放進這塊程式碼庫）；③具該語言 **readability（可讀性）認證者的核准**（符合語言風格與最佳實務）。三個角色常由同一人扮演，但拆開來才能**靈活又能擴展**——這正是規模的考量。

審查帶來六項效益：**檢查正確性、確保可理解性、強制一致性、心理上促成集體擁有感、促進知識分享、留下歷史紀錄**。書中明言：對 Google 而言，**檢查正確性並非首要效益**，「程式長期是否可被理解、可被維護」才更重要——因為**程式被讀的次數遠多於被寫**。最後是讓流程能擴展的最佳實務：**有禮且專業、寫小變更（約 200 行）、寫好的變更說明、審查者人數降到最低（多半剛好一人）、能自動化就自動化（presubmit）**。一句根本心法貫穿全章：**code is a liability（程式碼是負債）**。

## 為什麼要讀這一篇

- **你把 code review 當成「抓 bug 的關卡」。** 這章會翻轉你的認知：抓正確性其實**不是** Google 最看重的效益。真正的長期價值是**可理解性、一致性、知識分享、歷史紀錄**——這些都是「為時間與規模服務」的事。看懂這點，你寫 review 留言的角度會完全不同。
- **你的 review 又慢又痛，大家都怕被審。** 本章把「慢」歸因到**實作方式**而非審查本身，並給出可直接落地的解方：**寫小變更**（約 200 行、35% 變更只動一個檔案）、**24 個工作小時內回饋**、**別零碎地擠牙膏式給意見**、**審查者降到一人**（「最重要的是第一個 LGTM，後面的邊際效益遞減」）。
- **你想知道「擁有權」「readability」這些 Google 制度到底在解什麼問題。** 答案是 **scale（規模）**：把 LGTM／owner／readability 三個位元拆開，讓 intern 與 tech lead 走同一套流程卻有不同路徑；`OWNERS` 檔讓「誰負責這塊」沿目錄樹一查就到，無需中央登記。
- **你不確定不同類型的變更該用不同力道審。** greenfield（全新程式）要最嚴、bug fix 要最聚焦（別順手改別的）、機器產生的 LSC 只該挑「你這塊特有」的問題（不能對整個流程行使否決權）。把力道用對地方，是讓 review 能規模化的關鍵。

## 細節重點

### Code Review Flow（審查的流程）
核心主張：Google 採 **precommit review（提交前審查）**——程式進庫前必須先過審。一次典型流程六步：①作者在工作區寫好變更，上傳快照（patch＋描述），產生對程式碼庫的 **diff**；②作者可先做自審或套自動留言，滿意後把變更**寄（mail）**給一位以上的審查者；③審查者在工具裡對 diff 留言（有些要求明確解決、有些只是 FYI）；④作者依回饋改、再上傳新快照、回覆——③④可重複多次；⑤審查者滿意後標 **LGTM**（預設只需一個）；⑥標了 LGTM、解決所有留言、且取得 approval 後，作者才能提交。

工程意涵：**LGTM 是「必要的權限位元」**，但不是唯一條件——它要和下面的 owner／readability 位元組合，才放行提交。

### Code Is a Liability（程式碼是負債）
核心主張：**程式碼本身是負債，而非資產。** 就算是必要的負債，它對某時某地的某個人來說，終究是一項維護工作——像飛機載的燃油，有重量，雖然飛行需要它。新功能常是必要的，但**動手寫之前要先確認這功能真的值得**。重複的程式不只是浪費，甚至比「根本沒有這段程式」更花時間。「**如果你在從頭造輪子，你大概做錯了。**」

工程意涵：審查**不是**重開先前設計辯論的場合。設計決策該在設計審查／API 審查階段就定案；code review 既不該為「全新程式」感到意外（它應該事先溝通過），也不該被當成翻舊帳的機會。

### How Code Review Works at Google：三個批准位元（The three approval bits）
核心主張：任何變更在 Google 要被「批准」，牽涉三種審查：
- **正確性與可理解性檢查（LGTM bit）：** 由另一位工程師確認程式恰當、且確實做到作者宣稱的事。通常是隊友，但不必然。
- **Owner 的核准（ownership bit）：** 由該目錄的 owner 確認這段程式適合放進這塊程式碼庫（能 check-in 到某目錄）。Google 程式碼庫是**樹狀結構、目錄有階層式 owner**（見 [[ch16]]）；owner 是該目錄的守門人。若作者本身是 owner，這個位元可隱含成立。
- **語言 readability 的核准（readability bit）：** 由具該語言 readability 的工程師確認程式符合該語言風格與最佳實務。這些人來自**全公司**的 readability 認證池。若作者本身有該語言 readability，這位元也可隱含成立。

關鍵在於：**三個位元可任意組合**。一個既是 owner、又有該語言 readability 的 tech lead，只要再拿一個別人的 LGTM 就能提交；一個沒有這些權限的 intern 送同一個變更到同一塊程式碼庫，則需要再取得「具 readability 的 owner」核准。多數需要多重批准的審查走**兩步**：先拿 peer 的 LGTM（聚焦正確性與一般有效性），再找 owner／readability 核准（owner 更關心「這會不會難維護？會不會增加技術債？我們團隊有能力維護它嗎？」）。

工程意涵：**為什麼不乾脆讓一個全能審查者包辦全部？答案是 scale（規模）。** 把三個角色拆開帶來彈性，讓不同人專注不同面向、省下審查時間——這是讓「幾萬名工程師在數十億行單一程式碼庫上高效運作」的關鍵設計。

### Ownership（擁有權，by Hyrum Wright）
核心主張：小團隊在專屬 repo 裡常把全 repo 權限開給所有人；團隊一大，這招就**撐不住**。Google 用 **ownership（擁有權）** 記錄「誰對 repo 哪一塊有知識與責任」。**這不是「佔有」程式碼，而是一種 stewardship（受託管理）——以公司最大利益看顧某段程式碼**（作者甚至說，若能重來，「stewards（管家）」會是更好的詞）。

機制：特別命名的 **`OWNERS` 檔**列出對某目錄及其子目錄負有擁有責任者的帳號。檔案可引用其他 `OWNERS` 或外部 ACL，但最終解析成一份個人名單。每個子目錄可有自己的 `OWNERS`，關係**階層式相加**：一個檔案通常由其目錄樹上方所有 `OWNERS` 成員的聯集擁有。鼓勵名單**小而聚焦**，責任才清楚。

工程意涵：這套分散式擁有權支撐了書中許多其他實務——root `OWNERS` 名單可當大規模變更（[[ch22]]）的**全域核准者**，不必煩擾各地團隊；`OWNERS` 檔本身就是一種**文件**，沿目錄樹往上走就能找到負責人；新專案不需中央登記，**放一個新 `OWNERS` 檔就夠了**。這個機制簡單卻強大，已穩定擴展二十年。

### Code Review Benefits（審查的六項效益）
核心主張：一個設計良好、且被認真對待的審查流程，提供六項效益：

- **Code Correctness（正確性）：** 多一雙眼睛確認變更做到所宣稱的事——檢查測試是否齊全、設計是否得當、是否正確而有效率。IBM 研究指出**越早發現缺陷、後續修復越省時**。但有個前提：**流程本身必須輕量**；笨重、無法擴展的審查流程會變得不可持續。為避免正確性評估流於主觀，**作者對其特定做法享有 deference（尊重）**——審查者不該因個人偏好提替代方案，只有當替代方案能**改善可理解性（更簡單）或功能（更有效率）**時才提。鼓勵工程師**批准「能改善程式碼庫」的變更，而非苦等一個更「完美」的共識**。隨著靜態分析與自動測試變強，許多正確性檢查已自動化（屬「shift left」策略的一環，見 [[ch20]]）——但工具有極限，仍未完全取代人眼。**令人意外的是：正確性並非 Google 從審查得到的首要效益。**
- **Comprehension of Code（可理解性）：** 審查通常是「作者以外的人」第一次檢視變更，能提供**不受作者觀點偏誤影響**的回饋——這是再強的工程師也無法自己給自己的。**程式被讀的次數遠多於被寫**，所以可理解性至關重要。對「設計決策」該尊重作者，但對「可理解性的提問」要用「**顧客永遠是對的**」的格律來對待：你現在收到的每個疑問，未來會被放大許多倍，所以每個關於可理解性的問題都當成有效——這不代表你得改邏輯，但可能代表你得**解釋得更清楚**。正確性＋可理解性，正是 LGTM 的兩大判準。
- **Code Consistency（一致性）：** 規模化下，你寫的程式會被別人依賴、維護、重構（甚至由自動工具）。程式需符合某種一致性標準才好被理解與維護，也該**避免過度複雜**。一致與簡單的程式更易被工具更新（重構時）——**「若某模式在程式碼庫裡總是用同一種方式做，就更容易寫工具去重構它。」** 這就是為何 **LGTM 狀態（正確性＋可理解性）要與 readability 核准分離**：readability 只能由通過該語言可讀性訓練的人授予（如 Java 程式要由有「Java readability」的人核准）。一致性有時與「功能更好」衝突——readability 審查者可能偏好一個功能上未必更佳、但**更易理解**的較簡單寫法。
- **Psychological and Cultural Benefits（心理與文化效益）：** 審查強化「**程式不是『你的』，而是集體事業的一部分**」。它逼作者讓別人有輸入、為了整體利益而妥協。人天生對自己的手藝自豪、不願被批評；審查流程提供一個**緩衝機制**，把可能情緒化的互動轉成**規定好的、中性的**形式——「流程當壞人，審查者仍可當好人」（連工具都叫「Critique／批評」，你不能怪審查者盡責地挑剔）。另一個心理效益是**validation（肯定）**：再強的工程師也可能有冒牌者症候群，審查能成為對其工作的肯定與認可。此外，**發起審查這件事本身就逼作者多用點心**——「把所有 ducks 排好隊」，送出前那一刻的反思，正是檢查自己有沒有漏東西的最佳時機（「我單元測試還沒寫完，但晚點再補」這種偷懶會被擋下）。
- **Knowledge Sharing（知識分享）：** 這是**最重要卻最被低估**的效益。作者多半挑該領域的專家當審查者，審查過程讓專家把領域知識傳給作者（甚至直接在工具裡分享建議的程式編輯）。資訊常**雙向流動**：作者與審查者都能學到新技巧與模式。**「工程師未必讀每封寄給他的 email，但他們傾向回應每一個寄來的 code review。」** 這種知識傳遞跨時區、跨專案，及時又可行動（許多 Google 工程師是透過 code review **第一次「認識」彼此**的）。加上每次變更都進了程式碼庫，審查本身成為**歷史紀錄**——任何人都能考古某個模式何時被引入、調出當時那次審查。
- **歷史紀錄（Historical record）：** 見上；每次審查都留檔，常為遠多於原作者與審查者的人提供洞見。

### Code Review Best Practices（最佳實務）
核心主張：審查確實會帶來摩擦與延遲，但**多半是實作方式的問題，而非審查本身**。讓流程保持**敏捷、快速**，才能正確擴展。

- **Be Polite and Professional（有禮且專業）：** 建立在 Google 的信任與尊重文化上。回饋與批評**必須嚴守在專業範疇**。審查者一般該**尊重作者的做法**，只有在作者做法有缺陷時才指出替代；若數種做法同等有效，採作者偏好。**24 個工作小時內**給回饋；若做不完，至少回覆「已看到、會盡快」。**別零碎地擠牙膏式回饋**（最惹惱作者的，莫過於改完一輪又冒出不相關的新意見）。作者也要專業：**「你不是你的程式碼」**，這變更不是「你的」而是團隊的；把每個審查留言當成 **TODO**——可以不接受，但至少要回應；不同意就說清楚理由、別逕自把留言標成已解決；常用 **PTAL（please take another look，請再看一下）** 來保持討論文明。
- **Write Small Changes（寫小變更）：** **保持流程敏捷最重要的一條。** 一個審查最好聚焦單一議題、易消化。Google 流程**勸阻**「整包成形專案」式的巨大變更，審查者有權因「太大」而退件。小變更也讓人不必為大審查空等、且更容易定位 bug 來源。**「小」大致以約 200 行為界。** 多數變更**約一天內**會被審（指「一天內給初步回饋」，非審完）；**約 35% 的變更只動一個檔案**。代價是：小變更有時難以承載重大新功能（可用 integration branch、不同 diff base 等技巧，但有額外開銷）——把「偏好小變更」當成一種**最佳化**，允許偶爾的大變更。也因為變更小，**幾乎所有審查都由唯一一人審**；若全團隊都得對每個變更表態，流程根本無法擴展。
- **Write Good Change Descriptions（寫好變更說明）：** **第一行是黃金地段**——它是工具裡的摘要、email 主旨、Code Search（[[ch17]]）歷史摘要裡看得到的那行。描述要講清楚**改了什麼、為什麼**；「Bug fix」這種說明對審查者或未來的程式考古者毫無幫助。描述是這次變更的歷史紀錄。實作裡也要慷慨地下註解——**若審查者看不懂你為何這樣做（即使正確），就是該寫更好結構或註解的訊號**。
- **Keep Reviewers to a Minimum（審查者人數降到最低）：** Google 多數審查**剛好一人**。業界（與個人）常想找一群人取得一致同意，但這會**邊際效益遞減**——**「最重要的是第一個 LGTM，後續的並不如你想的那麼有價值；額外審查者的成本很快超過其價值。」** 確需多人時，那些人也該**各自聚焦不同面向**。
- **Automate Where Possible（能自動化就自動化）：** 人的輸入重要，但機械性的部分該自動化、投資工具會有回報。最重要的進展是對變更做**自動靜態分析**（[[ch20]]）——透過 **presubmit（提交前檢查）**：變更初次送審時跑檢查，在寄給審查者前就攔下各種問題、退回給作者先修（避免寄出一封尷尬的 email），讓審查者能專注在比格式更重要的事。

### Types of Code Reviews（審查的四種類型）
核心主張：**並非所有審查都一樣**，不同類型需要不同的關注重點。Google 的變更大致分四類：
- **Greenfield（全新程式）：** 最不常見、卻最該嚴審——這是評估「程式能否經得起時間與規模考驗」最重要的時機。全新程式不該突然冒出（應先過設計審查）；要確認 **API 符合議定的設計、被完整測試（所有 endpoint 都有測試、且當假設改變時測試會失敗）、有適當 owner（新專案最早的審查之一往往就是一個 `OWNERS` 檔）、有足夠註解與補充文件、必要時納入 CI（[[ch23]]）**。
- **Behavioral Changes, Improvements, Optimizations（行為變更、改良、最佳化）：** Google 多數變更屬此。沿用 greenfield 的判準：**這變更必要嗎？它改善了程式碼庫嗎？**（**最棒的修改有些其實是刪除！** 清掉死碼／過時碼是改善程式健康最好的方式之一。）任何行為變更都要連帶修訂相應測試；最佳化可能需附**效能基準測試（benchmark）**。
- **Bug Fixes and Rollbacks（修錯與回滾）：** 修 bug 時**抗拒「順手改別的」的誘惑**——那會放大審查範圍、妨礙回歸測試、也讓別人更難回滾你的變更。bug fix 應**只聚焦修這個 bug**，並（通常）更新相應測試以攔住這個錯。**Rollback（回滾）** 是一個「撤銷前一變更、回到已知狀態」的變更，可在數秒內產生，但**仍需審查**。因此**任何變更都該盡量小而原子化（atomic）**——Google 看過開發者在程式提交後極快就依賴它，回滾有時反而弄壞這些下游；小變更能緩解此問題。
- **Refactorings and Large-Scale Changes（重構與大規模變更，LSC）：** Google 許多變更是**機器自動產生**的（作者不是人、是機器，見 [[ch22]]），但**機器產生的變更一樣要審**。低風險者由具全庫核准權的指定審查者審；風險較高或需在地領域知識者，才請個別工程師審。審機器變更時，**只該標記「你這塊程式特有」的疑慮，別去質疑底層工具或 LSC 流程**——該流程已被審過、個別團隊不能對它行使否決權，否則無法跨組織擴展（有疑慮可另外向 LSC 監督小組反映）。也別藉機**擴大範圍**：跑工具的人可能同時有上百個變更在飛，任何一點不相關的留言都會限制其運作規模。

## 關鍵字補充

- **Precommit Review（提交前審查）** — 程式進入程式碼庫之前必須先被審。Google 對幾乎所有變更（不論大小）都強制執行，是少數的全面性流程之一。
- **LGTM（looks good to me，看起來沒問題）** — 審查者對「正確性＋可理解性」的同意，是一個必要的權限位元；預設只需一個 LGTM。
- **三個批准位元（The three approval bits）** — LGTM（正確性/可理解性）、owner approval（適不適合放這塊程式碼庫）、readability（符合語言風格）。可由一人或多人扮演、任意組合，拆開是為了彈性與規模。
- **OWNERS 檔（OWNERS file）** — 列出某目錄及子目錄負責人的檔案；階層式相加、可解析到個人。讓「誰負責」沿目錄樹一查就到，新專案放一個就成立，無需中央登記。
- **Ownership / Stewardship（擁有權／受託管理）** — 不是「佔有」程式碼，而是以公司最大利益看顧某段程式碼的責任（作者自承「stewards 管家」是更好的詞）。
- **Code Is a Liability（程式碼是負債）** — 程式本身是維護成本而非資產；故動筆前先確認功能值得，重複碼甚至比沒有更糟（「從頭造輪子大概就是做錯了」）。
- **Readability（可讀性認證）** — 經該語言可讀性訓練取得、能核准「程式符合語言風格與最佳實務」的資格；與 LGTM 分離，專責一致性與可維護性（見 [[ch03]]、[[ch08]]）。
- **Small Changes（小變更）** — 保持審查敏捷最重要的實務，約以 200 行為界；多數一天內獲初步回饋，約 35% 只動一個檔案；讓「一人審查」與「易回滾」成為可能。
- **PTAL（please take another look，請再看一下）** — 作者不同意審查者時，提出替代方案並請對方再看一次的禮貌用語，用來保持討論文明。
- **Presubmit（提交前檢查）** — 變更送審時自動跑的靜態分析／格式檢查，在寄給審查者前先攔問題、退回作者，讓人專注在比格式更重要的事（見 [[ch20]]）。
- **Rollback（回滾）** — 撤銷前一變更、回到已知狀態的變更；數秒可生成但仍需審查。故所有變更都應小而原子化，以免回滾連鎖弄壞下游。

## 延伸提問

1. 你團隊把 code review 的價值主要押在「抓 bug」上嗎？若 Google 說正確性**不是**首要效益，那你們從審查得到的**可理解性、一致性、知識分享、歷史紀錄**有被認真對待嗎？下次寫 review 留言時，你會不會多問一句「半年後的人讀得懂嗎」？
2. 把你最近送審的幾個變更量一量行數：有多少超過約 200 行？哪些其實可以**切成數個小而原子化的變更**？回想一次「為了等大 review 而卡住後續工作」的經驗——小變更能不能根治它？
3. 你們的審查需要幾個人點頭才放行？對照「**最重要的是第一個 LGTM**」：你們是不是在用「找一群人取得一致同意」來換取安心，卻付出了遞延與遞減效益的代價？哪些變更其實一人審就夠？
4. 套用「**三個批准位元**」想像你的團隊：你們有沒有把「正確性」「這塊程式碼庫的擁有權」「語言風格一致性」**混成一團、由同一種人一次審完**？把它們拆開，會不會讓 intern 與資深工程師都更好走？你們的 `OWNERS` 是清楚的嗎？
5. 你修 bug 時會不會「順手」改其他東西？回想一次因此**難以回滾或回歸測試**的痛。你能不能養成「bug fix 只修這個 bug＋補一個能攔住它的測試」的紀律？對照「程式碼是負債」——你上次審查時，有沒有想過「**最好的修改其實是刪除**」？
