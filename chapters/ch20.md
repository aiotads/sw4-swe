# 第 20 章｜Static Analysis（靜態分析）

> *"Focus on developer happiness."*
> 「以開發者的幸福感為念。」

## 摘要

**靜態分析（static analysis）** 是「分析原始碼」（而非執行中的程式，後者稱 dynamic 動態分析）以找出 bug、反模式與其他可在不執行下診斷的問題。它能在程式進入 production 前**及早抓 bug**（如溢位的常數運算式、永不執行的測試、會崩潰的 logging 格式字串）；但用途遠不只抓 bug——在 Google 還用來**編碼最佳實務、讓程式跟上現代 API、預防/減少技術債**（檢查命名慣例、標記 deprecated API、指出更簡潔的等價寫法），更是**API 棄用流程**裡防止「回潮」的關鍵工具（[[ch15]]、[[ch22]]），甚至能**教育開發者、預防反模式進入程式碼庫**。

讓靜態分析有效的兩大特性：**Scalability（可擴展）**——工具要 shardable、incremental，只分析待審變更**所影響的檔案/行**，而非整個專案；也要能擴展**分析的數量與種類**（向全公司徵集貢獻），並**直接把結果秀給相關工程師**而不形成瓶頸。**Usability（可用性）**——權衡使用者的成本效益，**聚焦於新引入的警告**（對沒在改的、運作正常的程式，「修」它可能反而引入 bug、效益不明），**能自動修就自動修**，且平順整合進工作流。

由此導出三條關鍵教訓：**①Focus on developer happiness（以開發者幸福為念）**——量測、維持低 false-positive 率、建立反饋迴路、贏得**使用者信任**；關鍵概念是 **effective false positive（有效誤報）**：只要開發者沒採取正面行動，就算技術上正確也算誤報。**②Make static analysis part of the core workflow（成為核心工作流的一部分）**——主要整合點是 **code review**（「sweet spot」：開發者已在改動心態、被審查阻塞時有時間跑分析、有同儕壓力、替審查者省時）。**③Empower users to contribute（賦能使用者貢獻）**——讓領域專家用簡單 API（如 Refaster）寫分析，把專業規模化。

Google 的平台是 **Tricorder**（與 Critique 整合，灰色框；微服務、每天分析 5 萬+ 變更）。新 check 四準則：**可理解、可行動且易修、effective false positive < 10%、對程式品質有顯著影響潛力**（整體有效誤報率略低於 5%）。其餘關鍵：**整合工具**（Error Prone/clang-tidy 的 AST 反模式、IfThisThenThat、binary size checker）、**反饋管道**（「Not useful」/「Please fix」按鈕；有時修好「訊息」就解決問題）、**suggested fixes**（可一鍵套用；風格問題該自動修）、**per-project（非 per-user）客製**、**presubmits**（擋提交）、**compiler 整合**（推到更早、break the build；**從不發 compiler warning**——要嘛當 error 要嘛不顯示，因為開發者會忽略警告）、IDE/瀏覽時的分析。

## 為什麼要讀這一篇

- **你導入過靜態分析工具，結果大家都嫌吵、然後忽略它。** 本章點破關鍵：**低 false-positive 率與使用者信任**才是成敗，而衡量標準是 **effective false positive**（只要開發者沒採取正面行動就算誤報）。建立**反饋迴路**、把誤報壓到 10% 以下，工具才會被真正使用。
- **你不知道靜態分析該在哪個環節跑。** 本章給你完整地圖：**IDE（要 < 100ms）→ code review（sweet spot）→ presubmit（擋提交）→ compiler（break the build）**——愈往左愈早、愈往右愈強制。並解釋為何 **code review 是甜蜜點**、為何 **compiler 從不發 warning**。
- **你想讓靜態分析規模化、又不想養一支龐大的工具團隊。** 本章的答案是 **empower users to contribute**：建一個「易於 plug-in 的生態」、用簡單 API（Refaster：給前後程式片段就能定義轉換）讓領域專家自己寫 check——Tricorder 100+ 分析器多數來自團隊外。
- **你在猶豫要不要讓使用者各自關掉煩人的檢查。** 本章的血淚教訓：**做 per-project 客製、不要 per-user**——當初移除 user 客製後，「煩人」的抱怨反而幫他們找出工具真正的 bug（C++ linter 跑在 Obj-C 上、HTML linter 高誤報）；user 客製只會藏起 bug 與壓住反饋。

## 細節重點

### Characteristics of Effective Static Analysis（有效靜態分析的特性）
核心主張：學界數十年聚焦於新分析技術，但**「可擴展性與可用性」是相對晚近的重點**。
- **Scalability（可擴展）：** 工具必須 **shardable、incremental**——不分析整個大專案，而是**聚焦於待審變更所影響的檔案**，通常只對**編輯過的檔案或行**顯示結果。規模也有好處：程式碼庫夠大，能找的 bug「低垂果實」很多。還要擴展**分析的數量與種類**（向全公司徵集），並透過**直接把結果秀給相關工程師**避免瓶頸。
- **Usability（可用性）：** 要考慮使用者的**成本效益**——成本可能是開發者時間或程式品質（修一個 dead code 警告可能讓未測的程式突然執行）。故**聚焦於新引入的警告**（既有、運作正常的程式通常只在特別重要時——安全、重大修正——才值得修），這也讓看到警告的開發者**握有最相關的脈絡**。開發者時間寶貴：**能自動修就自動修**，只秀真正會負面影響品質的問題，並**平順整合進工作流**（同一工作流讓專屬工具團隊能與程式碼同步演進工具）。

### 三條關鍵教訓
- **① Focus on Developer Happiness（以開發者幸福為念）：** 量測工具表現（不量就無法修）；**只部署低 false-positive 率的工具**；即時徵集並回應反饋，養出「工具使用者↔工具開發者」的良性循環——**使用者信任對成功極其重要**。**false negative（漏報）vs false positive（誤報）**：研究傳統聚焦於減少漏報，但**實務上低誤報率才是開發者願意用的關鍵**（誰想在數百個假報告裡撈幾個真的？）。**Perception（感知）** 也是關鍵——技術上正確但被誤解（訊息混亂）、或正確但無關緊要的警告，使用者的反應都跟誤報一樣，故引入 **effective false positive（有效誤報）**：**只要開發者看了沒採取正面行動，就算有效誤報**。反之，若分析報的不完全對、但開發者仍樂意改以提升可讀性（如 `contains` →`containsValue` 更清楚），就**不算**有效誤報。
- **② Make Static Analysis Part of the Core Developer Workflow（成為核心工作流的一部分）：** Google 透過與 **code review** 整合來做。幾乎所有程式提交前都會被審——開發者已在「改動心態」，故工具的建議能無太多干擾地被採納；他們送審後通常 context switch、被審查者阻塞，**有時間讓分析跑**（即使要幾分鐘）；有**同儕壓力**促使處理警告；分析還能自動標出常見問題、**替審查者省時、幫審查擴展**。**Code review 是分析結果的甜蜜點。**
- **③ Empower Users to Contribute（賦能使用者貢獻）：** Google 有許多領域專家——靜態分析是把其專業**規模化**的機會（讓他們寫新分析器或單一 check）。除了專家，發現某 bug 的開發者也會貢獻 check 以防同類 bug 再現。重點是**建一個「易於 plug-in 的生態」**而非整合一小組現成工具；提供**簡單 API**（如 **Refaster**：只要給前/後程式片段示範期望的轉換，就能寫出分析器）讓全公司工程師都能寫分析——不只分析或語言專家。

### Tricorder：Google's Static Analysis Platform
核心主張：**Tricorder** 是 Google 靜態分析的核心平台，源於數次「整合靜態分析到工作流」的失敗——與前嘗試的關鍵差異是**死磕「只交付有價值的結果」**。它與主審查工具 **Critique（[[ch19]]）整合**（警告以灰色留言框顯示）；用**微服務架構**（透過 FUSE 檔案系統讀變更版本、存取快取的 build 輸入輸出），每天分析 **5 萬+ 變更**、常每秒跑數個分析。

新 Tricorder check 的**四準則**：**①Be understandable（任何工程師都看得懂輸出）②Be actionable and easy to fix（含如何修的指引）③effective false positive < 10%（至少 90% 時間指出真問題）④對程式品質有顯著影響潛力**。Tricorder 報告 **30+ 種語言**、含 **100+ 分析器（多數來自團隊外）**，其中 7 個本身是有數百個 check 的 plug-in 系統；**整體有效誤報率略低於 5%**。

- **整合工具：** **Error Prone（Java）、clang-tidy（C++）** 擴充編譯器找 AST 反模式（例：對 `int` 型別做 `f >>> 32` 右移是 no-op，啟用此 check 時修了 31 處）；**Deleted Artifact Analyzer**（刪了被別處引用的檔就警告）、**IfThisThenThat**（指定兩檔須同改）、**Chrome Finch analyzer**（A/B 實驗設定，會 RPC 其他服務）、**binary size checker**。幾乎全是 **intraprocedural（程序內）**。
- **整合反饋管道：** 顯示「**Not useful**」按鈕（可直接對分析器作者開 bug）；審查者可按「**Please fix**」要求作者處理。團隊追蹤高「not useful」率的分析器、不改善就停用。建立並調校這個反饋迴路花了大量工夫，但回報巨大。**有時修好「訊息」就解決問題**（printf `%s` 例子：改了診斷文字後，誤報的 bug 回報就停了）——好訊息在最相關的時間點解釋了「哪裡錯、為何、怎麼修」。
- **Suggested fixes（建議修正）：** 可從 Critique 直接套用、或用 CLI 套用整個變更；**風格問題尤其該自動修**（formatter）。審查者每天按「Please fix」數千次、作者每天套用自動修正約 **3,000 次**、「Not useful」每天約 **250 次**。
- **Per-project（非 per-user）客製：** 在建立信任後加入「optional」分析器（如 **Proto Best Practices**——只對有儲存序列化資料的專案才需要）。**關鍵洞見：做專案層級客製、不要使用者層級**——確保全隊對結果有**一致的視圖**，避免「一人在修、另一人在引入」。血淚教訓：當初移除 Critique 的 user 客製後招來抱怨，但**追問「為何煩」反而找出 linter 的真 bug**（C++ linter 跑在 Obj-C 上產生無用結果、HTML linter 高誤報被普遍壓掉）——**user 客製只會藏起 bug 與壓住反饋**。
- **Presubmits（提交前檢查）：** 因開發者可忽略 code review 的警告，Google 另有能**擋住提交**的 presubmit（檢查「DO NOT SUBMIT」、測試檔有附、測試通過、無特定 Tricorder 問題、格式正確）。團隊可寫自訂 presubmit（比全公司更嚴），但會讓 LSC 更難，故描述含 `CLEANUP=` 的變更會跳過部分（見 [[ch22]]、[[ch23]]）。
- **Compiler 整合：** 更早通知更好——**break the build 是無法忽略的警告**，但非總可行。高度機械化、無 effective false positive 的分析（如 Error Prone 的「ERROR」check）全部啟用進 Java 編譯器。準則：**可行動易修、無 effective false positive、只報正確性（非風格/最佳實務）**。啟用新 check 前須先**清掉程式碼庫裡所有既有案例**（用 MapReduce 跑遍全庫、必須能自動產生 fix）。**我們也力求從不發 compiler warning**——因為**開發者會忽略警告**，故要嘛當 error（break build）要嘛不顯示（Go 甚至把未用變數/import 當 error）。
- **編輯與瀏覽時的分析：** IDE 整合要**極快（理想 < 100ms）**、且要確保多個 IDE 跑出一致結果，加上 IDE 會起落，故比接入審查流程更麻煩。Code review 還有特定好處（能納入整個變更脈絡、想忽略結果得說服審查者）。但對某些分析（如安全），開發者確實想在**瀏覽程式碼時看整個程式碼庫的結果**（規劃清理時也是）。

工程意涵（結論）：靜態分析能改善程式碼庫、及早抓 bug，**讓更昂貴的流程（人工審查、測試）聚焦於非機械可驗證的問題**。靠提升基礎設施的**可擴展性與可用性**，Google 把靜態分析變成軟體開發的有效一環。

## 關鍵字補充

- **Static Analysis（靜態分析）** — 分析原始碼（非執行中程式）以找 bug/反模式/問題；及早抓 bug，也用於編碼最佳實務、跟上現代 API、預防技術債、防棄用回潮。
- **Scalability（可擴展）** — 工具 shardable/incremental，只分析待審變更所影響的檔案/行；也擴展分析的數量與種類、直接秀給相關工程師避免瓶頸。
- **Usability（可用性）** — 權衡成本效益；聚焦新引入的警告、能自動修就自動修、平順整合進工作流。
- **Effective False Positive（有效誤報）** — 以使用者行動定義的誤報：開發者看了沒採取正面行動就算；感知與訊息清晰度同樣重要。
- **三條關鍵教訓** — Focus on developer happiness（信任、低誤報、反饋迴路）、Make it part of the core workflow（code review 是甜蜜點）、Empower users to contribute（簡單 API、領域專家）。
- **Tricorder** — Google 靜態分析平台，與 Critique 整合、微服務、每天 5 萬+ 變更；新 check 四準則（可理解/可行動易修/<10% 有效誤報/顯著影響潛力）。
- **Error Prone / clang-tidy** — 擴充編譯器找 Java/C++ 的 AST 反模式；Error Prone 的「ERROR」check 啟用進編譯器、防同類錯誤再現。
- **Refaster** — 用前/後程式片段示範期望轉換就能寫分析器；讓非分析/語言專家也能貢獻。
- **Suggested Fixes（建議修正）** — 可從 Critique 或 CLI 一鍵套用；風格問題該自動修（formatter）。
- **Per-project Customization（專案層級客製）** — 客製化做在專案層級而非使用者層級，確保全隊一致視圖；user 客製只會藏 bug、壓反饋。
- **Presubmit（提交前檢查）** — 能擋住提交的檢查（DO NOT SUBMIT、測試、格式、特定 Tricorder 類別）；`CLEANUP=` 可跳過部分以利 LSC。
- **Never Issue Compiler Warnings（從不發編譯器警告）** — 開發者會忽略警告，故要嘛當 error（break build）要嘛不顯示；Go 把未用變數/import 當 error。

## 延伸提問

1. 你導入的靜態分析工具，**false-positive 率**是多少？你用「技術正確性」還是 **effective false positive（開發者有沒有採取正面行動）** 來衡量？有沒有「技術正確卻因訊息混亂被當誤報」的 check？你有沒有一條**反饋迴路**讓使用者能回報「沒用」並真的據以調整？
2. 你的靜態分析在**哪個環節**跑？沿著 **IDE→code review→presubmit→compiler** 這條線，你目前落在哪、該往哪推？哪些檢查機械到可以**break the build**、哪些該留在 code review 當「甜蜜點」？對照 [[ch19]]、[[ch23]]。
3. 你的團隊是不是讓使用者**各自關掉**煩人的檢查？回想本章「移除 user 客製反而找出真 bug」的教訓——你能不能改成**專案層級**客製，並把「煩人」當成找 bug 的線索而非壓住反饋？
4. 你有沒有把領域專家的知識**規模化**？哪些「你修過一次、卻可能在別處再現」的 bug，值得寫成一個 check 防止回潮？有沒有像 Refaster 那樣**夠簡單的 API**讓非專家也能貢獻分析？
5. 你的工具有**自動修正**嗎？「能自動修就自動修」——哪些目前靠人工審查者一一指出的問題（格式、風格、deprecated 用法）其實該被 formatter/auto-fix 取代，把人的時間留給非機械可驗證的問題？對照 [[ch09]]。

## 跨章連結

- [[ch19]] Critique — 靜態分析的主要整合點；Tricorder 結果以灰色 chip 在 Critique 的 diff 上呈現、附 suggested fix；「Please fix」按鈕串起審查者與作者。
- [[ch09]] Code Review — 「code review 是分析結果的甜蜜點」「替審查者省時、幫審查擴展」直接呼應第 9 章；自動修風格問題免得浪費人工審查時間。
