# 第 12 章｜Unit Testing（單元測試）

> *"Test behaviors, not methods. … Follow DAMP over DRY when sharing code for tests."*
> 「測試『行為』，而不是『方法』。……在測試裡共用程式碼時，奉行 DAMP（描述性而有意義）而非 DRY（不要重複自己）。」

## 摘要

接續 [[ch11]] 的 size／scope 兩軸，本章聚焦 **unit test（單元測試，範圍窄、通常 size 小）**——Google 最常見的測試類型（建議比例約 **80% 單元／20% 較廣範圍**，工程師一天常跑數千個）。整章的核心只有一個詞：**maintainability（可維護性）**。可維護的測試「就是會動」——寫完後你不必再想它，直到它失敗，而失敗時指向的是**有清楚成因的真實 bug**。本章用 Mary 的故事點出反例：她改了幾行程式卻被測試系統噴一整頁錯誤，逐一修了一整天——這些測試沒抓到任何真 bug，只是壞在它們對程式**內部結構**的假設上。

可維護性的兩大敵人是**brittle（脆弱）** 與 **unclear（不清晰）**。防脆弱有三招：**①追求「不會變」的測試（strive for unchanging tests）**——四種程式變更裡，只有「**改變既有行為**」才該動到既有測試（純重構、加新功能、修 bug 都不該）；**②透過 public API 測試**（用使用者的方式呼叫系統，而非戳實作細節，「先走正門」）；**③測『狀態』而非『互動』**（state 而非 interaction；過度依賴 mocking 框架是脆弱測試的元兇，能用真實物件就用）。

寫清晰測試的要訣：**完整且精簡（complete & concise）**、**測行為而非方法（behaviors not methods，用 given/when/then 結構、以行為命名）**、**別在測試裡放邏輯（no logic，一看就對）**、**寫清楚的失敗訊息**。最後是共用程式碼的哲學：**DAMP, not DRY**——測試裡**一點重複是可以的**，只要它讓測試更具描述性與意義；DAMP 與 DRY 互補而非取代。

## 為什麼要讀這一篇

- **你（像 Mary 一樣）每次小改動都被一堆測試噴錯，明明沒introduce 任何 bug。** 那是 **brittle tests**。本章給你根因（測試綁死了實作細節）與三帖藥：**追求不會變的測試、透過 public API 測、測狀態而非互動**。讀完你會知道怎麼寫「改 production 不必跟著改」的測試。
- **你習慣「一個 method 配一個 testXxx」。** 本章告訴你這會隨 method 變複雜而失控；改成**測行為（behaviors）**——「given… when… then…」——每個測試只管一個行為，讀起來像自然語言、失敗時一眼知道哪個行為壞了。
- **你以為測試也該奉行 DRY、把重複抽乾。** 本章反直覺地主張測試該 **DAMP**：因為**測試本身沒有測試**，複雜度的代價在測試裡更高；把資訊藏進 helper 反而讓測試「不完整」、藏住 bug。一點重複換來「整段讀完就懂、不必跳出去」的清晰，很划算。
- **你被 flaky/慢/難懂的測試拖垮，甚至想直接刪掉看不懂的測試。** 本章提醒：刪掉看不懂的測試＝在覆蓋率上開個你不知道的洞，且代表它可能**多年來價值為零**。與其如此，不如一開始就寫**清晰、不放邏輯、失敗訊息明確**的測試。

## 細節重點

### The Importance of Maintainability（可維護性為何最重要）
核心主張：抓 bug 之後，測試最重要的目的是**提升工程師生產力**。unit test 因為「小、快、確定、易與程式同時寫、易懂、可當文件」而特別擅長這件事。但 Mary 的故事顯示反面：**寫得差的測試會讓生產力不增反減**——她的改動沒引入任何真 bug，卻因測試綁死了程式內部結構而被迫逐一修補，且補丁讓測試未來更難懂。

工程意涵：壞測試**必須在 check-in 前就修好**，否則會拖累未來的所有工程師。Mary 遇到的問題分兩類——**brittle（脆弱：被無關、無害的改動弄壞）** 與 **unclear（不清晰：壞了也搞不懂它想測什麼、怎麼修）**。本章其餘部分就是對治這兩者。

### Preventing Brittle Tests（防止脆弱測試）
核心主張：brittle test＝在「不引入任何真 bug 的無關改動」下卻失敗的測試（注意：這和 **flaky** 不同——flaky 是 production 程式沒變卻非確定性地時過時不過）。在小團隊裡每次改動微調幾個測試或許還好，但**在 Google 規模下尤其致命**：一個工程師一天跑數千測試、一次 large-scale change（[[ch22]]）可觸發數十萬測試——即使只有一小比例假性失敗，也會吃掉巨量工程時間。

#### Strive for Unchanging Tests（追求「不會變」的測試）
核心主張：理想的測試是**不會變的**——寫完後，除非「被測系統的需求」改變，否則永遠不需要改它。對照工程師對 production 程式做的四種變更：
- **純重構（Pure refactorings）：** 不改介面、只改內部 →**測試不該變**（測試的角色正是確認重構沒改行為；若要改測試，要嘛它不是純重構、要嘛測試寫在錯誤的抽象層級）。
- **新功能（New features）：** 既有行為不受影響 →只需**為新行為加新測試**，不該動既有測試。
- **修 bug（Bug fixes）：** bug 的存在代表初始測試**漏了一個 case** →補上那個 case，同樣不該動既有測試。
- **行為變更（Behavior changes）：** **唯一**該更新既有測試的情況——且通常**遠比其他三種昂貴**（使用者依賴現有行為，改它需與使用者協調；改測試代表你正在打破系統的「明示契約」）。

工程意涵：寫完測試後，重構／修 bug／加功能都不該回去碰它——這正是「能在規模上工作」的關鍵：擴充系統只需就改動寫少量新測試，而非可能得碰過每一個曾寫過的測試。

#### Test via Public APIs（透過 public API 測試）
核心主張：確保「測試不隨實作變」最重要的一招，就是**用使用者的方式呼叫系統——打它的 public API，而不是實作細節**（「先走正門 / use the front door first」）。如此一來，能弄壞測試的改動，**按定義也會弄壞某個使用者**；這種測試還能順便當使用者的範例與文件。書中的 transaction 例子：與其偷拿掉 `private`、直接測 `isValid()`／`saveToDatabase()`（任何重新命名、抽 helper、改序列化格式都會弄壞測試），不如只透過 `setAccountBalance`／`processTransaction`／`getAccountBalance` 這些公開方法達到同樣的覆蓋。

工程意涵：「public API」沒有死板定義，問題直指「unit 到底是什麼」——指的是該 unit **對團隊外第三方暴露的 API**（不必然等同語言的 `public`）。判準：只為支援一兩個類別的 **helper class** 不算獨立 unit（透過那些類別測）；任何人不必問 owner 就能用的、或設計成通用的 **support library**，就該被當 unit 直接測（即使會與使用者的測試有些冗餘——這冗餘有價值）。

#### Test State, Not Interactions（測狀態，而非互動）
核心主張：驗證系統行為有兩種方式——**state testing（狀態測試）**：呼叫後觀察系統自身變成什麼樣；**interaction testing（互動測試）**：檢查系統對其協作者做了哪一連串動作。**互動測試更脆弱**，理由與「測私有方法比測公開方法脆弱」相同：互動測試檢查的是「系統如何抵達結果」，但你通常只該在乎「結果是什麼」。書中 `verify(database).put("foobar")` 的例子有兩種出錯方式：① bug 把記錄寫完馬上刪掉，測試卻仍通過（該失敗卻沒）；②重構成呼叫等價的另一個 API，測試卻失敗（該通過卻沒）。改成測狀態 `assertThat(accounts.getUser("foobar")).isNotNull()` 就穩健得多。

工程意涵：互動測試出問題最常見的原因是**過度依賴 mocking 框架**（它讓你輕易做出「記錄並驗證每一次呼叫」的 double）——故 Google **偏好真實物件**，只要它們夠快且確定（詳見 [[ch13]]）。

### Writing Clear Tests（寫清晰的測試）
核心主張：再怎麼避免脆弱，測試遲早會失敗——而失敗是好事（提供有用訊號）。失敗只有兩種原因：**①被測系統有問題**（測試正在做它該做的事）；**②測試本身有瑕疵**（系統沒事，是測試寫錯了；若是既有測試，那就是 brittle）。工程師第一份工作是判斷屬於哪一種、再診斷真正問題——速度取決於測試的**clarity（清晰度）**。清晰尤其重要，因為**測試常比寫它的工程師活得更久**：一個多年前、由早已離隊的人寫的失敗測試，你可能永遠搞不懂它的目的（不像 production 程式可從「誰呼叫它、拿掉會壞什麼」反推）。最糟的下場是看不懂就刪掉——這不只開了覆蓋率的洞，也代表它**可能存在多年卻價值為零**。

- **完整且精簡（Complete & Concise）：** 測試**完整**＝本體含讀者理解「如何得到結果」所需的一切資訊；**精簡**＝沒有其他分散注意力的無關資訊。反例（12-6）：建構子塞一堆無關參數、關鍵數字藏在 helper 裡（「這個 5 哪來的？」）。改法：用 helper 隱藏無關的建構細節（更精簡），同時把關鍵輸入講明（更完整）。
- **測行為，而非方法（Test Behaviors, Not Methods）：** 很多人本能地「一個 production method 配一個 test method」，但 method 一複雜，測試就跟著失控。更好的做法是**為每個行為寫一個測試**——**behavior（行為）** 是系統對「在某狀態下、給一串輸入」會如何回應的任何保證，常可用 **given / when / then** 表達（「given 帳戶為空，when 試圖提款，then 交易被拒」）。method 與 behavior 是**多對多**。把一個測多個訊息的 method-driven 測試拆成數個 behavior-driven 測試，雖多了 boilerplate 卻清晰得多：讀起來像自然語言、因果更清楚、也鼓勵新增精簡的測試而非往舊的上面堆。
- **結構上凸顯行為：** 每個行為三部分——**given（如何設定系統）／when（對系統採取的動作）／then（驗證結果）**（亦稱 arrange/act/assert）。結構明確時最清晰（Cucumber、Spock 直接內建 given/when/then；其他語言用空白行＋註解）。這讓測試可在**三種粒度**閱讀：①測試方法名（粗略行為）②given/when/then 註解（正式描述）③實際程式（精確表達）。多步驟流程可交替 when/then、用「and」拆長區塊，但**每個測試只該涵蓋單一行為**（絕大多數只需一個 when、一個 then）。
- **以行為命名：** 測試名常是失敗報告裡第一個、甚至唯一可見的 token，是溝通問題的最佳機會。好名字同時描述「採取的動作」與「預期結果」。卡住時的訣竅：用「**should**」開頭，配上被測類別名就讀成一句話——`BankAccount` 的 `shouldNotAllowWithdrawalsWhenBalanceIsEmpty` 讀作「BankAccount should not allow withdrawals when balance is empty」。若名字裡需要「**and**」，很可能你在測多個行為、該拆成多個測試。
- **別在測試裡放邏輯（Don't Put Logic in Tests）：** 清晰的測試**一看就對**。複雜度多半來自邏輯（運算子、迴圈、條件）——一有邏輯，你就得在腦中算一遍而非直接讀出答案。書中 URL 例子：僅僅一個字串串接就藏住了「兩條斜線」的 bug；把整個字串寫死，bug 立刻現形。教訓：測試裡**寧用直線程式碼也別耍小聰明**，必要時容忍一點重複以換取描述性（見下方 DAMP）。
- **寫清楚的失敗訊息：** 理想上，工程師光讀 log 裡的失敗訊息就能診斷、不必翻測試本身。好訊息清楚區分**預期 vs 實際**並給脈絡。`Test failed: account is closed`（含糊）→ `Expected an account in state CLOSED, but got {…state: "OPEN"}`（清楚）。好的斷言庫有幫助：JUnit 的 `assertTrue(colors.contains("orange"))` 只能說「expected true but was false」；Google 的 **Truth** 庫 `assertThat(colors).contains("orange")` 能說「`<[red, green, blue]>` should have contained `<orange>`」。Go 則慣用 `t.Errorf("Add(2, 3) = %v, want %v", result, 5)`。

### Tests and Code Sharing: DAMP, Not DRY（共用程式碼：DAMP 而非 DRY）
核心主張：多數軟體追求 **DRY（Don't Repeat Yourself）**——每個概念只在一處表述、把重複降到最低，好處是改一處即可。但在**測試**裡這套利弊不同：好測試本就該穩定、且你通常**希望**它在系統改變時壞掉，所以 DRY 的好處在測試裡沒那麼大；同時複雜度的代價更高（production 有測試保護，**測試卻只能靠自己**，不夠「自明地正確」就有 bug 風險）。因此測試應追求 **DAMP——Descriptive And Meaningful Phrases（描述性而有意義的語句）**：**一點重複是可以的**，只要它讓測試更簡單清晰。書中對照：過度 DRY 的測試把細節藏進 `createUsers`／`validateForumAndUsers` 等 helper（還藏了 bug、本體雖簡潔卻不完整）；改成 DAMP 後雖較長、有重複，但每個測試**整段讀完就懂、不必離開本體**。

工程意涵：**DAMP 不是取代 DRY，而是互補**。helper 與測試基礎設施仍能藉「抽掉與當前行為無關的重複步驟」讓測試更清晰——重點是**為了描述性與意義而抽，而非單純為了消除重複**。共用程式碼的常見模式：
- **Shared Values（共用值）：** 與其用語意含糊的共用常數（`ACCOUNT_1`／`ACCOUNT_2`，得往上捲才知道是什麼），不如用**帶預設值的 helper method**（具名參數，或用 Builder 模式模擬）——讓每個測試只指定它在乎的值。甚至可**輕微隨機化未明設的預設值**，避免兩個實例意外相等、或工程師硬寫死對預設的依賴。
- **Shared Setup（共用初始化）：** `@Before` 等 setup 方法適合建構**被測物件與其協作者**（多數測試不在乎其建構參數時）。風險：若測試開始**依賴 setup 裡的特定值**就會變不完整（讀者得去翻 setup 才知道「Donald Knuth」哪來的）——這類測試該**直接在本體覆寫**它在乎的值。
- **Shared Helpers and Validation（共用 helper 與驗證）：** 用 helper 簡潔建值是好的；但**每個測試結尾呼叫一個固定的 `validate()`** 是壞習慣（測試變得不以行為驅動、難判斷意圖、bug 一來就讓大量測試一起失敗）。較好的 validation helper **只斷言「單一概念事實」**，且適合「概念簡單但需迴圈/條件實作、放進本體會傷清晰」的情況。
- **Defining Test Infrastructure（定義測試基礎設施）：** 跨多個 test suite 共用的程式碼。它更像 **production 程式**（有許多呼叫者、難改不破壞），故**必須有自己的測試**、被當成獨立產品對待。多數測試基礎設施其實是知名第三方庫（如 JUnit）——組織應**儘早、普遍地標準化**（Google 多年前**強制 Mockito 為唯一 Java mocking 框架**、禁止新測試用其他，當時招怨，今日公認是好決定）。

## 關鍵字補充

- **Unit Test（單元測試）** — 範圍窄（單一 class/method）、通常 size 小的測試；Google 最常見、建議約 80% 比例；首要價值是可維護地提升生產力。
- **Maintainability（可維護性）** — 可維護的測試「就是會動」：寫完不必再想，直到失敗，且失敗指向有清楚成因的真實 bug。本章核心。
- **Brittle Test（脆弱測試）** — 在不引入真 bug 的無關改動下仍失敗的測試；多因綁死實作細節。與 flaky（非確定性失敗）不同。
- **Unchanging Tests（不會變的測試）** — 理想測試：除非系統需求改變否則永不需動。四種變更中只有「改變既有行為」該更新既有測試。
- **Test via Public APIs（透過 public API 測試）** — 用使用者的方式呼叫系統（走正門），而非戳實作細節；弄壞測試＝弄壞使用者，形成明示契約。
- **State vs Interaction Testing（測狀態 vs 測互動）** — state＝觀察呼叫後系統變成什麼；interaction＝檢查對協作者做了哪些呼叫。互動測試較脆弱（在乎「如何」而非「結果」）。
- **Behaviors, Not Methods（行為而非方法）** — 為每個行為寫一個測試而非每個方法；behavior 是「在某狀態下對輸入的回應保證」，用 given/when/then 表達，method↔behavior 多對多。
- **Given / When / Then（arrange / act / assert）** — 行為的三部分結構：設定系統／採取動作／驗證結果；讓測試可在名稱、註解、程式三種粒度閱讀。
- **DAMP（Descriptive And Meaningful Phrases）** — 測試的程式共用哲學：容忍一點重複以換取描述性與意義；與 DRY 互補而非取代（因測試本身無測試、複雜度代價更高）。
- **Truth** — Google 的 Java 斷言庫，能產生比 JUnit `assertTrue` 更有用的失敗訊息（明確點出 subject 與預期）。
- **Test Infrastructure（測試基礎設施）** — 跨多 suite 共用的程式碼；更像 production（有許多呼叫者），須有自己的測試、儘早標準化（如強制 Mockito）。

## 延伸提問

1. 拿你團隊最近「改 production 卻被迫跟著改一堆測試」的經驗：那些測試是不是綁死了**實作細節**（戳 private、verify 互動）？若改成**透過 public API 測狀態**，這次改動還會弄壞它們嗎？對照「四種變更」——你動到的是哪一種？
2. 翻一個你寫的 `testXxx`（對應某個 method）：它是不是其實在測**多個行為**？試著用 **given/when/then** 把它拆成數個 behavior-driven 測試，並用「should…」重新命名——拆完後，失敗報告會不會更快告訴你「哪個行為壞了」？
3. 你的測試裡有沒有**邏輯**（迴圈、條件、字串拼接、計算）？挑一段把邏輯攤平成直線程式碼——會不會像書中 URL 例子一樣，攤平後反而看見一個原本藏住的 bug？
4. 你的測試套件是「太 DRY」還是「DAMP」？有沒有把關鍵資訊藏進 `createX`／`validateX` 之類 helper，導致讀單一測試得跳出去好幾處才懂？哪些 helper 其實該**內聯回測試本體**以換取清晰？
5. 你團隊的**測試基礎設施**（共用 fixture、自製 framework）有沒有自己的測試？mocking／斷言庫有沒有**標準化成一套**？回想一次「因測試框架太多而互相看不懂」的痛——值不值得像 Google 強制 Mockito 那樣收斂？對照 [[ch13]]。
