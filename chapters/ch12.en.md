# Chapter 12 — Unit Testing

> *"Test behaviors, not methods. … Follow DAMP over DRY when sharing code for tests."*
> Test what the system promises, not how it's built — and let tests repeat themselves a little if it makes them clear.

## Summary

Continuing [[ch11]]'s size/scope axes, this chapter focuses on the **unit test** (narrow scope, usually small size) — Google's most common kind (recommended mix ≈ **80% unit / 20% broader**, with engineers running thousands a day). The whole chapter turns on one word: **maintainability.** Maintainable tests "just work" — after writing them you don't think about them again until they fail, and a failure points to a **real bug with a clear cause.** The chapter uses Mary's story as the anti-example: she changes a few lines and gets a screenful of test errors, spending a day fixing them one by one — none caught a real bug; they merely broke on assumptions about the code's **internal structure.**

Maintainability's two enemies are **brittleness** and **unclarity.** Three ways to prevent brittleness: **① strive for unchanging tests** — of the four kinds of production change, only "changing existing behavior" should require touching existing tests (pure refactorings, new features, and bug fixes shouldn't); **② test via public APIs** (call the system the way users do, not via implementation details — "use the front door first"); **③ test state, not interactions** (state over interaction; over-reliance on mocking frameworks is the chief cause of brittle tests — use real objects when feasible).

Keys to writing clear tests: **complete and concise**, **test behaviors not methods** (given/when/then structure, named after the behavior), **don't put logic in tests** (obviously correct on inspection), and **write clear failure messages.** Finally, the code-sharing philosophy: **DAMP, not DRY** — a little duplication in tests is fine if it makes them more descriptive and meaningful; DAMP complements DRY rather than replacing it.

## Why read this

- **You (like Mary) get a wall of test failures from every small change, even with no bug introduced.** Those are **brittle tests.** The chapter gives the root cause (tests coupled to implementation details) and three cures: **strive for unchanging tests, test via public APIs, test state not interactions.** You'll learn to write tests that don't need to change when you change production code.
- **You write one `testXxx` per method.** That spirals as the method grows complex; switch to **testing behaviors** — "given… when… then…" — where each test covers one behavior, reads like natural language, and points at the broken behavior on failure.
- **You assume tests should be DRY too.** Counterintuitively, tests should be **DAMP**: because **tests aren't themselves tested**, complexity costs more in test code; hiding info in helpers makes tests "incomplete" and can hide bugs. A little duplication buys "read it top to bottom and understand it" — worth it.
- **You're dragged down by flaky/slow/unintelligible tests, even tempted to delete the ones you can't understand.** The chapter warns: deleting an unintelligible test opens an unknown coverage hole and signals it may have provided **zero value for years.** Better to write **clear, logic-free tests with explicit failure messages** from the start.

## Key details

### The Importance of Maintainability
The core claim: after catching bugs, a test's most important purpose is **improving engineer productivity.** Unit tests are great at this because they're small, fast, deterministic, easy to write alongside the code, easy to understand, and serve as documentation. But Mary's story shows the flip side: **badly written tests drain productivity** — her change introduced no real bug, yet tests coupled to internal structure forced her to patch them one by one, and the hacks made them harder to understand later.

Takeaway: bad tests **must be fixed before check-in**, lest they drag on every future engineer. Mary's problems split into two categories — **brittle** (broke on a harmless, unrelated change) and **unclear** (when it broke, hard to tell what it was testing or how to fix it). The rest of the chapter addresses both.

### Preventing Brittle Tests
The core claim: a brittle test fails on "an unrelated change that introduces no real bug" (distinct from **flaky** — which fails nondeterministically with no production change). Tweaking a few tests per change may be fine on a small team, but it's **acute at Google's scale**: an engineer runs thousands of tests a day, and a single large-scale change ([[ch22]]) can trigger hundreds of thousands — even a small percentage of spurious breakages wastes enormous time.

#### Strive for Unchanging Tests
The core claim: the ideal test is **unchanging** — after writing it, it never needs to change unless the *requirements* of the system under test change. Map this against the four kinds of production change:
- **Pure refactorings** (internals change, interface doesn't): **tests shouldn't change** (their role is to confirm the refactor preserved behavior; if a test must change, either it wasn't a pure refactor or the test was at the wrong abstraction level).
- **New features**: existing behaviors are unaffected → **add new tests** for new behaviors, don't touch existing ones.
- **Bug fixes**: the bug means a case was **missing from the suite** → add that case; again, don't change existing tests.
- **Behavior changes**: the **only** case that should update existing tests — and typically **far more expensive** than the others (users rely on current behavior; changing it needs coordination; changing a test here means breaking an *explicit contract*).

Takeaway: after writing a test, refactoring/fixing/adding shouldn't make you touch it — that's the key to working at scale: expanding a system requires only a few new tests, not touching every test ever written.

#### Test via Public APIs
The core claim: the single most important way to keep tests from changing with the implementation is to **call the system the way its users do — its public API, not implementation details** ("use the front door first"). Then a change that breaks a test **by definition also breaks a user**; such tests also serve as examples and documentation. In the book's transaction example: rather than dropping `private` to test `isValid()`/`saveToDatabase()` (any rename, helper extraction, or serialization-format change breaks the test), achieve the same coverage through the public `setAccountBalance`/`processTransaction`/`getAccountBalance`.

Takeaway: "public API" has no rigid definition — the question gets at what a "unit" is: the API a unit exposes to **third parties outside the owning team** (not necessarily the language's `public`). Rules of thumb: a **helper class** supporting just one or two others isn't its own unit (test it through them); a package usable by anyone without consulting owners, or a general-purpose **support library**, is a unit to test directly (even at the cost of some valuable redundancy).

#### Test State, Not Interactions
The core claim: there are two ways to verify behavior — **state testing** (observe what the system looks like after invoking it) and **interaction testing** (check the sequence of actions it took on its collaborators). **Interaction tests are more brittle** for the same reason testing a private method is more brittle than a public one: they check *how* the system reached its result, but you usually care only *what* the result is. The book's `verify(database).put("foobar")` can go wrong two ways: ① a bug deletes the record right after writing, yet the test still passes (should've failed); ② a refactor calls an equivalent API, and the test fails (should've passed). Testing state — `assertThat(accounts.getUser("foobar")).isNotNull()` — is far more robust.

Takeaway: the most common cause of bad interaction tests is **over-reliance on mocking frameworks** (easy to make doubles that record and verify every call), so Google **prefers real objects** when they're fast and deterministic (see [[ch13]]).

### Writing Clear Tests
The core claim: however well you avoid brittleness, tests will eventually fail — and failure is good (a useful signal). Failures happen for one of two reasons: **① the system under test has a problem** (exactly what tests are for) or **② the test itself is flawed** (the system is fine; if it's an existing test, it's brittle). The engineer's first job is to tell which — speed depends on the test's **clarity.** Clarity matters because **tests often outlive the engineers who wrote them**: a years-old failing test by a departed author may leave no way to discern its purpose (unlike unclear production code, whose purpose you can reverse-engineer from callers). Worst case, you delete what you can't understand — opening a coverage hole and signaling it may have given **zero value for years.**

- **Complete & concise:** a test is **complete** when its body holds all the info to understand how it reaches its result; **concise** when it holds nothing distracting. The anti-example stuffs irrelevant args into the constructor and hides the key number in a helper ("where did this 5 come from?"). Fix: a helper hides irrelevant construction details (more concise) while spelling out key inputs (more complete).
- **Test behaviors, not methods:** the instinct is one test method per production method, but that spirals as methods grow complex. Better: **a test per behavior** — a **behavior** is any guarantee about how the system responds to inputs in a given state, often expressed as **given / when / then** ("given an empty account, when withdrawing, then the transaction is rejected"). Methods↔behaviors is **many-to-many.** Splitting a method-driven test into behavior-driven ones costs boilerplate but reads like natural language, expresses cause/effect, and encourages adding new tests over piling onto old ones.
- **Structure to emphasize behaviors:** each behavior has three parts — **given (set up the system) / when (act) / then (validate)** (a.k.a. arrange/act/assert). Tests are clearest when this is explicit (Cucumber/Spock bake in given/when/then; other languages use blank lines and comments). This lets a test be read at **three granularities**: the name (rough behavior), the given/when/then comments (formal), the code (precise). Multistep tests may alternate when/then and split long blocks with "and," but **each test should cover a single behavior** (most need one when, one then).
- **Name after the behavior:** the test name is often the first or only token in a failure report — your best chance to communicate the problem. A good name describes the action and the expected outcome. Stuck? Start with "**should**," so with the class name it reads as a sentence — `BankAccount`'s `shouldNotAllowWithdrawalsWhenBalanceIsEmpty`. If you need "**and**" in the name, you're probably testing multiple behaviors — split them.
- **Don't put logic in tests:** clear tests are **trivially correct on inspection.** Complexity comes mostly from logic (operators, loops, conditionals) — with logic, you must compute the result mentally instead of reading it. The book's URL example: a single string concatenation hides a double-slash bug; writing the whole string out reveals it instantly. Lesson: prefer **straight-line code over clever logic**, tolerating some duplication for descriptiveness (see DAMP).
- **Write clear failure messages:** ideally an engineer diagnoses from the log alone, never opening the test. A good message distinguishes **expected vs actual** with context. `Test failed: account is closed` (ambiguous) → `Expected an account in state CLOSED, but got {…state: "OPEN"}` (clear). Good assertion libraries help: JUnit's `assertTrue(colors.contains("orange"))` can only say "expected true but was false"; Google's **Truth** — `assertThat(colors).contains("orange")` — says "`<[red, green, blue]>` should have contained `<orange>`." Go conventionally uses `t.Errorf("Add(2, 3) = %v, want %v", result, 5)`.

### Tests and Code Sharing: DAMP, Not DRY
The core claim: most software pursues **DRY (Don't Repeat Yourself)** — each concept canonical in one place, minimal duplication, so you change one spot. But the calculus differs for **tests**: good tests should be stable, and you usually *want* them to break when the system changes, so DRY's benefit shrinks; meanwhile complexity costs more (production has a test suite to protect it, but **tests stand alone**, risking bugs if not self-evidently correct). So tests should be **DAMP — Descriptive And Meaningful Phrases**: **a little duplication is fine** if it makes the test simpler and clearer. The book contrasts an over-DRY test that hides details (and a bug) in helpers like `createUsers`/`validateForumAndUsers` (concise bodies but incomplete) with the DAMP rewrite — longer and more repetitive, but each test is **understandable end to end without leaving the body.**

Takeaway: **DAMP doesn't replace DRY; it complements it.** Helpers and infrastructure still aid clarity by factoring out repetitive steps irrelevant to the behavior under test — the point is to refactor **toward descriptiveness and meaning, not solely to cut repetition.** Common sharing patterns:
- **Shared values:** instead of ambiguously named shared constants (`ACCOUNT_1`/`ACCOUNT_2` — scroll up to learn what they are), use **helper methods with defaults** (named parameters, or the Builder pattern), letting each test specify only the values it cares about. You can even **slightly randomize unset defaults** so two instances don't accidentally compare equal.
- **Shared setup:** `@Before`-style methods suit constructing the **object under test and its collaborators** (when most tests don't care about the args). Risk: tests that **depend on specific setup values** become incomplete (where did "Donald Knuth" come from?) — such tests should **override the value directly in the body.**
- **Shared helpers and validation:** helpers for concise construction are good; a fixed `validate()` at the end of every test is a bad habit (less behavior-driven, intent unclear, and one bug fails many tests). Better validation helpers **assert a single conceptual fact**, useful when that fact is conceptually simple but needs looping/conditional logic that would hurt clarity inline.
- **Defining test infrastructure:** code shared across suites is more like **production code** (many callers, hard to change without breakage), so it **must have its own tests** and be treated as a separate product. Most is well-known third-party libraries (JUnit) — standardize **early and universally** (years ago Google **mandated Mockito as the only Java mocking framework**; grumbled about then, universally seen as good now).

## Key terms

- **Unit test** — narrow-scope (one class/method), usually small-size test; Google's most common, ~80% recommended; its prime value is improving productivity maintainably.
- **Maintainability** — maintainable tests "just work": forgotten after writing until they fail, with failures pointing to real bugs with clear causes. The chapter's core.
- **Brittle test** — fails on an unrelated change introducing no real bug; usually coupled to implementation details. Distinct from flaky (nondeterministic failure).
- **Unchanging tests** — the ideal: never need changing unless requirements change. Only "changing existing behavior" should update existing tests.
- **Test via public APIs** — call the system as a user would (front door), not via implementation details; breaking the test = breaking a user, forming an explicit contract.
- **State vs interaction testing** — state = observe what the system becomes; interaction = check calls made on collaborators. Interaction tests are more brittle (care about "how," not "result").
- **Behaviors, not methods** — a test per behavior, not per method; a behavior is a "guarantee about response to inputs in a state," expressed as given/when/then; method↔behavior is many-to-many.
- **Given / when / then (arrange / act / assert)** — the three parts of a behavior: set up / act / validate; lets tests be read at name, comment, and code granularities.
- **DAMP (Descriptive And Meaningful Phrases)** — the test code-sharing philosophy: tolerate some duplication for descriptiveness and meaning; complementary to DRY (tests aren't themselves tested).
- **Truth** — Google's Java assertion library, producing more useful failure messages than JUnit's `assertTrue` (names the subject and expectation).
- **Test infrastructure** — code shared across suites; more like production (many callers), so it needs its own tests and early standardization (e.g., mandating Mockito).

## Questions to push your thinking

1. Take a recent "changed production, forced to change many tests" experience: were those tests coupled to **implementation details** (poking privates, verifying interactions)? If you tested **state via the public API**, would this change still break them? Per the "four kinds of change" — which was yours?
2. Pick a `testXxx` (mapped to a method): is it actually testing **multiple behaviors**? Split it into behavior-driven tests with **given/when/then**, renamed with "should…" — would failure reports now tell you faster *which behavior* broke?
3. Is there **logic** in your tests (loops, conditionals, string concat, math)? Flatten one into straight-line code — like the book's URL example, does flattening reveal a bug that was hidden?
4. Is your suite "too DRY" or "DAMP"? Have you hidden key info in `createX`/`validateX` helpers, so reading one test means jumping to several places? Which helpers should be **inlined back** for clarity?
5. Does your **test infrastructure** (shared fixtures, homegrown frameworks) have its own tests? Are your mocking/assertion libraries **standardized to one**? Recall the pain of "too many frameworks to read each other's tests" — worth converging like Google's Mockito mandate? Per [[ch13]].
