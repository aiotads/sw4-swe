# 第 19 章｜Critique：Google's Code Review Tool（Critique：Google 的程式碼審查工具）

> *"Trust and communication are core to the code review process. A tool can enhance the experience, but it can't replace them."*
> 「信任與溝通，是程式碼審查流程的核心。工具能增強這份體驗，卻無法取代它們。」

## 摘要

[[ch09]] 談的是 code review 的**流程**，本章談支撐流程的**工具**——Google 自家、廣受喜愛的 **Critique**（公司內最受歡迎的審查工具，差距極大）。一個好的審查工具不只實作流程，更要由文化形塑出設計原則。Critique 的四條指導原則：**①Simplicity（簡單）**——UI 快、導航易、有清楚的視覺狀態標記，「沒有不必要的選擇」（對工具影響最大的一條）；**②Foundation of trust（信任的地基）**——審查不是為了拖慢別人，而是賦能；信任作者會處理小意見，不必再加一輪複查；**③Generic communication（通用的溝通）**——溝通問題鮮少靠工具解決，故鼓勵用一般留言把想說的講清楚，而非把資料模型搞複雜；**④Workflow integration（與工作流整合）**。Simplicity 與 integration 有張力——Google 曾考慮做一個「Code Central」（編輯＋審查＋搜尋三合一）但**刻意放棄**，讓 code review 保持為主要焦點（功能用連結而非嵌入）。

Critique 支撐**提交前審查（precommit review）** 的六階段：**①建立變更（上傳快照→觸發自動分析器）②請求審查（mail 給審查者）③留言（unresolved 預設＝作者必須處理；resolved＝選擇性；原子式 publish；任何人可 drive-by review）④修改並回覆（可比對任意兩個快照；③④可重複）⑤核准（LGTM；UI 標綠）⑥提交（presubmits 通過後）**。其餘關鍵：強大的 **diffing**（LCS＋語法高亮＋Kythe 交叉引用＋intraline diff＋忽略空白＋move detection＋並排）、分析結果以 chip 呈現（紅/黃/灰、binary 可行動性、附 fix suggestion）、**緊密的工具整合**（Cider/Code Search/Tricorder/Rapid/Zapfhahn，偏好連結而非嵌入）、**GwsQ** 自動指派審查者、**attention set（「輪到誰」）**、**Changelist Search** 驅動的 dashboard。評分模型簡化成三件事：**LGTM＋Approval＋unresolved comments**——每個變更都需至少一個 LGTM（兩雙眼睛），LGTM/Approval 永遠是正面的（不能無理由 thumbs-down）。結論：**信任與溝通是核心，工具能增強卻無法取代；緊密整合是 Critique 成功的關鍵**。

## 為什麼要讀這一篇

- **你在挑或做一個 code review 工具，糾結要加多少功能。** Critique 的頭號原則是 **Simplicity**——「沒有不必要的選擇」。它甚至**刻意放棄**把編輯/審查/搜尋做成三合一的「Code Central」。看懂「為何少即是多」，你會更敢於拒絕誘人但增複雜度的功能。
- **你的審查常卡在「現在到底輪到誰？」** Critique 的 **attention set（「輪到誰」）** 是個小到不行卻效果驚人的功能——用粗體標出「change 卡在誰身上」，實作後大家都無法想像沒有它的日子。小工作流優化能大幅減少摩擦。
- **你的評分/核准機制很複雜（多種分數、否決、來回複查）。** Critique 把它簡化成 **LGTM＋Approval＋unresolved comments** 三件事，並讓 **LGTM/Approval 永遠是正面的**（負面回饋必須綁定具體的 unresolved comment）——這不只簡化流程，還**正面影響了審查文化**。
- **你想把審查工具接進整個開發生態。** Critique 的成功關鍵之一是**緊密的工具整合**（跳到 Code Search、在 Cider 編輯、看 Tricorder 分析、看 Rapid 是否已發布）——但用**連結而非嵌入**，以免犧牲核心的審查體驗。學會這個分寸。

## 細節重點

### Code Review Tooling Principles（審查工具的原則）
核心主張：Critique 為何成功？因為它被 Google「把 code review 當工作流核心」的文化形塑出四條原則：
- **Simplicity（簡單）：** UI 快載入、導航易（支援 hotkey）、有清楚的視覺狀態標記；「不做不必要的選擇」。**這條對工具影響最大**——有很多有趣功能被考慮過，但為了不讓模型為一小撮使用者變複雜而放棄。
- **Foundation of trust（信任的地基）：** 審查不是為了拖慢別人，而是**賦能**；盡量信任同事——例如信任作者會處理小意見、不必再加一輪複查；變更也對全 Google 開放可看可審。
- **Generic communication（通用的溝通）：** **溝通問題鮮少靠工具解決**；Critique 優先用一般留言（甚至建議編輯），而非把資料模型與流程搞複雜——「再好的工具，溝通仍可能出錯，因為使用者是人」。
- **Workflow integration（與工作流整合）：** 與其他核心工具有多個整合點（跳到 Code Search、在 Cider 編輯、看測試結果）。

工程意涵：**Simplicity 與 integration 有張力**——Google 曾考慮做一個「Code Central」（編輯＋審查＋搜尋三合一），但**刻意放棄**，讓 code review 保持為主要焦點；功能從 Critique **連結出去**、實作在別的子系統。

### Code Review Flow（六階段流程）
核心主張：Critique 支撐**提交前審查（precommit review）**，典型六步：
1. **Create a change（建立變更）：** 作者在 workspace 寫好變更，上傳**快照（snapshot）**到 Critique，觸發自動分析器（[[ch20]]）。
2. **Request review（請求審查）：** 對 diff 與分析結果滿意後，mail 給一位以上審查者。
3. **Comment（留言）：** 審查者草擬留言；**預設 unresolved（作者必須處理）**，也可加 resolved（選擇性/資訊性）；留言**原子式 publish**（審完整個變更才一次送出完整想法）；**任何人都能 drive-by review**。
4. **Modify & reply（修改並回覆）：** 作者依回饋改、上傳新快照、回覆；處理（至少）所有 unresolved 留言；可比對**任意兩個快照**的 diff；③④可重複多次。
5. **Change approval（核准）：** 審查者滿意就 **LGTM**（可附帶留言）；可提交時 UI **明顯標綠**。
6. **Commit（提交）：** 核准後作者觸發提交；presubmits 無問題就 commit。

工程意涵：系統提供大量偏離常規流程的彈性（審查者可取消指派/轉派、作者可延後）；緊急時作者可**強制提交、事後再審**。**通知**模型讓 Critique 專注當主要審查工具：發出事件，其他系統（Chrome 擴充、email）據以建構——有人愛即時通知、有人嫌打斷心流而不用。

### Stage 1：Diffing 與 Analysis Results（差異與分析結果）
核心主張：理解程式變更本身是審查核心，故**最佳化 diff 是好工具的核心要求**。Critique 的 diffing 從最佳化的 **LCS（最長共同子序列）** 演算法出發，再加上：**語法高亮、Kythe 交叉引用（[[ch17]]）、intraline diff（字元級、考慮詞界）、可調程度地忽略空白、move detection（被搬移的程式碼標為「移動」而非「這刪那加」）**。可並排（side-by-side）或 overlay 檢視——為了並排，他們簡化了 diff 結構（無邊框、無 padding）並反覆調字型大小以容下 Java 的 100 字元行限。提供快照版本 widget（拖放選版本比對、自動收合相似快照）；**prefetch 一切**讓載入極快。

上傳快照觸發**程式分析器（[[ch20]]）**，結果以 **chip** 摘要呈現（紅＝highlight、黃＝進行中、灰＝其他；**actionability 是 binary**，不搞更多選項）；finding 可顯示在 diff 裡（樣式與留言不同以區別），常附 **fix suggestion** 可預覽並一鍵套用（例：linter 抓到行尾多空白 →兩下看到問題、一下套用修正）。

**緊密的工具整合：** Critique 是接觸這些工具的「melting pot」——**Cider（雲端 IDE）、Code Search、Tricorder（靜態分析）、Rapid（發布）、Zapfhahn（測試覆蓋率）**。但**偏好連結而非嵌入**以免分散核心審查體驗（唯一例外是測試覆蓋率——用 gutter 背景色顯示某行是否被測涵蓋）。能緊密整合，是因 workspace 存在 **FUSE-based 檔案系統**、Source of Truth 在雲端、所有工具都可存取。

### Stage 2：Request Review（請求審查）
核心主張：作者要挑審查者——小團隊也該把審查平均分配、考慮誰在休假。團隊可提供一個 **email alias**，由 **GwsQ**（以最初用此技術的團隊命名）依 alias 設定自動指派具體審查者。Google 規模下「找誰最適合審你專案外的變更」是難題——Critique 的審查者建議考慮：**誰擁有被改的程式（owner）、誰最熟（最近改過）、誰有空（不在休假、最好同時區）、GwsQ 設定**。

工程意涵：指派審查者會觸發 **presubmits（提交前 hook）**：自動加 email list 提高透明、跑專案測試套件、強制專案特定的程式/描述不變式（如產生 release notes）。因跑測試耗資源，測試是 presubmit 的一部分（請求審查與提交時跑），而非像 Tricorder 每個快照都跑。失敗的 presubmit 會**擋住**變更送審或提交，並 email 通知作者。

### Stages 3 & 4：Commenting 與理解狀態
核心主張：**留言是 Critique 第二常見的動作**（僅次於看變更），且**人人皆可留言**（不限作者與指定審查者）。提供 per-person 狀態（審查者可勾選「已審」個別檔；作者一改該檔就清空所有人的勾選）。看到分析 finding 可按「**Please fix**」建一條 unresolved 留言；也可 inline 編輯建議修正（Critique 轉成附 fix 的留言）；常見回應有 **Done**（已處理）/ **Ack**（已讀，用於資訊性留言）快捷鈕。留言**草擬中、原子式 publish**。

**理解變更狀態的兩機制：**
- **「Whose turn」/ Attention set（「輪到誰」/注意力集合）：** 多位審查者時，知道「現在輪到你」能加速審查。attention set 是「變更目前卡在誰身上」的人；在裡面的人該及時回應；Critique 用**粗體**標出這些名字。實作後使用者「無法想像以前怎麼活的」——它也凸顯了 code review 的**回合制**本質：總是至少有一個人該行動。
- **Dashboard 與 Changelist Search：** Critique 的著陸頁是使用者 dashboard，由 **Changelist Search** 驅動（索引全 Google 所有變更的最新狀態、支援 regex 查詢、夠快可互動）。預設第一區顯示「需要你注意的變更」（attention set），但可自訂（如 LSC 的全域審查者會自訂以免被洪水淹沒）。

### Stage 5：Change Approvals（評分一個變更）
核心主張：除了用留言給具體建議，還需要一個高層次的「OK」機制。Google 把評分簡化成**三件事**：**LGTM（looks good to me）＋Approval（核准）＋unresolved comments（未解決留言數）**。
- **LGTM** ＝「我審過了，相信它符合標準，處理完 unresolved 留言後可提交」。
- **Approval** ＝「身為守門人，我允許這變更進這塊程式碼庫」。
- **每個變更都需至少一個 LGTM**（不論 approval 狀態），確保**至少兩雙眼睛**看過。

工程意涵：曾有「Needs More Work」與「LGTM++」評分，後來**簡化**為「LGTM/Approval 永遠是正面的」——若確實需要二審，主審查者可加留言但不給 LGTM。這也**正面影響了審查文化**：審查者**不能無理由地 thumbs-down**，所有負面回饋都必須綁定具體可修的東西（unresolved comment，連用詞都選得相對委婉）。**Hard requirement（LGTM/Approval，只有審查者能給/撤）vs soft requirement（unresolved comment，作者可標 resolved）** 的區分，促進並仰賴**信任**——審查者可在附 unresolved 留言下就 LGTM、不必事後逐一檢查是否真的處理了，這份信任在跨時區時特別省時。

### Stage 6 與 After Commit、Gerrit
核心主張：
- **Stage 6（提交）：** Critique 有個提交按鈕，免得切換到 CLI。
- **After Commit（追蹤歷史）：** Critique 也用於**程式考古（change archaeology）**——任何人都能瀏覽變更歷史（含留言與演進），用於日後稽核、理解「為何這樣改」「bug 怎麼引入」，code review 資料聚合後還用於製作教材。也支援**提交後留言**（事後發現問題時）與**回滾**（並可看某變更是否已被回滾）。
- **Case Study｜Gerrit：** Critique 因與 monorepo 及內部工具緊密耦合而**不對外**；做開源專案（Chrome/Android）或不想放進 monorepo 的團隊用 **Gerrit**——獨立、開源、與 Git 緊密整合，有細粒度權限模型、更精巧（也更可設定）的評分系統（審查者可用 **−2 否決**）。兩者模型相同（每個 commit 各自審）。

工程意涵（結論）：用審查工具有許多隱含取捨。**花在 review 的時間是沒花在寫程式的時間**，故任何流程最佳化都是公司的生產力增益；多數情況只需兩人（作者＋審查者）同意就能提交，保持高 velocity。Critique 在多處需應對**規模**（大量審查請求不掉速、在 critical path 上要快載入、能處理異常大的變更、幫忙找審查者而不必自己搞懂 ownership）。它**偏好有主見的流程與簡單介面**，但允許部分客製（自訂分析器/presubmit、團隊政策如要求多個 LGTM）。**信任與溝通是核心，工具能增強卻無法取代；緊密整合是成功關鍵。**

## 關鍵字補充

- **Critique** — Google 自家、最受歡迎的程式碼審查工具；支撐提交前審查的六階段流程，與其他開發工具緊密整合。
- **四條原則（Simplicity / Trust / Generic communication / Workflow integration）** — Critique 的設計指導；Simplicity 影響最大（曾為此放棄「Code Central」三合一）。
- **Precommit Review（提交前審查）** — 變更進程式碼庫前先審；六階段：建立→請求→留言→修改→核准→提交。
- **Snapshot（快照）** — 變更在某時間點的 patch；上傳快照觸發自動分析器，可比對任意兩快照的 diff。
- **Diffing（差異）** — LCS＋語法高亮＋Kythe 交叉引用＋intraline diff＋忽略空白＋move detection＋並排；最佳化 diff 是好審查工具的核心要求。
- **Analyzer Chips / Tricorder** — 分析結果以 chip 呈現（紅/黃/灰、binary 可行動性、附 fix suggestion）；Tricorder 是靜態分析工具（見 [[ch20]]）。
- **GwsQ** — 依 email alias 自動指派審查者的工具；審查者建議考慮 owner/熟悉度/可用性/GwsQ 設定。
- **Presubmits（提交前 hook）** — 請求審查與提交時跑（加 email list、跑測試套件、強制不變式）；失敗會擋住變更。
- **Attention Set / "Whose Turn"（注意力集合/輪到誰）** — 標出「變更目前卡在誰身上」（粗體）；凸顯 code review 的回合制本質；小優化大幅減摩擦。
- **Changelist Search** — 驅動 dashboard 的搜尋系統，索引全 Google 變更最新狀態、支援 regex、夠快可互動。
- **LGTM / Approval / Unresolved Comments（三件式評分）** — LGTM＝符合標準可提交、Approval＝守門人允許進這塊、unresolved＝作者必須處理；每變更至少一個 LGTM（兩雙眼睛）。
- **Gerrit** — 獨立開源、與 Git 緊密整合的審查工具；用於開源/非 monorepo 專案；有 −2 否決與更可設定的評分。

## 延伸提問

1. 你用/做的審查工具，是偏向 **Simplicity（沒有不必要的選擇）** 還是不斷加功能？有沒有「為了一小撮使用者而讓模型變複雜」的功能其實該砍？你敢不敢像 Google 放棄「Code Central」那樣，**克制把所有東西塞進一個工具**？
2. 你的團隊審查時常卡在「現在輪到誰？」嗎？你能不能引入類似 **attention set**（明確標出「卡在誰身上」）的小機制？回想一次「change 在作者與審查者間空等、因為沒人知道該誰動」的經驗——這個小優化能省多少時間？
3. 你的核准/評分機制是不是太複雜（多種分數、否決、來回複查）？能不能簡化成像 **LGTM＋Approval＋unresolved comments** 的三件式、並讓**負面回饋必須綁定具體可修的東西**？這對你的審查**文化**會有什麼影響？
4. 你的審查工具與其他工具（搜尋、IDE、CI、發布、覆蓋率）整合到什麼程度？哪些該**連結**、哪些值得**嵌入**？回想 Critique「偏好連結、唯覆蓋率嵌入」的分寸——你有沒有為了整合而犧牲了核心體驗的簡單？對照 [[ch17]]、[[ch20]]。
5. 「**信任作者會處理小意見、不必事後逐一檢查**」在你的團隊行得通嗎？你們的審查是在**賦能**還是在**拖慢**？跨時區時，這份信任能省多少來回？對照 [[ch09]]、[[ch02]] 的 HRT。
