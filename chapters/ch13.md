# 第 13 章｜Test Doubles（測試替身）

> *"A real implementation should be preferred over a test double. … Interaction testing should be avoided when possible."*
> 「應該優先用真實實作，而非測試替身。……能避免就避免互動測試。」

## 摘要

**Test double（測試替身）** 是在測試中頂替真實實作的物件或函式——就像電影裡的替身演員（stunt double）。它讓你能寫出大量「小、快、不 flaky」的單元測試，去測那些本來得跨行程/跨機器、會慢又不穩的程式（如打外部伺服器、寫資料庫）。但用替身會帶來三項需要取捨的衝擊：**testability（可測試性，程式得設計成能抽換依賴）、applicability（適用性，誤用會造成脆弱、複雜、低效的測試）、fidelity（保真度，替身的行為有多接近真實實作）**。

本章最重要的立場（也是 Google 用血淚換來的教訓）：**優先用真實實作（real implementation），這叫 classical testing；而非以 mocking 框架為主的 mockist testing**——後者難以規模化。Google 曾把 mocking 框架當成「萬用鐵鎚」，數年後才發現這些測試**好寫卻要不斷維護、又很少抓到 bug**，鐘擺如今盪回「寫更真實的測試」。**偏好真實勝於隔離（realism over isolation）**：用真實依賴能跑到 production 會跑的程式碼，更有信心；真實實作有 bug 時測試會失敗——這是好事。

使用替身有三種技術：**faking（用輕量假實作，如記憶體資料庫）、stubbing（硬寫函式的回傳值）、interaction testing（驗證某函式有沒有被正確呼叫，而不真的執行它）**。優先序：**真實實作 ＞ fake ＞（節制使用的）stub／interaction**。決定要不要用真實實作，看三件事：**執行時間、確定性（determinism）、依賴建構難度**。fake 若不存在，最好由**擁有真實實作的團隊**來寫與維護、且 **fake 自己要有測試（contract test）**。stubbing 過度使用會讓測試**不清晰、脆弱、低效**；interaction testing 應盡量避免（它把實作細節洩漏進測試，淪為「change-detector test」），只在**狀態改變函式**（如 `sendEmail()`）上做、且避免 overspecification。

## 為什麼要讀這一篇

- **你一遇到「依賴難搞」就 `@Mock` 一切。** 本章用 Google 的血淚教訓告訴你代價：mock 出來的測試好寫、卻**要不斷維護又很少抓到 bug**，還會在 API owner 想改實作時擋路（被 mock 成千上萬次）。優先序很明確：**真實 ＞ fake ＞ stub／interaction**。
- **你分不清 fake／stub／mock 差在哪。** 本章把「替身」拆成三種**技術**：fake（輕量真實作）、stubbing（硬寫回傳值）、interaction testing（驗證有沒有被呼叫）。搞懂這三者，你才知道面對某個依賴時該用哪一招。
- **你的測試是「change-detector」——改 production 一定跟著紅，即使行為沒變。** 那多半是 interaction testing 過度使用。本章教你**偏好 state testing**（驗證結果/狀態，而非驗證「呼叫了什麼」），只在狀態改變函式上做互動測試。
- **你想讓團隊的測試又快又真。** 關鍵是**為依賴寫一個高保真的 fake**——一個好 fake 能大幅提升整個組織的開發速度；而它該由**真實實作的 owner** 維護、且自己有 contract test 保證不與真實實作走鐘。

## 細節重點

### The Impact of Test Doubles（替身對開發的三項衝擊）
核心主張：用替身會引入需要取捨的三件事——**testability（可測試性）**：程式得設計成能把真實實作抽換成替身（事後才想加測試，往往得大改）；**applicability（適用性）**：用得好能大幅提速，用得差（尤其大規模誤用）會造成脆弱、複雜、低效的測試，**很多情況下根本不該用替身、該用真實實作**；**fidelity（保真度）**：替身行為與真實實作的接近程度——保真度太低的替身幾乎沒價值（想像一個「忽略所有寫入、永遠回傳空」的假資料庫），但**完美保真往往不可行**，故用替身的單元測試常需更大範圍的測試來補。

工程意涵（Google 的教訓）：**過度依賴 mocking 框架很危險**。它初看像「萬用鐵鎚」——不必煩惱怎麼建構依賴就能寫高度聚焦的測試；但數年後 Google 發現這些測試**好寫卻要不斷維護、又很少抓到 bug**，鐘擺已盪回「避開 mocking 框架、寫更真實的測試」。

### Basic Concepts：Seam、Mocking Framework（基本概念）
核心主張：程式「可測試（testable）」指它寫得能讓人為它寫單元測試。**Seam（接縫）** 是讓程式可測試的手段——它讓你能為被測系統替換不同的依賴（測試用替身，而非 production 用的真實依賴）。引入 seam 最常見的技術是 **dependency injection（依賴注入，DI）**：類別需要的依賴是**被傳進來**而非自己 `new`，於是測試時能換成替身。Java 常用自動 DI 框架 **Guice、Dagger**；動態語言（Python/JavaScript）能直接抽換函式，故對 DI 的依賴較低。**寫可測試的程式是一筆前期投資**，且愈早做愈省——愈晚才考慮可測試性，愈難回頭加。

**Mocking framework（mock 框架）** 是讓你在測試裡更易建立替身的函式庫，可把物件換成 **mock**（行為在測試裡 inline 指定的替身），省去每次都定義新類別的樣板。Google 用 **Mockito（Java）、googlemock（C++）、unittest.mock（Python）**。但**過度使用會讓程式碼庫更難維護**。

### Techniques for Using Test Doubles（三種技術）
核心主張：用替身有三種主要技術，懂得區分才知道何時用哪招：
- **Faking（假實作）：** 一個輕量、行為類似真實實作但不適合 production 的實作（如記憶體資料庫）。**需要替身時通常是最理想的技術**，但可能不存在、且寫一個有挑戰（得確保現在與未來都與真實行為相似）。
- **Stubbing（樁）：** 給一個本身沒有行為的函式硬塞行為——明確指定它該回傳什麼（`when(...).thenReturn(...)`）。通常透過 mock 框架做以省樣板。
- **Interaction testing（互動測試）：** 驗證某函式「有沒有被正確呼叫」而不真的執行它（沒被呼叫、被呼叫太多次、用錯參數都該失敗，`verify(...)`）。**它有時被稱為 mocking**，但本章避用這詞（易與 mock 框架混淆——後者也能做 stubbing）。**能避免就避免**，過度使用易造成脆弱測試。

### Real Implementations：Prefer Realism Over Isolation（優先真實實作）
核心主張：儘管替身是寶貴工具，**測試的第一選擇是用依賴的真實實作**（與 production 同一份）。理由是 **fidelity（保真度）**——執行 production 會執行的程式，更有信心。偏好真實的這種風格叫 **classical testing**；相對的 **mockist testing**（偏好用 mock 框架取代真實實作）即使有人實踐（包括第一批 mock 框架的作者），Google 發現**難以規模化**（它要求工程師在設計被測系統時遵守嚴格準則）。

**偏好真實勝於隔離（realism over isolation）：** 用真實依賴讓被測系統更真實（依賴裡的程式都會被執行）；過度用替身則把被測系統與依賴**任意隔離**，且把測試**綁死在「作者剛好放進這個類別的實作」**上——但好測試應**獨立於實作、以被測 API 來表述**。真實實作有 bug 時測試會失敗——**這是好事**（代表你的程式在 production 也不會正常運作）；有好的 CI 工具，通常容易追出是哪個變更造成的。

> **Case Study｜@DoNotMock：** Google 見過太多過度依賴 mock 的測試，於是在 ErrorProne 靜態分析工具裡做了 **`@DoNotMock` 標註**，讓 API owner 宣告「這型別不該被 mock，因為有更好的替代」。為什麼 owner 要在乎？因為**每次用 mock 框架 stub 或做互動測試，都在重複（duplicate）API 提供的行為**；當 owner 想改 API，可能發現它已被 mock 了**成千上萬次**，這些替身很可能違反 API 契約（如對「永不回 null」的方法卻回 null）。若當初用的是真實實作或 fake，owner 就能放手改實作、不必先修數千個有瑕疵的測試。

#### How to Decide When to Use a Real Implementation（何時用真實實作）
核心主張：真實實作在它**快、確定、依賴簡單**時優先（如 value object——金額、日期、地址、list/map）。更複雜的程式則常不可行，要權衡三點：
- **執行時間（Execution time）：** 單元測試要快。多慢算太慢？沒有確切答案（取決於有沒有感到生產力損失、多少測試用它——5 個測試每個多 1 秒尚可，500 個就不行）。邊界情況常用「先用真實實作，直到太慢再換替身」。平行化、Bazel 的快取可緩解（但真實實作也增加 build 時間）。
- **確定性（Determinism）：** 同版本系統每次跑結果都一樣＝確定。非確定性導致 flaky（見 [[ch11]]）。真實實作較複雜，更可能非確定（多執行緒、依賴外部服務即非 hermetic、依賴系統時鐘）。flaky 頻繁時就該換替身或用 **hermetic instance（封閉實例，由測試掌控其生命週期）**（見 [[ch14]]）。
- **依賴建構（Dependency construction）：** 用真實實作得建構它**整棵依賴樹**（`new Foo(new A(new B(new C())...))`），耗時且建構子簽章一改就得跟著改。替身常無依賴故建構簡單（`@Mock Foo mockFoo`）——但理想解是**用 production 同一套物件建構碼**（factory method 或自動 DI），讓它彈性到能換替身。

### Faking（假實作）
核心主張：真實實作不可行時，**fake 通常是最佳選擇**——因為它行為類似真實實作，被測系統甚至分不出在跟真的還是 fake 互動。
- **何時寫 fake：** fake 需要更多心力與領域知識（要與真實行為相似），且真實行為一變 fake 就得跟著更新——故**應由擁有真實實作的團隊來寫與維護**。要不要寫是個取捨：幾個使用者可能不值得，數百個使用者就明顯划算。為減少要維護的 fake 數，**fake 應只建在「不適合用於測試的程式的根部」**（如為資料庫 API 本身寫 fake，而非為每個呼叫資料庫的類別都寫）。
- **fake 的保真度：** 最重要的概念——**fake 應對真實實作的 API 契約保持保真**（同樣輸入給同樣輸出、同樣狀態變化）。完美保真不總是可行（假資料庫存記憶體而非硬碟）；換個角度：**fake 只需從「測試的視角」對真實實作完美保真**（hashing fake 不必產生與真實相同的 hash 值，只要對同輸入唯一即可——只要契約沒保證具體值）。延遲、資源消耗通常不需保真（但若要測這些就不能用 fake）。fake 不必有 100% 功能；不支援的路徑該 **fail fast**（拋錯，告訴工程師此處不適用 fake）。
- **fake 自己要被測試：** 否則行為會隨真實實作演進而走鐘。一種做法是寫 **contract test**：對 API 公開介面寫測試，同時跑在真實實作與 fake 上（對真實實作那組較慢，但只需 fake 的 owner 跑）。
- **沒有 fake 怎麼辦：** 先請 API owner 寫一個（他們可能不知道 fake 的好處）；不行就自己寫（把所有對該 API 的呼叫包進一個類別、再做它的 fake，常比 fake 整個 API 簡單，因你多半只用到 API 的子集）；再不行就回去用真實實作或其他替身技術。fake 可視為一種**最佳化**：若加速幅度不抵維護成本，就維持用真實實作。

### Stubbing：The Dangers of Overusing（樁與過度使用的危險）
核心主張：stubbing 把函式行為硬寫在測試裡，快又簡單，但**過度使用會嚴重損及生產力**：
- **測試變不清晰：** 額外的 stub 設定碼模糊了測試意圖——若你得在腦中走過被測系統才懂為何某些函式被 stub，就是個警訊。
- **測試變脆弱：** stubbing 把實作細節洩漏進測試；production 實作一改，測試就得跟著改（好測試應只在 API 的對外行為改變時才需要改）。
- **測試變低效：** 無法保證被 stub 的函式行為與真實一致（`when(stubCalculator.add(1,2)).thenReturn(3)` 硬寫了契約卻無從保證正確）；且 stubbing **無法存狀態**（不像 fake 能 `save` 後再 `get`）。

工程意涵：書中對照——一個用 5 個 `when(...).thenReturn(...)` 設定、最後只能 `verify(...).pay()` 的測試（無從得知付款是否真的成功）；重寫成「用真實/fake 信用卡伺服器、直接查狀態 `getMostRecentCharge`」後更短、不洩漏實作細節。**何時用 stubbing 才恰當：** 當你需要函式回傳特定值以把被測系統帶入某狀態（如要它回傳非空交易清單），或模擬真實/ fake 難以觸發的回傳值或錯誤；且**每個 stub 都該與測試斷言有直接關係、數量要少**（要 stub 很多函式＝過度使用或系統太複雜該重構）。

### Interaction Testing：Prefer State Testing（互動測試：偏好狀態測試）
核心主張：**偏好 state testing（狀態測試）勝於 interaction testing**。state testing 呼叫被測系統後驗證「回傳值正確」或「某狀態被正確改變」；interaction testing 只能驗證「某些函式有被如預期呼叫」。書中排序例子：state testing 能確認數字真的被排序好；interaction testing 只能說「系統試圖呼叫 quicksort」，**無從得知數字真的排好沒**。

工程意涵：互動測試的兩大問題——①**它無法告訴你系統真的正常運作**，只能驗證函式有被呼叫，逼你對行為做假設（「呼叫了 `database.save(item)` 就假設存進去了」）；②它**洩漏被測系統的實作細節**，使測試脆弱——Google 戲稱過度互動測試的測試為 **change-detector test（變更偵測器）**：production 一改它就紅，即使行為沒變。**何時才恰當：** ①無法做 state testing（無真實實作也無 fake）時當 fallback；②函式呼叫的**次數或順序**會造成不良行為（如驗證快取確實減少了對資料庫的呼叫次數，`verify(databaseReader, atMostOnce()).selectRecords()`）。互動測試不能取代狀態測試——若單元測試只能做互動測試，強烈建議用更大範圍的測試（如對真實資料庫做狀態測試）來補（見 [[ch14]]）。

**最佳實務：** ①**只對「狀態改變函式」做互動測試**（有副作用者：`sendEmail()`、`saveRecord()`、`logAccess()`；對無副作用的 `getUser()`、`findResults()`、`readFile()` 做互動測試多半冗餘、又使測試脆弱）。②**避免 overspecification（過度指定）**：呼應 [[ch12]]「測行為而非方法」——別一次驗證多個函式與所有參數；用 `eq()`／`any()` 等 matcher 只驗證與當前行為相關的最小集合，讓測試更清晰、對範圍外的改動更有韌性。

## 關鍵字補充

- **Test Double（測試替身）** — 在測試中頂替真實實作的物件/函式（如替身演員）；讓你寫大量小而快、不 flaky 的單元測試。「mocking」一詞本章刻意避用（語意太多重）。
- **Classical vs Mockist Testing（古典派 vs mock 派）** — classical 偏好真實實作（Google 採用）；mockist 偏好 mock 框架取代真實實作（Google 認為難以規模化）。
- **Fidelity（保真度）** — 替身行為與真實實作的接近程度；fake 應對真實實作的 **API 契約**保真，但只需從測試的視角看，不必完美。
- **Seam（接縫）** — 讓程式可測試的手段：可為被測系統替換不同依賴（測試用替身）。最常透過 dependency injection 引入。
- **Dependency Injection（依賴注入，DI）** — 依賴被傳進來而非自己 new，使測試能抽換成替身；Java 常用 Guice／Dagger，動態語言對 DI 依賴較低。
- **Fake（假實作）** — 輕量、行為類似真實但不適合 production 的實作（如記憶體資料庫）；需替身時的理想選擇；應由真實實作的 owner 維護、且自己要有測試。
- **Stubbing（樁）** — 給本無行為的函式硬寫回傳值（`when(...).thenReturn(...)`）；過度使用使測試不清晰、脆弱、低效、無法存狀態。
- **Interaction Testing（互動測試）** — 驗證函式有沒有被正確呼叫而不真的執行它（`verify(...)`）；應盡量避免，只對狀態改變函式做、並避免 overspecification。
- **Change-Detector Test（變更偵測器測試）** — 過度互動測試的後果：production 一改就失敗、即使行為沒變；洩漏實作細節、脆弱。
- **@DoNotMock** — ErrorProne 的標註，讓 API owner 宣告「此型別不該被 mock，有更好替代」；保護 owner 改實作的自由。
- **Contract Test（契約測試）** — 對 API 公開介面寫測試、同時跑在真實實作與 fake 上，確保 fake 不與真實實作走鐘。
- **State vs Interaction Testing（測狀態 vs 測互動）** — state＝驗證回傳值/狀態改變；interaction＝驗證函式被呼叫。偏好 state（更能確認系統真的正常）。

## 延伸提問

1. 翻你團隊測試裡的 `@Mock`／`when().thenReturn()`：有多少其實可以換成**真實實作或 fake**？哪些依賴是「快、確定、依賴簡單」的 value object（金額、日期），其實根本不該被 mock（試試 `@DoNotMock` 的精神）？
2. 你的某個測試是不是「change-detector」——改 production 一定跟著紅、即使行為沒變？它是不是在做 **interaction testing**？能不能改成 **state testing**（驗證結果/狀態而非「呼叫了什麼」）？
3. 你最依賴、卻最難在測試裡用的依賴（資料庫、外部服務）有沒有 **fake**？若沒有，該請它的 owner 寫、還是你自己包一層？這個 fake 有沒有 **contract test** 確保它不與真實實作走鐘？
4. 用「執行時間／確定性／依賴建構」三準則檢視一個你猶豫「要真實還是替身」的測試：真實實作慢在哪？非確定來自哪（時鐘？多執行緒？非 hermetic？）？建構整棵依賴樹有多痛？哪一點才是你真正該用替身的理由？
5. 你做 interaction testing 時有沒有 **overspecify**（驗證了一堆與當前行為無關的函式與參數）？哪些其實是**非狀態改變函式**（`getUser`、`readFile`）根本不必驗證？用 `any()`／`eq()` 收斂到最小集合後，測試會不會更耐改？對照 [[ch12]]。
