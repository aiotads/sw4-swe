# 第 18 章｜Build Systems and Build Philosophy（建置系統與建置哲學）

> *"Limiting engineers' power and flexibility can improve their productivity."*
> 「限制工程師的權力與彈性，反而能提升他們的生產力。」

## 摘要

問 Google 工程師最愛公司什麼（除了免費食物），你可能意外聽到：**他們愛這套建置系統**。Google 從零打造了 **Blaze**（2015 開源版為 **Bazel**），目標是讓建置**快（單一指令、幾秒得到結果）且正確（任何人、任何機器、相同輸入永遠得到相同結果）**——老式系統常在「快」與「正確」間走捷徑而導致不一致的 build。而**Google 絕大多數的 build 是自動觸發的**（CI、code review、合併前、LSC），幾乎所有開發工具都掛在建置系統上。

沒有建置系統會怎樣？單一目錄用 `javac *.java` 就夠；但一旦跨語言、跨編譯單元、有外部依賴，就得管理依賴、按序建置——shell 腳本能撐一陣，最終卻淪為**冗長、慢、不可重現、新人 bootstrap 痛苦**的經典規模問題。一個反覆出現的主題：**管理自己的程式碼很直接，管理它的依賴才是難題**——而**管理依賴是建置系統最根本的工作**。

本章的核心對比是兩種建置系統哲學：
- **Task-based（任務式，如 Ant/Maven/Gradle）**：基本單元是 **task（任意腳本）**。極有彈性，但**給工程師太多權力、給系統太少**——系統不知道腳本在幹嘛，故難平行化、難做增量 build、腳本難維護除錯（race condition、非確定性）。
- **Artifact-based（產物式，如 Blaze/Bazel/Pants/Buck）**：BUILD 檔是**宣告式的 manifest**——工程師說「要建什麼（what）」，系統決定「怎麼建（how）」。如同函數式程式設計，系統因此能**保證正確性、並大幅平行化與重用**（改一個檔只重建最小集合、第二次 build 一秒內回「up to date」）。

由此再推展出**分散式建置**（遠端快取 ObjFS ＋遠端執行 Forge，自 2008 年起）——把工作分散到整個運算叢集。建置系統的取捨是**時間、規模、彈性**：DIY 腳本只適合最小專案；task-based 換來規模但要寫 build 檔；artifact-based 解鎖巨大規模但犧牲彈性。**Google 相信幾乎每個新專案從一開始就該用 artifact-based**。最後談模組與依賴：**偏好細粒度模組（1:1:1 規則）、最小化 visibility、嚴格傳遞依賴（strict transitive deps）、手動而非自動管理外部依賴版本、One-Version Rule**。

## 為什麼要讀這一篇

- **你的 build 又慢又愛因「換台機器就壞」而抓狂。** 本章診斷根因：你用的可能是 **task-based**（shell 腳本/Ant/Gradle），系統不知道腳本在幹嘛，故無法安全平行化或做可靠的增量 build。看懂 **artifact-based** 的思路，你會明白為何「**限制彈性反而提升生產力**」。
- **你以為「有 compiler 就夠」或「寫個 build 腳本就好」。** 本章用「經典規模問題」的故事帶你走過：跨語言、外部依賴、新人 bootstrap、自動化建置——每一步都讓 DIY 腳本崩潰。它幫你判斷「何時該投資真正的建置系統」。
- **你被鑽石依賴、版本衝突、`latest` 害慘過。** 本章給你三帖藥：**手動釘版本（不要 `latest`）、One-Version Rule、嚴格傳遞依賴**。理解「依賴是 target 的公開契約」，你會重新設計你的依賴管理。
- **你在決定模組要粗還是細。** 本章主張**細粒度模組擴展更好**（1:1:1：每目錄一個 package/target/BUILD）——更快的分散式 build、更少重建、測試時只跑受影響的子集。配合工具自動管理 BUILD 檔以減輕負擔。

## 細節重點

### Purpose of a Build System（建置系統的目的）
核心主張：所有建置系統的根本目的都一樣——**把工程師寫的原始碼轉成機器可讀的可執行檔**。好的建置系統最佳化兩個性質：**Fast（快——單一指令、幾秒回傳結果）** 與 **Correct（正確——任何人、任何機器、相同輸入永遠得到相同結果）**。老式系統常在兩者間走捷徑而導致不一致的 build；**Bazel 的主要目標就是不必在快與正確間二選一**。

工程意涵：建置系統不只為人——**Google 絕大多數的 build 是自動觸發的**（自動建置/測試/推到 production；code review 時自動測；合併前再測；底層庫作者跨整個程式碼庫測；LSC 一次動數萬檔）。幾乎所有開發工具都掛在它上面——這一切都建立在 Google 對建置系統的投資上。

### What Happens Without a Build System?（沒有建置系統會怎樣）
核心主張：**「我只需要一個 compiler！」** 在單一目錄用 `javac *.java` 確實夠；但一旦程式擴張——跨語言、跨編譯單元、有外部依賴（第三方 JAR）——建置就不再是一步：得想清楚依賴、按序建置、依賴變了要重來，否則依賴到 stale binary。**「Shell 腳本來救援？」** 能撐一陣，最終卻撞上：①**冗長**（花在 build 腳本的時間快跟寫程式一樣多）；②**慢**（為防 stale 每次全建）；③發布、災難復原（硬碟掛了，那些下載的庫找得回同版本嗎？）、新人 bootstrap、自動化建置（昨天某人改的東西在他機器能跑、在自動建置上掛）全是痛。這是**經典的規模問題**：單人寫幾百行幾週用 compiler 就夠，但一旦要跨多開發者與機器協調，再完美的腳本也不夠——該投資真正的建置系統了。

### It's All About Dependencies（一切都是依賴）
核心主張：所有問題反覆指向同一主題——**管理自己的程式碼很直接，管理它的依賴才難得多**（[[ch21]] 專論此）。依賴有很多種：對**任務**的依賴（「發布前先推文件」）、對**產物**的依賴（「要先有最新的電腦視覺庫才能建我的程式」）、**內部**依賴（程式碼庫的另一部分）、**外部**依賴（別的團隊或第三方）。但「我得先有那個，才能有這個」的概念反覆出現——**管理依賴是建置系統最根本的工作**。

### Task-Based Build Systems（任務式建置系統）
核心主張：基本單元是 **task（任務，一段可執行任意邏輯的腳本）**，任務以其他任務為「必須先跑」的依賴，形成無環圖（acyclic graph）。Ant/Maven/Gradle/Grunt/Rake 都是任務式（Ant 用 XML buildfile，target＝我們說的 task）。比起 shell 腳本，已大有進步（可連結多個 buildfile、任意加依賴）。

工程意涵（黑暗面）：它們讓工程師把任意腳本定義成任務，**極有彈性，但代價是給工程師太多權力、給系統太少**——系統不知道腳本在幹嘛：
- **難平行化：** A 依賴 B 與 C，但 B、C 能否同時跑？系統無從得知（若兩者碰同一檔就衝突），只好保守地單執行緒跑——浪費多核、也排除了分散式建置。
- **難做增量 build：** 任務能做任何事，系統無法判斷它是否已做過（也許下載了會變的檔、也許寫了時間戳）；為保正確只好每次全跑。工程師被迫每次 `clean`，完全違背增量的初衷。
- **難維護除錯腳本：** build 腳本也是程式碼、也藏 bug——任務間隱性依賴一改就壞、對機器環境的假設、非確定性元件、多依賴的 race condition。**只要工程師能寫任意程式在 build 時跑，系統就無法總是又快又正確。**

### Artifact-Based Build Systems（產物式建置系統）
核心主張：解方是**把權力從工程師手中拿回給系統**——別讓工程師定義任務，而是由系統定義少數任務、工程師有限地設定它。建置系統的首要任務就是**建程式碼**：工程師說「**要建什麼（what）**」，「**怎麼建（how）**」交給系統。這正是 Blaze 及其後裔（Bazel/Pants/Buck）的做法——BUILD 檔不是命令式腳本，而是**宣告式的 manifest**（描述要建的產物、其依賴、有限的選項）。系統因此**完全掌控何時跑什麼工具**，能做更強的保證、更有效率又仍正確。

工程意涵：
- **函數式視角：** 任務式像命令式語言（一串依序執行的語句）；產物式像函數式語言（描述計算、把「何時/如何執行」交給編譯器）。整個建置系統本質就是一個**數學函式**：輸入原始碼＋工具，輸出 binary——以函數式精神為基礎很自然。
- **Bazel 具體運作：** BUILD 檔定義 **target**（`java_binary`/`java_library`），每個 target 對應一個可建的產物，有 `name`／`srcs`／`deps`；依賴可在同 package、同源樹的別的 package、或源樹外的第三方產物；每個源樹是一個 **workspace**（根有 WORKSPACE 檔）。因 Bazel 知道每個 target 只會跑 Java compiler，故**安全地平行跑**（多核上有數量級的提速）；且因輸出只取決於輸入，**輸入沒變就重用輸出**（第二次 build 一秒內回 up-to-date；改一個檔只重建最小集合）。**以產物而非任務來重新框定建置，雖微妙卻強大。**
- **Bazel 的巧招：** **①Tools as dependencies（工具即依賴）**——target 隱性依賴 compiler，變了就重建；用 **toolchain** 達成平台獨立。**②Extending（擴展）**——用 custom rule 加新 target 類型，rule 產生 **action**（最低階可組合單元，宣告輸入輸出）。**③Isolating the environment（隔離環境）**——用 **sandboxing（LXC，Docker 同技術）** 讓 action 只能看見宣告的輸入輸出，使衝突不可能、也限制網路。**④Deterministic external deps（讓外部依賴確定）**——用 workspace-wide manifest 記錄每個外部依賴的**密碼學雜湊（cryptographic hash）**；hash 不符就 build 失敗、除非更新 manifest（須核准並 check-in），確保「checkout 上週的 commit 就拿到上週的依賴」。

### Distributed Builds（分散式建置）
核心主張：Google 程式碼庫逾 20 億行、依賴鏈極深（簡單 binary 常依賴數萬個 build target）——這規模下單機**不可能**在合理時間建完（物理定律），唯一解是**分散式建置**：把工作分散到任意可擴展數量的機器上。
- **Remote caching（遠端快取）：** 最簡單的分散式建置。所有建置者（開發者工作站、CI）共享一個遠端快取服務；要建某產物時先問快取有沒有，有就下載、沒有就自己建再上傳。**前提是 build 完全可重現**（相同輸入在任何機器產生完全相同輸出）——故每個產物以「target ＋輸入的 hash」為 key。但前提是「下載比建快」（注意網路延遲，要做實驗）。
- **Remote execution（遠端執行）：** 才是真正的分散式建置。每台機器的 build 工具把請求送到中央 **build master**，master 把請求拆成 action、排程到可擴展的 worker pool；worker 把結果寫進共享快取、也從快取讀依賴。**這要求前述 artifact-based 的一切到位**：環境自我描述、流程自我包含、輸出完全確定——這些保證對 task-based 幾乎不可能，故在其上幾乎無法建可靠的遠端執行。
- **Google 的實作（自 2008）：** 遠端快取 **ObjFS**（後端把 build 輸出存在分散於整個機隊的 Bigtable，前端 `objfsd` FUSE daemon 讓人按需瀏覽輸出——建置速度快一倍）；遠端執行 **Forge**（Blaze 裡的 Distributor 把 action 送到資料中心的 Scheduler，Scheduler 有 action 結果快取，Executor pool 持續執行並把結果存進 ObjFS）。每天數百萬次 build、數百萬測試案例、PB 級輸出。

### Time, Scale, Trade-Offs（時間、規模、取捨）
核心主張：建置系統都是為了「讓程式碼在規模與時間上更易處理」，而選哪種有取捨：**DIY 腳本**只適合最小、不需長期維護的專案（或像 Go 內建建置）；**task-based** 換來規模（自動化複雜建置、跨機重現），代價是要花心思結構化、寫 build 檔；**artifact-based** 解鎖全新規模（分散到多機、上千工程師更確信 build 一致可重現），代價是**缺彈性**（不能用真正的程式語言寫泛用任務）。改建置系統很貴、且隨專案變大更貴——故 **Google 相信幾乎每個新專案從一開始就該用 Bazel 這類 artifact-based 系統**（Google 從小實驗到 Google Search 全用 Blaze）。

### Dealing with Modules and Dependencies（模組與依賴）
核心主張：
- **細粒度模組與 1:1:1 規則：** 結構化 artifact-based build 的第一問是「一個模組該含多少功能」——單一巨模組（永不必動 BUILD 但無法平行/快取）vs 一檔一模組（最大彈性但維護依賴清單累）之間取捨。**Google 偏好遠比 task-based 小得多的模組**（典型 production binary 依賴數萬 target）；Java 等強封裝語言每目錄一個 package/target/BUILD（Pants 稱 **1:1:1 規則**）。細粒度的好處在規模下顯現：更快的分散式 build、更少重建，**測試時只跑受影響的子集**；用工具自動管理 BUILD 檔減輕負擔。
- **最小化 visibility：** 每個 target 可宣告誰能依賴它（public／private／明確名單）；visibility 是依賴的反面。如同程式語言，**盡量最小化**——只有廣用的庫才設 public。
- **內部依賴：** 從原始碼建（非下載預建產物）、**無「版本」概念**（target 與其內部依賴永遠在同一 commit 建）。要小心**傳遞依賴**：A→B→C，A 該能用 C 的 class 嗎？底層工具沒問題，但若 B 重構成不再依賴 C，A 就壞了——**依賴變成 target 的公開契約、永遠無法安全更改**，依賴累積、build 變慢。Google 解法是 **strict transitive dependency mode（嚴格傳遞依賴模式）**：偵測「引用了沒直接依賴的符號」就失敗、並給出自動補上依賴的指令（花數年重構數百萬 target，但很值得——build 更快、可安心移除不需要的依賴）。Bazel 對 Java 預設啟用。
- **外部依賴：** 有版本、且獨立於專案原始碼。**手動 vs 自動管理版本**——自動（指定可接受版本範圍、永遠下載最新，如 Gradle 的 `1.+`）對小專案方便，但對稍大專案是災難（無法控制何時更新、第三方破壞性更新讓昨天能跑今天壞）；**手動**（須改 source control）可發現、可回滾、可 checkout 舊版搭舊依賴——**Bazel 要求所有依賴版本手動指定**。
  - **One-Version Rule：** 對所有第三方依賴強制單一版本，避免**鑽石依賴**（A 依賴 B 與外部庫 v1，B 後來加依賴 v2 → A 隱性依賴兩個版本而壞）。
  - **傳遞外部依賴：** Maven/Gradle 預設遞迴下載每個傳遞依賴（方便，但必然違反 One-Version、招致鑽石依賴）——故 **Bazel 不自動下載傳遞依賴**，改用一個列出全 repo 每個外部依賴與明確版本的全域檔（有工具能自動產生）。
  - **安全與可靠：** 第三方來源有可用性風險（掛了 build 停擺）與安全風險（被入侵則注入任意程式）；緩解＝**鏡像到自控伺服器、要求 hash、或 vendoring（把依賴 check-in 到 `third_party`，把外部依賴變成內部依賴）**——Google 用 vendoring（但這只因其源碼系統為超大 monorepo 量身打造）。

工程意涵（結論）：建置系統是工程組織最重要的部分之一（每位開發者一天互動數十上百次、常是生產力的速率限制步驟）。Google 學到一個意外的教訓：**限制工程師的權力與彈性反而提升生產力**——不是給工程師自由定義怎麼建，而是用高度結構化的框架把最有趣的決策交給自動化工具。「**能信任 build 是強大的**——增量 build 就是會動，幾乎不需要清快取或跑 clean。」

## 關鍵字補充

- **Blaze / Bazel** — Google 內部建置系統 Blaze 與其 2015 開源版 Bazel；artifact-based，目標是兼顧快與正確。
- **Fast & Correct（快與正確）** — 好建置系統的兩大目標：單一指令幾秒回傳；任何人任何機器相同輸入永遠相同輸出。
- **Task-Based Build System（任務式）** — 基本單元是 task（任意腳本），如 Ant/Maven/Gradle；彈性高但給系統太少資訊，難平行化/增量/維護。
- **Artifact-Based Build System（產物式）** — BUILD 檔是宣告式 manifest，工程師說 what、系統決定 how（Blaze/Bazel/Pants/Buck）；能平行化、重用、保證正確。
- **BUILD / Target / Workspace** — BUILD 檔定義 target（java_binary/java_library，有 name/srcs/deps）；源樹根有 WORKSPACE 檔。
- **Tools as Dependencies / Toolchains** — 把 compiler 等工具當依賴（變了就重建）；toolchain 達成平台獨立。
- **Sandboxing（沙箱）** — 用 LXC（Docker 同技術）讓每個 action 只見宣告的輸入輸出，使衝突不可能、並限制網路。
- **Remote Caching / Remote Execution（遠端快取／遠端執行）** — 共享快取避免重建（ObjFS）；把 action 分散到 worker pool（Forge）；前提是 build 完全可重現。
- **1:1:1 Rule（1:1:1 規則）** — 每目錄一個 package/target/BUILD（Pants 命名）；偏好細粒度模組以利分散式 build、增量與精準測試。
- **Strict Transitive Dependencies（嚴格傳遞依賴）** — 引用沒直接依賴的符號就失敗；避免「依賴變成 target 的公開契約」、讓 build 更快。
- **One-Version Rule / Diamond Dependency（單一版本規則／鑽石依賴）** — 第三方依賴只准一個版本，避免 A 同時依賴 v1 與 v2 而壞（見 [[ch21]]）。
- **Vendoring / third_party** — 把外部依賴 check-in 到源樹（`third_party`），把外部依賴變成內部依賴；Google 用之（仰賴超大 monorepo）。

## 延伸提問

1. 你的 build 是 **task-based 還是 artifact-based**？它能安全地**平行化**與做可靠的**增量 build** 嗎？還是你常被迫 `clean`、或因「換台機器就壞」而抓狂？看懂「系統不知道腳本在幹嘛」這個根因，你會怎麼重構你的 build？
2. 回想你「沒有真正建置系統」時的痛（跨語言、外部依賴、新人 bootstrap、自動建置在他機器掛）：你目前落在 DIY 腳本／task-based／artifact-based 的哪一檔？「改建置系統的成本隨專案變大」對你意味著什麼——是不是該趁早投資？
3. 你的外部依賴是**手動釘版本還是 `latest`/範圍**？有沒有被鑽石依賴或第三方破壞性更新坑過？套用 **One-Version Rule＋手動管理＋hash**：哪些依賴是顆未爆彈？對照 [[ch16]]、[[ch21]]。
4. 你的模組是**粗還是細**？算算「測試時只跑受影響子集」對你的 CI 時間意味多少節省？你能不能往 **1:1:1** 靠、並用工具自動管理依賴清單以減輕負擔？對照 [[ch11]]、[[ch23]]。
5. 本章說「**限制彈性反而提升生產力**」。你的工具/流程裡，有沒有「給了人太多權力、卻讓系統無法保證正確或高效」的地方？把哪個決策從「人自由發揮」收回給「自動工具」，會讓大家其實更輕鬆？
