# 第 25 章｜Compute as a Service（運算即服務）

> *"I don't try to understand computers. I try to understand the programs." —Barbara Liskov*
> 「我不試著去理解電腦，我試著去理解程式。」——Barbara Liskov

## 摘要

寫完程式後，你需要硬體來跑它——於是你買或租硬體。**這本質上就是 Compute as a Service（CaaS，運算即服務）**：「Compute（運算）」是「實際執行你程式所需算力」的簡稱。本章（全書最後一章）追蹤 Google 內部 **Borg** 系統（Kubernetes、Mesos 等今日 CaaS 架構的先驅）的演化，講「給我硬體跑我的東西」這個簡單訴求，如何長成一套能隨組織成長而存活、擴展的系統，分四部分：**①馴服運算環境（雜務自動化→自動排程→容器化與多租戶→rightsizing/autoscaling）②為受管運算寫軟體（為失敗而設計、batch vs serving、管理狀態、服務連線、一次性程式）③CaaS 隨時間與規模（容器作為抽象、一統天下的單一服務、提交式設定）④選擇運算服務（集中 vs 客製、抽象層級/serverless、公有 vs 私有）**。

貫穿全章的是「**cattle, not pets（把機器當牛、不當寵物）**」——這被視為 Google 過去 15 年成功的根本工具。寵物壞了要人 panic 地搶救、難替換；牛（replica001~100）壞了自動化會移除並補一台新的，**關鍵特徵是「能全自動 stamp out 一個新實例、不需人工設定」**，由此得到自癒（self-healing）。但「把機器當牛」只保證能自動回到健康狀態，要在持續故障下平順運作還需額外設計（動態分配 work chunk、serving job 收到下線預告後優雅排空）。把機器當牛的最大敵人是**狀態（state）**：行程內狀態與本機儲存都該視為短暫（transient），「真正的儲存」要外置到有複本同步的持久化系統（Google 為此造了 GFS、Bigtable、Spanner）。其他關鍵：服務發現（service discovery，靠一層間接定址而非寫死 hostname）、冪等性（idempotency，retry 與重複請求的解藥，client 指派 ID）、快取要「按延遲目標配置、按總負載配置核心路徑」。

**容器**最初為隔離/多租戶而生（cgroups 由 Google 2007 貢獻進 Linux kernel），但更重要的是成了**抽象邊界**——機器變了只有容器層要適配，應用層不動；檔案系統抽象讓外來軟體易上船、依賴可預先打包；具名資源（如網路埠）也被抽象（Borg 早期沒做，導致 `PickUnusedPortOrDie` 在 C++ codebase 用了逾 2 萬次）。但 **Hyrum's Law 在容器抽象上特別猛**：Borg 把 PID 限在 0–32,000 成了沒人承諾卻被依賴的「API」，log 系統假設 PID 只有五位數、`{hostname, timestamp, pid}` 唯一識別行程——修這個的工程八年後仍在進行。**Borg 一統 batch 與 serving 兩種工作負載**帶來兩大效率：serving 機器變成牛（避免每隊各管一池的線性管理成本）；batch 填進 serving job 為應對尖峰而保留的閒置容量，使「batch 幾乎免費跑」。**選擇運算服務是高 lock-in 的取捨**（Hyrum's Law＋周邊工具生態）：集中 vs 客製（Borg 為 Cloud VM 與 Search 磁碟做的 bifurcation 讓 API 變得龐雜難測）、抽象層級（serverless 帶來 autoscaling/縮到零，但要求真正無狀態、犧牲環境控制權；Google 選擇不重押 serverless，因 Borg 已涵蓋多數好處，且許多應用無法真正無狀態）、公有 vs 私有（外包管理 overhead、易擴展，但怕 lock-in；用開源架構/多雲/混合雲緩解）。結論：規模需要**統一的共用運算基礎設施**；容器是共享機器與提供抽象層的關鍵；用好它要把應用設計成「牛」模型。

## 為什麼要讀這一篇

- **你還在 SSH 進機器、手動部署與重啟行程。** 本章從 Jeff Dean 2002 年那段「在 50+ 台機器上手動啟動、監控、出事還得手動遷移」的惡夢講起，給你一條清楚的演化路徑：簡單自動化（部署腳本、自癒重啟）→ 自動排程（中央服務管機器清單）→ 容器化多租戶 → rightsizing/autoscaling。看完你會知道「naive 解法在什麼規模、為什麼會崩」。
- **你的服務是「寵物」，壞了得有人 panic 地救。** 「cattle, not pets」是本章（也是規模化）的核心心法：把伺服器設計成能全自動 stamp out 替換的牛，維護負擔才不會隨 fleet 大小線性甚至超線性成長。但本章也誠實告訴你：當牛還不夠，**狀態**才是真正的難關——你得學會把本機狀態當短暫、把真儲存外置。
- **你在為團隊/組織選運算服務（VM？容器？serverless？公有雲？）。** 本章最後一節就是寫給你的：講清楚**高 lock-in**的本質（Hyrum's Law 讓你用上系統每個性質、周邊工具生態難遷移）、集中 vs 客製的張力、serverless 的真正比較對象（不是「VM 當寵物」而是「持久容器」）、以及公有/多雲/混合雲緩解 lock-in 的手法。
- **你想理解 Hyrum's Law 在基礎設施層有多可怕。** PID 32,000 那個案例是全書 Hyrum's Law 最生動的體現：一個沒人承諾的實作細節（PID 上限）被無數系統當成保證依賴，修起來是跨越八年、分三階段、還要 backport 資料的工程。這會永久改變你對「可觀察行為＝API」的警覺。

## 細節重點

### Taming the Compute Environment（馴服運算環境）
核心主張：從大學機房的 SFTP+SSH+手動編譯執行出發，naive 解法在規模下崩潰。Jeff Dean 2002 的設計文件描述了痛點：在 50+ 台機器上手動啟動、ad hoc 監控、機器掛了沒有自動遷移、還要靠人工維護的「sign-up 檔」節流——導致排程不佳、資源爭用。演化路徑：
- **Simple automations（簡單自動化）：** 部署可用 shell script（再進化成更穩健、可並行的程式）；監控可自動化（行程匯出存活/處理量等 metric 到共享儲存或監控服務，如 Grafana/Prometheus）；自癒——用 agent 偵測異常並殺掉行程、用 `while true; do run && break; done` 包住執行（雲端等價物＝設 autohealing policy：健康檢查失敗就殺並重建 VM/容器）。
- **Automated scheduling（自動排程）：** 下一步是自動化機器分配——這是第一個真正的「服務」、最終長成 CaaS。中央服務知道所有可用機器、按需挑空閒機器並自動部署你的 binary，取代手工 sign-up 檔（很像早期的分時架構）。可結合機器故障反應（掃 log 找壞健康徵兆、標記需修、先試自動修如重開機/磁碟掃描再找人）；機器壞了就由排程器配新機重啟工作、棄置舊的。這些都在系統性地對付組織成長的規模。引文出自 2002「Global WorkQueue」設計文件。
- **Containerization and Multitenancy（容器化與多租戶）：** 「一機一程式」在資源（RAM/CPU）上極浪費（工作類型遠多於機器類型、機器部署慢但程式需求成長快、舊機器還得管成異質 fleet）。自然解法：為每個程式指定資源需求（CPU/RAM/磁碟），讓排程器把複本 **bin-pack（裝箱）** 到機器池。但若大家不守規矩（bug 或自然成長吃超量）就會壞事——CPU 超用害鄰居 serving job 延遲抖動、RAM 超用引發 OOM kill（Google 一律選 OOM kill＋遷移，不忍受 disk swap 的延遲）；還有依賴版本衝突、`/tmp` 等系統資源獨佔、安全隔離。故多租戶運算服務必須提供**隔離（isolation）**。VM 是經典隔離解但開銷大（要跑整個 OS、開機慢），不適合小資源足跡、短執行的 batch job——這讓 Google 2003 設計 Borg 時轉向**容器**（基於 cgroups〔Google 2007 貢獻進 Linux kernel〕＋chroot jail/bind mount/union fs 做檔案系統隔離；開源實作有 Docker、LMCTFY）。隔離失敗會隨時間陸續被發現（如 2011 的 PID 耗盡）。
- **Rightsizing and autoscaling（合適尺寸與自動擴縮）：** 2006 的 Borg 按工程師在設定裡填的複本數與資源需求排程，但「叫人去定資源數字」本身有缺陷——這不是人類日常打交道的數字，定起來費時、隨服務增多成本上升，且程式會成長而設定不跟進，最終演成「保留的 slack 被新版吃掉、真尖峰來時不夠」的 outage。自然解法是自動化設定，但出乎意料地難做好——Google 直到近期才達到「整個 Borg fleet 過半資源用量由 rightsizing 自動化決定」。但即使只佔一半用量，卻是更大比例的「設定數」，意味多數工程師不必再煩惱裝箱——這是「**簡單的事該簡單、複雜的事該可能**」的成功應用。
- **Summary：** 組織成長有三軸（要管的應用數、單一應用的複本數、最大應用的大小）；要有效管理規模就需自動化，且自動化本身會隨時間變複雜（如過去十年 Borg 加入 GPU/TPU 排程）。連資料中心開設都從「數週、需專門技能、有風險的手動流程」轉向自動化（Google 仍在進行中）。

### Writing Software for Managed Compute（為受管運算寫軟體）
核心主張：從手管機器清單轉到自動排程/rightsizing，讓管理 fleet 容易得多，但也深刻改變了寫軟體的方式。
- **Architecting for Failure（為失敗而設計）／Cattle vs Pets：** 把 100 萬份文件分到 200 台機器（從 12 天降到 100 分鐘），但 Borg 排程器可能單方面殺掉某個 worker 移到別台（原因如更新 kernel、磁碟壞、重新裝箱優化——身為軟體作者你不該需要知道為什麼）。「**cattle, not pets**」：寵物壞了人 panic 搶救、難替換；牛（replica001–100）壞了自動化移除並補新的，關鍵是**能全自動 stamp out 新實例、不需人工**，由此自癒。寵物 → 維護負擔隨 fleet 線性甚至超線性成長；牛 → 故障後系統能自動回穩。但當牛還不夠：要在中等故障率下平順運作，需把工作切成可動態分配的小 chunk（100 萬份切成 1,000 個 1,000 份的 chunk，worker 做完一塊報告再領下一塊，故障最多丟一塊；正好契合 Google 當時 MapReduce/Flume 的資料處理架構）；serving job 則要能在 Borg 預告下線時拒新請求、排空在途請求，並要 load-balancer 懂「我不能收新請求」這個回應。
- **Batch vs Serving：** batch job（資料處理/ML 訓練）關心**吞吐量**、短命（分鐘到小時）；serving job（從預建索引回答搜尋查詢）關心單一請求**延遲**、長命（只在新版發布時重啟）、啟動時間較長。Batch 靠 MapReduce→Flume 把工作切小動態分配以抗失敗；serving 天生較適合抗失敗（請求自然切小、靠 load balancing），但有不自然的例外：「leader」型 server（維護系統狀態，掛了新實例重建不出）、把資料分片到 100 台各持 1%（一台掛就暫時失去那部分資料）、以及被別處用 hostname 寫死的 server。
- **Managing State（管理狀態）：** 狀態是把工作當牛的麻煩源——替換牛就丟掉行程內狀態與本機儲存。最簡單解法是把所有儲存外置到外部系統；本機狀態若全為不可變（immutable），抗失敗就相對輕鬆。「那持久儲存本身是牛嗎？」答案應是 yes——靠**狀態複本（state replication）**（類比 RAID：把磁碟當短暫卻仍維持狀態；通常複製 3–5 份，寫入需共識處理，故 Google 造了 GFS/Bigtable/Spanner）。其他本機儲存用途：**快取**（短暫本機儲存，賭狀態多半還在以提升平均效能——關鍵教訓：**按延遲目標配置快取、按總負載配置核心應用**，這樣快取層掛了也不會 outage，只是延遲變高）；warm-up 時從外部拉資料進本機；**批次寫入**（如監控資料，可容忍一小部分遺失或可重建；長計算可用週期性 checkpoint 切成小時間窗）。
- **Connecting to a Service（連線到服務）：** 任何東西寫死你的 host name（甚至當啟動參數傳入），你的複本就不是牛了。解法是加一層**間接（service discovery，服務發現）**：別的應用用一個跨重啟仍持久的識別碼指你，由排程器寫入的另一系統解析；client 通常在啟動時查地址、建連線、背景監控（多數方案還含 load balancing）。後果是有時得**重試請求**（你連的 server 可能還沒回應就被下線了——retry 要正確：backoff、graceful degradation、jitter 避免級聯故障，該放進 RPC 函式庫而非各自手刻）。對 mutating 請求，重複請求很棘手，要保證某種 **idempotency（冪等性）**——issue 兩次等於一次；好用工具是 **client 指派的識別碼**（如送披薩訂單由 client 給 ID，server 見過就當重複請求回報成功）。還有個驚奇案例：排程器因網路問題與某機失聯、判定工作全失而重排到別機——然後那機又回來了！兩台都自認是「replica072」，靠「定址系統指的是哪一個」消歧（另一個自我終結），這又是一個冪等性的理由。
- **One-Off Code（一次性程式）：** 工程師也要跑一次性分析、探索原型、客製管線。常常工作站就夠（撈 1 GB log 寫個 Python script 跑一兩分鐘）；但撈 1 TB（一年份）等一天不可接受——能在分散式環境用幾百核幾分鐘跑完，就是「現在有結果 vs 明天才有」、甚至「做得完 vs 根本做不完」的差別。有人擔心讓工程師隨意跑會浪費資源，但這是該有意識做的取捨：**工程師的時間幾乎一定比那點運算貴**（千核小時遠不及一天工程工時的成本，類比書開頭的「麥克筆」）。差別在運算很容易不小心拿太多（有人讓 1,000 台 GCE VM 在他放假時空轉、新人在工作站 debug 卻在背景生 8,000 個整機 worker）——解法是個人配額，或如 Google 用「低優先 batch 幾乎免費」提供近乎無限的低優先配額。

### CaaS Over Time and Scale（CaaS 隨時間與規模）
核心主張：幾個 CaaS 架構如何影響軟體跨時間與規模的層面值得細看。
- **Containers as an Abstraction（容器作為抽象）：** 容器最初是隔離/多租戶機制，但更重要的是成了**部署軟體與實際機器之間的抽象邊界**——機器隨時間變，只有容器層（由單一團隊管）要適配，應用層（隨組織成長由各團隊管）可不動。兩個例子：**檔案系統抽象**讓不在公司內寫的軟體（開源、收購來的）不必管客製機器設定就能上船，並讓軟體預先宣告/打包依賴（否則依賴機器上裝的東西＝漏抽象，逼大家用同版預編譯庫、升級難如登天）；**具名資源**（網路埠、GPU 等加速器）——Borg 早期沒把網路埠納入容器抽象，binary 得自己找空埠，導致 `PickUnusedPortOrDie` 在 Google C++ codebase 用了**逾 2 萬次**；Docker 用 namespace 給容器虛擬 NIC、Kubernetes 更要求把容器（pod）當「真」IP，讓每個 app 都能監聽任意埠不怕衝突。
- **Containers and implicit dependencies（容器與隱式依賴）：** Hyrum's Law 在容器抽象上**比平常更猛**（用戶極多＋用戶根本不覺得自己在用 API，如用檔案系統時）。**PID 32,000 案例：** Linux PID 預設在 [0, PID_MAX−1]、PID_MAX 預設 32,000；可改大——但 Borg 上 PID 限在 0–32,000 的事實成了被依賴的隱式 API 保證（log 儲存假設 PID 五位數、六位數就爆）。修法是漫長的多階段工程：①先給單容器 PID 數設暫時上限（免單一漏執行緒的 job 拖垮整機）②拆分執行緒與行程的 PID 空間（少有人依賴執行緒 PID 的 32,000、故執行緒可放寬而行程維持）③為 Borg 引入 PID namespace 給每容器完整 PID 空間——但又有一堆系統假設 `{hostname, timestamp, pid}` 唯一識別行程，會被打破，這修正工程**八年後仍在進行**。教訓不是「你該用 PID namespace」，而是：設計能長期維護的容器系統有多難，因而**用更廣社群開發/使用的容器系統有其價值**（這類問題別人已踩過、教訓已被納入）。
- **One Service to Rule Them All（一統天下的單一服務）：** 原 WorkQueue 只管部分 batch job、serving job 各跑自己的專屬機器池（開源等價＝每種負載各跑一個 Kubernetes cluster）。2003 啟動的 Borg 把這些分散的池併成一個大池、同時涵蓋 serving 與 batch，成為每個資料中心唯一的池。兩大效率：①**serving 機器變成牛**（「機器是匿名的：程式不在乎跑在哪台，只要特性對」）——否則每個管 serving job 的隊都要管自己的池，管理實務隨時間發散、讓全公司級變更（換 server 架構、換資料中心）越來越複雜；統一管理避免這個線性因子（沒有 n 套管理實務，只有 Borg）。②**batch 與 serving 需求互補**：serving 必須 over-provision（要有容量應付尖峰/局部故障而不增延遲）故機器被低度利用；硬塞別的 serving job 會抵消 slack 的目的，但可以塞 **batch job 進那 70% 閒置**，政策是 serving 一需要就從 batch 收回（CPU 凍結、RAM 殺掉）——因 batch 關心的是聚合吞吐、複本本來就是牛，樂於吸收這些閒置。結果是 batch 幾乎免費跑（Google 多數時候如此），或 serving 只付實際用量、不付為抗失敗保留的 slack。
- **Multitenancy for serving jobs（serving job 的多租戶）：** 把受管運算擴及 serving 有新要求：**重排要節流**（殺 50% batch 複本只是處理短暫抖動，但殺 50% serving 複本剩下的撐不住用戶流量）；**batch 通常可無預警殺**（丟的處理可重做），但 serving 被無預警殺會害用戶流量回錯誤/增延遲，**該給幾秒預告**讓它排空在途請求、不收新的。
- **Submitted Configuration（提交式設定）：** Borg 排程器以 RPC 內容收取設定；可用 CLI 送 RPC、參數存在文件或腦中——但**依賴文件與部落知識（tribal knowledge）通常不好**（會隨時間腐化，見 [[ch03]]），把 CLI 包進本地腳本也仍劣於用**專屬設定語言**。隨時間，一個邏輯服務會在多軸長大（跨多資料中心、分出 staging/dev 環境、加掛 memcached 等附屬服務），用標準化設定語言能輕鬆表達標準操作（如「更新到新版但任何時刻下線不超過 5% 容量」）、讓別隊易於 include 標準設定（否則每隊各寫一段 memcached 程式，全組織級換實作或推安全更新就很難）；標準化設定語言也是部署自動化（見 [[ch24]]）的前提。

### Choosing a Compute Service（選擇運算服務）
核心主張：少有組織會像 Google 從零自建；今日有開源（Kubernetes、Mesos、OpenWhisk、Knative）與公有雲受管方案（GCP MIG、AWS EC2 autoscaling、AKS/GKE、AWS Lambda/Cloud Functions）。但選擇有**高 lock-in**：因 Hyrum's Law，程式會用上系統每個性質（選 VM 就調 VM image、選容器就呼叫 cluster manager API；架構若允許把 VM/容器當寵物，團隊就會這麼幹，日後遷到「當牛」的方案就難）。Borg 用 ash 取代 bash 省記憶體的小改動就翻車——有些團隊早已在自訂 fs overlay 把 bash 換成自寫程式，Borg 改用 ash（未被覆寫）反讓他們記憶體上升、觸發告警回滾。且任何運算服務終會被一大圈周邊工具（logging/監控/debug/告警/視覺化/設定語言/UI）包圍，換服務得重寫、連列舉都難。故選擇很重要，是取捨：
- **Centralization vs Customization（集中 vs 客製）：** 管理 overhead 與資源效率上，最好是全 fleet 採單一 CaaS、大家只用那裡的工具（Borg 路線）。但成長的組織需求漸雜：Google 2012 推 GCE（VM-as-a-Service）時 VM 也由 Borg 管（每 VM 在獨立容器），但「當牛」不適合 Cloud 負載（用戶把 VM 當寵物）——Cloud 得支援 **VM live migration（即時遷移）**、Borg 得改成不隨意殺含 VM 的容器並優化降低重排風險，只對 cloud 機器推出，造成內部運算的小分叉（bifurcation）。另一例：2011 Search 某容器在本機磁碟建巨大索引（多顆硬碟、數小時填資料），但 Borg 當時假設容器任一磁碟壞就得重排——高磁碟故障率害可用性慘，得讓容器能自行處理磁碟故障、Search 得能在部分資料遺失下續跑。多個分叉（檔案系統形狀/存取、記憶體控制、CPU/記憶體 locality、特殊硬體、特殊排程約束）讓 Borg 的 API 變得龐大難測、行為交集難預測。2012 後 Borg 團隊花大量時間清理 API（發現有些功能根本沒人用、有些是 config 互抄而擴散的 power-user 功能——引入 whitelist 限制擴散），但清理八年後仍未完（如用 label 識別容器群仍沒全做完）。**Kubernetes 受惠於清理 Borg 的經驗、又沒有龐大既有用戶包袱，一開始就現代得多**（但廣為採用後也開始遭遇同樣問題）。
- **Level of Abstraction: Serverless（抽象層級：serverless）：** 故事易被讀成「抽象越多越好」，但沒這麼簡單。從 bare-metal 寵物 → 容器當牛，中間還有 VM 路線（從 IaaS 如 GCE/EC2 當更彈性的 bare metal 替代，到加 autoscaling/rightsizing 當較重的容器替代）。Google 經驗：管牛不管寵物是規模化的解；選了管牛後容器是自然選擇（更輕、可設定到能對特定負載開特殊硬體存取）；VM 當牛的優勢主要在「能帶自己的 OS」與既有 VM 經驗/負載。**Serverless** 是更高抽象：用戶只寫框架的 Action/Handler 函式（控制反轉 inversion of control），框架 server 本身也多租戶、動態載入/卸載 action 程式並導流，個別團隊不再跑 server（故曰 serverless）。多數討論拿 serverless 比「VM 當寵物」（那當然像革命：autoscaling、低 overhead、免顯式 provision），但既然該往「共享多租戶當牛」走，serverless 的正確比較對象是 **Borg/K8s/Mesos 這類持久容器架構**。
  - **Pros/cons：** serverless 要求程式**真正無狀態**（跑不了用戶 VM 或 Spanner，前述所有管本機狀態的法子都不適用，一切要 request-scope 設定）。實務上多數組織有無法被無狀態負載服務的需求 → 要嘛靠特定方案（如受管資料庫，常與公有雲 serverless 搭配）、要嘛同時有容器與 serverless 兩套（很多 serverless 框架本就建在別的運算層上：AppEngine 跑在 Borg、Knative 在 K8s、Lambda 在 EC2）。serverless 在**低流量端**有吸引力（能縮到零、成本隨流量；K8s 持久容器縮不到零、有「光是可用」的最低成本）；高流量端則受底層基礎設施限制（要 10 萬核就得有 10 萬實體核）；中間兩者都能撐、但 serverless 擴縮更即時、更細粒度。serverless 也意味**失去部分環境控制權**（少了管理 overhead，但缺框架沒有的功能時就麻煩——如 Code Jam 團隊得自寫腳本在比賽前用假流量尖峰預熱足夠 AppEngine 實例）。
  - **取捨：** Google 選擇不重押 serverless——Borg 已夠先進、能提供多數 serverless 好處（autoscaling、各類框架、部署/統一 logging 監控工具），唯一缺的是更激進的擴縮（尤其縮到零），但 Google 資源足跡多來自高流量服務、over-provision 小服務相對便宜；同時 Google 跑很多無法真正無狀態的應用（GCE、BigQuery、Spanner、長尾搜尋）——故「一套統一架構涵蓋全部」的好處勝過為部分負載另立 serverless stack。但這不必是每個組織的正解：serverless 對**小組織/小團隊**才是主要吸引力（比較天生不公平：serverless 讓供應商承擔大部分管理 overhead，在共享 cluster 沒被多隊真正共享時，用 Lambda/Cloud Run 比自建 GKE/AKS cluster 簡單便宜得多）。注意組織長大後可能會長出純 serverless 的限制，故**有「破繭路徑」的方案**（如 KNative→Kubernetes）有吸引力。
- **Public vs Private（公有 vs 私有）：** 公有雲＝把部分管理 overhead 外包給供應商（讓你專注本業、不必養龐大基礎設施專長；雲商雖收費高於裸機成本以回收管理費，但專長已具備且攤給多客戶），也更易擴展（從 colocation 簽租約 → 跑 CLI 多開幾台 VM → autoscaling 隨流量自動變動資源足跡；對年輕組織/產品尤其重要，因資源需求難預測）。主要顧慮是 **lock-in**（供應商可能突然漲價或倒掉——如早期 serverless 供應商 Zimki 2007 給三個月通知就關門）。緩解：用跑在開源架構（如 Kubernetes）上的公有雲方案以保有遷移路徑（但 Hyrum's Law 讓「保證沒用到供應商特有部分」很難）；兩種延伸——用低階公有雲（EC2）跑高階開源方案（OpenWhisk/KNative）於其上，或跑**多雲**（multicloud，如 GKE＋AKS 都跑 K8s）；還有**混合雲**（hybrid cloud，部分私有、部分公有，可用公有雲處理溢流 overflow）——但多雲/混合雲都要求多環境良好連通（直接網路連線＋兩邊都有的共用 API）。

工程意涵（全書收束）：規模需要**統一的共用運算基礎設施**（每區一個或少數共享 cluster），帶來管理與資源成本的效率、並讓共用工具能建在其上；**容器**是讓不同任務共享機器（資源效率）並在應用與 OS 間提供抽象層（時間上的韌性）的關鍵工具；用好容器架構要把應用設計成「**牛**」模型（節點可輕易自動替換才能擴到數千實例），而這需要不同的思考方式——把所有本機儲存（含磁碟）當短暫、別寫死 hostname。Google 對自己的架構選擇滿意且成功，但別的組織會從寵物、牛、serverless 的廣大光譜中（受管與開源皆有）做出屬於自己的複雜取捨。

## 關鍵字補充

- **Compute as a Service（CaaS，運算即服務）** — 「給我硬體跑我的東西」這個訴求長成的受管運算系統；Borg 是 Kubernetes/Mesos 等的先驅。
- **Cattle, not Pets（把機器當牛、不當寵物）** — 把伺服器設計成能全自動 stamp out 替換的牛而非要人搶救的寵物；規模化與自癒的根本，Google 15 年成功的核心工具。
- **Bin-packing（裝箱）／Multitenancy（多租戶）** — 按程式資源需求把複本裝箱到機器池共享機器；需要隔離以免互相干擾。
- **Isolation（隔離）／Container（容器）** — 多租戶必備的保證；容器是比 VM 輕量的隔離（cgroups 由 Google 2007 貢獻進 Linux kernel）＋應用與機器間的抽象邊界。
- **Rightsizing / Autoscaling（合適尺寸／自動擴縮）** — 自動決定資源需求與複本數，免去人工填數字的費時易錯；「簡單的事該簡單、複雜的事該可能」。
- **Architecting for Failure（為失敗而設計）** — 工作切成可動態分配的小 chunk、serving job 收下線預告後優雅排空；當牛還不夠，要能在故障率下平順運作。
- **Batch vs Serving** — batch 關心吞吐、短命、可無預警殺；serving 關心延遲、長命、重排要節流且需下線預告。
- **State Replication（狀態複本）／Transient Local Storage** — 把本機狀態當短暫、真儲存外置到有複本同步的持久系統（GFS/Bigtable/Spanner）；快取按延遲配置、核心按總負載配置。
- **Service Discovery（服務發現）／Idempotency（冪等性）** — 用一層間接定址取代寫死 hostname；retry 與重複請求的解藥是冪等（client 指派 ID）。
- **Hyrum's Law（在容器抽象上）** — PID 32,000 案例：沒人承諾的實作細節被當 API 依賴，修正跨八年三階段；故用廣社群開發的容器系統有價值。
- **One Service to Rule Them All** — Borg 把 batch 與 serving 併成單一池：serving 變牛省線性管理成本＋batch 填 serving slack 幾乎免費跑。
- **Serverless / Lock-in** — 只寫 Handler、框架 server 多租戶、能縮到零；要求真正無狀態、犧牲環境控制權；比較對象是持久容器而非 VM 寵物。運算服務選擇高 lock-in（Hyrum's Law＋周邊工具生態），用開源/多雲/混合雲緩解。

## 延伸提問

1. 你的服務是「寵物」還是「牛」？你能不能做到「一台壞了，自動化全自動 stamp out 一台新的、不需任何人 SSH 進去設定」？若還不行，卡在哪——是部署不自動、還是藏了**狀態**（行程內/本機磁碟/寫死的 hostname）讓它無法被替換？對照 [[ch22]] 也用了 cattle vs pets。
2. 你把本機狀態當「短暫」了嗎？哪些東西其實該外置到持久儲存？你的快取是「按延遲目標配置、核心路徑按總負載配置」，還是把快取當成不能失去的依賴（一掛就 outage）？你的 mutating API 有沒有 **idempotency**（retry／重複請求不會出錯）？
3. 你的基礎設施裡有沒有「PID 32,000」式的隱式依賴——某個沒人承諾的實作細節（埠範圍、ID 格式、檔案佈局），被無數地方當保證依賴？一旦要改，你估得出代價嗎？這正是 [[ch01]] Hyrum's Law 在基礎設施層的體現。
4. 你在為團隊/組織選運算服務嗎？你評估過 **lock-in** 的真實代價（Hyrum's Law 讓你用上系統每個性質、周邊工具生態難遷移）嗎？你的需求是真正無狀態（適合 serverless）還是有狀態（需持久容器）？小團隊用公有雲 serverless、大組織用統一容器——你在哪一格？
5. 你的組織是把 batch 與 serving 分開各管一池，還是有沒有機會像 Borg 那樣**統一**以吃到「serving 變牛省管理成本＋batch 填 slack 幾乎免費」的效率？你算過各隊各管一池的線性管理成本嗎？對照 [[ch07]] 的生產力量測。
