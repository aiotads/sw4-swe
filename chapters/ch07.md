# 第 7 章｜Measuring Engineering Productivity（衡量工程生產力）

> *"Success means giving a stakeholder the data they need to make a decision."*
> 「所謂成功，是把決策者做決定所需的資料交到他手上。」

## 摘要

這是「文化」篇的收尾，主題是 Google 如何**用資料、而非直覺**去衡量並改善工程生產力。作者 Ciera Jaspan 來自 Google 內部專門研究工程生產力的團隊——刻意混入認知心理學、行為經濟學等社會科學家，因為要研究的不只是工程師產出的「軟體製品」，還有「人」這一側（動機、誘因結構、處理複雜任務的策略）。

全章用一條真實案例貫穿：[[ch03]] 提過的 **readability（可讀性）制度到底值不值得？** 流程包含三個關卡：
1. **Triage（分流）：值得測嗎？** 測量本身昂貴，且可能改變工程師行為而掩蓋真相。先把問題寫成**具體問句**，再追問五題——你預期什麼結果、為什麼？若資料支持你的預期會採取什麼行動？若得到**負面**結果會採取行動嗎？誰、何時會根據結果行動？很多專案就死在「負面結果其實不會改變決定」這題——那就不值得測。**只有當結果會導向一個具體決定時，才測。**
2. **選有意義的指標：GSM 框架。** Goal（目標：想理解的最終結果，不含測量方式）→ Signal（訊號：你怎麼知道達成了，未必可測）→ Metric（指標：訊號的可測代理）。由上而下定義能避免「**路燈效應**」（只在燈下找鑰匙——只用手邊好測的指標）、避免事後挑指標的偏誤，並保持**可追溯性**。目標要涵蓋生產力的五個面向——**QUANTS**（Quality 品質、Attention 專注、iNtellectual complexity 智識複雜度、Tempo/velocity 步調與速度、Satisfaction 滿意度）——它們彼此**取捨**，以免你改善速度卻拖垮品質（「我能讓你的審查速度飛快：把 code review 整個拿掉就好」）。
3. **用資料驗證指標、採取行動。** 量化指標給你規模與信心，但沒有脈絡與敘事；質化研究（訪談、調查）才能解釋「為什麼」。兩者衝突時，**通常是量化指標沒抓對**。最終 readability 的結論是「**確實值得**」：有 readability 的作者程式碼被審查與提交都更快（即使已扣掉不再需要那麼多審查者）、滿意且覺得有學到東西。

## 為什麼要讀這一篇

- **你想推一個「能提升生產力」的工具或流程改動，卻拿不出證據說服主管。** 這章給你 GSM 這套可防禦的方法：先講清楚目標，再導出訊號與指標，讓利害關係人**事先**同意「這就是衡量它的最佳指標組」，杜絕事後「換個指標重算」的扯皮。
- **你或你的主管很愛用 LOC、commit 數、修了幾個 bug 當生產力指標。** 這章用 Dijkstra 的話告訴你 LOC 是「**spent（花掉的）**」不是「produced（產出的）」，並警告「路燈效應」與**用生產力指標考核個人**會立刻被 gaming、從此失去衡量價值。
- **你常常還沒想清楚「測來幹嘛」就先開始收數據。** Triage 的五個問題會逼你先問「拿到結果會採取什麼行動？」——若答案是「什麼都不會改」，那這次測量從一開始就是浪費。**測量的成功 = 決策者真的用了那份資料。**
- **你只信「硬數據」、看不起「軼事/問卷」。** 這章示範了量化＋質化「三角驗證」：median build latency 的真實烏龍（自動建置被誤計）正是靠質化抽樣訪談才抓到。質化指標也是指標，而且常常是它在救你的量化指標。

## 細節重點

### Why Should We Measure Engineering Productivity?（為何要衡量工程生產力）
核心主張：當組織規模**線性**成長時，溝通成本卻是**二次方（quadratically）**成長（引 Brooks《人月神話》）。所以加人雖能擴大業務範圍，但溝通開銷不會線性擴展——你無法把業務範圍線性地擴展到工程組織的規模。

另一條路：**讓每個工程師更有生產力**——這樣就能擴大業務範圍而不必付出對應的溝通開銷。為此要：理解什麼讓工程師有生產力 → 找出流程中的低效 → 修掉 → 在「持續改進循環」中重複。但**改進循環本身也要消耗人力**：若用 50 人/年去理解與修復、卻只換來相當於 10 人/年的提升，就不划算。所以目標是「**不只要提升生產力，還要有效率地提升**」。Google 的解法：成立專責研究團隊，混入社會科學家，用資料驅動的方法衡量與改善。

工程意涵：這一章把全書的 Scale 主軸用在「人」身上——個人生產力的次線性槓桿，是對抗「溝通成本二次方爆炸」的關鍵。

### Triage: Is It Even Worth Measuring?（分流：這東西值得測嗎？）
核心主張：決定「怎麼測」之前，先決定「**值不值得測**」。測量本身昂貴（要人去測、分析、散播），可能拖慢工程組織，甚至**光是追蹤就會改變工程師行為、反而掩蓋了底層問題**。先把想測的東西寫成一個**具體問句**（越具體越可能真的得到好處），再追問五題：

- **你預期什麼結果，為什麼？** 承認自己並非中立——先講出預設立場，才能對抗偏誤、避免事後硬掰解釋。（readability 團隊的答案是「不確定」：自動格式化與靜態分析普及後，沒人敢肯定它還值不值。）
- **若資料支持你的預期，會採取什麼行動？** 不會採取行動就別測。（會把研究與資料放上 readability 的 FAQ 來設定期待。）
- **若得到負面結果，會採取適當行動嗎？** ——**這題殺掉了研究團隊最多的專案**。很多時候決策者只是想知道結果，但基於其他理由不會改變方向；那就不值得測。（readability 團隊承諾：若成本大於效益或效益微不足道，就**砍掉這個流程**，且按語言逐一評估。）
- **誰、在什麼時候會根據結果採取行動？** 確保提出測量需求的人有權行動，並了解**什麼形式的資料能說服他**（訪談故事？問卷？logs？複雜統計？）。若決策者原則上不信那種形式的結果，測了也沒用。（Java、C++ 團隊主動求助；決策者「快樂與學習」信工程師自陳，但「速度與品質」要看 logs 的「硬數字」→ 質化與量化都要做。）

**不值得測的常見情況**：①現在無力改變流程/工具（更廣的脈絡完全正當化「延後行動」）；②結果很快會被其他因素作廢（如重組前、棄用系統的技術債）；③決策者已有不可動搖的成見、你給不出能改變他的證據；④結果只會被當成**虛榮指標（vanity metrics）**去支持一個你本來就要做的決定（最常見的勸退理由）；⑤可用的指標不夠精確、會被其他因素混淆（如改用 LOC——結果無法解讀）。

工程意涵：**測量的成功，不是證明假設對或錯，而是把決策者所需的資料交到他手上；如果他不會用那份資料，這個專案就是失敗。**

### Selecting Meaningful Metrics with Goals and Signals（用目標與訊號選出有意義的指標）
核心主張：決定要測之後，用 **GSM 框架**選指標——LOC 顯然不行：
- **Goal（目標）：** 想要的最終結果，用「想理解的高層性質」描述，**不得提到任何具體測量方式**。目標本身不可測，但好的一組目標是大家在進到訊號與指標前都能同意的。
- **Signal（訊號）：** 你怎麼知道達成了目標。訊號是「我們想測的東西」，但**未必本身可測**。目標與訊號**非一對一**：每個目標至少一個訊號，可有多個，也可共用。
- **Metric（指標）：** 訊號的**可測代理（proxy）**，實際能量到的東西，未必完美。一個訊號可能配多個指標，以**三角驗證**底層訊號。

GSM 帶來三個好處：①**先目標、再訊號、最後指標**，避免「**路燈效應（streetlight effect）**」——只在路燈下找鑰匙（只用手邊好測的指標）；②**事先用有原則的方法選好指標**，避免 metrics creep（指標蔓延）與 metrics bias（指標偏誤）——否則結果不如預期時，利害關係人會提議換指標，而你當初沒原則就無法反駁；③**顯示覆蓋缺口**——跑完 GSM 會發現有些訊號根本不可測（沒關係！至少你知道哪裡缺）。核心是**可追溯性（traceability）**：每個指標都能追回它代理的訊號、以及它要量的目標。

#### Goals（目標：QUANTS 五面向）
核心主張：要先選對「該測的目標」。團隊常忘了把生產力裡的**取捨**全部納入，導致誤測（例：只顧把 review 變快，忘了 code quality——「我能讓審查飛快：把 code review 整個拿掉」）。為此把生產力切成五個彼此取捨的核心成分，用口訣 **QUANTS** 記憶：
- **Quality of the code（程式碼品質）：** 產出的程式碼品質如何？測試夠不夠防回歸？架構能否化解風險與變更？
- **Attention from engineers（工程師的專注）：** 多常進入心流？被通知打斷多少？工具是否鼓勵頻繁切換情境？
- **iNtellectual complexity（智識複雜度）：** 完成任務需多少認知負荷？問題本身的固有複雜度？有沒有不必要的複雜？
- **Tempo and velocity（步調與速度）：** 多快能完成任務、推出版本？單位時間完成多少任務？
- **Satisfaction（滿意度）：** 對工具多滿意？工具多符合需求？對工作與產出多滿意？工程師會不會 burnout？

（readability 的目標就分布在 Quality／Intellectual／Tempo／Satisfaction 四面向；**Attention 沒有目標——沒關係！不是每個問題都涉及全部五面向。**）

#### Signals（訊號）
核心主張：訊號是「我們怎麼知道達成了目標」的方式（Table 7-1）。例：「有 readability 的工程師，自評程式碼品質高於沒有的人」「有 readability 的人寫的 CL 比沒有的人更快被審查」。

#### Metrics（指標）
核心主張：指標是訊號的可測代理，**不是訊號本身**。例：要測「拿到 readability 後程式碼是否被更快審查」，可同時用**問卷資料＋logs 資料**——兩者都不提供底層真相，但若**兩者結果不一致，就代表可能有一個錯了、得再深挖；若一致，就更有信心觸及了某種真相**。有些訊號可能**沒有任何指標**（暫時不可測）——例如 **code quality**：學術界提了很多代理但都沒真正抓到它，最後 readability 決定**不把它當量化指標**（只請工程師自評品質）。

### Using Data to Validate Metrics（用資料驗證指標）
核心主張：照 GSM 跑完仍可能「指標沒抓對訊號」，所以要用**質化資料驗證量化指標**。
- **median build latency 的烏龍：** 想抓工程師建置延遲的「典型體驗」，做了 **experience sampling（經驗抽樣）研究**——建置一啟動就推一份小問卷。結果有工程師回「我根本沒啟動建置！」——原來是**自動化工具**在跑建置，工程師沒被卡住、不該計入「典型體驗」。於是調整指標、排除這類建置。
- **量化 vs 質化：** 量化給你**規模與信心**（跨全公司、長時間），但**不提供脈絡與敘事**——它不解釋工程師「為什麼」用了過時工具、走了不尋常的流程、繞過了標準程序。只有質化研究能給這些洞見與下一步。Google 的經驗：**量化與質化衝突時，通常是量化沒抓到預期結果。**
- **三個資料來源：** readability 最終結合 ①readability 流程**專屬問卷**（流程後立即填，避免 recall bias，但引入 recency bias 與 sampling bias）、②大規模**季度問卷**（追蹤非 readability 專屬、但 readability 預期會影響的指標）、③開發工具的**細粒度 logs**（如 median review time、shepherding time、time to submit）。
- **鐵則（footnote 9）：** **絕不要用生產力指標去考核個人、辨識高低績效者**——一旦用於績效考核，工程師會立刻 gaming 指標，指標從此失去衡量與改善組織生產力的價值。「唯一能讓這些測量有效的方法，是**放下『衡量個人』的念頭、擁抱衡量總體效應**。」

### Taking Action and Tracking Results（採取行動並追蹤結果）
核心主張：研究後一定要備出一份**改進建議清單**（新功能、降低工具延遲、改善文件、移除過時流程、甚至改變誘因結構）。理想上建議要「**工具驅動（tool-driven）**」：光叫工程師改變思維或流程、卻沒有工具支援是沒用的——**假設工程師在有正確資料與合適工具時，就會做出恰當的取捨**。

readability 的結果：整體**值得**。有 readability 的工程師滿意、覺得學到東西；logs 顯示他們的程式碼被審查更快、提交更快（**即使已扣掉不再需要那麼多審查者**）。研究也指出可改進處（痛點），語言團隊據此改善了工具與流程的速度與透明度。

工程意涵：這正是 [[ch03]] readability「用線性換一致性」取捨的**資料結局**——一個成熟組織如何用 GSM＋QUANTS＋量化質化三角，把「值不值得」這種爭論從午餐桌的口水戰，變成可被決策者採用的證據。

## 關鍵字補充

- **GSM Framework（Goals/Signals/Metrics）** — 由上而下選指標：目標（想理解的結果）→ 訊號（怎麼知道達成）→ 指標（可測代理）。避免路燈效應、保持可追溯性。
- **QUANTS（生產力五面向）** — Quality 品質、Attention 專注、iNtellectual complexity 智識複雜度、Tempo/velocity 速度、Satisfaction 滿意度。彼此取捨，確保不會改善一項卻拖垮另一項。
- **Streetlight Effect（路燈效應）** — 只用手邊好測的指標（在路燈下找鑰匙），而非真正能回答目標的指標。GSM 的由上而下流程就是為了避免它。
- **Traceability（可追溯性）** — 每個指標都能追回它代理的訊號與要量的目標。確保你知道在測什麼、為何測。
- **Triage（測量分流）** — 在投入測量前，用五個問題判斷「值不值得測」。核心是「拿到結果（含負面）會不會導向具體行動」。
- **Vanity Metrics（虛榮指標）** — 只用來支持一個你本來就要做的決定的指標。Google 最常用來勸退測量的理由。
- **Quantitative vs Qualitative（量化 vs 質化）** — 量化給規模與信心、沒有脈絡；質化給「為什麼」與敘事。兩者衝突時，通常是量化沒抓對。質化指標也是指標。
- **Experience Sampling（經驗抽樣）** — 在工程師正做某任務的當下打斷、問幾題，以捕捉「情境中的真實體驗」。用來驗證 median build latency 指標、抓出自動建置的誤計。
- **Recall / Recency / Sampling Bias（回憶／近因／抽樣偏誤）** — 問卷時點與對象造成的三種記憶/取樣偏誤；流程後立即問可避回憶偏誤，但引入近因與抽樣偏誤。
- **Tool-Driven Recommendation（工具驅動的建議）** — 把改進內建進開發者的日常工作流與誘因結構，而非只叫人改變思維；改變才更可能真的發生。

## 延伸提問

1. 挑一個你或你主管最近用來談「生產力／績效」的指標（LOC、commit 數、velocity、修了幾個 bug、PR 數）。它是 Goal、Signal 還是 Metric？它能**追溯**回哪個你真正在乎的目標嗎？還是只是「路燈下」最好測的那個？
2. 把你最近一次想做的「流程/工具改動」套上 Triage 五問。最關鍵的一題：**如果量出來是負面結果，你（或決策者）真的會改變決定嗎？** 如果不會，這次測量是不是其實是 vanity metric？
3. 用 QUANTS 檢查你團隊現行的「效率改善」目標：你是不是只盯著 Tempo/velocity，卻沒在量 Quality 或 Satisfaction？有沒有可能你正在「把 code review 拿掉式」地犧牲另一個面向？
4. 你的組織有沒有把某個生產力/品質指標**用在個人績效考核**上？它有沒有被 gaming 的跡象（為衝指標而寫的水 code、灌的 commit）？對照本章「衡量總體、不衡量個人」。
5. 回想你上次「只信硬數據、不信問卷/訪談」的時刻。有沒有可能你錯過了 median build latency 那種「量化其實沒抓對」的烏龍？你能不能設計一個**質化抽樣**去驗證你最依賴的那個量化指標？對照 [[ch06]] Web Search 用資料平衡延遲。
