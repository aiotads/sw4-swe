# Chapter 20 — Static Analysis

> *"Focus on developer happiness."*
> Build the feedback loop and earn the trust — that's what makes static analysis actually get used.

## Summary

**Static analysis** analyzes *source code* (not a running program — that's *dynamic* analysis) to find bugs, antipatterns, and issues diagnosable without execution. It catches bugs **early**, before they're committed (overflowing constant expressions, never-run tests, crash-prone logging format strings); but it's for far more than bugs — at Google it also **codifies best practices, keeps code current with modern APIs, and prevents/reduces technical debt** (checking naming conventions, flagging deprecated APIs, pointing out simpler equivalents), and is a key tool in **API deprecation** to prevent "backsliding" ([[ch15]], [[ch22]]) — it can even **educate developers and prevent antipatterns from entering the codebase.**

Two properties make static analysis effective: **Scalability** — tools must be shardable and incremental, analyzing only the **files/lines affected by a pending change** rather than whole projects; and scaling the **number and variety** of analyses (soliciting contributions company-wide) while **showing results directly to relevant engineers** to avoid bottlenecks. **Usability** — weigh the cost-benefit for users, **focus on newly introduced warnings** (for working code that isn't being changed, "fixing" a warning may introduce a bug for unclear benefit), **fix automatically whenever possible**, and integrate smoothly into the workflow.

From these come three key lessons: **① Focus on developer happiness** — measure, keep false-positive rates low, build a feedback loop, and earn **user trust**; the key concept is the **effective false positive**: any issue is one if the developer took no positive action after seeing it, even if technically correct. **② Make static analysis part of the core workflow** — the main integration point is **code review** (a "sweet spot": developers are in a change mindset, have time to run analyses while blocked on reviewers, feel peer pressure, and reviewers save time). **③ Empower users to contribute** — let domain experts write analyses via simple APIs (like Refaster), scaling expertise.

Google's platform is **Tricorder** (integrated with Critique as gray boxes; microservices; 50,000+ changes analyzed per day). New checks must meet four criteria: **understandable, actionable and easy to fix, < 10% effective false positives, and potential for significant impact on code quality** (the overall effective-false-positive rate is just under 5%). Other essentials: **integrated tools** (Error Prone/clang-tidy AST antipatterns, IfThisThenThat, a binary-size checker), **feedback channels** ("Not useful"/"Please fix" buttons; sometimes fixing the *message* fixes the problem), **suggested fixes** (one-click apply; style should be auto-fixed), **per-project (not per-user) customization**, **presubmits** (blocking commits), **compiler integration** (push earlier, break the build; **never issue compiler warnings** — make it an error or don't show it, because developers ignore warnings), and IDE/browsing-time analysis.

## Why read this

- **You've rolled out a static-analysis tool and everyone found it noisy and ignored it.** The chapter names the key: **low false-positive rates and user trust**, measured by the **effective false positive** (any issue where the developer took no positive action). Build a **feedback loop** and drive false positives under 10%, and the tool actually gets used.
- **You don't know where to run static analysis.** Here's the full map: **IDE (< 100ms) → code review (sweet spot) → presubmit (block commit) → compiler (break the build)** — earlier to the left, more forcing to the right. It explains why **code review is the sweet spot** and why **the compiler never issues warnings.**
- **You want to scale static analysis without a huge tools team.** The answer is **empower users to contribute**: build an "easy-to-plug-into ecosystem" with simple APIs (Refaster: define a transform from before/after code snippets) so domain experts write checks — most of Tricorder's 100+ analyzers come from outside the team.
- **You're tempted to let users turn off annoying checks individually.** The hard-won lesson: do **per-project, not per-user** customization — after removing user customization, the "annoying" complaints actually helped find real tool bugs (a C++ linter running on Obj-C, a high-false-positive HTML linter); user customization only hides bugs and suppresses feedback.

## Key details

### Characteristics of Effective Static Analysis
The core claim: decades of research focused on new techniques, but **scalability and usability are a relatively recent focus.**
- **Scalability:** tools must be **shardable and incremental** — not analyzing entire large projects but **focusing on the files affected by a pending change**, typically showing results only for **edited files or lines.** Scale helps: a huge codebase has lots of low-hanging-fruit bugs. Also scale the **number and variety** of analyses (solicited company-wide), and avoid bottlenecks by **showing results directly to relevant engineers.**
- **Usability:** consider the user's **cost-benefit** — the cost may be developer time or code quality (fixing a dead-code warning could make untested code suddenly run). So **focus on newly introduced warnings** (existing, working code is usually worth fixing only when especially important — security, significant bugs), which also gives the viewing developer the **most relevant context.** Developer time is valuable: **fix automatically whenever possible**, show only quality-impacting issues, and integrate smoothly (one workflow lets a dedicated tools team evolve tools in tandem with the code).

### The Three Key Lessons
- **① Focus on Developer Happiness:** measure tool performance (you can't fix what you don't measure); **only deploy tools with low false-positive rates**; solicit and act on feedback in real time, building a virtuous tool-user↔tool-writer loop — **user trust is extremely important.** **False negative (a missed issue) vs false positive (a wrongly-flagged one)**: research traditionally focused on reducing false negatives, but **in practice low false-positive rates are critical** for adoption (who wants to wade through hundreds of false reports for a few true ones?). **Perception** matters too — warnings that are technically correct but misunderstood (confusing messages) or unimportant draw the same reaction as false positives, hence the **effective false positive**: **if a developer takes no positive action after seeing it, it's an effective false positive.** Conversely, if an analysis isn't quite right but the developer happily fixes it for readability (`contains` → `containsValue` is clearer), that's **not** an effective false positive.
- **② Make Static Analysis Part of the Core Developer Workflow:** Google integrates via **code review.** Almost all code is reviewed before commit — developers are in a "change mindset," so suggestions land with little disruption; after sending for review they context-switch and are blocked on reviewers, leaving **time for analyses to run** (even minutes); there's **peer pressure** to address warnings; and analysis **saves reviewer time and helps review scale.** **Code review is the sweet spot for analysis results.**
- **③ Empower Users to Contribute:** Google has many domain experts — static analysis is a chance to **scale their expertise** (writing new analyzers or checks). Beyond experts, a developer who found a bug contributes a check to prevent its recurrence. The focus is an **"easy-to-plug-into ecosystem"** over a few existing tools, with **simple APIs** (like **Refaster**: write an analyzer by giving before/after code snippets) usable by any engineer, not just analysis/language experts.

### Tricorder: Google's Static Analysis Platform
The core claim: **Tricorder** is the core platform, born from several failed attempts to integrate static analysis into the workflow — the key difference being a **relentless focus on delivering only valuable results.** It's **integrated with Critique ([[ch19]])** (warnings shown as gray boxes), using a **microservices architecture** (reading change versions via a FUSE filesystem, accessing cached build inputs/outputs), analyzing **50,000+ changes a day**, often several analyses a second.

The **four criteria** for new Tricorder checks: **① be understandable (any engineer can read the output), ② be actionable and easy to fix (with guidance), ③ < 10% effective false positives (right at least 90% of the time), ④ potential for significant impact on code quality.** Tricorder covers **30+ languages** with **100+ analyzers (most contributed from outside the team)**, 7 of which are plug-in systems with hundreds of checks; the **overall effective-false-positive rate is just under 5%.**

- **Integrated tools:** **Error Prone (Java), clang-tidy (C++)** extend the compiler to find AST antipatterns (e.g., `f >>> 32` on an `int` is a no-op — enabling the check fixed 31 occurrences); **the Deleted Artifact Analyzer** (warns if a referenced file is deleted), **IfThisThenThat** (two files must change in tandem), **Chrome's Finch analyzer** (A/B config, RPCing other services), a **binary-size checker.** Almost all are **intraprocedural.**
- **Integrated feedback channels:** a "**Not useful**" button (file a bug directly against the analyzer writer); reviewers click "**Please fix**." The team tracks high "not useful" rates and disables analyzers that don't improve. Building/tuning this loop took work but paid off greatly. **Sometimes fixing the *message* fixes the problem** (the printf-`%s` example: after rewording the diagnostic, the false-report bugs stopped) — a good message explains "what's wrong, why, and how to fix it" right where it's most relevant.
- **Suggested fixes:** apply from Critique or via a CLI across a whole change; **style issues in particular should be auto-fixed** (formatters). Reviewers click "Please fix" thousands of times a day; authors apply auto-fixes ~**3,000 times a day**; "Not useful" gets ~**250 clicks a day.**
- **Per-project (not per-user) customization:** after building trust, optional analyzers were added (e.g., **Proto Best Practices** — only needed by projects that store serialized data). **The key insight: customize at the project level, not the user level** — ensuring the whole team has a **consistent view** and avoiding "one developer fixing while another introduces." The hard lesson: after removing Critique's user customization, the complaints **revealed real linter bugs** (a C++ linter running on Obj-C, a high-false-positive HTML linter usually suppressed) — **user customization only hides bugs and suppresses feedback.**
- **Presubmits:** since developers can ignore code-review warnings, Google also has **commit-blocking** presubmits (checking "DO NOT SUBMIT," that test files are included, that tests pass, that there are no specific Tricorder issues, and that code is formatted). Teams can write custom presubmits (stricter than company-wide), but they make LSCs harder, so changes with `CLEANUP=` skip some (see [[ch22]], [[ch23]]).
- **Compiler integration:** earlier is better — **breaking the build is an unignorable warning**, but isn't always feasible. Highly mechanical analyses with no effective false positives (Error Prone "ERROR" checks) are all enabled in Google's Java compiler. The criteria: **actionable and easy to fix, no effective false positives, correctness only (not style/best practices).** Enabling a new check first requires **cleaning up all existing instances** (running compilers over the whole codebase via MapReduce, with fixes that automate cleanup). And **never issue compiler warnings** — because **developers ignore warnings**, so either make it an error (break the build) or don't show it (Go even treats unused variables/imports as errors).
- **Analysis while editing/browsing:** IDE integration needs to be **very fast (ideally < 100ms)** and consistent across IDEs, which rise and fall — messier than the review process. Code review has benefits (the full change context; ignoring results requires convincing reviewers). But for some analyses (security), developers do want to **view results across the whole codebase while browsing** (also when planning a cleanup).

Takeaway (conclusion): static analysis improves a codebase, finds bugs early, and **lets more expensive processes (human review, testing) focus on what isn't mechanically verifiable.** By improving the scalability and usability of the infrastructure, Google made static analysis an effective part of development.

## Key terms

- **Static analysis** — analyzing source code (not a running program) to find bugs/antipatterns/issues; catches bugs early, and also codifies best practices, keeps code current with APIs, prevents tech debt, prevents deprecation backsliding.
- **Scalability** — tools are shardable/incremental, analyzing only the files/lines affected by a pending change; scale the number and variety of analyses, show results directly to relevant engineers.
- **Usability** — weigh the cost-benefit; focus on newly introduced warnings, fix automatically when possible, integrate smoothly into the workflow.
- **Effective false positive** — a false positive defined by user action: any issue where the developer took no positive action; perception and message clarity matter as much as correctness.
- **The three key lessons** — focus on developer happiness (trust, low false positives, feedback loop), make it part of the core workflow (code review is the sweet spot), empower users to contribute (simple APIs, domain experts).
- **Tricorder** — Google's static-analysis platform, integrated with Critique, microservices, 50,000+ changes a day; four criteria for new checks (understandable / actionable & easy to fix / < 10% effective false positives / significant impact potential).
- **Error Prone / clang-tidy** — extend the compiler to find Java/C++ AST antipatterns; Error Prone "ERROR" checks are enabled in the compiler to prevent recurrence.
- **Refaster** — write an analyzer from before/after code snippets; lets non-analysis/language experts contribute.
- **Suggested fixes** — apply from Critique or a CLI in one click; style issues should be auto-fixed (formatters).
- **Per-project customization** — customize at the project, not user, level, ensuring a consistent team view; user customization only hides bugs and suppresses feedback.
- **Presubmit** — a commit-blocking check (DO NOT SUBMIT, tests, formatting, specific Tricorder categories); `CLEANUP=` can skip some to ease LSCs.
- **Never issue compiler warnings** — developers ignore warnings, so make it an error (break the build) or don't show it; Go treats unused variables/imports as errors.

## Questions to push your thinking

1. What's your static-analysis tool's **false-positive rate**? Do you measure by "technical correctness" or the **effective false positive (did the developer take positive action)**? Are there checks that are technically correct but treated as false positives due to confusing messages? Do you have a **feedback loop** for users to report "not useful" and actually retune?
2. Where does your static analysis run? Along **IDE → code review → presubmit → compiler**, where are you and where should you push? Which checks are mechanical enough to **break the build**, and which belong in code review's "sweet spot"? Per [[ch19]], [[ch23]].
3. Does your team let users **turn off** annoying checks individually? Recall the lesson that "removing user customization revealed real bugs" — could you switch to **project-level** customization and treat "annoying" as a bug-finding clue rather than suppressing feedback?
4. Have you **scaled** domain-expert knowledge? Which bugs you "fixed once but could recur elsewhere" deserve a check to prevent backsliding? Is there an API simple enough (like Refaster) for non-experts to contribute analyses?
5. Does your tooling **auto-fix**? "Anything that can be fixed automatically should be" — which problems your human reviewers point out one by one (formatting, style, deprecated usage) should be replaced by a formatter/auto-fix, freeing human time for the non-mechanical? Per [[ch09]].
