# Chapter 18 — Build Systems and Build Philosophy

> *"Limiting engineers' power and flexibility can improve their productivity."*
> Counterintuitively, taking power away from engineers and giving it to the system makes everyone faster.

## Summary

Ask Google engineers what they love about the company (besides the free food), and you might hear something surprising: **they love the build system.** Google built **Blaze** from scratch (open-sourced in 2015 as **Bazel**), aiming for builds that are **fast (one command, a result in seconds) and correct (any developer, any machine, same inputs → same result)** — older systems often cut corners between speed and correctness, producing inconsistent builds. And **the vast majority of Google's builds are triggered automatically** (CI, code review, pre-merge, LSCs); nearly every dev tool hangs off the build system.

What happens without a build system? `javac *.java` suffices for one directory; but once you have multiple languages, compilation units, and external dependencies, building is no longer one step — you must manage dependencies and build in order. Shell scripts last a while but become **verbose, slow, unreproducible, and a bootstrapping nightmare** — a classic problem of scale. One theme recurs: **managing your own code is straightforward; managing its dependencies is the hard part** — and **managing dependencies is a build system's most fundamental job.**

The chapter's core contrast is two build philosophies:
- **Task-based (Ant/Maven/Gradle)**: the unit is a **task (an arbitrary script)**. Very flexible, but it **gives engineers too much power and the system too little** — the system doesn't know what scripts do, so it can't parallelize, can't do reliable incremental builds, and the scripts are hard to maintain/debug (race conditions, nondeterminism).
- **Artifact-based (Blaze/Bazel/Pants/Buck)**: BUILD files are a **declarative manifest** — engineers say *what* to build, the system decides *how*. Like functional programming, this lets the system **guarantee correctness and heavily parallelize and reuse** (a one-file change rebuilds the minimal set; a second build returns "up to date" in under a second).

From there comes **distributed builds** (remote caching, ObjFS, + remote execution, Forge, since 2008) — spreading work across a whole compute cluster. The trade-offs are **time, scale, flexibility**: DIY scripts suit only the smallest projects; task-based buys scale but requires build files; artifact-based unlocks huge scale at the cost of flexibility. **Google believes almost every new project should use an artifact-based system from the start.** Finally, modules and dependencies: **prefer fine-grained modules (the 1:1:1 rule), minimize visibility, enforce strict transitive deps, manage external dependency versions manually (not automatically), and follow the One-Version Rule.**

## Why read this

- **Your build is slow and maddening because "it breaks on another machine."** The chapter diagnoses the root cause: you're probably **task-based** (shell scripts/Ant/Gradle), and the system doesn't know what scripts do, so it can't safely parallelize or do reliable incremental builds. Grasp the **artifact-based** approach and you'll see why "**limiting flexibility improves productivity.**"
- **You think "a compiler is all I need" or "a build script will do."** The chapter's "classic scale problem" story walks you through it: multiple languages, external dependencies, newcomer bootstrapping, automated builds — each breaks DIY scripts. It helps you judge "when to invest in a real build system."
- **You've been burned by diamond dependencies, version conflicts, and `latest`.** Three cures: **pin versions manually (no `latest`), the One-Version Rule, strict transitive dependencies.** Understand "a dependency is a target's public contract" and you'll redesign your dependency management.
- **You're deciding coarse vs fine modules.** The chapter argues **fine-grained modules scale better** (1:1:1 — one package/target/BUILD per directory) — faster distributed builds, fewer rebuilds, and running only the affected subset of tests. Tools auto-manage BUILD files to ease the burden.

## Key details

### Purpose of a Build System
The core claim: every build system's fundamental purpose is to **turn the source code engineers write into machine-readable executables.** A good one optimizes two properties: **Fast (one command, a result in seconds)** and **Correct (any developer, any machine, same inputs → same result).** Older systems cut corners between the two, producing inconsistent builds; **Bazel's main objective is to avoid choosing between speed and correctness.**

Takeaway: build systems aren't just for humans — **the vast majority of Google's builds are triggered automatically** (auto build/test/push to production; auto-test at code review; re-test before merge; low-level library authors test across the whole codebase; LSCs touch tens of thousands of files). Nearly every dev tool hangs off it — all built on Google's investment in the build system.

### What Happens Without a Build System?
The core claim: **"All I need is a compiler!"** — `javac *.java` does suffice for one directory; but once code expands (multiple languages, compilation units, external dependencies like third-party JARs), building is no longer one step — you must reason about dependencies, build in order, and rebuild when dependencies change, or depend on stale binaries. **"Shell scripts to the rescue?"** — they last a while, then hit: ① **tedium** (almost as much time on build scripts as on code), ② **slowness** (full rebuilds to avoid staleness), and the pain of releases, disaster recovery (your disk crashes — can you find the same versions of those downloaded libraries?), newcomer bootstrapping, and automated builds (yesterday's change works on one machine, fails on the build server). This is the **classic problem of scale**: a compiler suffices for one developer writing a few hundred lines for a couple weeks, but coordinating across developers and machines defeats even a perfect script — time to invest in a real build system.

### It's All About Dependencies
The core claim: every problem points to one theme — **managing your own code is straightforward; managing its dependencies is much harder** ([[ch21]] is devoted to this). Dependencies come in kinds: on a **task** ("push the docs before marking a release done"), on an **artifact** ("I need the latest computer-vision library to build"), **internal** (another part of your codebase), **external** (another team or third party). But "I need that before I can have this" recurs throughout — **managing dependencies is a build system's most fundamental job.**

### Task-Based Build Systems
The core claim: the unit is a **task (a script running arbitrary logic)**, with tasks naming other tasks as "must run first" dependencies, forming an acyclic graph. Ant/Maven/Gradle/Grunt/Rake are task-based (Ant's XML buildfile, where a target = our "task"). They improve on shell scripts (link multiple buildfiles, add dependencies arbitrarily).

Takeaway (the dark side): letting engineers define any script as a task is **very flexible, but gives engineers too much power and the system too little** — the system doesn't know what scripts do:
- **Hard to parallelize:** A depends on B and C, but can B and C run together? The system can't know (they might touch the same file), so it conservatively runs single-threaded — wasting cores and ruling out distributed builds.
- **Hard to do incremental builds:** a task can do anything, so the system can't tell if it's already done (it might download a changing file or write a timestamp); to stay correct it reruns everything. Engineers end up running `clean` every time, defeating the point.
- **Hard to maintain/debug scripts:** build scripts are code too and hide bugs — implicit inter-task dependencies break on change, assumptions about the machine, nondeterministic components, race conditions among multiple dependencies. **As long as engineers can write arbitrary build-time code, the system can't always be fast and correct.**

### Artifact-Based Build Systems
The core claim: the fix is to **take power back from engineers and give it to the system** — instead of letting engineers define tasks, the system defines a few tasks engineers configure in limited ways. A build system's primary task is to **build code**: engineers say **what to build**, the system handles **how**. This is exactly Blaze and its descendants (Bazel/Pants/Buck) — a BUILD file isn't an imperative script but a **declarative manifest** (the artifacts to build, their dependencies, limited options). With full control over what tools run when, the system can make far stronger guarantees, being more efficient while staying correct.

Takeaway:
- **A functional perspective:** task-based is like imperative languages (a list of statements run in order); artifact-based is like functional languages (describe a computation, leave when/how to the compiler). The whole build system is effectively a **mathematical function**: inputs (source + tools) → outputs (binaries) — a natural fit for functional tenets.
- **Concretely, Bazel:** BUILD files define **targets** (`java_binary`/`java_library`), each corresponding to a buildable artifact, with `name`/`srcs`/`deps`; dependencies can be in the same package, another package in the same source hierarchy, or a third-party artifact outside it; each source hierarchy is a **workspace** (a WORKSPACE file at the root). Because Bazel knows each target only runs the Java compiler, it can **safely run them in parallel** (an order-of-magnitude speedup on multicore); and because output depends only on input, it **reuses output when input is unchanged** (a second build returns up-to-date in under a second; a one-file change rebuilds the minimal set). **Reframing the build around artifacts rather than tasks is subtle but powerful.**
- **Bazel's nifty tricks:** **① Tools as dependencies** — targets implicitly depend on the compiler (changing it rebuilds); **toolchains** achieve platform independence. **② Extending** — custom rules add target types; rules generate **actions** (the lowest-level composable unit, declaring inputs/outputs). **③ Isolating the environment** — **sandboxing (LXC, the tech behind Docker)** lets an action see only its declared inputs/outputs, making conflicts impossible and restricting the network. **④ Deterministic external deps** — a workspace-wide manifest records a **cryptographic hash** for each external dependency; a hash mismatch fails the build unless the manifest is updated (which must be approved and checked in), ensuring "checking out last week's commit gets last week's dependencies."

### Distributed Builds
The core claim: Google's codebase exceeds 2 billion lines with deep dependency chains (simple binaries often depend on tens of thousands of targets) — at this scale, a single machine **can't** build in reasonable time (physics), so the only answer is a **distributed build**: spread the work across an arbitrary, scalable number of machines.
- **Remote caching:** the simplest distributed build. All builders (workstations, CI) share a remote cache; to build an artifact, first check the cache — download if present, else build and upload. **The prerequisite is fully reproducible builds** (same inputs → identical output on any machine) — so each artifact is keyed on "target + a hash of its inputs." But it only helps if "downloading beats building" (watch network latency; experiment).
- **Remote execution:** the true distributed build. Each machine's build tool sends requests to a central **build master**, which breaks them into actions scheduled over a scalable worker pool; workers write results to and read dependencies from the shared cache. **This requires all the artifact-based pieces**: self-describing environments, self-contained processes, fully deterministic outputs — guarantees nearly impossible for task-based systems, making a reliable remote-execution system nigh-impossible to build on one.
- **Google's implementation (since 2008):** remote cache **ObjFS** (a backend storing build outputs in Bigtables across the fleet, a frontend `objfsd` FUSE daemon serving file contents on demand — twice as fast to build); remote execution **Forge** (Blaze's Distributor sends actions to the datacenter Scheduler, which caches action results, and an Executor pool runs them and stores results in ObjFS). Millions of builds, millions of test cases, petabytes of output per day.

### Time, Scale, Trade-Offs
The core claim: build systems make code easier to work with at scale and over time, and choosing one involves trade-offs. **DIY scripts** suit only the smallest, short-lived projects (or languages like Go with a built-in build system); **task-based** buys scale (automate complex builds, reproduce across machines) at the cost of structuring your build and writing build files; **artifact-based** unlocks new scale (distribute across machines, thousands of engineers confident builds are consistent) at the cost of **flexibility** (no generic tasks in a real programming language). Changing a build system is expensive and gets more so as the project grows — so **Google believes almost every new project benefits from an artifact-based system like Bazel from the start** (everything from tiny experiments to Google Search is built with Blaze).

### Dealing with Modules and Dependencies
The core claim:
- **Fine-grained modules and the 1:1:1 rule:** the first question is "how much functionality per module" — a single huge module (never touch BUILD but no parallelism/caching) vs one module per file (max flexibility but tedious dependency lists). **Google favors far smaller modules than task-based systems** (a typical production binary depends on tens of thousands of targets); for strong-packaging languages like Java, one package/target/BUILD per directory (Pants's **1:1:1 rule**). Fine grains pay off at scale: faster distributed builds, fewer rebuilds, and **running only the affected subset of tests**; tools auto-manage BUILD files to ease the burden.
- **Minimize visibility:** each target declares who may depend on it (public/private/an explicit list); visibility is the opposite of a dependency. As in most languages, **minimize it** — make targets public only for widely-used libraries.
- **Internal dependencies:** built from source (not downloaded as prebuilt artifacts), with **no notion of "version"** (a target and its internal deps are always built at the same commit). Handle **transitive dependencies** carefully: A→B→C — should A use C's classes? The underlying tools are fine, but if B is refactored to no longer depend on C, A breaks — **a target's dependencies become part of its public contract and can never be safely changed**, so dependencies accumulate and builds slow. Google's fix is **strict transitive dependency mode**: detect "referencing a symbol without depending on it directly" and fail with a command to auto-insert the dependency (a multiyear refactor of millions of targets, but worth it — faster builds, safe dependency removal). Bazel enables this for Java by default.
- **External dependencies:** versioned, independent of the project's source. **Manual vs automatic version management** — automatic (a version range, always download the latest, e.g. Gradle's `1.+`) is convenient for small projects but a disaster at scale (no control over when versions update; third-party breaking changes break yesterday's build); **manual** (a source-control change) is discoverable, rollback-able, and lets you check out an old version with old dependencies — **Bazel requires all versions specified manually.**
  - **The One-Version Rule:** enforce a single version for all third-party dependencies, avoiding the **diamond dependency** (A depends on B and external lib v1; B later adds v2 → A implicitly depends on two versions and breaks).
  - **Transitive external dependencies:** Maven/Gradle recursively download each transitive dependency by default (convenient, but necessarily violates One-Version and invites diamond dependencies) — so **Bazel doesn't auto-download transitive deps**, using instead a global file listing every external dependency and its explicit version (with a tool to generate it).
  - **Security and reliability:** third-party sources carry availability risk (down → build halts) and security risk (compromised → arbitrary code injected); mitigate by **mirroring to servers you control, requiring hashes, or vendoring (checking dependencies into `third_party`, converting external to internal deps)** — Google vendors (but only because its source system is built for an extremely large monorepo).

Takeaway (conclusion): a build system is one of the most important parts of an engineering org (each developer interacts dozens or hundreds of times a day; often the rate-limiting step for productivity). Google's surprising lesson: **limiting engineers' power and flexibility improves productivity** — not free rein in defining builds, but a highly structured framework that leaves the interesting decisions to automated tools. "**Being able to trust the build is powerful** — incremental builds just work, and there's almost never a need to clear caches or run a clean step."

## Key terms

- **Blaze / Bazel** — Google's internal build system Blaze and its 2015 open-source version Bazel; artifact-based, aiming for both speed and correctness.
- **Fast & correct** — a good build system's two goals: one command, a result in seconds; any developer/machine, same inputs → same output.
- **Task-based build system** — the unit is a task (an arbitrary script), like Ant/Maven/Gradle; flexible but gives the system too little info, making parallelism/incrementality/maintenance hard.
- **Artifact-based build system** — BUILD files are a declarative manifest; engineers say what, the system decides how (Blaze/Bazel/Pants/Buck); enables parallelism, reuse, and guaranteed correctness.
- **BUILD / target / workspace** — BUILD files define targets (java_binary/java_library with name/srcs/deps); a WORKSPACE file marks the source-hierarchy root.
- **Tools as dependencies / toolchains** — treat tools like the compiler as dependencies (changing them rebuilds); toolchains give platform independence.
- **Sandboxing** — LXC (Docker's tech) lets each action see only its declared inputs/outputs, making conflicts impossible and restricting the network.
- **Remote caching / remote execution** — a shared cache avoids rebuilds (ObjFS); actions spread across a worker pool (Forge); prerequisite is fully reproducible builds.
- **1:1:1 rule** — one package/target/BUILD per directory (Pants's term); prefer fine-grained modules for distributed builds, incrementality, and precise testing.
- **Strict transitive dependencies** — fail when a symbol is referenced without a direct dependency; avoids "dependencies as a public contract" and speeds builds.
- **One-Version Rule / diamond dependency** — only one version of each third-party dependency, avoiding A depending on both v1 and v2 (see [[ch21]]).
- **Vendoring / third_party** — checking external dependencies into the source tree (`third_party`), converting external to internal deps; used by Google (relying on a huge monorepo).

## Questions to push your thinking

1. Is your build **task-based or artifact-based**? Can it safely **parallelize** and do reliable **incremental builds**, or are you forced to `clean` and maddened by "breaks on another machine"? Understanding "the system doesn't know what scripts do," how would you restructure your build?
2. Recall your "no real build system" pain (multiple languages, external deps, newcomer bootstrapping, builds failing on the server): where are you — DIY scripts / task-based / artifact-based? "Changing the build system gets more expensive as the project grows" — should you invest early?
3. Are your external dependencies **pinned manually or `latest`/range**? Burned by diamond dependencies or third-party breaking changes? Apply **One-Version + manual + hash**: which dependencies are a time bomb? Per [[ch16]], [[ch21]].
4. Are your modules **coarse or fine**? Tally what "run only the affected test subset" would save your CI. Could you move toward **1:1:1** and auto-manage dependency lists with tooling? Per [[ch11]], [[ch23]].
5. The chapter says "**limiting flexibility improves productivity.**" Where in your tooling/process did you "give people too much power while the system can't guarantee correctness or efficiency"? Which decision, taken back from "free rein" and given to "automated tools," would make everyone's life easier?
