# 第 21 章｜Dependency Management（依賴管理）

> *"Prefer source control problems to dependency management problems."*
> 「能用版本控制問題就別用依賴管理問題。」

## 摘要

**依賴管理**——管理「我們不控制的函式庫、套件、依賴所構成的網路」——是軟體工程中**最不被理解、也最具挑戰**的問題之一。它與**版本控制**最相關，兩者的分野是一個問題：「**這個子專案的開發/更新/管理，是不是由我們的組織控制？**」由此導出本章最強的單一建議：**在其他條件相同下，寧可面對版本控制問題，也不要面對依賴管理問題**——能把「組織」定義得更廣（整間公司而非一個團隊），通常是划算的取捨。

核心難題是要**以「網路」而非「單一依賴」來思考**。經典症狀是 **diamond dependency（鑽石依賴）**：libbase 被 liba 與 libb 用，兩者又被 libuser（你的程式）用；若 libbase 有不相容變更、liba 升到新版而 libb 還在舊版，**就沒有通用辦法把一切兜在一起**。語言對此容忍度不同（Java 可 shading 改名、C++ 因 One Definition Rule 幾乎零容忍），但**只要型別會跨依賴傳遞，所有技倆都失效**。

**匯入依賴**在「程式設計」幾乎免費（重用勝過重造），但在「軟體工程」（要考慮時間與維護）就有隱藏成本。依賴提供者該講清**相容性承諾**：C++ 標準庫（近乎無限期向後相容＋ABI＋SD-8）、Go（原始碼相容、無二進位相容）、**Abseil**（不保證 ABI，但保證「破壞性 API 變更必附自動重構工具」）、**Boost**（不保證版本間相容，是實驗試煉場）。Google 的 `third_party` 故事（Alice/Bob）顯示：**「我讓它能動」≠「這是被支援地運作」**——這正是 Hyrum's Law。

理論上的四種模型：**①Nothing Changes（靜態，多數新組織的正確起點，但夠久就會失效）②SemVer（語意化版本，業界事實標準，靠 SAT-solver；解不出就是「dependency hell」）③Bundled Distribution（捆綁發行，如 Linux distro，由 distributor 找出相容集合）④Live at Head（永遠依賴最新、不釘版本、靠 CI 與自動重構工具，是 trunk-based 的延伸）**。本章深入剖析 **SemVer 的侷限**（它是「估計」非「證明」、會 overconstrain 也會 overpromise、誘因不對；**MVS 最小版本選擇**提升保真度）並提出「無限運算資源」的思想實驗——以**跑下游測試的實證**取代維護者的相容性估計。最後談**匯出依賴**（gflags、AppEngine 的血淚）：對外發布 API 比對內昂貴得多。

## 為什麼要讀這一篇

- **你被鑽石依賴、版本衝突、「dependency hell」反覆折磨。** 本章把它從「單一依賴」拉高到「**依賴網路**」的層次，解釋為何衝突需求是核心難題、為何語言技倆（shading）只能救函式救不了型別。看懂後你會更敢於**把問題收進組織內變成版本控制問題**。
- **你不知道該不該、以及如何引入一個外部依賴。** 本章給你 Google 的判準清單（有測試嗎？通過嗎？誰提供？相容承諾？多久？多常破壞性變更？），並用 Alice/Bob 的 `third_party` 故事點破：**匯入依賴不是免費的，它是一份需要長期維護的契約**。
- **你把 SemVer 當成絕對真理。** 本章拆穿它只是「**人對風險的有損估計**」：會 overconstrain（明明能動卻被擋）也會 overpromise（patch 版本照樣因 Hyrum's Law 弄壞別人）。理解這點，你選依賴與設計版本策略會更清醒（並認識 **MVS** 與「以測試取代估計」的方向）。
- **你在考慮把團隊的程式碼開源/對外發布。** 本章用 gflags 與 AppEngine 的真實教訓警告你：**「丟過牆就忘」會損及聲譽、無計畫的對外發布會變成工程效率的稅**；「別發布一個你沒有長期支援計畫（與授權）的東西」。

## 細節重點

### Why Is Dependency Management So Difficult?（為何這麼難）
核心主張：許多半吊子方案聚焦於太窄的問題（「怎麼匯入一個套件？」）——真正的難題是**管理一個依賴網路及其隨時間的變化**。網路的某些節點是你的第一方程式直接需要的，有些只由傳遞依賴拉進來；夠久之後所有節點都會有新版、有些更新很重要。**衝突需求與鑽石依賴**是核心：當網路中兩個節點有衝突需求、而你兩者都依賴時就出事。
- **Diamond dependency（鑽石依賴）：** libbase 被 liba 與 libb 用，兩者又被 libuser 用；libbase 不相容變更 → liba 升新版、libb 還舊版 → libuser 無法兜起來（需至少三層）。
- **語言容忍度：** Java 可 **shading**（改名隔離多版本）；C++ 因 **One Definition Rule** 幾乎零容忍（觸發 UB）。但**只要型別會在依賴間傳遞**（如 libbase v1 的 map 傳給 libbase v2 的 API），所有改名技倆都失效——「最多是部分解」。
- **唯一容易的答案：** 在版本上前進或後退找相容點；不行就只能在本地 patch（特別難，因為發現者通常不熟兩邊）。**依賴管理的政策與技術，根本上都歸結為：「如何在允許非協調群體各自改變的同時，避免衝突需求？」**

### Importing Dependencies（匯入依賴）
核心主張：重用勝過重造（別讓每個新手重寫 JSON parser）。但**一旦考慮時間**就有取捨：避開開發成本不代表匯入就對——還有持續的維護成本（安全漏洞、平台變動、依賴網路演進都可能逼你升級，不管你原本想不想）。
- **相容性承諾（各家不同）：** **C++ 標準庫**＝近乎無限期向後相容（含 ABI；SD-8 列出可做的少數變更）；**Go**＝原始碼相容、無二進位相容；**Abseil**＝不保證 ABI、但保證「**不做破壞性 API 變更而不附自動重構工具**」（~2.5 億行 C++ 依賴它，故不輕易改 API 但必須可能）；**Boost**＝不保證版本間相容（是實驗試煉場，目標與標準庫根本不同）。**這些是軟體工程議題、非程式設計議題**——你可以把無相容承諾的 Boost 深埋進關鍵系統、它今天會正常運作；問題全在「隨時間如何變」。
- **匯入前該問（Google 鼓勵的清單）：** 有測試嗎？通過嗎？誰提供（聲譽值得查）？目標相容性？支援哪種用法？多熱門？我們要依賴多久？多常破壞性變更？以及對內：自己實作多複雜？有何誘因保持更新？誰來升級？升級多難？
- **Google 怎麼做（「我們可以做得更好」）：** 絕大多數依賴是**內部開發**的——所以大部分「依賴管理」其實是版本控制（這是設計使然）。外部專案放進 monorepo 的 `third_party`。**Alice/Bob 故事：** Alice 趕 demo 下載一個 OSS 套件、走簡單檢查清單（能用我們的 build、無既有版本、兩個 OWNERS），兩人對實作一無所知。之後它對其他團隊可見、被 Charlie 加進 Search 基礎設施的深處——**從無害變成關鍵基礎設施，卻沒有任何訊號提示 Charlie**。某天安全漏洞逼出關鍵升級：Bob 轉管理、Alice 換隊、沒人改 OWNERS、數千專案間接依賴它、沒人懂實作細節——**一次昂貴又困難的升級**（還因橫跨許多小版本而更難）。`third_party` 政策對這常見場景失靈。

### Dependency Management, In Theory（理論上的四種模型）
核心主張：好方案必須**對時間與規模都有彈性**——不能假設任何節點無限期穩定，也不能假設不會加新依賴。能避免衝突需求就是好方案；若還不假設版本穩定、依賴扇出、組織間協調或大量運算資源，就是好得不得了的方案。四種常見模型：
- **① Nothing Changes（靜態依賴模型）：** 永不改（只允許不會弄壞任何使用者的 bug 修正）。因假設無限期穩定而不理想，但**這是每個組織的起點、也大概是多數新組織的正確模型**——除非你已證明專案壽命長到非變不可。缺點：夠久就會失效，且沒有「還能假裝穩定多久」的明確警示。無版本選擇。
- **② SemVer（語意化版本）：** 業界事實標準——`major.minor.patch`（major＝破壞性、minor＝純新增、patch＝低風險修正）；需求常表為「≥ 某版」。把網路視為節點＋需求邊，找出滿足所有傳遞需求的相容版本集合是 **SAT-solver** 問題（SemVer 約束套在網路上已被證明是 **NP-complete**）；解不出就是俗稱的 **「dependency hell」**。
- **③ Bundled Distribution（捆綁發行）：** 由 **distributor（發行者）** 蒐集一組依賴、找出相容集合、當成單一單元發行（如 Linux distro——不保證各件來自同一時間點）。對外使用者把「依賴 72 個庫」變成「依賴 RedHat 版本 N」。版本選擇由專職 distributor 處理。
- **④ Live at Head（活在主幹頂端）：** Google 部分人推動、理論健全但對參與者代價高——是 **trunk-based development 在上游依賴的延伸**。前提：**unpin 依賴、放棄 SemVer、依賴提供者在提交前對整個生態跑測試**。永遠依賴最新版、且絕不以「下游難以適應」的方式改東西；非預期改 API/行為的變更會被下游 CI 抓到而不該提交；必須破壞時（如安全），要嘛先更新下游、要嘛**提供自動重構工具就地升級**（對閉源下游尤其關鍵）。版本選擇＝「所有東西的最新穩定版」。

### The Limitations of SemVer（SemVer 的侷限）
核心主張：SemVer 的點分三段數字到底是**承諾還是估計**？其實是估計——你無法只看原始碼 API 就證明相容性，必須知道「跟什麼東西相容」。把估計與自我聲明當成絕對，地基本就不穩：
- **可能 overconstrain（過度約束）：** libbase 只有 Foo 與 Bar 兩函式，liba/libb 只用 Foo；libbase 對 Bar 做破壞性變更必須 bump major，於是 SAT-solver 拒絕讓 liba/libb 用新版——**即使它們其實完美相容**（只有沒用到的 Bar 改了）。→ 報「無法更新」即使忽略 SemVer 一切都能動，這正是 dependency hell 的一大來源。
- **可能 overpromise（過度承諾）：** 假設變更只分三桶（破壞/純新增/不影響 API）。但加 1 毫秒延遲、改 logging 格式、改 import 順序、改「無序」串流的回傳順序算哪一桶？**Hyrum's Law**：「使用者夠多，你系統的每個可觀察行為都會被某人依賴」——所謂「安全」的 patch 變更，在規模下必然弄壞某些依賴假設。**「breaking change」一詞本身就誤導**：有理論上破壞但實務安全（移除沒用的 API），也有理論上安全卻弄壞客戶（Hyrum's Law 例）。一個變更是否 breaking，**只能在「它如何被使用」的脈絡下評估**——而這資訊不在 SemVer 公式裡。
- **誘因（Motivations）：** 維護者只承擔版本 bump 成本的一小部分（多數使用者是間接的、不訂閱通知），故系統性誘因不利於「不做破壞性變更」。**Go/Clojure 的做法很正義**：major bump 等於一個全新套件（你既然要破壞向後相容，為何假裝是同一組 API？）。加上人類易錯——語意變更和語法變更一樣需要 bump，但「這是行為變更還是 bug 修正」在計算上不可判定。
- **Minimum Version Selection（MVS，最小版本選擇）：** Russ Cox 提出——liba 要求 libbase ≥1.7 時，就用 **1.7**（即使有 1.8）。這「產生高保真的 build，盡量貼近作者開發時所用的版本」——因為 liba 說要 ≥1.7，幾乎確定代表開發者裝的是 1.7（至少有基本互通測試的軼事證據）。**走最小的一步**（如同寧可一次提交一小時的工作而非一次倒一年）——比 SemVer 多一點實務保真，但沒修好其根本的理論與誘因問題。
- **SemVer 到底行不行？** 在小規模行得通，前提是：**提供者準確負責、依賴夠細粒度、所有 API 用法都在預期內**。但 Google 經驗顯示**在規模下這三者難以同時成立並持續**——保真度的複合損失會主宰，同時出現 false positive（理論不相容實則可動）與 false negative（相容卻被擋＝dependency hell）。

### Dependency Management with Infinite Resources（無限運算資源下的依賴管理）
核心主張：思想實驗——若有無限運算資源會怎樣？業界用 SemVer 有三個理由（只需本地資訊、不假設有測試/CI、是既有實踐），但這些都非必要：依賴網路只在兩種環境形成（單一組織內、OSS 生態），兩者都有下游用法資訊可用。SemVer 的本意是「我估計這變更好不好採用」——**有更好的方式傳達它：用實證**（對提案變更跑下游依賴的測試）。若 OSS 生態走向「每個變更都附帶『是否安全』的證據」，相容性的真相就來自**跑受影響的下游測試**（可依使用度/維護品質/歷史信號優先排序，把目標從「絕不破壞任何人」改成「風險緩解」的統計論證）。需要的改變不少：所有依賴要有單元測試、依賴網路（含反向邊）要可查、要有 CI 運算資源、依賴不可 pin、CI 計算要納入歷史與聲譽。簡化後——**這實際上就是 Live at Head 模型**。

### Exporting Dependencies（匯出依賴）
核心主張：發布軟體當依賴有兩種損失：**①聲譽**（做不好或不維護就拖累組織——「community over code」）；**②工程效率的稅**（不同步、夠久所有 fork 都變貴）。
- **gflags 案例（聲譽損失）：** 2006 開源 C++ 旗標庫本是善舉，卻因「當時無法做大規模重構、無法把程式移到新位置、OSS 接受外部貢獻有法律問題」而注定變成「丟過牆」或斷裂的 fork；隨原作者離開、優先序轉移，內外版本緩慢分歧、最終被外部 fork。**「別發布一個你沒有長期支援計畫（與授權）的東西。」** 後來 C++ 庫團隊調整了「可觀察但沒文件」的內部 flag 實作，依賴內外 fork 等價性的少數專案突然爆掉——一個值數千 CPU 的最佳化被顯著延遲，全因 Hyrum's Law（連跨組織維護的 forked API 都中招）。
- **AppEngine 案例（更嚴重的技術依賴風險）：** 2014 把 Python runtime 與 C++ 編譯器/標準庫一起大升級，等於把「用新 C++ 編譯器的程式」綁到「用新 Python 版」。但 AppEngine 發現許多**付費客戶**無法/不願更新（不想換 Python 版、或負擔不起 32→64-bit 的資源變化），因有大客戶付大錢，AppEngine 得以延後強制切換——這逼得 AppEngine 依賴傳遞閉包裡的**每一段 C++ 程式都得相容舊編譯器與標準庫**，這狀態持續了近三年。

工程意涵（結論）：**對外發布 API 會讓你暴露於外部群體的競爭優先序與不可預見的約束**（尤其是付你錢的）。「使用者夠多，你系統的任何『可觀察』都會被某人依賴」——在 Google 內部，monorepo 與程式碼索引讓變更仍可能；一旦失去可見性、或受外部競爭優先序左右，純工程取捨就難得多。**外部使用者的維護成本遠高於內部使用者**；評估是否發布時，要意識到長期風險。

## 關鍵字補充

- **Dependency Management（依賴管理）** — 管理我們不控制的函式庫/套件/依賴網路；軟體工程最不被理解、最具挑戰的問題之一。
- **Source Control vs Dependency Management** — 分野是「這子專案是否由我們組織控制」；寧可面對版本控制問題（更易、更便宜）。
- **Diamond Dependency（鑽石依賴）** — libbase 被 liba/libb 用、兩者又被 libuser 用；libbase 不相容變更使兩條路徑要求不同版本而無法兜起。
- **Shading（遮蔽）/ One Definition Rule** — Java 可改名隔離多版本；C++ 因 ODR 幾乎零容忍；但型別跨依賴傳遞時所有技倆失效。
- **Compatibility Promise（相容性承諾）** — C++ 標準庫（近無限期＋ABI＋SD-8）、Go（原始碼相容無二進位）、Abseil（破壞性 API 變更必附自動重構工具）、Boost（不保證、實驗試煉場）。
- **third_party / 「能動 ≠ 被支援地運作」** — Google 放外部依賴的目錄；Alice/Bob 故事顯示匯入依賴是需長期維護的契約（Hyrum's Law）。
- **Nothing Changes（靜態模型）** — 永不改；多數新組織的正確起點，但夠久就失效。
- **SemVer（語意化版本）** — major/minor/patch＋「≥」需求＋SAT-solver；解不出＝dependency hell；約束套在網路上是 NP-complete。
- **Bundled Distribution（捆綁發行）** — distributor 找出相容集合當單一單元發行（Linux distro）；把「依賴 N 個庫」變成「依賴版本 X」。
- **Live at Head（活在主幹頂端）** — unpin、放棄 SemVer、提供者提交前對整個生態跑測試、破壞時附自動重構工具；trunk-based 的依賴延伸。
- **MVS（Minimum Version Selection，最小版本選擇）** — 要求 ≥1.7 就用 1.7（非 1.8）；高保真、貼近作者測過的版本；走最小的一步。
- **Exporting Dependencies（匯出依賴）** — 對外發布的兩種損失：聲譽、工程效率的稅；外部使用者維護成本遠高於內部（gflags、AppEngine 教訓）。

## 延伸提問

1. 你最近一次依賴衝突/「dependency hell」，是不是 **diamond dependency**？你能不能把它從「依賴管理問題」**收進組織內變成版本控制問題**（把「組織」定義得更廣、或把該依賴 vendor 進來）？對照 [[ch16]]、[[ch18]] 的 One-Version Rule。
2. 用 Google 的清單檢視你最近引入的一個外部依賴：有測試嗎？誰提供、相容承諾是什麼？你預期依賴它多久、由誰升級、多難升？它有沒有像 Alice/Bob 的 `third_party` 一樣**悄悄從無害變成關鍵基礎設施**卻沒人維護？
3. 你把 SemVer 當「承諾」還是「估計」？回想一次「版本號說相容卻爆了」（overpromise／Hyrum's Law）或「明明能動卻被 SAT-solver 擋住」（overconstrain）——你的依賴策略有沒有為這兩種失真做準備？**MVS** 對你適用嗎？
4. 四種模型（Nothing Changes／SemVer／Bundled／Live at Head）裡，你的專案目前在哪一個、該在哪一個？你的專案預期壽命多長——久到「Nothing Changes 會失效」了嗎？你有沒有測試與 CI 來支撐 Live at Head 的方向？對照 [[ch23]]。
5. 你正考慮把團隊的程式碼**對外發布**嗎？回想 gflags「丟過牆就忘」與 AppEngine「付費客戶不肯升級拖三年」的教訓——你有**長期支援的計畫與授權**嗎？你準備好承擔「外部使用者透過 Hyrum's Law 依賴你沒承諾的行為」的代價嗎？
