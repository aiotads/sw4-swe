# 第 24 章｜Continuous Delivery（持續交付）

> *"Counterintuitively, faster is safer."*
> 「違反直覺地說：更快，反而更安全。」

## 摘要

**Continuous Delivery（CD，持續交付）** 的核心信念，與 Agile 一脈相承：**長期而言，較小批次的變更帶來較高品質——也就是「faster is safer（更快反而更安全）」**。這乍看極具爭議，尤其在 CI 與測試等前置條件還沒到位時；所以 Google 把 CD 拆成幾個能**獨立交付價值**的面向逐步推進：**敏捷（Agility，頻繁小批次發布）、自動化（Automation，消除重複發布開銷）、隔離（Isolation，模組化架構讓除錯更易）、可靠（Reliability，量測 crash/延遲等健康指標並持續改善）、資料驅動決策（A/B 測試）、分階段推出（Phased rollout）**。一個組織的競爭優勢在於上市速度（velocity），而 velocity 的瓶頸是部署時間——**程式碼的價值不在提交時實現，而在功能送到使用者手上時才實現**；縮短「code complete → 使用者回饋」的時間，就是把在製品（work-in-progress）的成本壓到最低。

本章用 Google 真實產品串起 CD 的關鍵槓桿。**Velocity 是團隊運動**：團隊長大時的反模式是「拉分支避免互踩」，但日後整合與找元兇都很痛——Google 寧可大家**在共享 codebase 的 head 上開發**，靠 CI、自動回退、culprit finding 快速抓問題（YouTube 那種 50 小時人工回歸、滿是 cherry-pick 的痛苦發布是反例）。對「發布又貴又險」的直覺反應是「放慢、拉長穩定期」，但那只換來短期穩定、長期拖垮 velocity；**正解是降低成本、提高紀律、讓風險更增量化**，並抗拒表面的營運修補、投資長期架構（**遷移到微服務、甚至打掉重寫以建立模組化**）。**Flag-guarding（旗標守護）** 讓每個變更被功能旗標包住，**把單一功能的命運與整體發布解耦**（新舊 codepath 並存、出事用動態 config 改旗標即可、發布前一刻才開旗標以配合新聞稿）。**Release train（發布列車）**（Search 從一週一次、靠運氣，做到每兩天穩定發布）教兩件事：**沒有完美的 binary**（用有明確門檻的 KPI 讓「不完美但夠好」的功能能上）與**錯過班次就等下一班**（deadline 是確定的、人生不是；規律發布把開發者的恐慌與 release 工程師的爆肝都降下來）。**只出貨被用到的東西**（動態部署＋A/B，避免 bloat 讓使用者裝置付空間/流量代價）。**Shift left（左移）＋把現實當基準**：客戶端多樣性「不是問題、是事實」——做代表性測試、分階段推出、自動化 A/B 發布（甚至 A/B「部署」本身，因為光是推一個更新就會造成統計顯著的指標變化）。文化上「**Always Be Deploying（持續部署）**」＋「一項發布職責是保護產品不受開發者傷害」。結論：faster 不只更安全、也更便宜；而且**光是把能持續部署的結構建起來，就產生了大部分的價值——即使你不真的每天推給使用者**。

## 為什麼要讀這一篇

- **你的發布又貴又險，本能是「放慢、拉長穩定期」。** 本章直球打臉這個直覺：放慢只給短期穩定、長期拖垮 velocity 與士氣。正解反而是**降低每次發布的成本、提高紀律、把風險切成增量**——並抗拒表面的營運修補，投資長期架構（微服務化/重寫）。看完你會重新評估「該慢還是該快」。
- **你的團隊長大了，有人想拉分支「避免互踩」。** 本章點名這是反模式：分支當下舒服，日後整合與 culprit finding 會痛。應該**在共享 head 上開發＋CI＋自動回退**。對照 [[ch23]] 的 TAP/Build Cop——這是 velocity 作為「團隊運動」的技術底座。
- **你不知道怎麼把「發布功能」與「發布 binary」解耦。** **Flag-guarding** 是答案：新舊 codepath 並存、用動態 config 控制、出事改旗標不必重發 binary、發布前一刻才開旗標配合新聞稿。理解這個槓桿，你才能獨立管理每個功能的命運與風險。
- **你在為「要不要為某個 bug／某個功能延遲發布」而焦慮。** 本章給你兩條鐵律：**沒有完美的 binary**（用有門檻的 KPI 與 error budget 思維做取捨，連「只影響一個菲律賓小島方言」的 bug 都值得想清楚）＋**錯過發布列車就等下一班**（規律發布讓「等下一班」只是幾小時、而非幾天）。

## 細節重點

### Idioms of Continuous Delivery at Google（Google 的 CD 慣用法）
核心主張：CD 與 Agile 的核心信條——**較小批次的變更隨時間帶來較高品質，即「faster is safer」**。這對前置條件（CI、測試）還沒到位的團隊乍看很爭議，故 Google 把 CD 拆成能**獨立交付價值**的面向逐步推進：**Agility（頻繁小批次）、Automation（消除重複開銷）、Isolation（模組化以隔離變更、易除錯）、Reliability（量測並改善 crash/延遲）、Data-driven（A/B）、Phased rollout（先給少數人）**。
- 頻繁發布乍看有風險（用戶變多、來不及窮盡測試），但這正是 CD 能幫的地方：**兩次發布間變更夠少，除錯就變得 trivial**。極限上每個變更都過 QA 管線並自動部署到 production；多數團隊做不到，故常以「**能隨時部署但不一定真的部署**」為文化變革的中間步驟，累積未來更頻繁發布的信心。

### Velocity Is a Team Sport（Velocity 是團隊運動）
核心主張：團隊長大或拆子團隊時的反模式——**子團隊拉分支避免互踩，日後卻苦於整合與 culprit finding**。Google 偏好大家**持續在共享 codebase 的 head 上開發**，並架 CI 測試、自動回退、culprit finding 快速抓問題（見 [[ch23]]）。
- **YouTube 反例：** 大型單體 Python 應用，發布耗力（Build Cop、release manager、志工），幾乎每次發布都多次 cherry-pick 與 respin，外加遠端 QA 對每次發布跑 **50 小時人工回歸**。發布營運成本一高 → 想多測一點就拖著不發 → 有人想再塞一個「快好了」的功能 → 發布變得耗力、易錯、又慢 → 上次做發布的專家爆肝離隊、沒人會 troubleshoot 那些怪 crash → 一想到按發布鈕就恐慌。
- **正解不是放慢：** 發布又貴又險時，本能是放慢 cadence、拉長穩定期，但那只給**短期穩定**、長期拖垮 velocity 並激怒團隊與用戶。答案是**降低成本、提高紀律、讓風險更增量化**——且要**抗拒表面的營運修補**（退回傳統規劃、加治理與審查、獎勵低風險常即低價值的功能），**投資長期架構**：**遷移到微服務**（讓大產品團隊保持靈活創新同時降風險），有時甚至**打掉重寫**以把模組化建進新架構。短期痛、但在數年壽命裡，營運成本與認知簡潔的回報會划算。

### Evaluating Changes in Isolation: Flag-Guarding（用旗標守護、隔離評估變更）
核心主張：可靠連續發布的關鍵是工程師**為所有變更加 flag guard（旗標守護）**。產品長大時，binary 裡會並存多個處於不同開發階段的功能；旗標可逐功能控制功能碼的納入/表現，且在 release build 與 development build 可不同表達（旗標關閉時，若語言允許，build 工具可把該功能從 build 中剝除）。
- **運作：** 已上線的穩定功能在 dev 與 release build 都開；開發中的功能只在 dev 開（保護用戶不碰未完成功能）。新功能碼與舊 codepath **並存於 binary**、由旗標守護；新碼可用就移除舊 codepath、下次發布完整推出；出問題就用**動態 config 更新**獨立於 binary 發布去改旗標值。
- **解耦發布與新聞稿：** 舊的 binary 發布世界要把新聞稿與 rollout 緊密對時（功能得先在野外才能宣布，有被提前發現的風險）；有了旗標，可在新聞稿前一刻才開旗標，**把洩漏功能的風險降到最低**。
- **警告：** 旗標守護不是真正敏感功能的完美安全網（程式碼仍可被 scrape/分析，若沒好好混淆）；不是所有功能都能藏在旗標後而不增大量複雜度；**旗標設定變更本身也要小心 rollout**（一次對 100% 用戶開旗標不是好主意，值得投資能安全 rollout 設定的 config 服務）。但「把單一功能的命運與整體產品發布解耦」這個控制力，是應用長期永續的強力槓桿。

### Striving for Agility: Setting Up a Release Train（追求敏捷：建立發布列車）
核心主張：以 Google **Search** binary（最古老、可追溯到 2003 年甚至更早的程式碼）為例——智慧手機興起後，行動功能被硬塞進原為 server 部署寫的程式毛球裡，部署可行 build 越來越難，一度**一週只能發布一次、且常靠運氣**。Sheri Shipe 接手提升發布 velocity 時，每個發布週期要一組工程師花數天（建 binary、整合資料、測試、每個 bug 手動 triage 確認不影響 Search 品質/UX/營收），不 scale，開發者永遠不知功能何時上 production。歷經數年，一組專責工程師做出連續發布流程——**自動化、設功能提交 deadline、簡化 plug-in 與資料整合，最終能穩定每兩天發布一次 Search binary**。換得可預測性的兩大取捨：
- **No Binary Is Perfect（沒有完美的 binary）：** 尤其是納入數十上百名開發者獨立開發數十項主要功能的 build。不可能修掉每個 bug，得不斷權衡（一條線左移兩像素會不會影響廣告顯示與營收？一個框的陰影微調會不會害視障用戶讀不到字？）。**軟體本質複雜、沒有完美 binary**；用**有明確門檻的 KPI 指標**讓不完美但夠好的功能能上（呼應 SRE 的 **error budget**——完美鮮少是最佳目標），也為爭議性的發布決策帶來清晰。但有個 bug 是：某菲律賓小島才講的稀有方言，用該方言問就回空白頁——團隊跑遍各辦公室查有多少人講、是否每次都發生、這些人是否常用 Google，最後把問題丟給 Search 資深 VP：該為只影響一個菲律賓小島的 bug 延遲關鍵發布嗎？**結論：無論你的島多小，都該得到可靠準確的搜尋結果——延遲發布、修了 bug**。
- **Meet Your Release Deadline（趕上你的發布 deadline）：** 錯過發布列車，它就不等你開走。「deadline 是確定的，人生不是」；發布時間軸某一點要插旗、把開發者與新功能擋在外。一般而言過了 deadline 再怎麼求也進不了今天的發布（罕見例外：週五晚六名工程師衝進 release manager 隔間，剛完成與 NBA 的合約功能、明天大賽前必須上線否則違約——但 release 工程師搖頭說切+測新 binary 要四小時，今天還是他小孩生日、得去買氣球）。**規律發布意味錯過列車只需等幾小時而非幾天**，大幅降低開發者恐慌、改善 release 工程師的工作生活平衡。

### Quality and User-Focus: Ship Only What Gets Used（只出貨被用到的東西）
核心主張：**Bloat（臃腫）** 是多數開發生命週期的不幸副作用，產品越成功通常程式越臃腫；高效發布列車反而放大 bloat。尤其客戶端應用（如行動 App）會讓**使用者裝置為從沒用過的功能付出空間、下載、流量代價**，開發者則付出更慢的 build、更複雜的部署、罕見 bug。
- **原生 App 的取捨：** 原生 App 較高效、較能抗時好時壞的連線，但較難更新、較易受平台層問題影響；反對原生 App 頻繁連續部署的常見論點是用戶不喜歡頻繁更新、要付流量與中斷代價。
- **關鍵是「讓選擇是刻意的」：** 有了順暢的 CD 流程，**「多久產生一個可行發布」可以和「用戶多久收到一個」分開**——你可以達到「能每週/每天/每小時部署」卻不真的那麼做，並依用戶需求與組織目標**刻意選擇**發布流程與人力/工具模型。
- **動態部署＋A/B：** 保持程式模組化，就能做**動態、可設定的部署**，更好地利用受限資源（如裝置空間）——否則每個用戶都得收到他永遠用不到的程式（不需要的翻譯、為別種裝置設計的架構）。動態部署讓 App 保持小體積、只把對該用戶有價值的程式送上裝置；A/B 實驗則讓「功能成本 vs 對用戶與業務的價值」做刻意取捨。前期有建置成本，但長期在風險管理、velocity、快速創新上的贏面夠大，值得。

### Shifting Left: Making Data-Driven Decisions Earlier（左移：更早做資料驅動決策）
核心主張：為所有用戶開發，意味客戶端橫跨 smart screen、speaker、Android/iOS 手機平板——光 Android 就 >20 億台、多樣性大到讓「驗證一個發布」令人卻步。一位 release manager 的智慧扭轉了局面：**客戶端市場的多樣性不是「問題」，而是「事實」**。接受後，發布驗證模型可改為：
- **窮盡測試不可行時，改做代表性測試（representative testing）**。
- **分階段推出（staged rollout）** 給逐步增加的用戶比例，允許快速修補。
- **自動化 A/B 發布**用統計顯著結果證明發布品質，不必靠疲累的人盯 dashboard 做決定。
- **Android 實務：** 用專門測試軌道＋分階段推給漸增的用戶流量、嚴密監控；Play Store 提供無限測試軌道，可在每個要上市的國家設 QA 團隊、全球隔夜周轉測試關鍵功能。
- **A/B 測「部署」本身：** 在 Android 上發現——**光是推一個更新就會造成統計顯著的用戶指標變化**（即使產品沒改）。所以對小流量 canary 能告訴你 crash/穩定問題，卻幾乎不能告訴你新版是否真的比舊版好。較大的 App 因此會**A/B 測部署**：同時送出兩版——「想要的更新」與當基準的「安慰劑（把舊版再發一次）」——對夠大的相似用戶群同時 rollout、比較兩版，幾天甚至幾小時內得統計顯著結果；自動化指標管線可在「確定不影響 guardrail 指標」時盡快把發布推給更多流量。用戶群不夠大時，最佳實務是**追求 change-neutral 發布**：所有新功能都 flag guard，rollout 期間唯一被測的是**部署本身的穩定性**。

### Changing Team Culture: Building Discipline into Deployment（改變團隊文化：把紀律建進部署）
核心主張：「**Always Be Deploying（持續部署）**」處理 velocity，也有處理規模的實務。初始團隊可能 <10 人輪流值部署與 production 監控；長到數百人、子團隊各管功能後，**每次部署的變更數與每次發布的風險會超線性（superlinearly）增加**——每次發布都含數月血淚，讓發布變高接觸、勞力密集；開發者常陷入兩難：放棄含一季新功能與修補的發布，還是沒信心地硬推。
- 規模下複雜度常表現為**發布延遲增加**：即使每天發布，一個發布可能要一週以上才安全完整 rollout，害你 debug 時落後一週。這正是「Always Be Deploying」讓專案回到有效狀態之處——**頻繁發布列車讓你與「已知良好位置」的偏離最小**，變更的近期性有助解決問題。
- **Google Maps 觀點：** 功能很重要，但**極少有功能重要到該為它扣住發布**。若發布頻繁，一個功能錯過發布的痛，遠小於整批新功能為延遲所受的痛，尤其小於「把沒準備好的功能趕著塞進去」害到用戶的痛。
- **一項發布職責是保護產品不受開發者傷害：** 做取捨時，開發者對推出新功能的熱情與急迫，永遠不能凌駕既有產品的使用者體驗。這意味新功能必須**用強契約的介面與其他元件隔離、關注點分離、嚴格測試、早而頻繁的溝通、以及新功能驗收的慣例**。

> **ZH:** 「如果你趕不上發布列車，它會自己開走、不等你。……deadline 是確定的，人生不是。」
> *語境/註解：Meet Your Release Deadline——規律發布把「等下一班」從幾天縮成幾小時，降開發者恐慌、護 release 工程師的工作生活平衡。*

> **EN:** "Simply having the structures in place that enable continuous deployment generates the majority of the value, even if you don't actually push those releases out to users."
> **ZH:** 「光是把那些能支撐持續部署的結構建立起來，就已經產生了絕大部分的價值——即使你並沒有真的把那些發布推給使用者。」
> *語境/註解：結論——不必每天推一個天差地別的 Search/Maps/YouTube；重點是「能隨時做到」所需的流程、即時指標、清楚政策與可在 production 設定的 binary。*

## 關鍵字補充

- **Continuous Delivery（CD，持續交付）** — 持續組裝 RC 並晉級測試、隨時能可靠發布的實踐；核心信條是「小批次帶來高品質」。
- **Faster Is Safer（更快反而更安全）** — CD 全章主旨；小批次、頻繁發布讓變更增量、除錯 trivial、適應更快，且每次發布成本與被放棄的成本都更低。
- **Velocity（速度）／Launch and Iterate** — 組織競爭力在上市速度、瓶頸是部署時間；價值在功能到用戶手上才實現，故「早發布、常發布、邊做邊改」。
- **Flag-Guarding（旗標守護）／Feature Flag** — 用功能旗標包住每個變更，新舊 codepath 並存、用動態 config 控制；把單一功能的命運與整體 binary 發布解耦。
- **Release Train（發布列車）** — 規律、可預測的發布節奏；錯過就等下一班（只需數小時），降低恐慌與爆肝（Search 做到每兩天一班）。
- **No Binary Is Perfect（沒有完美的 binary）** — 軟體本質複雜，每次發布都要做取捨；用有明確門檻的 KPI（呼應 error budget）讓不完美但夠好的功能能上。
- **Bloat（臃腫）／Ship Only What Gets Used** — 產品越成功越臃腫；動態部署＋A/B 讓 App 只把對用戶有價值的程式送上裝置，避免裝置付空間/流量代價。
- **Dynamic Deployment（動態部署）** — 靠模組化做可設定部署，把「多久產生可行發布」與「用戶多久收到」解耦，讓選擇是刻意的。
- **Reality as Benchmark（把現實當基準）／Representative Testing** — 客戶端多樣性是事實而非問題；窮盡測試不可行就做代表性測試＋分階段推出。
- **A/B Deployment（A/B 部署）／Change-Neutral Release** — 同送更新版與安慰劑（舊版再發）比較，因光推更新就會動指標；用戶群不夠大時改追求 change-neutral（只測部署穩定性）。
- **Always Be Deploying（持續部署）** — 把部署變成高頻、低風險的日常紀律；頻繁列車讓與「已知良好位置」的偏離最小。
- **Protect the Product from the Developers（保護產品不受開發者傷害）** — 一項發布職責；開發者對新功能的熱情不能凌駕既有產品的 UX，新功能須以強契約介面隔離。

## 延伸提問

1. 你的發布「又貴又險」嗎？你的本能是放慢 cadence、拉長穩定期嗎？如果是，你願不願意接受本章的反直覺結論——**正解是降低成本、提高紀律、把風險增量化**，並投資長期架構（微服務化/重寫）而非表面營運修補？你的 YouTube 式 50 小時人工回歸在哪？
2. 你的團隊長大時，有沒有人想「拉分支避免互踩」？你能不能改成**在共享 head 上開發＋CI＋自動回退＋culprit finding**？你已具備 [[ch23]] 的哪些前置（presubmit、TAP 等價物、兩下回退）？
3. 你為所有變更加 **flag guard** 了嗎？你能不能做到「出事時改動態 config 旗標、不必重發 binary」「發布前一刻才開旗標配合公告」？你有沒有「能安全 rollout 設定」的 config 服務，避免一次對 100% 用戶開旗標？
4. 回想你上次為「某個 bug／某個功能」焦慮要不要延遲發布——你是用**有門檻的 KPI 與 error budget 思維**做取捨，還是憑感覺/最大聲的人？「菲律賓小島」那題你會怎麼判？你的「發布列車錯過只等幾小時」了嗎？
5. 你出貨了多少「沒人用」的東西（bloat）？你能不能用**動態部署＋A/B** 把「產生可行發布」與「用戶收到」解耦、只送有價值的程式？你量測過每個功能在野外的成本與價值嗎？對照 [[ch07]] 的生產力量測與 [[ch11]] 的測試取捨。
