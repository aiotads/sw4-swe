# 第 14 章｜Larger Testing（大型測試）

> *"Larger tests cover things unit tests cannot. … A good design includes a test strategy that identifies risks and larger tests that mitigate them."*
> 「大型測試涵蓋單元測試做不到的事。……好的設計包含一份測試策略：辨識風險，並用對應的大型測試去緩解它們。」

## 摘要

單元測試是地基，但**只靠它無法確認整個系統真的能運作**。本章談 **larger tests（大型測試）**——它們是健康軟體工程「風險緩解策略」的重要一環。大型測試不受小測試的限制，故可能**慢（預設逾時 15 分或 1 小時，有的跑數小時甚至數天）、非 hermetic（與他者共享資源）、非確定性**。存在的首要理由是 **fidelity（保真度）**：測試多大程度反映 **SUT（system under test，被測系統）** 的真實行為——保真度愈高，成本與風險也愈高（production 是最高保真，也最危險）。

大型測試補的是單元測試的五個系統性缺口：**①不忠實的替身（mock 與真實實作不符、且會 stale）、②設定問題（configuration 是 Google 重大故障的頭號原因，2013 一次未經測試的網路設定推送造成全球當機）、③負載下才現形的問題（效能/負載/壓力）、④未預期的行為與輸入（Hyrum's Law：有效契約是所有可見行為）、⑤湧現行為與「真空效應」（單元測試像真空中的理論物理，乾淨但脫離現實）**。但大型測試也違反「可靠、快、可擴展」三美德，還有**擁有權**與**標準化**兩大挑戰（缺標準化使它們在 LSC 時常被跳過）。

一個大型測試由四階段組成：**取得 SUT → 植入測試資料（seed）→ 用 SUT 執行動作 → 驗證行為**。SUT 由兩個常相衝突的因素衡量：**hermeticity（封閉性）與 fidelity（保真度）**；形態從 single-process、single-machine、multimachine 到 shared（staging/production）與 hybrid。驗證有三法：**manual、assertions、A/B diff（差異比對）**。書中列出十種大型測試，其中 **A/B diff 回歸測試是 Google 最常見的大型測試**（始於 2001）。最後談如何把大型測試納入開發流程：**盡量做到「可能的最小測試」、用輪詢取代 sleep、驅除 flakiness、讓失敗訊息可被理解、且每個大型測試都要有明確 owner**。

## 為什麼要讀這一篇

- **你寫了一堆單元測試，卻仍不敢說「整個系統能動」。** 本章點名單元測試的**五個系統性盲區**（不忠實的替身、設定、負載、未預期行為、湧現/真空效應）——尤其 **configuration 是 Google 重大故障的頭號原因**。看懂這些，你才知道哪些風險非得用大型測試補。
- **你的測試金字塔長成了「冰淇淋甜筒」。** 本章解釋成因：**從手動測試起家**（以為程式只活幾分鐘），手動 e2e 累積成主流、又難補單元測試——「幾天內就不小心造出了 legacy code」。解方是**頭幾天就往金字塔靠**。
- **你想知道大型測試到底有哪幾種、各自緩解什麼風險。** 本章給你十種類型的「SUT／資料／驗證」配方表（functional、perf/load、deployment config、exploratory、A/B diff、UAT、prober/canary、DiRT/chaos、user evaluation）——這正是 **Test Engineer 角色**要會的「測試策略」。
- **你的大型測試又慢又愛掛、沒人維護、壞了也看不懂。** 本章給實戰守則：**用輪詢取代 time-based sleep、降低內部 timeout、可預建依賴、用 Dapper 追蹤跨行程呼叫鏈、失敗訊息要帶上下文與聯絡人、每個測試都要有 OWNERS**——否則大型測試會「rot（腐爛）」。

## 細節重點

### What Are Larger Tests?（什麼是大型測試）
核心主張：小測試受限於**一執行緒、一行程、一機器**；大型測試不受這些限制，因而可能**慢、非 hermetic、非確定性**。為什麼要它們？因為你其實會「跑跑看真實 binary」來確認程式真的能動、確認升級（如 Google Maps API 換版）沒壞相容性——**單元測試給你對個別函式/模組的信心，大型測試給你對「整體系統如預期運作」的信心**，而自動化的測試能以手動測試做不到的方式擴展。

#### Fidelity（保真度）
核心主張：大型測試存在的**首要理由就是 fidelity**——測試反映 SUT 真實行為的程度。可從**環境**看：單元測試把測試與一小段程式綁成可跑單元（確保被測，但與 production 跑法很不同）；production 自然是最高保真；中間有一整條光譜。**提高保真度同時也提高成本與（在 production 的情況下）失敗風險**——大型測試的關鍵是找到恰當的落點。也可從**測試內容**看保真：手刻的測試資料若看起來不真實會被工程師棄用；從 production 複製的資料最忠實，但「新程式上線前如何造出真實流量」是難題（AI 尤甚，seed 資料常有內在偏誤）。手刻資料覆蓋面窄、且帶作者偏誤——沒覆蓋到的情境就是測試的**保真度缺口**。

#### Common Gaps in Unit Tests（單元測試的系統性缺口）
核心主張：大型測試補的是單元測試做不好的五處：
- **不忠實的替身（Unfaithful doubles）：** 替身與被替身者可能不一致；mock 多由「寫單元測試的工程師」定義，但他通常不是被 mock 之物的作者，可能誤解其真實行為——**契約理解一旦錯誤就失效**；且 mock 會 **stale**（真實實作改了、mock 測試卻無訊號跟上）。（若團隊為自家服務提供 fake，此憂大致緩解——見 [[ch13]]。）
- **設定問題（Configuration issues）：** binary 通常不自足，有部署設定/啟動腳本/設定資料庫。**configuration 是 Google 重大故障的頭號原因**——2013 一次「從未被測試」的壞網路設定推送造成全球當機。設定常用設定語言（非 production 語言）寫、推送週期更快、更難測。**把設定也納入版本控制**，才能在出包時快速辨識肇因、並納入大型測試。
- **負載下的問題（Issues under load）：** 效能/負載/壓力測試常需對某 binary 送大量流量（廣告即時競價動輒每秒數千至數百萬筆查詢），這在典型單元測試模型裡難以辦到。
- **未預期的行為/輸入/副作用：** 單元測試受限於工程師的想像力，只能測「預期」的行為與輸入；但使用者遇到的問題多半是**未預期**的。**Hyrum's Law**：即使 100% 符合明示契約，有效的使用者契約涵蓋**所有可見行為**，不只是聲明的那些（見 [[ch01]]）。
- **湧現行為與「真空效應」：** 單元測試刻意排除真實依賴、網路、資料的混亂以求快與可靠——**像真空中的理論物理問題，乾淨卻錯失某些缺陷類別**。

### Why Not Have Larger Tests?（為何不（多）用大型測試）
核心主張：開發者友善的測試需**可靠（不 flaky、給出有用的通過/失敗訊號）、快（不打斷工作流）、可擴展（presubmit/post-submit 都能高效跑）**——好單元測試三者皆備，**大型測試常三者皆違**（用更多基礎設施故更 flaky、設定與執行都更慢、因資源/時間且彼此會碰撞而難擴展）。另有兩大挑戰：**擁有權**（跨多個 unit、多個 owner，沒有明確 owner 測試就會 rot）與**標準化的缺乏**（不同團隊用不同平台/語言/框架；缺標準化使大型測試在 LSC（[[ch22]]）時常被跳過、也無法教給新人）。

### Larger Tests and Time / Scale（大型測試與時間、規模）
核心主張：測試類型該隨程式預期壽命改變。**冰淇淋甜筒反模式**常源於「**從手動測試起家**」：以為程式只活幾分鐘，手動 e2e 累積成主流；若程式又難單元測試，能寫的只剩 e2e——**幾天內就不小心造出了 legacy code**。健康之道是**頭幾天就往金字塔靠**（把單元測試設為提交門檻），再補上自動整合測試、減少手動 e2e。

規模上：微服務系統的互連像一張圖，**每加一個節點，端到端路徑數就乘法（甚至指數/組合）成長**（書中社群網路 SUT：2 群使用者、2 個 UI、3 個資料庫、索引管線、6 個伺服器、14 條邊，已難窮舉）——故須改用其他大型測試策略。但其價值也隨之上升：**保真度的乘法效應**——若用低保真替身（如各 10% 準確）替換兩個伺服器，出 bug 機率高達 99%（1 − 0.1×0.1）。故大型測試須**在規模上可行、又維持夠高的保真**。**「可能的最小測試」**：即使整合測試也是愈小愈好；可用 **chained tests（鏈式測試）**——把一個測試的輸出持久化成另一個測試的輸入，用多個小的成對整合測試代表整體情境。

### Structure of a Large Test（大型測試的結構）
核心主張：大型測試多遵循四階段工作流——**①取得 SUT → ②植入必要測試資料（seed）→ ③用 SUT 執行動作 → ④驗證行為**。

- **The System Under Test（SUT）：** 由兩個**常相衝突**的因素衡量——**hermeticity（封閉性：與其他元件/流量的隔離程度，愈高愈不受並行與基礎設施 flakiness 影響）** 與 **fidelity（保真度：愈接近 production 的二進位、設定、拓撲）**。SUT 範圍是大型測試範圍的主要驅動。形態：**single-process**（全打包成一個 binary，最不忠於 production 拓撲）、**single-machine**（多 binary 但同機，medium 測試）、**multimachine**（跨多機，最高保真但最易受網路/機器 flakiness）、**shared environments（staging/production）**（成本最低但會與他人衝突、且要等程式推送到該環境、production 還有影響使用者風險）、**hybrids**（部分自跑、後端共享——對 Google 這種規模幾乎必須混用）。
  - **hermetic SUT 的好處：** in-production 測試沒有額外環境成本，但**程式得先到 production 才能跑、故擋不住發布**（太遲）；共享 staging 環境也只能在程式可用時跑、且「預約時段」無法隨工程師/服務數成長而擴展。下一步是 **cloud-isolated / machine-hermetic SUT**，避開衝突與預約。
  - **Case Study｜Webdriver Torso：** 為驗證 YouTube production 影片轉檔，建了自動腳本上傳測試影片到一個**公開的** YouTube 頻道；被 Wired 報導、引發媒體解謎，最後 Google 自承並用 Rickroll、彩蛋玩了一把——提醒：**要預期使用者可能發現你放在 production 的任何測試資料**。
  - **在問題邊界縮小 SUT：** UI 測試出名地不可靠又貴（外觀改動就脆裂、非同步行為難測）——若後端有 public API，最好在 **UI/API 邊界**把測試拆開、用 public API 驅動 e2e。**第三方依賴**是另一個該拆的接縫（不建議讓自動測試打真實第三方 API）。把資料庫換成記憶體版、移除不關心的伺服器，可讓 SUT 縮到單機（最穩、最易跑的「整合」測試）。
  - **Record/replay proxies：** 用測試替身 double 整個伺服器/行程，但無法保證 double 符合真實契約。業界用 **consumer-driven contract tests**（Pact、Spring Cloud Contracts）；Google 因重度依賴 protocol buffers 而不用這些，改用 **record/replay**：跑大型測試時錄下對外部服務的流量（**Record Mode**，持續在 post-submit 跑、必須通過才能產生日誌），開發與 presubmit 時重放（**Replay Mode**）。因非確定性，請求須用 matcher 比對該回放哪個回應（類似 stub/mock 的引數比對）。客戶端行為大改時須重跑 Record——故要讓 Record 易跑、快、穩。

- **Test Data（測試資料）：** 大型測試需兩種——**seeded data（植入資料：測試開始時 SUT 的狀態）** 與 **test traffic（測試流量：執行期間送進去的資料）**。植入往往比單元測試複雜數個量級：**domain data（預填的設定資料，缺了 binary 可能啟動失敗）、realistic baseline（如社群網路需夠真實的社交圖）、seeding APIs（直接寫資料庫可能繞過真實 binary 的觸發器與檢查）**。產生方式：**handcrafted（手刻）、copied（從 production 複製，如地圖資料）、sampled（抽樣，「smart sampling」用最少資料達最大覆蓋）**。

- **Verification（驗證）：** 三法——**manual（人工，含回歸與探索式；不能次線性擴展，系統愈大愈耗人力）、assertions（如 `assertThat(response.Contains("Colossal Cave"))`）、A/B comparison（差異比對：跑兩份 SUT、送相同資料、比輸出；不需明示預期，但需人工判讀差異是否為預期）**。

### Types of Larger Tests（十種大型測試）
核心主張：把 SUT／資料／驗證的不同組合，配成各緩解不同風險的大型測試。**測試策略**（test plan 的核心）就是辨識主要風險向量、選對應的測試——這正是 Google **Test Engineer** 角色要會的事。
- **Functional（功能測試，1+ 互動 binary）：** SUT 單機 hermetic 或雲端隔離；手刻資料；assertions。微服務環境常把所有相關 binary 拉起來、透過已發布 API 互動。
- **Browser / Device（瀏覽器/裝置測試）：** functional 的特例——對使用者而言 public API 就是 app 本身，透過前端以第三方身分互動多一層覆蓋。
- **Performance / Load / Stress（效能/負載/壓力）：** SUT 雲端隔離；資料手刻或從 production 多工複製；diff（效能指標）。負載/壓力是「高度湧現」的系統屬性，要盡量像 production；可調整部署拓撲、或同機跑兩版以消噪。
- **Deployment Configuration（部署設定測試）：** SUT 單機 hermetic 或雲端隔離；無資料；assertions（不崩潰）。其實是 SUT 的 smoke test——啟動成功就通過。
- **Exploratory（探索式測試）：** SUT production/共享 staging；production 或已知測試宇宙的資料；人工。受訓使用者探索新路徑、找可疑行為/安全漏洞（像手動版的 fuzz）；**bug bash**（全員開會一起手動測）是常見形式。不能次線性擴展，找到的 bug 應補成自動測試。
- **A/B Diff 回歸測試：** SUT 兩個雲端隔離環境；資料多從 production 多工或抽樣；A/B diff 驗證。**Google 最常見的大型測試**（概念可溯 1998，Google 自 2001 起用於 Ads/Search/Maps）。變體：**A-A**（找非確定性/噪音/flakiness）、**A-B-C**（比上一個 production 版、baseline、待發布版）。挑戰：**approval（需有人判斷差異是否預期）、noise（噪音逼出更多人工調查）、coverage（造足夠流量難）、setup（同時維護兩個 SUT 複雜度加倍）**。
- **UAT（使用者驗收測試）：** SUT 機器 hermetic 或雲端隔離；手刻資料；assertions。對治「Working as implemented ≠ Working as intended」（單元測試由寫程式者寫，誤解會同時反映在程式與測試）；用 Cucumber/RSpec 等寫成「可執行規格」。**Google 其實做得不多**（產品多由工程師自己造、定義行為者本就懂程式語言）。
- **Probers / Canary Analysis：** SUT production；production 資料；assertions 與指標 A/B diff。**prober** 對 production 跑唯讀、確定性的 smoke 斷言（如搜 google.com 確認有結果回來）；**canary analysis** 在發布灰度推送時，比較 canary 與 baseline 的健康指標。極限：抓到的問題已在影響使用者；可變動（寫）動作會改 production 狀態。
- **Disaster Recovery / Chaos Engineering：** SUT production；production＋人造故障注入；人工與指標 A/B diff。Google 年度 **DiRT（Disaster Recovery Testing）** 戰演把故障注入到近乎行星級規模（曾模擬地震把 Mountain View 總部與公司隔離）；**chaos engineering**（Netflix 推廣）是「持續」注入背景故障，Google 用自製的 **Catzilla**，每週數千次。
- **User Evaluation（使用者評估）：** SUT production；production 資料；人工與指標 A/B diff。**dogfooding（自家員工先吃狗糧）、experimentation（不知情子群對照——著名故事：AdWords 廣告背景陰影色實驗使點擊顯著上升）、rater evaluation（人工評分，對搜尋/機器學習這類「沒有唯一正解、只有好壞」的系統至關重要）**。

### Large Tests and the Developer Workflow（納入開發流程）
核心主張：即使標準單元測試基礎設施不適用，仍**必須把大型測試整合進開發流程**。許多大型測試**不屬於 TAP**（非 hermetic、太 flaky、太耗資源），故有**獨立的 post-submit 持續建置**，也鼓勵 presubmit 跑（直接回饋作者）。需人工 bless 差異的 A/B diff 測試也能納入（presubmit 設成「核准 UI 差異」的 code review 要求；有的測試會自動為未解決差異開「擋發布」的 bug）。太大/太痛的測試只在 post-submit 與發布流程跑——代價是壞東西進了 monorepo、得回頭找肇因 rollback，這是「開發者痛苦 vs 變更延遲 vs 持續建置可靠性」的取捨。

- **Authoring（撰寫）：** 靠清楚的函式庫、文件、範例。Google 重用斷言庫、並造了與 SUT 互動／跑 A/B diff／植入資料／編排工作流的函式庫。A/B diff 與 production SUT 之所以受歡迎，是因為**驗證/維護的人力成本較低**——但要整體看，若人工調差或保護 production 測試的成本蓋過節省就不划算。
- **Speeding up（加速）：** 工程師不等慢測試。最好的加速是**縮小範圍或拆成可平行的兩個小測試**；**用輪詢（接近微秒、配 timeout）/事件處理/訂閱通知，取代 time-based sleep**（依賴 sleep 的測試會在測試機群過載時一起失敗、進而惡性循環）；**調低內部 timeout/delay**、**優化 build 時間**（可用已知良好版的預建依賴，更接近 production「不同服務不同版本」的現實）。
- **Driving out flakiness（驅除不穩定）：** 對大型測試 flakiness 可使其不可用，應列高優先。先**縮小範圍**（hermetic、單機 SUT 風險最低）；事件驅動可同時加速並去 flaky；內部 timeout 是取捨（調高減 flaky、調低恐增 flaky）。一個棘手點：production 為保護使用者會優雅處理內部問題（如無法及時供廣告就不供、而非回 500），這在 test runner 看來像壞了——要讓失敗模式明顯、且 timeout 可為測試調整。
- **Making tests understandable（讓測試可理解）：** 失敗的好大型測試應：**①訊息清楚指出失敗是什麼**（不要只有「Assertion failed」＋堆疊；要「預期 10 筆搜尋結果卻得 1 筆」）；**②最小化找出根因的努力**（堆疊追蹤對跨行程無用——用 **Dapper**：以單一 request ID 串起 RPC 呼叫鏈的所有日誌）；**③提供支援與聯絡資訊**。
- **Owning（擁有）：** 大型測試**必須有文件化的 owner**（能審查變更、失敗時提供支援），否則更難修改、更久才解、然後 rot。專案內整合測試由 project lead 擁有；跨服務的功能測試由 feature owner（可能是工程師、PM 或 test engineer）擁有。可用結構化資訊做自動化：**OWNERS 檔（[[ch09]]）** 或**逐測試標註（per-test annotations）**。

> *語境/註解：開章對「為何要大型測試」的核心回答；自動化測試能以手動測試做不到的方式擴展。*

> **EN:** "A unit test is like a problem in theoretical physics: ensconced in a vacuum, neatly hidden from the mess of the real world."

> *語境/註解：Emergent behaviors and the "vacuum effect"——這對速度與可靠很好，卻錯失了某些只在真實依賴/網路/資料下才現形的缺陷。*

## 關鍵字補充

- **Larger Test（大型測試）** — 不受小測試（一執行緒/行程/機器）限制的測試，可能慢、非 hermetic、非確定性；補單元測試的系統性盲區，是風險緩解策略的一環。
- **Fidelity（保真度）** — 測試（環境與內容）反映 SUT 真實行為的程度；大型測試存在的首要理由。保真度↑ ⇒ 成本與風險↑。
- **SUT（System Under Test，被測系統）** — 大型測試的核心；由 hermeticity 與 fidelity 兩個常衝突的因素衡量；形態：single-process / single-machine / multimachine / shared / hybrid。
- **Hermeticity（封閉性）** — SUT 與其他元件/流量的隔離程度；高封閉性最不受並行與基礎設施 flakiness 影響，但常以犧牲保真為代價。
- **冰淇淋甜筒（Ice Cream Cone）** — 大量 e2e、少單元的反模式；常源於「從手動測試起家」，幾天內就造出 legacy code。解方：頭幾天就往金字塔靠（見 [[ch11]]）。
- **A/B Diff 回歸測試** — 跑兩版 SUT、送相同流量、比輸出；Google 最常見的大型測試（始於 2001）；變體 A-A（找噪音/flaky）、A-B-C。挑戰：approval、noise、coverage、setup。
- **Record/Replay** — 跑大型測試時錄下對外部服務的流量（Record Mode，post-submit），開發/presubmit 時重放（Replay Mode）；Google 因重用 protobuf 而採此，取代 consumer-driven contract tests。
- **Seeded Data vs Test Traffic** — 植入資料（測試開始時 SUT 的狀態：domain data、realistic baseline）vs 測試流量（執行期送入）；產生方式 handcrafted/copied/sampled。
- **Prober / Canary Analysis** — prober 對 production 跑唯讀確定性 smoke 斷言；canary analysis 在灰度推送時比較 canary 與 baseline 的健康指標。
- **DiRT / Chaos Engineering / Catzilla** — DiRT 是 Google 年度災難演練（行星級故障注入）；chaos engineering（Netflix 推廣）是持續注入背景故障；Catzilla 是 Google 自製、每週數千次。
- **Dogfooding / Experimentation / Rater Evaluation** — production 使用者評估三法；experimentation 的著名故事是 AdWords 背景陰影色實驗使點擊顯著上升。
- **Dapper** — Google 以單一 request ID 串起 RPC 呼叫鏈所有日誌的追蹤框架；讓跨行程的大型測試失敗可被追根究柢。
- **Test Engineer（測試工程師）** — Google 的專門工程角色；核心能力之一是為產品擬定「測試策略」（辨識風險向量、選對應大型測試）。

## 延伸提問

1. 盤點你系統的「風險向量」：哪些風險是**單元測試結構上補不到**的（不忠實的替身？設定？負載？未預期輸入？湧現行為）？你有沒有為每個主要風險配一個對應的大型測試——也就是一份明確的**測試策略**？
2. 你的設定（config/部署腳本/feature flag）有沒有進**版本控制**、有沒有被**測試**？回想 Google「設定是重大故障頭號原因／2013 全球當機」——你上次出包，是程式還是設定造成的？
3. 你的測試金字塔像金字塔還是**冰淇淋甜筒**？你的專案是不是「從手動測試起家、幾天內就造出 legacy code」？哪些手動 e2e 該被自動整合測試取代？對照 [[ch11]]。
4. 挑一個你最大、最痛的 e2e 測試：它的 **SUT** 能不能縮小（在 UI/API 邊界拆開？第三方換 record/replay？資料庫換記憶體版？）？在 **hermeticity vs fidelity** 上，你目前落在哪、該往哪移？
5. 你的大型測試壞掉時，別人看得懂為什麼嗎（訊息有上下文嗎？有跨行程追蹤如 Dapper 嗎？知道找誰嗎）？它有**明確的 owner** 嗎？它是在 presubmit、post-submit、還是只在發布時跑——這個取捨對你的團隊對嗎？
