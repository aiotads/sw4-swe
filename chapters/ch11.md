# 第 11 章｜Testing Overview（測試總覽）

> *"Automated testing is foundational to enabling software to change. … If you liked it, you should have put a test on it."*
> 「自動化測試是『讓軟體能夠改變』的根基。……你要是喜歡它，當初就該為它寫一個測試。」

## 摘要

這是測試三部曲（[[ch11]]→[[ch12]]→[[ch13]]→[[ch14]]）的開場。核心論點翻轉了多數人的直覺：**測試的首要目的不是「抓 bug」，而是「支撐改變的能力」**——有了穩固的測試，你不必懼怕改動，反而能擁抱它。寫測試這個動作本身還會**改善系統設計**（測試是你程式的第一個客戶，會逼你面對耦合與邊界問題）。

關鍵案例是 **Google Web Server（GWS）**：2005 年它一度有**超過 80% 的生產推送含有需要回滾的、影響使用者的 bug**；導入「工程師驅動、自動化測試、新程式都要附測試並持續執行」的政策後，**一年內緊急推送減半**（即使變更量創新高），今日有數萬個測試、幾乎每天發布。它教會 Google 一件事：**光靠工程師個人能力無法避免缺陷**——100 人團隊就算人人每月只寫一個 bug，每個工作日仍冒出 5 個新 bug。自動化測試把「個人智慧」變成「全隊共享資源」。

本章給出兩個正交的設計維度：**Size（規模）**——小（單一行程）、中（單一機器）、大（多台機器）——衡量的是資源與**速度＋確定性（determinism）**；**Scope（範圍）**——narrow（unit）、medium（integration）、broad（e2e）——衡量驗證了多少程式。Google 建議的混合比例約 **80% 單元／15% 整合／5% 端到端**（測試金字塔），並警告兩種反模式：**ice cream cone（冰淇淋甜筒）** 與 **hourglass（沙漏）**。其餘關鍵：**Beyoncé Rule（「想留住它就為它寫測試」）**、flaky（不穩定）測試的昂貴（接近 1% 就失去價值）、**code coverage 不是測試品質的金標準**（80% 的地板會變成天花板）、以及把測試「**當成生產程式碼一樣對待**」。

## 為什麼要讀這一篇

- **你覺得「寫測試拖慢開發」。** 這章用 GWS 的真實數字反駁：測試不是為了抓 bug 而抓 bug，而是**讓你敢改、能快速迭代**——能更快迭代的公司，能更快適應技術、市場與顧客口味。看懂這點，測試從「成本」變成「速度的前提」。
- **你分不清 unit／integration／e2e 到底怎麼切。** 本章給你兩把更精準的尺：**Size（小/中/大＝單行程/單機/多機，管的是速度與確定性）** 與 **Scope（窄/中/廣＝驗證多少程式）**——兩者正交。理解這個區分，你才知道為何 Google 偏好「**範圍可以廣、但 size 要小**」（用 test double 把昂貴依賴換掉）。
- **你的測試套件又慢又愛亂掛（flaky），大家開始繞過它。** 本章點名這正是**最危險的失敗模式**：一個不被信任的測試套件比沒有更糟。給了硬指標（接近 1% flakiness 就失去價值、Google 維持在 ~0.15%）與藥方（hermetic、禁 `sleep()` 改用輪詢、把測試當生產程式碼）。
- **你被要求「測試覆蓋率要到 80%」。** 這章告訴你為何那是個糟糕的目標：coverage 只量「某行被執行」、不量「驗證了什麼」；80% 的**地板**會變成**天花板**。該問的不是「夠不夠多」，而是「**我顧客期待能動的，我有信心都會動嗎？**」

## 細節重點

### Why Do We Write Tests?（我們為什麼寫測試）
核心主張：最簡單的測試由四要素定義——**一個被測行為（通常是一個方法或 API）、一個特定輸入、一個可觀察的輸出/行為、一個受控環境（如單一隔離的行程）**。數百上千個這種簡單測試（test suite）合起來，告訴你整個產品多符合設計——**更重要的是，當它不符合時**。但「抓 bug」只是動機的一部分；同等重要的是**支撐改變**：你愈想頻繁、快速地改系統，就愈需要一個快速測試它的辦法。

工程意涵：測試的價值來自**工程師對它的信任**。若測試變成生產力黑洞、不斷製造 toil 與不確定，工程師會失去信任、開始找繞道——**一個壞的測試套件可能比沒有測試還糟**。所以「健康」比「數量」更重要。

#### The Story of GWS（GWS 的故事）
核心主張：2005 年的 Google Web Server（負責服務 Google 搜尋查詢，「對搜尋的重要性如同空管之於機場」）隨規模膨脹而生產力驟降、發布愈來愈 buggy——**一度超過 80% 的生產推送含有需回滾的使用者可見 bug**。TL 導入「工程師驅動的自動化測試」政策（新程式都要附測試、持續執行），**一年內緊急推送減半**（儘管每季變更量創新高）。

工程意涵：GWS 教會的關鍵洞見——**不能只靠工程師個人能力避免缺陷**。想像 100 人團隊，人人厲害到每月只寫一個 bug，這群神人每個工作日仍產出 **5 個新 bug**；更糟的是複雜系統裡修一個 bug 常引發另一個。自動化測試把「成員的集體智慧」轉成「全隊的資產」：一人寫的測試進入共享資源池，全隊都能跑、都受惠——與「每次出 bug 就有人付除錯成本」相比，工程資源的差距是天壤之別。

#### Write, Run, React（寫、跑、反應）
核心主張：自動化測試的純粹形式由三個活動構成——**寫測試、跑測試、對失敗做出反應**。今天建系統的工程師**親自**為自己的程式寫並跑自動測試（不再是昔日「一房間 QA 人工點過每個行為」）；在現代的速度與規模下，唯一跟得上的方法就是**把寫測試這件事分攤給整個工程團隊**。寫完要**頻繁地跑**（持續整合 CI，見 [[ch23]]；機器不會累不會膩，一天可跑數千次）；把測試表達成程式碼也讓它**易於模組化、跨環境執行**（Firefox 與 Chrome、日文與德文 UI 用同一份測試碼）。但真正讓流程有效的是**如何處理失敗**：讓失敗測試堆積會瞬間摧毀其價值——**在失敗後幾分鐘內修好**的團隊，才能維持高信任與快速的失敗隔離。

### Benefits of Testing Code（測試帶來的六項效益）
核心主張：對沒有測試文化的組織，「寫測試提升生產力」聽來矛盾（寫測試可能和實作功能一樣久、甚至更久）。但 Google 發現測試帶來六項關鍵效益：
- **更少除錯（Less debugging）：** 被測程式提交時缺陷更少，且**終其一生**缺陷更少（一段程式預期一生被改數十次）；一次寫好的測試持續配息。
- **對改變更有信心（Increased confidence）：** 重要行為被持續驗證，故能放心接受變更、**鼓勵重構**（純重構而不改行為，理想上不需動既有測試）。
- **更好的文件（Improved documentation）：** 文件出名地不可靠；清晰、一次只測一個行為的測試是**可執行的文件**——想知道某情況下程式做什麼，就看那個 case 的測試（呼應 [[ch10]]）。
- **更簡單的審查（Simpler reviews）：** 附上展示正確性、邊界與錯誤情況的測試，審查者就不必在腦中逐一走過每個 case（見 [[ch09]]）。
- **深思熟慮的設計（Thoughtful design）：** 為新程式寫測試是**檢驗其 API 設計**的實用手段——若難測，常因職責太多或依賴難管理；及早修設計問題＝日後更少返工。
- **快速、高品質的發布（Fast, high-quality releases）：** 有健康套件，數百工程師、每天數千變更的大專案也能**每天發布到生產**。

### Designing a Test Suite：Size 與 Scope（設計測試套件：兩個正交維度）
核心主張：早期教訓——工程師偏好寫大型、系統級測試，但這些**更慢、更不可靠、更難除錯**。為止痛，團隊逐步寫出愈來愈小的測試，發現它們**更快、更穩、更不痛**。由此釐清**每個測試有兩個截然不同的維度**：**Size（規模）**＝跑它需要的資源（記憶體、行程、時間）；**Scope（範圍）**＝驗證了哪些程式路徑。兩者相關但不同（執行一行 ≠ 驗證它如預期運作）。

#### Test Size（測試規模）
核心主張：Google 把每個測試分類成一個 size，並鼓勵**對任一功能都寫可能的最小測試**——size 不由行數決定，而由**它怎麼跑、被允許做什麼、消耗多少資源**決定。最看重的特質是**速度與確定性（determinism）**，與範圍無關。
- **Small（小）：** 最受限——必須在**單一行程**內跑（許多語言更進一步要求單一執行緒）；**不准 sleep、I/O、或任何阻塞呼叫**（即不准存取網路或磁碟）。要測依賴這些操作的程式，就用 **test double（[[ch13]]）** 把重量級依賴換成輕量的行程內依賴。這些限制像個 sandbox，讓你「很難搬石頭砸自己腳」——因為**網路與磁碟正是慢與不確定的主要來源**。
- **Medium（中）：** 可跨多行程、用執行緒、做阻塞呼叫——但**只能對 localhost**（必須侷限在單一機器內）。彈性換來風險：跨行程／阻塞呼叫依賴 OS 與第三方行程的快與確定性，無法保證。
- **Large（大）：** 移除 localhost 限制，測試與被測系統可**跨多台機器**（如對遠端叢集）。多保留給「**驗證設定**多過驗證程式碼」的全系統端到端測試、以及無法用 test double 的舊元件（見 [[ch14]]）；常與小/中測試隔離，只在 build/release 時跑，以免拖累開發流程。
（技術上 Google 有四種 size：small／medium／large／enormous；large 與 enormous 之差很細微、屬歷史因素。）

> **Flaky tests are expensive（不穩定測試很昂貴）：** 每測試 0.1% 機率亂掛 × 每天 10,000 個測試＝每天查 10 個 flake。**接近 1% flakiness 時測試開始失去價值**；Google 把 flaky 率壓在 **~0.15%**（仍意味每天數千個 flake），靠投入工程時數硬扛。可用「自動重跑」短期緩解（拿 CPU 換工程師時間），但那只是**延後處理根因**。

**所有 size 共通的特質：** 測試應**hermetic（自足封閉）**——自己包含 setup/執行/拆解環境所需的一切，對外部環境（如測試執行順序、共享資料庫）假設愈少愈好。一個測試只該含**驗證該行為所需的資訊**；**「測試應該一看就懂（obvious upon inspection）」**——因為**測試本身沒有測試**，需靠人工審查把關正確性。故**強烈勸阻在測試裡用條件、迴圈等控制流**（複雜流程自己可能藏 bug、也更難定位失敗）。記住：**程式被讀遠多於被寫，寫出你自己想讀的那個測試**。

#### Test Scope（測試範圍）
核心主張：scope＝一個測試驗證了多少程式。**narrow（窄，俗稱 unit test）** 驗證小而聚焦的部分（單一 class 或 method）；**medium（中，俗稱 integration test）** 驗證少數元件間的互動（如 server 與其資料庫）；**broad（廣，俗稱 functional/e2e/system test）** 驗證多個不同部分的互動或無法由單一 class 表達的「湧現行為」。**注意「窄」指的是被驗證的程式，不是被執行的程式**——一個 class 自然會呼叫它的依賴；Google 偏好**可行時保留真實依賴**（而非大量用 test double，見 [[ch13]]）。

工程意涵：**size 與 scope 大致相關但非綁定**——可以寫一個 scope 廣（涵蓋 server endpoint 的解析、驗證、商業邏輯）卻 size 小的測試（用 double 換掉所有跨行程依賴）；也可能有 scope 窄卻 size 中的測試（如測一個 date picker UI 元件得跑整個瀏覽器）。Google 鼓勵**既小又窄**。建議混合約 **80% 窄範圍單元測試（驗證多數商業邏輯）／15% 中範圍整合測試／5% 端到端測試**（依測試 case 數計）——可視為一個**金字塔**。兩種反模式：**ice cream cone（冰淇淋甜筒）**＝大量 e2e、很少整合/單元（慢、不可靠、難用，常見於趕著上線的原型）；**hourglass（沙漏）**＝大量 e2e＋大量單元、但很少整合（沒甜筒糟，但仍讓本可更快抓到的問題拖到 e2e 才爆；肇因於緊耦合難以單獨實例化依賴）。

混合比例由兩個主要目標決定：**工程生產力**與**產品信心**。偏重單元測試＝快速、早期、高信心；**較大的測試是 sanity check，不該被當成抓 bug 的主要手段**。沒有萬靈丹——好套件依當地架構與組織現實，混用不同 size 與 scope。

### The Beyoncé Rule（碧昂絲法則）& Testing for Failure
核心主張：新人常問「到底哪些行為該被測？」直白答案：**測你不想壞掉的一切**。要確信系統有某行為，唯一辦法就是為它寫自動測試——含效能、行為正確性、無障礙（accessibility）、安全，以及**系統如何處理失敗**這類較不明顯的性質。這套哲學叫 **Beyoncé Rule**：「**If you liked it, then you shoulda put a test on it.（你要是喜歡它，當初就該為它寫個測試。）**」常被負責全庫變更的基礎設施團隊援引：若不相干的基礎設施變更通過了你**所有**測試卻仍弄壞你的產品，那是**你**的責任去修、並補上那個缺的測試。

**為失敗而測（Testing for Failure）：** 失敗無可避免，但等真災難發生才知道系統如何應對是自找苦吃——應**主動寫模擬常見失敗的測試**（在單元測試模擬例外/錯誤，在整合與 e2e 注入 RPC 錯誤或延遲，甚至用 **Chaos Engineering** 影響真實生產網路）。對逆境有**可預測、受控的反應**，是可靠系統的標誌。

### A Note on Code Coverage（關於程式覆蓋率的提醒）
核心主張：code coverage 量的是「哪些行被哪些測試執行」（100 行測到 90 行＝90% 覆蓋）。它常被捧為測試品質的金標準，**這很不幸**：你可以用少數測試執行大量行卻從不檢查每行是否做了有用的事——**coverage 只量「某行被呼叫」，不量「結果發生了什麼」**（故建議**只從小測試量 coverage**，以免大測試造成覆蓋率灌水）。更陰險的是它**很快變成目標本身**：團隊定 80% 門檻，工程師卻把 80% 從**地板**當成**天花板**（「何必做超過指標要求的工？」）。更好的做法是思考**被測的行為**：「我顧客期待能動的，我有信心都會動嗎？我有信心能抓到依賴的破壞性變更嗎？我的測試穩定可靠嗎？」——用單一數字回答「夠不夠多」忽略了太多脈絡。

### Testing at Google Scale（Google 規模下的測試）
核心主張：要懂 Google 的測試，先懂其開發環境——**單一 monorepo（單體儲存庫）**，逾 **20 億行**程式，每週近 **2,500 萬行變更**（約一半來自數萬工程師、一半來自自動系統如 LSC，見 [[ch22]]）。**幾乎沒有團隊用分支**：所有變更都提交到 repo head、立即對所有人可見；所有 build 都用「測試基礎設施已驗證過的最後一次提交」、且幾乎每個依賴都從 head 原始碼建起。Google 靠 **CI 系統**管理此規模，其關鍵元件是 **TAP（Test Automated Platform）**（見 [[ch23]]）。

#### The Pitfalls of a Large Test Suite（大型測試套件的陷阱）
核心主張：寫得差的自動測試會**抵抗改變**。**Brittle tests（脆弱測試）**——過度指定預期結果、或依賴大量複雜樣板——會在不相干的變更時也亂掛（「改五行卻弄壞數十個不相關的測試」）；久之讓團隊**不敢做必要的重構**。脆弱測試最大的禍首之一是**濫用 mock**（Google 的程式碼庫深受其害，逼得一些工程師宣告「**no more mocks！**」；見 [[ch13]]）。其次是**慢**：套件愈慢、跑得愈少、效益愈低；常見元兇是 `sleep()`／`setTimeout()`（天真的「等一下再檢查」會在被廣用的工具裡累積成每次跑都多出好幾分鐘的閒置）——**更好的解法是以接近微秒的頻率主動輪詢狀態轉換、再配一個 timeout**。

工程意涵：**與大型測試套件共處的祕訣是「尊重它」**——獎勵工程師擁有「磐石般穩固的測試」一如獎勵漂亮的功能發布；設適當的效能目標、重構慢或邊際的測試；**把測試當成生產程式碼對待**。並投資測試基礎設施（linter、文件）、**減少需支援的框架與工具數量**（每語言一套標準測試框架＋一套標準 mock 函式庫）。否則工程師終會繞過甚至跳過測試——那雖危險該被勸阻，但「若測試帶來的害多於益，工程師最終總會找到沒測試也能交差的辦法」。

### History & Limits（歷史與極限）
核心主張：**2005 年前測試在 Google 比較像好奇心而非紀律**；2005–2006 一場「測試革命」改變了一切，催化劑正是 GWS。志工們（**Testing Grouplet**）用三項計畫把自動測試送進公司意識：**①Orientation Classes（新人訓練課）**——所有新人必經的咽喉點，把測試當成「公司標準做法」呈現，新人成了把觀念偷渡進團隊的「特洛伊木馬」；**②Test Certified（測試認證）**——五個等級、每級一季可達成的具體行動（L1：建持續 build、追 coverage、分類測試 size、辨識 flaky；到 L5：全自動、提交前跑、無不確定性、行為全覆蓋），內部儀表板用社會壓力促成團隊競賽，幫超過 **1,500 個專案**改善（2015 由自動化的 pH 取代）；**③Testing on the Toilet（TotT，廁所裡的測試）**——把單頁短文貼在廁所隔間，雖引發爭議但「抱怨的人也還在談測試」，至今數百集、影響最深遠。**為何不直接強制要求寫測試？** 因為任何「該如何寫程式」的強制都嚴重違反 Google 文化、會拖慢進度——信念是**成功的點子會自己擴散**，所以重點放在「展示成功」。今日：每個變更都走 code review、都被期待含功能碼＋測試，**缺測試足以擋下一個變更**；Test Certified 由 **pH（Project Health，1~5 分）** 接棒。

**自動測試的極限：** 不適用所有任務——搜尋結果品質常需**人的判斷**（Search Quality Raters 跑真實查詢記錄印象）、音訊/視訊品質難自動捕捉、複雜安全漏洞人類找得比機器好。更一般的詞是 **Exploratory Testing（探索式測試）**：把待測應用當成一個「待破解的謎題」，用意想不到的步驟或資料去探。但**一旦人找到問題，就該補上自動測試以防未來回歸**——用自動測試覆蓋「已充分理解的行為」，好讓昂貴的人力測試者聚焦在他們最能創造價值之處。結論：公司自旅程開始已成長近 **100 倍**，對品質與測試的承諾比以往更強。

## 關鍵字補充

- **Developer-driven Automated Testing（工程師驅動的自動化測試）** — 由建系統的工程師親自為自己的程式寫並跑自動測試；在現代速度與規模下，把寫測試分攤給全工程團隊是唯一跟得上的方式。
- **Support the Ability to Change（支撐改變的能力）** — 測試的首要目的不是抓 bug，而是讓人敢改、能快速迭代而有信心；呼應全書 Time 主軸。
- **Test Size（測試規模）** — small（單一行程、禁 I/O/sleep/網路）、medium（單機、只准 localhost）、large（跨多機）；衡量資源，核心訴求是**速度與確定性**。
- **Test Scope（測試範圍）** — narrow＝unit（單一 class/method）、medium＝integration（少數元件互動）、broad＝e2e/system（跨多部分或湧現行為）；「窄」指被驗證的程式，不是被執行的程式。
- **Test Pyramid（測試金字塔）** — Mike Cohn 的模型，Google 版約 80% 單元／15% 整合／5% 端到端（按測試 case 數）；底層單元測試快、穩、易診斷。
- **Ice Cream Cone / Hourglass（甜筒／沙漏反模式）** — 甜筒＝大量 e2e、少單元（慢且脆）；沙漏＝多 e2e＋多單元、缺整合（緊耦合所致）。
- **Beyoncé Rule（碧昂絲法則）** — 「想留住它就為它寫測試」；測你不想壞掉的一切，基礎設施變更弄壞你產品時責任在你補測試。
- **Flaky Test（不穩定測試）** — 同樣輸入卻時過時不過的測試；接近 1% flakiness 即失去價值，Google 維持在 ~0.15%；根因多為非確定性（時鐘、執行緒排程、網路延遲）。
- **Hermetic Test（自足封閉測試）** — 自己包含 setup/執行/拆解所需的一切，不依賴外部環境或共享資料庫、不假設執行順序；size 愈大愈難做到但仍要努力。
- **Code Coverage（程式覆蓋率）** — 量哪些行被測試執行；只量「行被呼叫」不量「結果」，不是測試品質金標準；80% 地板易淪為天花板（建議只從小測試量）。
- **Test Doubles（測試替身）** — fake/stub/mock，用輕量行程內依賴替換重量級依賴（資料庫、網路）；濫用 mock 是脆弱測試的禍首（見 [[ch13]]）。
- **TAP（Test Automated Platform）** — Google CI 系統的關鍵元件，在 monorepo head 上對每次變更跑驗證（見 [[ch23]]）。

## 延伸提問

1. 把你團隊對測試的**心智模型**攤開：你們把它當「抓 bug 的關卡」還是「**讓你敢改的安全網**」？回想一次「不敢重構，因為怕改壞」的經驗——若有穩固測試，那次會不會不一樣？對照全書 Time 主軸。
2. 拿你最近寫的幾個測試，分別標出它的 **size（小/中/大）** 與 **scope（窄/中/廣）**：它們是不是「不必要地大」（明明可用 test double 換掉網路/磁碟而變小）？你的套件接近金字塔，還是偏向**甜筒或沙漏**？
3. 量一量你套件的 **flaky 率**：接近 1% 了嗎？團隊是不是已經開始「不理會失敗的測試」或「提交時跳過測試」？你是在**修根因**，還是只用「自動重跑」延後問題？
4. 你被要求的覆蓋率目標（如 80%）正在當**地板還是天花板**用？如果改成問「**我顧客期待能動的，我有信心都會動嗎？**」「我能抓到依賴的破壞性變更嗎？」，你對自家測試的評價會不會不同？
5. 套用 **Beyoncé Rule**：你最近上線的功能裡，有哪個「你很在乎、卻沒有任何自動測試守護」的行為（效能？安全？失敗處理？）？你有沒有**為失敗而測**（模擬 RPC 錯誤/延遲）？哪些任務其實該留給人（探索式測試、品質判斷）？
