# Chapter 13 — Test Doubles

> *"A real implementation should be preferred over a test double. … Interaction testing should be avoided when possible."*
> Reach for the real thing first; a double is a trade-off, not a default.

## Summary

A **test double** is an object or function that stands in for a real implementation in a test — like a stunt double for an actor. It lets you write many small, fast, non-flaky unit tests of code that would otherwise be slow and unstable (calling external servers, hitting databases). But doubles introduce three trade-offs: **testability** (the code must be designed to swap dependencies), **applicability** (misuse leads to brittle, complex, ineffective tests), and **fidelity** (how closely the double's behavior matches the real implementation).

The chapter's central stance — learned the hard way at Google — is to **prefer real implementations (classical testing) over mocking-framework-heavy mockist testing**, which doesn't scale. Google once treated mocking frameworks as "a hammer fit for every nail," only to find years later that those tests were **easy to write but needed constant maintenance while rarely finding bugs**; the pendulum has swung back toward "more realistic tests." **Prefer realism over isolation**: real dependencies execute code the way production will, giving more confidence; when a real implementation has a bug, your test fails — which is good.

There are three techniques: **faking** (a lightweight working implementation, e.g., an in-memory database), **stubbing** (hardcoding a function's return value), and **interaction testing** (verifying a function was *called* correctly without executing it). The preference order: **real implementation > fake > (sparingly used) stub/interaction.** Deciding whether to use a real implementation turns on three things: **execution time, determinism, and dependency-construction cost.** If no fake exists, the **team that owns the real implementation** should write and maintain it, and the **fake itself needs tests (contract tests).** Over-using stubbing makes tests **unclear, brittle, and ineffective**; interaction testing should be avoided where possible (it leaks implementation details, becoming a "change-detector test") — do it only on **state-changing functions** (e.g., `sendEmail()`) and avoid overspecification.

## Why read this

- **You `@Mock` everything the moment a dependency is awkward.** Google's hard-won lesson: mocked tests are easy to write but **need constant maintenance while rarely finding bugs**, and they block API owners (mocked thousands of times) from changing implementations. The order is clear: **real > fake > stub/interaction.**
- **You can't tell fake / stub / mock apart.** The chapter splits "double" into three *techniques*: fake (lightweight working impl), stubbing (hardcode return), interaction testing (verify it was called). Knowing them tells you which to reach for.
- **Your tests are "change-detectors" — production changes always turn them red even when behavior is unchanged.** That's usually over-used interaction testing. Prefer **state testing** (validate the result/state, not "what was called"), and do interaction testing only on state-changing functions.
- **You want your team's tests fast *and* realistic.** The key is a **high-fidelity fake** for your dependency — one good fake boosts a whole org's velocity; it should be maintained by the **real implementation's owner** and protected by **contract tests** so it doesn't drift.

## Key details

### The Impact of Test Doubles
The core claim: doubles introduce three trade-offs — **testability**: the code must be designed to swap real implementations for doubles (retrofitting tests later often means a major refactor); **applicability**: used well they boost velocity, but used badly (especially at scale) they produce brittle, complex, ineffective tests — **in many cases doubles aren't suitable and you should use the real implementation**; **fidelity**: how closely the double behaves like the real thing — a low-fidelity double has little value (imagine a fake database that ignores writes and always returns empty), but **perfect fidelity is often infeasible**, so double-based unit tests often need larger-scope tests to supplement them.

Takeaway (Google's lesson): **over-using mocking frameworks is dangerous.** They first looked like "a hammer fit for every nail" — write highly focused tests without worrying about constructing dependencies; but years later Google found those tests **easy to write yet in constant need of maintenance while rarely finding bugs**, and the pendulum has swung back toward "avoid mocking frameworks, write more realistic tests."

### Basic Concepts: Seams, Mocking Frameworks
The core claim: code is **testable** if it's written so you can write unit tests for it. A **seam** makes code testable by allowing doubles — letting the system under test use different dependencies than production. The most common way to introduce a seam is **dependency injection (DI)**: a class's dependencies are **passed in** rather than constructed directly, so tests can substitute doubles. Java commonly uses the automated DI frameworks **Guice and Dagger**; dynamic languages (Python/JavaScript) can replace functions directly, so they rely less on DI. **Writing testable code is an upfront investment**, more critical early — the later you consider testability, the harder it is to add.

A **mocking framework** is a library that makes creating doubles easier, replacing an object with a **mock** (a double whose behavior is specified inline), cutting the boilerplate of defining a new class each time. Google uses **Mockito (Java), googlemock (C++), unittest.mock (Python)** — but **over-use makes a codebase harder to maintain.**

### Techniques for Using Test Doubles
The core claim: three primary techniques, and knowing them tells you which to use:
- **Faking:** a lightweight working implementation that behaves like the real one but isn't production-suitable (an in-memory database). **Often the ideal technique** when you need a double, but one may not exist, and writing one is challenging (must stay similar to the real behavior, now and later).
- **Stubbing:** giving behavior to a function that has none — specifying exactly what it returns (`when(...).thenReturn(...)`). Usually done through a mocking framework to cut boilerplate.
- **Interaction testing:** verifying a function was *called* correctly without executing it (fails if not called, called too often, or with wrong arguments — `verify(...)`). **Sometimes called mocking**, a term the chapter avoids (it's confused with mocking *frameworks*, which also do stubbing). **Avoid where possible** — over-use easily yields brittle tests.

### Real Implementations: Prefer Realism Over Isolation
The core claim: although doubles are valuable, **the first choice is real implementations of the system's dependencies** (the same ones used in production), for **fidelity** — executing what production executes gives more confidence. Preferring real implementations is **classical testing**; the opposite, **mockist testing** (preferring mocking frameworks), is practiced by some (including the first mocking frameworks' creators) but Google finds it **hard to scale** (it demands strict design discipline).

**Prefer realism over isolation:** real dependencies make the system under test more realistic (their code is actually executed); over-using doubles **arbitrarily isolates** the system and **couples the test to the implementation the author happened to put in the class** — but a good test should be **independent of implementation, written in terms of the API.** When a real implementation has a bug, the test fails — **good** (it means your code won't work in production); with good CI tooling it's usually easy to find the offending change.

> **Case Study | @DoNotMock:** having seen too many mock-over-reliant tests, Google built the **`@DoNotMock` annotation** into the ErrorProne static-analysis tool, letting an API owner declare "don't mock this type — better alternatives exist." Why does the owner care? Because **every time a mocking framework stubs or interaction-tests, it duplicates behavior the API provides**; when the owner wants to change the API, they may find it's been mocked **thousands or tens of thousands of times**, with doubles that likely violate the contract (returning null where the method never does). Had tests used the real implementation or a fake, the owner could change freely without first fixing thousands of flawed tests.

#### How to Decide When to Use a Real Implementation
The core claim: prefer a real implementation when it's **fast, deterministic, and has simple dependencies** (e.g., a value object — money, date, address, list/map). For more complex code it's often infeasible, so weigh three things:
- **Execution time:** unit tests must be fast. How slow is too slow? No exact answer (depends on felt productivity loss and how many tests use it — 1 extra second is fine for 5 tests, not 500). At the borderline, "use the real implementation until it's too slow, then switch." Parallelization and Bazel caching help (but real implementations also increase build time).
- **Determinism:** a test is deterministic if the same system version always gives the same result. Nondeterminism causes flakiness (see [[ch11]]). Real implementations are more complex and more likely nondeterministic (multithreading, non-hermetic external dependencies, the system clock). When flakiness is frequent, switch to a double or a **hermetic instance** (its lifecycle controlled by the test) (see [[ch14]]).
- **Dependency construction:** a real implementation requires constructing its **entire dependency tree** (`new Foo(new A(new B(new C())...))`) — time-consuming and brittle to constructor-signature changes. Doubles often have no dependencies (`@Mock Foo mockFoo`) — but the ideal is to **use production's own construction code** (a factory method or automated DI) flexible enough to substitute doubles.

### Faking
The core claim: when a real implementation is infeasible, a **fake is often the best option** — it behaves like the real one, so the system under test can't even tell it's a fake.
- **When to write one:** a fake needs more effort and domain knowledge (must stay similar to real behavior) and must be updated whenever the real behavior changes — so **the team owning the real implementation should write and maintain it.** It's a trade-off: not worth it for a handful of users, clearly worth it for hundreds. To minimize fakes, **create one only at the root of the un-testable code** (a fake for the database API itself, not for each class that calls it).
- **Fidelity of fakes:** the key concept — a **fake should maintain fidelity to the real implementation's API contracts** (same output and state changes for a given input). Perfect fidelity isn't always feasible (a fake DB stores in memory, not on disk); reframed, **the fake needs perfect fidelity only from the test's perspective** (a hashing fake needn't produce the real hash, only a unique value per input — if the contract doesn't promise specific values). Latency and resource use usually needn't be faithful (but then you can't use a fake to test *those*). A fake needn't have 100% functionality; unsupported paths should **fail fast** (raise an error, signaling the fake is inappropriate there).
- **Fakes should be tested:** otherwise behavior drifts as the real implementation evolves. One approach is **contract tests**: write tests against the API's public interface and run them against both the real implementation and the fake (the real-implementation run is slower, but only the fake's owners need to run it).
- **If no fake exists:** first ask the API owners to write one (they may not know the benefit); else write your own (wrap all calls to the API in one class and fake that — often simpler, since you use only a subset of the API); else fall back to the real implementation or another technique. A fake can be seen as an **optimization** — if the speedup doesn't outweigh maintenance, stick with the real implementation.

### Stubbing: The Dangers of Overusing
The core claim: stubbing hardcodes function behavior in a test — quick and easy, but **over-use severely hurts productivity**:
- **Tests become unclear:** extra stub-setup code obscures intent — if you must mentally step through the system under test to understand why functions are stubbed, that's a warning sign.
- **Tests become brittle:** stubbing leaks implementation details; change the production implementation and you must update the tests (a good test should change only when the API's user-facing behavior changes).
- **Tests become ineffective:** no way to ensure the stubbed function behaves like the real one (`when(stubCalculator.add(1,2)).thenReturn(3)` hardcodes the contract with no guarantee it's right); and stubbing **can't store state** (unlike a fake that can `save` then `get`).

Takeaway: the book contrasts a test set up with five `when(...).thenReturn(...)` that can only `verify(...).pay()` at the end (no way to know the payment actually succeeded) with a rewrite using a real/fake credit-card server queried directly (`getMostRecentCharge`) — shorter, leaking no implementation details. **When stubbing is appropriate:** when you need a function to return a specific value to get the system into a state (a non-empty transaction list), or to simulate returns/errors hard to trigger from a real implementation or fake; and **each stub should relate directly to an assertion, kept few** (needing to stub many functions signals over-use or an over-complex system to refactor).

### Interaction Testing: Prefer State Testing
The core claim: **prefer state testing over interaction testing.** State testing calls the system and validates the returned value or a changed state; interaction testing can only validate that certain functions were called as expected. In the book's sorting example, state testing confirms the numbers are actually sorted; interaction testing only shows the system "tried to call quicksort," **with no way to know the numbers are really sorted.**

Takeaway: interaction testing has two problems — ① **it can't tell you the system actually works**, only that functions were called, forcing an assumption ("calling `database.save(item)` means it's saved"); ② it **leaks implementation details**, making tests brittle — Google jokingly calls over-used interaction tests **change-detector tests**: they break on any production change even when behavior is unchanged. **When it's appropriate:** ① as a fallback when state testing is impossible (no real implementation, no fake); ② when the **number or order of calls** matters (verifying a cache really reduces database calls — `verify(databaseReader, atMostOnce()).selectRecords()`). It isn't a replacement for state testing — if a unit test can only do interaction testing, supplement with larger-scope tests doing state testing against a real database (see [[ch14]]).

**Best practices:** ① **interaction-test only state-changing functions** (side effects: `sendEmail()`, `saveRecord()`, `logAccess()`; doing it for non-state-changing `getUser()`, `findResults()`, `readFile()` is usually redundant and makes tests brittle). ② **Avoid overspecification:** echoing [[ch12]]'s "test behaviors, not methods," don't verify many functions and all arguments at once; use matchers like `eq()`/`any()` to verify only the minimal set relevant to the behavior, for clearer tests resilient to out-of-scope changes.

## Key terms

- **Test double** — an object/function standing in for a real implementation (like a stunt double); lets you write many small, fast, non-flaky unit tests. The chapter avoids the overloaded word "mocking."
- **Classical vs mockist testing** — classical prefers real implementations (Google's choice); mockist prefers mocking frameworks (Google finds it hard to scale).
- **Fidelity** — how closely a double behaves like the real implementation; a fake should honor the real implementation's **API contracts**, but only from the test's perspective, not perfectly.
- **Seam** — a way to make code testable by allowing different dependencies (doubles) for the system under test; most often introduced via dependency injection.
- **Dependency injection (DI)** — dependencies are passed in rather than constructed directly, so tests can substitute doubles; Java uses Guice/Dagger, dynamic languages rely on it less.
- **Fake** — a lightweight working implementation that behaves like the real one but isn't production-suitable (in-memory DB); the ideal double when needed; maintained by the real implementation's owner and itself tested.
- **Stubbing** — hardcoding a no-behavior function's return (`when(...).thenReturn(...)`); over-use makes tests unclear, brittle, ineffective, and stateless.
- **Interaction testing** — verifying a function was called correctly without executing it (`verify(...)`); avoid where possible — do it only on state-changing functions, without overspecification.
- **Change-detector test** — the result of over-used interaction testing: fails on any production change even when behavior is unchanged; leaks implementation details, brittle.
- **@DoNotMock** — an ErrorProne annotation letting an API owner declare "don't mock this type — better alternatives exist"; protects the owner's freedom to change the implementation.
- **Contract test** — tests against an API's public interface run against both the real implementation and the fake, ensuring the fake doesn't drift.
- **State vs interaction testing** — state = validate the returned value/changed state; interaction = validate a function was called. Prefer state (it confirms the system actually works).

## Questions to push your thinking

1. Scan your tests' `@Mock` / `when().thenReturn()`: how many could be **real implementations or fakes**? Which dependencies are "fast, deterministic, simple" value objects (money, date) that shouldn't be mocked at all (try the `@DoNotMock` spirit)?
2. Is one of your tests a "change-detector" — turning red on any production change even when behavior is unchanged? Is it doing **interaction testing**? Could it become **state testing** (validate the result/state, not "what was called")?
3. Does your hardest-to-test dependency (database, external service) have a **fake**? If not, should the owner write it, or should you wrap a layer? Does the fake have a **contract test** keeping it from drifting?
4. Apply the "execution time / determinism / dependency construction" criteria to a test where you're torn between real and double: where is the real implementation slow? Where does nondeterminism come from (clock? threads? non-hermetic?)? How painful is the dependency tree? Which is the real reason to use a double?
5. In your interaction tests, are you **overspecifying** (verifying functions/arguments unrelated to the behavior)? Which are **non-state-changing** (`getUser`, `readFile`) and needn't be verified? After narrowing to the minimal set with `any()`/`eq()`, are the tests more change-resilient? Per [[ch12]].
