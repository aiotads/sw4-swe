# 第 22 章｜Large-Scale Changes（大規模變更）

> *"Making LSCs means making a habit of making LSCs."*
> 「要會做大規模變更，靠的是養成『常做大規模變更』的習慣。」

## 摘要

**Large-Scale Change（LSC，大規模變更）** 指任何**邏輯上相關、但實務上無法當成單一原子提交（atomic commit）一次送出**的變更集合——可能因為觸及的檔案多到工具一次提交不了，可能因為改動大到必然產生 merge conflict，也可能因為儲存庫拓樸（分散式/聯邦式 repo）讓跨 repo 的原子變更根本不可能。本章最反直覺的觀察是：**隨著程式碼庫與工程師人數成長，「可能的最大原子變更」反而縮小**——光是讓變更裡每個檔案都保持最新、跑完所有受影響的 presubmit 與測試，就越來越難。

既然原子變更撐不住，Google 改以「**分而治之**」的方式做 LSC：用自動化工具產生一份巨大的全域變更，再用 **Rosie** 依專案邊界與 OWNERS 規則**切成可獨立提交的小碎片（shard）**，每片各自跑 test→mail→submit 管線。測試靠 **TAP train（測試列車）**——把多個 LSC 併車、共享測試結果以攤平成本。文化上，Google 把這些 commit 當成「**牛（cattle）而非寵物（pets）**」：無名、可隨時被回退、丟幾頭也不心疼，因為自動化能廉價重生。

本章貫穿三主軸：**Scale（規模）**是核心——同一套工具的人力成本必須**次線性**於 repo 大小（產生全部變更花的人時，不該隨 repo 變大而暴增）；**Time（時間）**——LSC 讓「技術決策不再永遠不可變」（符號名稱、類別位置都能事後改），並對抗「**鬧鬼墓園（haunted graveyard）**」這種沒人敢碰的老系統；**Trade-offs（取捨）**——明知小碎片會讓同一批測試重跑多次，仍刻意換取「工程師除錯時間遠比 CPU 時間貴」的划算。誰來做？**基礎設施團隊**（他們有領域知識、能攤平成本、避免「無預算的強制令」unfunded mandate）。流程四階段：**授權（Authorization）→ 變更產生（Creation）→ 碎片管理（Shard management）→ 清理（Cleanup）**。案例橫跨 `scoped_ptr→std::unique_ptr`（>50 萬處引用）、單字元檔名修正（填坑），到 Operation RoseHub（>50 名 Googler 手動發 >2,600 個 patch 修補開源社群的 Mad Gadget 漏洞）。

## 為什麼要讀這一篇

- **你的「最大可原子提交」其實比你想的小，而且會越來越小。** 開篇那串問題（你能可靠地在單一提交裡更新幾個檔案？緊急時做得到嗎？）逼你正視一個反直覺事實：**repo 與團隊越大，原子變更的上限越低**。本章給你「放棄原子性、改用分片」的完整心法與工具觀，讓你的程式碼庫**對基礎設施變更保持靈活**。
- **你被一個沒人敢碰的老系統卡住（鬧鬼墓園）。** 本章把 SRE 的「No Haunted Graveyards」搬到程式碼層次，並給出唯一解藥：**扎實的測試**——有了測試，再老再複雜的系統都能放心改。這直接影響你「該不該補測試、補哪裡」的決策。
- **你想推一個更好的新系統，卻指望「使用者自己遷移」。** 本章解釋為何**有機遷移（organic migration）幾乎不會完成**（工程師照抄舊範例、好處分散到沒人有動機），以及為何該由**擁有舊系統的基礎設施團隊**負責遷移——這是把「無預算強制令」的外部性內部化。
- **你把每個 commit 都當寶貝。** 「cattle vs pets」會改變你對自動化變更的心態：當變更由機器大量產生，**個別 commit 該是可丟棄的牛、不是寵物**；被回退別往心裡去。這是大規模重構能不能跑起來的心理門檻。

## 細節重點

### What Is a Large-Scale Change?（什麼是大規模變更）
核心主張：LSC ＝**邏輯相關、但無法當成單一原子單元提交**的變更集合。原因可能是檔案太多、改動太大必有 merge conflict，或 repo 拓樸（分散式/聯邦式）讓跨 repo 原子變更不可能。
- **LSC 幾乎都由自動化工具產生。** 動機分幾類：用全庫分析工具**清理常見反模式**、**替換已棄用的函式庫特性**、**底層基礎設施升級（如編譯器升版）**、**把使用者從舊系統搬到新系統**。
- **多數 LSC 近乎零功能影響：** 傾向是為了清晰、最佳化、未來相容而做的大面積文字更新（但理論上不限於這種保行為/重構類）。
- **規模感：** 在 Google 這種規模，基礎設施團隊常要改動數十萬個對舊模式/符號的引用，最大案例觸及**數百萬處**。工具不僅讓改數千檔有效率，**縮小到數十檔也運作良好**——投資 LSC 工具同時嘉惠做小變更的人。

### Who Deals with LSCs?（誰來做）
核心主張：**基礎設施團隊**負責多數 LSC 的執行（工具與資源全公司可用）。為什麼不能「推出新系統、命令所有舊用戶自己搬」？因為那不擴展：
- **領域知識集中：** 建/管底層系統的團隊才有修正數十萬處引用的脈絡；讓每個消費團隊各自重新學習是全域低效。集中化也讓錯誤更快復原（錯誤通常落在少數類別，可備有 playbook）。
- **沒人喜歡「無預算強制令（unfunded mandate）」：** 新系統就算客觀更好，好處分散到各團隊都沒動機自己升級。遷移成本總得有人承擔——集中遷移＋核算成本，幾乎總是比指望各團隊有機遷移更快更省。
- **誘因對齊：** 有機遷移很難全成（工程師照抄既有程式當範例）；讓「有意願移除舊系統」的團隊負責遷移，才確保真的做完。為這種遷移編列人力看似額外成本，其實只是把無預算強制令造成的外部性內部化，還附帶規模經濟。

### Barriers to Atomic Changes（原子變更的障礙）
核心主張：理想世界裡每個邏輯變更都能獨立打包成單一原子提交來測試、審查、提交；但隨 repo 與人數成長，這越來越不可行（用分散式/聯邦 repo 時甚至小規模就不可行）。障礙有：
- **技術限制：** 多數 VCS 的操作隨變更大小**線性**擴展；小提交（數十檔）沒問題，但可能沒有足夠記憶體/算力一次原子提交數千檔。集中式 VCS 的大提交還會卡住其他寫入者（甚至讀取者）。把大變更切成獨立小塊可繞過，代價是執行更複雜。
- **Merge conflict（合併衝突）：** 變更越大，遇到 merge conflict 的機率越高，且被 repo 內工程師人數放大。小公司也許能挑沒人開發的週末偷渡一個碰每個檔案的變更，或用「傳令牌搶全域鎖」的土法；在 Google 這種全球公司，**總有人在改 repo**，這些招都不可行。變更檔案少 → 衝突機率小 → 更可能順利提交。
- **No Haunted Graveyards（不要鬧鬼墓園）：** SRE 的箴言。鬧鬼墓園＝老到、晦澀到、複雜到**沒人敢進去**的系統，常是業務關鍵卻被時間凍結（一改就可能以無法理解的方式壞掉、花真金白銀）。程式碼裡也有：老舊無人維護、卻在重要營收路徑上、堆滿官僚防止改動。**解藥是老派的測試**——測試夠扎實，任何年紀/複雜度的系統都能放心改，把「鬧鬼墓園」這概念本身送進墳墓。
- **Heterogeneity（異質性）：** LSC 只在「大部分工作能由電腦而非人完成」時才行得通；電腦需要**一致的環境**才能正確套用轉換。若組織有多種 VCS、CI、專案專屬工具、格式規範，就難以全庫掃改。Google 選擇**擁抱部分一致性**（如把 presubmit 測試標準化），並建議團隊在 LSC 觸及其程式時**略過自家特殊檢查**。
- **Testing（測試）：** 每個變更都該測，但越大越難測。Google CI 會跑「**直接受影響＋傳遞依賴**」的所有測試（廣覆蓋，但離受影響檔越遠的測試，越不可能是這變更弄壞的）。小而獨立的變更易驗證——在 25 檔的變更裡找出失敗根因很直觀，在 10,000 檔裡找就像大海撈針。**取捨：** 小碎片會讓同一批測試重跑多次（尤其依賴大片程式碼的測試），但**工程師追查失敗的時間遠比多跑測試的算力貴**，所以這是 Google 願意做的刻意取捨（你的組織未必相同，值得自行衡量）。

### Case Study: Testing LSCs — Riding the TAP Train（測試 LSC：搭 TAP 列車）
核心主張：今天一個專案裡常有**雙位數百分比（10–20%）的變更來自 LSC**——意味大量程式碼被「本職與該專案無關」的人改動。沒有好測試，這種工作根本不可能，Google 的程式碼庫會在自身重量下迅速萎縮。
- **TAP train（測試列車）的核心洞見：** LSC 彼此鮮少互動，且多數受影響測試對多數 LSC 都會通過。利用兩個事實——**①LSC 多是純重構、範圍窄、保留局部語意；②個別變更通常更簡單且被高度審視，正確率高**——就能**同時測多個變更、減少總測試次數**。
- **列車五步（每三小時發一班）：** ①對車上每個變更，跑**隨機抽樣的 1,000 個測試**；②把通過的變更全併成一個 uber-change（即「列車」）；③跑這群變更**直接影響的所有測試的聯集**（夠大或夠底層的 LSC 可能要跑遍 Google 每一個測試，可耗時 >6 小時）；④對每個失敗的非 flaky 測試，**逐一對車上每個變更重跑**以定位元兇；⑤為每個上車的變更產出報告，列出通過/失敗目標，當作「此 LSC 可安全提交」的證據。
- **Code Review：** LSC 同樣需審查；大型提交審起來繁瑣、易錯（尤其手工產生的——應避免）。把 LSC 切成碎片讓審查容易得多。

### Case Study: scoped_ptr to std::unique_ptr（智慧指標遷移）
核心主張：C++11 引入 `std::unique_ptr`，嚴格優於 Google 自製的 `scoped_ptr`，但程式碼庫有**逾 50 萬處** `scoped_ptr` 引用，散落於數百萬源檔——這是當時 Google 嘗試過**最大的 LSC**。
- 數名工程師平行進攻數月，用大規模遷移基礎設施把 `scoped_ptr` 引用改成 `std::unique_ptr`、並逐步讓 `scoped_ptr` 行為貼近 `unique_ptr`。**高峰期每天持續產生、測試、提交逾 700 個獨立變更、觸及逾 15,000 檔**；今天精煉工具後有時能達**十倍吞吐**。
- 像幾乎所有 LSC，有一條很長的尾巴：追查各種微妙的行為依賴（又一個 Hyrum's Law 的體現）、與其他工程師搶 race、以及自動工具偵測不到的生成程式碼用法——這些都靠人工處理。
- `scoped_ptr` 曾用作某些常用 API 的參數型別，使小而獨立的變更很難做。最終解法：**先把 `scoped_ptr` 變成 `std::unique_ptr` 的型別別名（type alias）→ 在新舊別名間做文字替換 → 最後移除舊別名**。

### LSC Infrastructure（LSC 基礎設施）
核心主張：Google 投入大量基礎設施使 LSC 可行（變更產生、變更管理、變更審查、測試的工具），但**最重要的支撐其實是圍繞 LSC 演化出的文化規範與監督**。
- **Policies and Culture（政策與文化）：** monorepo＋人人可見，意味任何工程師都能改任何檔送審；但每個編輯都有產生與審查成本，過去這成本對稱、限制了單人能造成的變更範圍。工具改善後，單人能極廉價地產生大量變更、也能輕易把負擔丟給全公司大量審查者——所以需要**輕量的核准流程**（一個約十多人、熟悉各語言細節的委員會監督），目標不是禁止 LSC，而是幫作者產出最佳變更。文化上：code owner 要學會 LSC 是 Google 擴展工程實踐的重要一環，**本地擁有者理解變更很重要，但對更廣的 LSC 沒有否決權**。
- **Codebase Insight（程式碼庫洞察）：** 需要對程式碼庫做大規模分析——文字層（傳統工具）與語意層（如語意索引工具 **Kythe**，可回答「誰呼叫這函式？」「哪些類別繼承自它？」）。也用編譯器索引做 AST 分析與轉換（**ClangMR、JavacFlume、Refaster** 等高度可平行化的工具）。小變更可用 perl/sed/regex/shell。**關鍵原則：變更產生工具的人力必須次線性於 repo 大小**——無論 repo 多大，產生「全部所需變更」花的人時應大致不變，且要全庫涵蓋。經驗法則：**一個變更若需 >500 處編輯，通常學工具比手改更划算**。
- **Change Management（變更管理）—— Rosie：** 最重要的一塊。Rosie 把一份 master change **切成可原子提交的小碎片**，並管理各片的測試、寄送、審查、提交。它不只是工具，更是「讓 LSC 在 Google 規模化」的整個平台。
- **Testing / Language Support：** 測試 LSC 不只確保大 master change 不出錯，更要確保**每個碎片都能獨立安全提交**（因碎片可含任意檔案，不用標準專案 presubmit，而是跑該片可能影響的測試的傳遞閉包）。語言層面：**type aliasing 與 forwarding functions** 對「讓舊用戶持續運作、同時非原子地遷移」極有價值；**靜態型別語言遠比動態型別語言易做自動大改**（編譯器工具＋強靜態分析能在測試前就擋掉無效轉換，故 Python/Ruby/JS 對維護者格外難）。**自動格式化工具（clang-format、google-java-format）是 LSC 基礎設施的關鍵一環**——讓自動產生的變更「看起來像人寫的」，否則大規模自動變更不會被接受為常態。

### Case Study: Operation RoseHub（RoseHub 行動）
核心主張：LSC 已是 Google 內部文化的大部分，並開始影響更廣的世界。2017 初，Apache Commons 函式庫的漏洞（**Mad Gadget**）讓任何 classpath 裡有脆弱版本的 Java 應用都可能被遠端執行——它甚至讓駭客加密了**舊金山交通局（SFMTA）**系統、癱瘓其營運。因為只要 classpath 裡有錯版本就中招，許多依賴 GitHub 開源專案的東西都脆弱。一群 Googler 發起自己的 LSC 流程：用 **BigQuery** 找出受影響專案，**寄出逾 2,600 個 patch** 把 Commons 升到修補版。這次不是自動工具管流程，而是**逾 50 名人類**手動完成這個 LSC。

### The LSC Process（LSC 流程四階段）
核心主張：有了基礎設施，LSC 流程大致分四階段（彼此界線模糊）：**授權 → 變更產生 → 碎片管理 → 清理**。理想上這些發生在新系統/類別/函式寫好之後，但**設計新系統時就該把遷移路徑想進去**（Google 設計後繼系統時就以「能把用戶自動搬過來」為念）。
- **Authorization（授權）：** 作者填一份簡短文件說明變更理由、估計影響（會切出多少碎片）、預設 FAQ；並取得被重構 API 擁有者的「領域審查」。提案送交約十多人的監督委員會討論回饋。最常見的建議之一是**把所有審查導向單一「全域核准者（global approver）」**（多數機械式 LSC，由一位專家理解變更本質並圍繞它建自動審查，比讓每個本地擁有者各審更便宜）。委員會歷來核准很寬鬆，也常一次核准一整類相關變更；明顯變更可快速通道。委員會也是衝突的升級仲裁者（實務上鮮少需要）。被明確否決的多是**危險**（如把所有 NULL 轉 nullptr）或**極低價值**（如英式↔美式拼字互換）的變更。
- **Change Creation（變更產生）：** 取得核准後產出實際編輯，盡量自動化（以便在使用者「倒退（backslide）」回舊用法或發生文字 merge conflict 時更新母變更）。罕見無法自動產生時，會像 RoseHub 那樣**把變更產生分片給人**。為維持人類可讀性，產出要盡量像人寫的——這帶出風格指南與自動格式化工具的必要。
- **Sharding and Submitting（切片與提交）—— Rosie：** Rosie 依專案邊界與擁有權規則把全域變更切成可原子提交的碎片，再讓每片走獨立的 test-mail-submit 管線。它會**限制任一 LSC 的未結碎片數、以較低優先序運行、並與基礎設施溝通可接受的負載**。
  - **Cattle vs Pets（牛 vs 寵物）：** 典型手作變更像養寵物——工程師花數天數週親手打磨、瞭若指掌、提交時很自豪。但有效處理 LSC 需要高度自動化、產出海量個別變更；此時該把變更當「**牛**」——無名、可隨時被回退或拒絕、丟幾頭成本很低（除非整群受影響）。有自動化，工具可更新、新變更可極低成本重生，所以偶爾損失幾頭牛不是問題。**別把回退當成對你個人的否定**。
  - **Testing（碎片測試）：** 每片透過 TAP 跑「傳遞依賴的所有測試」，常造成 CI 高負載。但實務上**絕大多數碎片影響的測試 <1,000 個**（在全庫數百萬測試中）；影響多的可分組（先跑所有碎片受影響測試的聯集，再對每片只跑「其受影響測試 ∩ 第一輪失敗測試」），因聯集幾乎跑遍全庫，**多加幾個變更幾乎免費**。大規模跑測試的缺點：低機率事件在足夠大規模幾乎必然發生——**flaky/brittle 測試**對 LSC 作者特別棘手。Google 對「語意保留、機器產生的單一變更」的正確性信心，已高於「近期有 flaky 史的測試」——以致提交時**會忽略近期 flaky 的測試**（理論上可能放過一個只被 flaky 轉 failing 抓到的回歸，但實務罕見到用人際溝通處理即可）。每個碎片必須能獨立提交（無互相依賴，或由切片機制把相依變更如標頭檔與實作綁在一起）。
  - **Mailing reviewers（寄送審查者）：** Rosie 用理解 **OWNERS 檔**的擁有者偵測服務、按「審查該碎片的預期能力」加權挑審查者；擁有者沒回應就自動加審查者。也跑各專案的 precommit 工具，但**對 LSC 選擇性停用某些檢查**（如非標準變更描述格式）。也**積極忽略變更前就存在的 presubmit 失敗**——本地擁有者依「社會契約」有責任保持自己程式碼庫無既有失敗。
  - **Reviewing（審查）：** 理想上 LSC 該像其他變更一樣被嚴格審；實務上本地擁有者太信任 LSC 作者，常只草草審。Google 演化成**只在「需要本地脈絡」時才送本地擁有者**，其餘交給全域核准者。全域核准者不逐一審，而是用**基於模式的工具自動核准符合預期的變更**，只人工檢查因 merge conflict 或工具異常而異常的小子集——讓流程良好擴展。
  - **Submitting（提交）：** 最後個別變更被提交（提交前一樣過各專案 precommit）。有了 Rosie，每天能跨全庫有效建立/測試/審查/提交**數千個變更**——曾經最終不可變的技術決策（廣用符號的名稱、熱門類別的位置）**不再需要是最終的**。
- **Cleanup（清理）：** 不同 LSC 對「完成」有不同定義（從完全移除舊系統，到只遷移高價值引用、讓舊的有機消失）。幾乎所有情況都要有**防止舊符號/系統再被引入**的機制——Google 用 **Tricorder** 框架在審查時標記「引入了已棄用物件的新用法」，有效防止倒退。

工程意涵（結論）：**LSC 讓某些技術決策不再不可變**——設計時就能知道「有些決定日後可改」，因而開放更多可能；也讓核心基礎設施維護者能把大片程式碼從舊系統/語言版本/函式庫慣用法遷到新的，讓程式碼庫**在空間與時間上都保持一致**。而這一切只靠**數十名工程師支撐數萬名其他工程師**。無論組織大小，都值得想想你會如何對你的源碼做這種掃改——有此能力會讓組織擴展時更靈活、源碼隨時間更可塑。

## 原文引用

> **EN:** "An LSC is any set of changes that are logically related but cannot practically be submitted as a single atomic unit."
> **ZH:** 「LSC 是任何在邏輯上相關、但實務上無法當成單一原子單元提交的變更集合。」
> *語境/註解：全章的定義句。判準不是「改了幾行」，而是「能不能一次原子提交」——這把問題從程式碼大小轉向工具與 repo 拓樸的限制。*

> **EN:** "As a codebase and the number of engineers working in it grows, the largest atomic change possible counterintuitively decreases."
> **ZH:** 「隨著程式碼庫與在其中工作的工程師人數成長，可能的最大原子變更反而會縮小。」
> *語境/註解：本章最反直覺的核心觀察，也是「為何需要 LSC＝放棄原子性、改用分片」的根本理由。*

> **EN:** "A haunted graveyard ... is a system that is so ancient, obtuse, or complex that no one dares enter it. ... we've found the counter to this to be good, ol'-fashioned testing."
> **ZH:** 「鬧鬼墓園……是一個老到、晦澀到、複雜到沒人敢踏進去的系統。……我們發現對付它的解藥，是老派而扎實的測試。」
> *語境/註解：Barriers to Atomic Changes 一節。鬧鬼墓園阻擋一切有意義的進展；唯一解藥是測試——有測試就能放心改任何年紀的系統。*

> **EN:** "It's useful to treat specific changes as cattle: nameless and faceless commits that might be rolled back or otherwise rejected at any given time with little cost."
> **ZH:** 「把個別變更當成『牛』來看待是有用的：那是無名無臉、隨時可能被回退或拒絕、代價極低的提交。」
> *語境/註解：Cattle vs Pets。LSC 的心理門檻——當變更由機器大量產生，個別 commit 該是可丟棄的牛、不是寵物；被回退別往心裡去。*

> **EN:** "Making LSCs means making a habit of making LSCs."
> **ZH:** 「要會做大規模變更，靠的是養成『常做大規模變更』的習慣。」
> *語境/註解：TL;DR 收尾。LSC 是一種需要持續練習與投資工具/文化的肌肉，不是偶一為之的英雄壯舉。*

## 關鍵字補充

- **Large-Scale Change（LSC，大規模變更）** — 邏輯相關但無法當單一原子單元提交的變更集合；幾乎都由自動化工具產生，目的多為清理反模式、替換棄用特性、基礎設施升級、遷移使用者。
- **Atomic Change（原子變更）** — 能一次測試、審查、提交的單一不可分割變更；隨 repo 與人數成長，可能的最大原子變更反而縮小。
- **Diamond / Merge Conflict 障礙** — 大變更必然撞上 VCS 的線性擴展極限與合併衝突機率上升（被工程師人數放大）。
- **Haunted Graveyard（鬧鬼墓園）** — SRE 箴言「No Haunted Graveyards」；老到沒人敢碰的關鍵系統，阻擋遷移與升級；解藥是扎實測試。
- **Unfunded Mandate（無預算強制令）** — 外部強加要求卻不給對應補償；解釋為何該由基礎設施團隊（而非各消費團隊）負責 LSC、把外部性內部化。
- **TAP Train（測試列車）** — 把多個 LSC 併車共享測試以攤平成本；五步、每三小時一班，產出「可安全提交」的證據。
- **Rosie** — Google 把 master change 切成可原子提交碎片、並管理各片 test-mail-submit 的平台；LSC 規模化的核心。
- **Shard（碎片）** — 由 Rosie 依專案邊界/OWNERS 切出的可獨立提交小變更；必須能獨立提交、各自過 presubmit。
- **Cattle vs Pets（牛 vs 寵物）** — 對待變更的心態對比；LSC 的個別 commit 是可丟棄的牛，不是親手打磨的寵物。
- **Global Approver（全域核准者）** — 有全庫核准權、用基於模式的工具自動核准 LSC 碎片的專家，取代逐一找本地擁有者。
- **Kythe / ClangMR / Refaster** — 語意索引與 AST 層的分析/轉換工具，提供 LSC 工具所需的程式碼庫洞察。
- **Type Alias / Forwarding Function** — 讓舊用戶持續運作、同時非原子遷移到新系統的語言特性（`scoped_ptr→unique_ptr` 即靠型別別名過渡）。
- **Tricorder** — 在審查時標記「引入已棄用物件新用法」的框架，防止 LSC 清理後的倒退。

## 延伸提問

1. 回到開篇那串問題：**你能可靠地在單一原子提交裡更新幾個檔案？** 限制因素是什麼（VCS 算力？merge conflict？presubmit？）？緊急時你做得到嗎？如果這數字遠小於你的程式碼庫大小，你的「分而治之」工具（誰是你的 Rosie？）準備好了嗎？
2. 你的程式碼庫裡有沒有**鬧鬼墓園**——老到沒人敢碰、卻在關鍵路徑上的系統？擋住你升級編譯器/函式庫或移除舊系統的，是不是它？你願不願意先投資補測試，把它從墓園裡解放出來？對照 [[ch11]]、[[ch14]]。
3. 你最近想推一個更好的新系統，是指望使用者**有機遷移**，還是由你（系統擁有者）負責 LSC？回想「無預算強制令」與「工程師照抄舊範例」——有機遷移在你的組織真的會完成嗎？對照 [[ch15]]。
4. 你對 LSC 的個別 commit 抱「牛」還是「寵物」心態？若你的自動化還做不到「廉價重生變更」，你卡在哪——是工具（產生）、測試（TAP train 等價物），還是文化（本地擁有者的否決權與信任）？
5. 你的變更產生工具，人力成本是否**次線性**於 repo 大小？換句話說，repo 長大十倍，產生「全部所需變更」的人時會跟著暴增嗎？那條「>500 處編輯就該學工具」的經驗法則，在你的團隊該設在哪？對照 [[ch17]]、[[ch20]]。

## 跨章連結

- [[ch15]] Deprecation — LSC 是「遷移使用者、移除舊系統」的執行引擎；棄用流程的「先遷移再移除」全靠 LSC 能力，Tricorder 防倒退兩章共用。
- [[ch11]] Testing Overview — 「No Haunted Graveyards」的唯一解藥是測試；TAP train 與「碎片跑傳遞閉包測試」是測試文化的規模化體現。
- [[ch14]] Larger Testing — LSC 的廣覆蓋測試（直接＋傳遞依賴）與 flaky 測試對吞吐的傷害，是大型測試議題的延伸。
- [[ch21]] Dependency Management — Abseil「破壞性 API 變更必附自動重構工具」、Live at Head 的「破壞時就地升級」都仰賴本章的 LSC 能力。
- [[ch16]] Version Control and Branch Management — 「寧可面對版本控制問題」與 monorepo＋trunk-based 是 LSC 可行的前提；分散式/聯邦 repo 讓原子變更不可能。
- [[ch09]] Code Review — LSC 同樣需審查；全域核准者、OWNERS 偵測、基於模式的自動核准都是審查流程的規模化變體。
- [[ch01]] What Is Software Engineering? — Hyrum's Law（任何可觀察行為都會被依賴）是 `scoped_ptr` 長尾與 LSC 必須廣測的根源；「整合時間維度」正是 LSC 對抗的對象。
