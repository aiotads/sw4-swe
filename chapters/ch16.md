# 第 16 章｜Version Control and Branch Management（版本控制與分支管理）

> *"One-Version Rules are surprisingly important for organizational efficiency. … Developers within an organization must not have a choice where to commit, or which version of an existing component to depend upon."*
> 「『單一版本規則』對組織效率的重要性出乎意料地高。……一個組織裡的工程師，不該有『要把程式提交到哪』或『要依賴某元件的哪個版本』的選擇。」

## 摘要

這是「工具」篇（Part IV）的開場。版本控制（VCS）大概是軟體工程界**最普遍採用**的工具。本章先回答它為何成為近乎普世的規範：把全書主軸具象化——**軟體工程＝整合了時間維度的程式設計**，而 VCS 正是工程師管理「原始碼 × 時間」的主要工具。可把 VCS 想成擴充的檔案系統：`VCS(filename, time, branch) => 檔案內容`。對只寫過「一個人、寫幾週」程式的新人而言，VCS 像是個多餘的「整個專案的 undo 鍵」——但它真正的價值是**讓團隊與組織能擴展**。

VCS 有兩大架構：**centralized（集中式，如 Subversion、Perforce、Google 的 Piper）** 與 **distributed（分散式 DVCS，如 Git、Mercurial）**。Git 是當今主流（「沒把握就用它」）。但本章強調：**所有現代 VCS 核心功能相同，差別在 UX；真正重要的是你往裡面放什麼、怎麼用它**——而「怎麼用」幾乎全是**政策與慣例**。最關鍵的政策概念是 **Source of Truth（單一事實來源）**：集中式把它內建進設計，DVCS 則需靠政策明訂「哪一個 repo 的哪一條 branch」是事實來源（否則退回「Presentation v5 - final - redlines - Josh's version v2」的混亂）。

本章的核心立場：**trunk-based development（主幹開發：一個 repo、沒有開發分支）是特別能擴展的政策**。長壽的 **dev branch（開發分支）** 被認為是「以為能換來穩定、其實是生產力拖累」的安全錯覺——小合併比大合併容易、由原作者合併比批次合併容易；正解是**重度依賴測試與 CI、保持 build 綠燈、用 runtime flag 關閉未完成功能**（DORA 研究證實 trunk-based 與高效組織強正相關）。**release branch（發布分支）** 則相對良性（會被拋棄、只做最小 cherry-pick）。Google 在 Piper monorepo（約 5 萬工程師、每工作日 6–7 萬次提交）上奉行 **One-Version Rule（單一版本規則）**：**工程師加依賴時，絕不能有「選哪個版本」的選擇**——否則 fork 會造成鑽石依賴、build 失敗或難解的執行期 bug。monorepo 讓 One Version「想違反都難」，但它**不是唯一正解**；重點不在 monorepo 本身，而在**盡可能堅守 One-Version**（manyrepo 或 VMR 也行）。

## 為什麼要讀這一篇

- **你還在開「這週合併策略怎麼喬」的會、有人被指派當「合併協調員」。** 本章直接點名這是**長壽 dev branch 造成的純開銷**：你以為分支換來穩定，其實只是把症狀（合併風險）拖慢、放大。正解是 **trunk-based development＋CI＋feature flag**——「branches are a drag on productivity」。
- **你的團隊為「該用哪個版本的某函式庫」吵過、或被鑽石依賴坑過。** 本章給你終結這類問題的政策：**One-Version Rule——加依賴時不該有選擇**。看懂 fork 為何危險（shading 能騙函式卻騙不了型別），你會明白「一致性的 choke point」價值何在。
- **你在猶豫 monorepo 還是 manyrepo、Git 還是別的。** 本章把這降級成「次要問題」：**真正重要的是堅守 One-Version 與 Source of Truth**，而非選哪個 repo 結構或 VCS。它也介紹 **VMR（虛擬 monorepo）** 這個「兩全其美」的方向。
- **你的新人覺得「回退到舊版本來 undo」是高成本的奢侈品。** 那是把「程式設計」誤當「軟體工程」。本章用「VCS＝管理原始碼 × 時間的主要工具」一句話，幫你向團隊說清楚為何版本控制不是可有可無。

## 細節重點

### What Is Version Control?（什麼是版本控制）
核心主張：VCS 追蹤檔案隨時間的版本（revisions），讓多位開發者同時在同一組檔案上工作。早期靠**一次一人鎖檔**來建立「哪個較新」的次序；更進階的系統確保「一次提交的多個檔案被當成單一單元」（**atomicity 原子性**）——沒有原子性的 CVS 會損毀、遺失變更。可把 VCS 想成擴充檔案系統的對映：**`VCS(filename, time, branch) => 檔案內容`**，把「時間」變成明確的一部分（在程式設計裡多餘，在軟體工程裡關鍵）。

工程意涵：對 VCS 的（少數）抗拒，幾乎直接來自**混淆了「程式設計」與「軟體工程」**——新人可能沒寫過「多人、跨數週」的程式，於是把「整個專案的 undo」看成多餘的高開銷奢侈品。但 VCS 真正的價值是**橋接單人與多人流程、讓團隊與組織能擴展**（即使我們很少真把它當 undo 鍵用）。它還移除了「哪個較新？」的問題、自動化易錯的「哪些變更已套用」追蹤，並順帶滿足稽核/法規（每一行每次變更的正式紀錄）與觸發提交前的反思時刻（跑 static analysis、檢查覆蓋率、跑測試——見 [[ch20]]）。

### Centralized vs Distributed VCS（集中式 vs 分散式）
核心主張：**所有現代 VCS 核心功能相同**（只要有「對一批檔案原子提交」的概念，其餘都是 UI）——爭論哪個「比較好」主要是 UX、命名、邊角功能與效能之爭。選 VCS 像選檔案系統格式：在夠現代的選項間差異不大，**真正重要的是你放什麼進去、怎麼用**。但架構大差異（集中 vs 分散）會讓設定、政策、擴展決策更容易或更難：
- **Centralized（集中式）：** 單一中央 repo（RCS 鎖檔 → CVS 批次、多人同時 checkout → Subversion 真原子提交）。
- **Distributed（DVCS）：** Git、Mercurial。核心差異是「**你能提交到哪？哪些副本算 repo？**」——有 clone/fork 就有可提交的 repo。中心性純屬概念與政策，非技術強制。DVCS 利於離線與協作、在開源世界採用尤廣。

工程意涵：**Git 是當今主流，沒把握就用它**（從眾有其價值）。但 Google 的主 repo 是自製的（巨大）集中式 VCS——遷往 Git 的嘗試被「程式碼庫與使用者規模」以及 Hyrum's Law 效應卡住（多數工具撐不住 5 萬工程師、數千萬次提交；DVCS 要傳整個歷史與 metadata，在此規模幾乎全是浪費）。**在 Google 的工作流裡，集中＋雲端儲存對擴展至關重要**（任一開發者其實只用到極小比例的檔案與版本，傳整份程式碼庫是浪費）。

### Source of Truth（單一事實來源）
核心主張：集中式 VCS 把「事實來源」**內建進設計**——trunk 最新提交就是當前版本。但 DVCS **沒有內建**哪份副本是唯一事實來源；理論上可讓各分支無協調地擴散，風險就是退回「Presentation v5 - final - redlines - Josh's version v2」的混亂。**管理良好的 DVCS 專案會明訂「某 repo 的某 branch」為事實來源**（GitHub/GitLab 的實務正是如此）。書中設想「沒有明確事實來源」的後果：怎麼確保發布 build 含過去幾週每個人開發的功能？什麼**次線性**的非中心化機制能做到？答案大概沒有——終究會有人去維護一份「哪些功能準備好了」的清單，**而那份記帳就是在重造一個集中式事實來源**。

工程意涵：Source of Truth 有**相對性**（同一專案，Google、RedHat、Linus 各有不同的事實來源——DVCS 在「組織與其事實來源是階層式且彼此不可見」時運作良好）。但只要在組織內**沒有「該推去哪」的選擇或不確定**，就能避開一大類混亂的擴展問題。技術只是其中一部分；任一組織幾乎總有等量的**政策與慣例**疊在上面——而分支管理是政策與慣例最多的主題。

### Branch Management（分支管理）
核心主張：
- **WIP ≈ 一個分支：** 組織裡每一份進行中的工作，等同於一個分支（DVCS 尤其明顯；集中式裡未提交的本地變更與分支上的提交概念上無異）。這在重構時特別相關——「把 Widget 改名為 OldWidget」依政策不同有數種解讀（只改 trunk？改所有分支？連別人未提交的變更也改？）。
- **Dev Branches（開發分支）：** 在「一致單元測試」普及前，怕任何變更回歸破壞他處，故對 trunk 特別小心、用功能開發分支。**本章認為「重度用 dev branch 換產品穩定」根本是誤導**——同一批 commit 終究要合併進 trunk；**小合併比大合併容易、由原作者合併比批次合併容易**；長壽分支讓回歸更難隔離與修復。組織如何上癮：看到「合併長壽分支降低了穩定性」就結論「分支合併有風險」，於是放慢、協調症狀（合併），愈來愈多分支、愈來愈多協調，某個倒楣工程師變成「合併協調員」、開每週合併策略會——**這些全是純開銷**。正解：**trunk-based development、重度依賴測試與 CI、保持 build 綠燈、用 runtime flag 關閉未完成/未測功能**。
- **Release Branches（發布分支）：** 若發布週期長於幾小時，建一條代表「進入發布 build 的確切程式」的發布分支是合理的；發現關鍵缺陷可從 trunk **cherry-pick（最小、針對性的合併）** 過去。**相對良性**——與 dev branch 的差別在預期終局：dev branch 預期合回 trunk，release branch 預期**最終被拋棄**。達成持續部署（CD，一天從 trunk 發布多次）的高效組織幾乎不用 release branch（直接加修正再重新部署更容易）。

### Version Control at Google：Piper、One Version、Monorepo
核心主張：Google 絕大多數原始碼在**單一 monorepo**、約 5 萬工程師共享，跑在自製集中式 VCS **Piper**（作為分散式微服務、80TB+ 內容與 metadata、每工作日 **6–7 萬次提交**、建 client＋加檔＋提交約 **15 秒**）。因 Piper 自製，可內建並強制源碼政策——如**階層式 `OWNERS`**（檔案樹每層列出可核准提交者，見 [[ch09]]）。

- **One-Version Rule（單一版本規則）：** 在 Single Source of Truth 之上再加一層——**「對程式碼庫裡的每個依賴，只能有一個版本可選」**。第三方套件：穩態下只有單一版本 checked in；內部套件：不可 fork 而不重新命名/打包。**根本規則：工程師加新依賴時，絕不能有「該依賴哪個版本」的選擇。** 對個人像是任意的阻礙，但對組織是高效擴展的關鍵。
  - **Fork 的危險（多版本場景）：** 某團隊發現 Abseil/Guava 的 bug，不就地修而是 fork、又不改名——若某專案同時依賴原版與 fork，最好的情況是 build 失敗、最壞是難解的執行期 bug（連入兩個不匹配的同名庫）。這替程式碼庫加上「分區/著色」屬性，使「加一個新依賴」可能得跑全庫測試。**Shading（Java 把內部依賴改名以隱藏）對函式技術上可行，對「會跨套件傳遞的型別」則理論與實務皆失敗**——任何「同 binary 多版本共存」的技倆都只能救函式、救不了 vocabulary type。
- **(幾乎)沒有長壽分支：** dev branch 應極少、極短命（呼應 Agile、DORA trunk-based、Phoenix Project「減少在製品 WIP」）。新 Widget 該**提交到 trunk、runtime 關閉、用 visibility 對其他人隱藏**，而非活在平行分支讓「選擇」偷溜回來。Google 約 1000 個團隊中，只有不到 10 個有長壽 dev branch（多為「跨時間相容性」這種不尋常需求，如資料格式或 API 相容）；且有 **build horizon**：每個 production job 最多每 6 個月就要重建重部署，封頂了版本偏移。
- **Monorepo：** One Version 在 monorepo 裡**想違反都難**（無需決定哪個版本「官方」、哪個 repo 重要；一致性放大新工具的影響）。但**不是唯一正解**——開源世界用 manyrepo 也運作良好。重點不在 monorepo，而在**盡可能堅守 One-Version**。工具正逐漸提供「混合細粒度 repo 與 monorepo」的體驗（Git submodules、Bazel 外部依賴、CMake subprojects），可合成**接近 monorepo 的 VMR（virtual monorepo，虛擬 monorepo）**——若各專案保密/法規需求一致，真 monorepo 很好；否則追求 monorepo 的「功能」、但用更彈性的方式實現。

### Future of Version Control（版本控制的未來）
核心主張：Microsoft、Facebook、Netflix、Uber 都公開談過 monorepo；DORA 也大量發表。反對 monorepo 的論點多是技術限制（大 repo 慢）——但 Git 正持續改進（shallow clone、sparse branch），「repo 要小」的重要性會遞減。另一反論「不合開源世界」也成立，但開源的做法源於「優先自由、缺協調、缺算力」，而組織內可假設算力、協調與某種集中權威。作者預測未來 10–20 年：VCS 既支援更大 repo、也提供更好的「跨專案/組織縫合」機制；有人（可能是套件管理或 Linux 發行版）會催生 de facto 的 VMR 標準。一句收束：**版本號其實是時間戳；允許版本偏移就是加進了「時間」這個額外的維度複雜度，代價高昂——而我們能學會避免它。一切始於某種邏輯上像 monorepo 的東西。**

## 關鍵字補充

- **Version Control System（VCS，版本控制系統）** — 追蹤檔案隨時間的版本；可視為 `VCS(filename, time, branch) => 內容` 的擴充檔案系統。工程師管理「原始碼 × 時間」的主要工具。
- **Centralized vs Distributed（集中式 vs 分散式）** — 集中式（Subversion/Perforce/Piper）有單一中央 repo；DVCS（Git/Mercurial）任一 clone 都是可提交的 repo。核心功能相同，差別在 UX。
- **Source of Truth（單一事實來源）** — 哪個 repo 的哪條 branch 是權威版本；集中式內建、DVCS 需靠政策明訂；有相對性（不同組織可不同）。
- **Trunk-based Development（主幹開發）** — 一個 repo、沒有（長壽）開發分支；小增量直接提交 trunk＋CI＋feature flag。DORA 證實與高效組織強正相關。
- **Dev Branch（開發分支）** — 長壽功能分支；以為換來穩定、實為生產力拖累（大合併難、難隔離回歸）。本章認為應極少、極短命。
- **Release Branch（發布分支）** — 代表進入發布 build 的確切程式；相對良性、預期被拋棄、只做最小 cherry-pick；CD 高效組織幾乎不用。
- **One-Version Rule（單一版本規則）** — 加依賴時不該有「選哪個版本」的選擇；每個依賴只能有一個版本可選。對組織高效擴展至關重要。
- **Fork without renaming（不改名的分叉）** — 違反 One Version 的典型；造成鑽石依賴、build 失敗或執行期 bug（見 [[ch21]]）。
- **Shading（遮蔽）** — Java 把內部依賴改名以隱藏；對函式可行，對跨套件傳遞的型別失敗——只是在 patch「需要多版本」的底層問題。
- **Monorepo / VMR（單體儲存庫／虛擬 monorepo）** — monorepo 讓 One Version 想違反都難但非唯一正解；VMR 用 Git submodules/Bazel/CMake 縫合細粒度 repo，逼近 monorepo 的功能。
- **Piper** — Google 自製集中式 VCS，作為分散式微服務、80TB+、約 5 萬工程師、每工作日 6–7 萬次提交。
- **Build Horizon（建置地平線）** — 每個 production job 最多每 6 個月重建重部署；封頂了潛在的版本偏移。

## 延伸提問

1. 你的團隊是不是在開「合併策略會」、有「合併協調員」、或維護長壽 dev branch？把這些開銷攤出來——若改成 **trunk-based development＋CI＋feature flag**，哪些會消失？你怕的「trunk 不穩」能不能用測試而非分支來解？對照 [[ch23]]。
2. 你的程式碼庫裡，工程師加依賴時**有沒有「選哪個版本」的選擇**？有沒有過 fork 不改名、或鑽石依賴的痛？用 **One-Version Rule** 檢查：哪些第三方/內部套件其實存在多版本、是顆未爆彈？對照 [[ch21]]。
3. 你們有明確的 **Source of Truth**（某 repo 的某 branch）嗎？還是「pull 完同事的 repo 也不確定有沒有拿到所有變更」？新人從哪裡拿到一份「已知良好」的程式？這個機制隨團隊長大還能**次線性**運作嗎？
4. 你在選 monorepo 還是 manyrepo、Git 還是別的——本章說這是**次要問題**。把焦點移回「能否堅守 One-Version 與 Source of Truth」：你目前的 repo 結構幫你還是害你做到這件事？**VMR** 對你是不是兩全其美？
5. 你的團隊把版本控制當「整個專案的 undo 鍵」這種奢侈品，還是「管理原始碼 × 時間的主要工具」？有沒有人把「軟體開發（坐下寫程式）」誤當「軟體工程（讓程式長期可用）」？這個心智模型差異，正是 VCS 價值的根源（回扣 [[ch01]]）。

## 跨章連結

- [[ch01]] What Is Software Engineering? — 「軟體工程＝整合時間維度的程式設計」是 VCS 為何重要的根本；Hyrum's Law 也卡住了 Google 遷往 Git。
- [[ch21]] Dependency Management — One-Version Rule、fork 的鑽石依賴、shading 的限制都在第 21 章深入；版本控制與依賴管理「概念高度相似」。
- [[ch23]] Continuous Integration — trunk-based development 的前提是「重度依賴測試與 CI、保持 build 綠燈」；Kythe/TAP 都建立在 monorepo 之上。
- [[ch22]] Large-Scale Changes — 「broad/shallow 全庫變更」在 trunk-based＋monorepo 上才可行；長壽分支會是 LSC 的沉重負擔。
- [[ch09]] Code Review — 階層式 `OWNERS` 由 Piper 在提交時強制；版本控制的提交時刻是 review 與 static analysis 的觸發點。
- [[ch24]] Continuous Delivery — CD（一天從 trunk 發布多次）讓 release branch 幾乎變得多餘；release branch 的取捨直接連到 CD。
- [[ch18]] Build Systems and Build Philosophy — 「分散式可重現的 build 系統比傳整份程式碼庫更能擴展」「VMR 靠 Bazel 外部依賴縫合」直接連到建置系統。
