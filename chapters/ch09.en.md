# Chapter 9 — Code Review

> *"Code review acts as the glue connecting engineers with one another, and the code review process is the primary developer workflow upon which almost all other processes must hang."*
> Code review is the connective tissue: testing, static analysis, and CI all hang off it.

## Summary

Code review is "a process in which code is reviewed by someone other than the author, often before the introduction of that code into a codebase." Industry practice varies enormously; Google's version is unusually thorough: **essentially every change, no matter how small, is reviewed before being committed (a precommit review), and every engineer both initiates reviews and reviews others' changes.** This chapter is about the *process*, not the tool — the tool, Critique, gets its own chapter ([[ch19]]).

The end goal of a review is another engineer's **LGTM ("looks good to me")**. But Google splits "approval" into **three independent permission bits**: ① an **LGTM** from another engineer (the code is correct and comprehensible); ② **approval from a code owner** (this code is appropriate for this part of the codebase); ③ approval from someone with language **readability** (it conforms to the language's style and best practices). One person often plays all three roles, but keeping them separable is what makes the process **flexible *and* able to scale** — the short answer to "why split them" is literally *scale*.

Review yields six benefits: **correctness, comprehension, consistency, the psychological sense of collective ownership, knowledge sharing, and a historical record.** Strikingly, the book says correctness is **not** the primary benefit Google accrues — whether code stays *understandable and maintainable* over time matters more, because **code is read far more than it is written.** Finally, the best practices that let the process scale: **be polite and professional, write small changes (~200 lines), write good change descriptions, keep reviewers to a minimum (usually exactly one), and automate where possible (presubmits).** One root idea runs through it all: **code is a liability.**

## Why read this

- **You treat code review as "the bug-catching gate."** This chapter flips that: catching correctness is **not** the benefit Google values most. The real long-term payoff is **comprehension, consistency, knowledge sharing, and a historical record** — all in service of *time* and *scale*. Internalize that and the way you write review comments changes.
- **Your reviews are slow and painful and everyone dreads them.** The chapter pins the "slow" on the chosen *implementation*, not on review itself, and gives directly actionable fixes: **write small changes** (~200 lines; ~35% of changes touch a single file), **give feedback within 24 working hours**, **don't review piecemeal**, and **keep reviewers to one** ("the most important LGTM is the first one; subsequent ones add less than you think").
- **You want to know what problems Google's "ownership" and "readability" institutions actually solve.** The answer is **scale**: splitting LGTM / owner / readability into three bits lets an intern and a tech lead run the same process by different paths; `OWNERS` files let "who's responsible for this" be answered by walking up the directory tree, with no central registry.
- **You're unsure how hard to review different kinds of change.** Greenfield (brand-new code) gets the strictest look; a bug fix stays laser-focused (don't fix unrelated things); a machine-generated LSC should only get comments specific to *your* code (you can't veto the whole process). Putting effort where it belongs is how review scales.

## Key details

### Code Review Flow
The core claim: Google does **precommit review** — code is reviewed before it can be committed. A typical flow has six steps: ① the author writes a change, uploads a snapshot (a patch + description), producing a **diff** against the codebase; ② the author may self-review or apply automated comments, then **mails** the change to one or more reviewers; ③ reviewers comment on the diff (some comments require resolution, some are merely informational/FYI); ④ the author revises, uploads new snapshots, and replies — steps ③④ may repeat; ⑤ when happy, reviewers mark **LGTM** (only one is required by default); ⑥ after LGTM, all comments resolved, and approval, the author may commit.

Takeaway: **LGTM is a *necessary* permission "bit," but not the only one** — it combines with the owner and readability bits below before a commit is allowed.

### Code Is a Liability
The core claim: **code itself is a liability, not an asset.** Even a necessary liability is, by itself, a maintenance task for someone somewhere down the line — like the fuel an airplane carries: it has weight, though flight needs it. New features are often necessary, but **take care before writing code at all to ensure the feature is warranted.** Duplicated code isn't just wasted effort — it can cost *more* time than not having the code at all. As some put it: "**If you're writing it from scratch, you're doing it wrong!**"

Takeaway: a review is **not** the occasion to rehash earlier design decisions. Those belong in design / API reviews. A code review of brand-new code shouldn't come out of the blue (it should have been communicated beforehand), and review shouldn't be treated as a chance to relitigate the past.

### How Code Review Works at Google: the three approval bits
The core claim: three aspects require "approval" for any change at Google:
- **A correctness & comprehension check (the LGTM bit):** another engineer confirms the code is appropriate and does what the author claims. Often a teammate, but it needn't be.
- **Approval from a code owner (the ownership bit):** an owner of the directory confirms the code is appropriate for this part of the codebase. Google's codebase is a **tree with hierarchical owners of directories** (see [[ch16]]); owners are gatekeepers for their directories. This bit can be implicit if the author is an owner.
- **Approval from someone with language readability (the readability bit):** an engineer with that language's readability confirms it follows the language's style and best practices. These come from a **company-wide pool** of readability-certified engineers. Implicit if the author has readability.

The key: **the three bits combine in any combination.** A tech lead who owns a project *and* has readability needs only one other engineer's LGTM to submit; an intern without those rights sending the same change needs approval from an owner who has readability. Most reviews needing more than one approval go **two steps**: get a peer's LGTM (focused on correctness and general validity), then seek owner/readability approval (the owner cares more about "Will this be hard to maintain? Does it add technical debt? Do we have the expertise to maintain it?").

Takeaway: **Why not just have one all-powerful reviewer do everything? The short answer is scale.** Separating the roles adds flexibility, lets different people focus on different aspects, and saves review time — it's the design that lets tens of thousands of engineers operate on billions of lines in one repository.

### Ownership (by Hyrum Wright)
The core claim: small teams often grant everyone access to everything in a repo; as the team grows, that **fails to scale.** Google records "who has what knowledge and responsibility for which part of the repo" via **ownership.** **This isn't *possession* of source code; it's a sense of *stewardship* — acting in the company's best interest over a section of the codebase** (the author admits "stewards" would have been the better word).

Mechanism: specially named **`OWNERS` files** list the usernames responsible for a directory and its children. They may reference other `OWNERS` files or external ACLs but ultimately resolve to a list of individuals. Each subdirectory may have its own `OWNERS`, and the relationship is **hierarchically additive**: a file is generally owned by the *union* of all `OWNERS` members above it in the tree. Keep lists **small and focused** so responsibility is clear.

Takeaway: this distributed ownership underpins many other practices — the root `OWNERS` set can act as **global approvers** for large-scale changes ([[ch22]]) without bothering local teams; `OWNERS` files double as **documentation** (walk up the tree to find who's responsible); new projects need no central registry — **a new `OWNERS` file suffices.** Simple yet powerful, it has scaled well for two decades.

### Code Review Benefits (the six)
The core claim: a well-designed review process, taken seriously, provides six benefits:

- **Code correctness:** a second set of eyes confirms the change does what was intended — checking tests, design, correctness, and efficiency. An IBM study found **finding defects earlier costs less to fix later** — *provided* the process stays **lightweight**; heavyweight, non-scaling review becomes unsustainable. To keep correctness objective, **authors get deference on their approach** — reviewers shouldn't propose alternatives from personal opinion, only when they improve **comprehension (simpler) or functionality (more efficient).** Engineers are encouraged to **approve changes that improve the codebase rather than wait for consensus on a more "perfect" solution.** As static analysis and automated testing strengthen, many correctness checks are automated (part of a "shift left" strategy, see [[ch20]]) — but tooling has limits and hasn't fully replaced human eyes. **Surprisingly, correctness is not the primary benefit Google accrues.**
- **Comprehension of code:** review is usually the first time someone *other than the author* inspects a change, giving feedback **unbiased by the author's perspective** — something even the best engineer can't do for themselves. Since **code is read far more than written**, comprehension is critical. Give authors deference on *design*, but treat *comprehension* questions by the maxim "**the customer is always right**": every question now will be multiplied many-fold over time, so treat each as valid — you may not need to change your logic, but you may need to **explain it more clearly.** Correctness + comprehension are the two criteria for an LGTM.
- **Code consistency:** at scale, your code will be depended on, maintained, and refactored by others (and by tools). It must meet consistency standards to be understood and maintained, and should **avoid being overly complex.** Consistent, simple code is easier for tools to update during refactoring — "if a pattern is always done one way, it's easier to write a tool to refactor it." This is why **the LGTM state (correctness + comprehension) is separated from readability approval**: readability can only be granted by those who passed that language's readability training (Java code needs a "Java readability" approver). Consistency sometimes clashes with functionality — a readability reviewer may prefer a less-complex change that's easier to understand even if not functionally "better."
- **Psychological and cultural benefits:** review reinforces that **code is not "yours" but part of a collective enterprise.** It forces authors to accept input and compromise for the greater good. It's human to be proud of one's craft and reticent about criticism; the process provides a **buffer**, turning a potentially charged interaction into a **prescribed, neutral** one — "the process is the bad cop, so the reviewer can stay the good cop" (the tool is even called "Critique" — you can't fault a reviewer for being critical). Another benefit is **validation**: even capable engineers suffer imposter syndrome, and review recognizes their work. And **initiating a review makes authors take extra care** — getting "all your ducks in a row"; that moment of reflection before sending is the perfect time to catch what you missed (no more "I'll add the unit tests later").
- **Knowledge sharing:** the **most important yet most underrated** benefit. Authors usually pick expert reviewers, who impart domain knowledge (even sharing suggested edits in the tool). Information flows **both ways** — author and reviewer both learn. "**An engineer may not read every email sent to them, but they tend to respond to every code review sent.**" This transfer crosses time zones and projects, timely and actionable (many engineers first "meet" colleagues through reviews).
- **Historical record:** because each change joins the codebase, review is a **historical record** — anyone can do "code archeology" to find when a pattern was introduced and pull up the original review, often informing far more people than the original author and reviewer.

### Code Review Best Practices
The core claim: review can add friction and delay, but that's **usually the chosen implementation, not review per se.** Keep it **nimble and quick** to scale properly.

- **Be polite and professional:** built on Google's culture of trust and respect; keep all feedback **strictly professional.** Reviewers should **defer to authors' approaches**, flagging alternatives only when the approach is deficient; if several are equally valid, accept the author's preference. Give feedback within **24 working hours**; if you can't finish, at least say you've seen it. **Don't review piecemeal** (nothing annoys an author more than addressing feedback only to get unrelated new comments). Authors must be professional too: "**you are not your code**," and this change isn't "yours" but the team's. Treat each comment as a **TODO** — you needn't accept it, but you must address it; if you disagree, say why and don't mark it resolved unilaterally; use **PTAL ("please take another look")** to keep debates civil.
- **Write small changes:** **probably the most important practice** for keeping review nimble. A review should focus on a single issue and be easy to digest. Google **discourages massive changes**, and reviewers can reject ones too large. Small changes also avoid waiting on big reviews and make bugs easier to localize. **"Small" is about 200 lines of code.** Most changes get **initial feedback within ~a day** (not "review done in a day"); **~35% of changes touch a single file.** The cost: small changes can be hard to reconcile with major features (techniques like integration branches exist, with overhead) — treat "prefer small" as an **optimization**, allowing occasional larger changes. Because changes are small, **almost all reviews are done by exactly one person** — if a whole team had to weigh in, it couldn't scale.
- **Write good change descriptions:** the **first line is prime real estate** — it's the summary in the tool, the email subject, and the line seen in Code Search history ([[ch17]]). The description should say **what changed and why**; "Bug fix" helps no reviewer or future archeologist. The description is the historical record. Comment liberally in the implementation too — **if a reviewer doesn't understand why you did something (even if correct), that signals a need for better structure or comments.**
- **Keep reviewers to a minimum:** most Google reviews have **exactly one reviewer.** There's a tendency to seek a cross-section's unanimous consent, but it hits **diminishing returns** — "**the most important LGTM is the first one, and subsequent ones don't add as much as you might think; the cost of additional reviewers quickly outweighs their value.**" When multiple reviewers are truly needed, each should **focus on a different aspect.**
- **Automate where possible:** human input matters, but automate the mechanical parts — tooling investments pay dividends. The biggest advance is **automatic static analysis** of changes ([[ch20]]) via **presubmits**: run when a change is first sent, they catch problems *before* the change reaches a reviewer (avoiding an awkward email), letting reviewers focus on things more important than formatting.

### Types of Code Reviews
The core claim: **not all reviews are alike** — different types need different focus. Google's changes fall into four buckets:
- **Greenfield (brand-new code):** least common, most important to review hard — the best time to judge whether code will **stand the test of time and scale.** New code shouldn't surprise (it should have had a design review). Ensure the **API matches an agreed design, is fully tested (every endpoint has tests that fail when assumptions change), has proper owners (one of the first reviews in a new project is often a single `OWNERS` file), is well-commented and documented, and enters CI ([[ch23]]) if needed.**
- **Behavioral changes, improvements, optimizations:** most Google changes. Apply the greenfield criteria: **is this necessary, and does it improve the codebase?** (**Some of the best changes are deletions!** Removing dead/obsolete code is among the best ways to improve code health.) Behavioral changes need revised tests; optimizations may need **performance benchmarks.**
- **Bug fixes and rollbacks:** when fixing a bug, **resist the temptation to address other issues** — that bloats the review, hinders regression testing, and makes your change harder to roll back. A bug fix should **fix only the indicated bug** and usually update a test to catch it. A **rollback** is a change that undoes a prior change to a known state — created in seconds but **still requires review.** Hence **all changes should be as small and atomic as possible** — Google has seen developers depend on new code very quickly, so a rollback can break them; small changes mitigate this.
- **Refactorings and large-scale changes (LSCs):** many Google changes are **machine-generated** (the author is a machine, see [[ch22]]) — but **machine-generated changes still require review.** Low-risk ones go to designated reviewers with codebase-wide approval; riskier ones needing local expertise go to individual engineers. When reviewing machine changes, **only flag concerns specific to *your* code — not the underlying tool or LSC process** (that process was already reviewed; individual teams can't veto it, or it couldn't scale; escalate concerns to the LSC oversight group instead). Also **don't expand scope** — the human running the tool may have hundreds of changes in flight, and even small percentages of unrelated comments limit the scale they can operate at.

## Key terms

- **Precommit review** — code must be reviewed before entering the codebase. Google mandates it for almost every change regardless of size — one of its few blanket processes.
- **LGTM ("looks good to me")** — a reviewer's agreement on correctness + comprehension; a necessary permission bit. Only one LGTM is required by default.
- **The three approval bits** — LGTM (correctness/comprehension), owner approval (fit for this part of the codebase), readability (conforms to language style). Played by one or more people, combined freely; split for flexibility and scale.
- **OWNERS file** — lists the people responsible for a directory and its children; hierarchically additive, resolving to individuals. Answers "who's responsible" by walking the tree; a new project just adds one — no central registry.
- **Ownership / stewardship** — not *possession* of code but responsibility to steward a section of the codebase in the company's interest (the author admits "stewards" was the better word).
- **Code is a liability** — code is a maintenance cost, not an asset; confirm a feature is warranted first, and duplicated code can be worse than none ("writing from scratch, you're doing it wrong").
- **Readability** — a certification, earned via that language's readability training, to approve that code follows the language's style and best practices; separated from LGTM, dedicated to consistency and maintainability (see [[ch03]], [[ch08]]).
- **Small changes** — the most important practice for nimble review, bounded at ~200 lines; most get initial feedback within a day, ~35% touch one file; what makes one-reviewer review and easy rollback possible.
- **PTAL ("please take another look")** — a polite cue from an author who disagrees, offering an alternative and asking the reviewer to look again — keeps debates civil.
- **Presubmit** — automated static-analysis/format checks run when a change is sent, catching problems before they reach a reviewer (see [[ch20]]).
- **Rollback** — a change that undoes a prior one to a known state; created in seconds but still requires review. Hence all changes should be small and atomic to avoid cascading breakage.

## Questions to push your thinking

1. Does your team pin code review's value mostly on "catching bugs"? If Google says correctness is **not** the primary benefit, are you taking **comprehension, consistency, knowledge sharing, and the historical record** seriously? Next time you write a comment, will you also ask "can someone read this in six months"?
2. Measure the line count of your recent changes: how many exceed ~200 lines? Which could be **split into several small, atomic changes**? Recall a time follow-on work stalled waiting on a big review — would small changes have cured it?
3. How many approvals do your changes need to land? Against "**the most important LGTM is the first one**": are you buying reassurance with "a group's unanimous consent" at the cost of delay and diminishing returns? Which changes really need only one reviewer?
4. Apply the **three approval bits** to your team: do you lump "correctness," "ownership of this codebase," and "language-style consistency" into one review by one kind of person? Would splitting them ease the path for both interns and seniors? Are your `OWNERS` clear?
5. Do you "while I'm here" fix unrelated things in a bug fix? Recall the pain of a change that was then **hard to roll back or regression-test.** Can you build the discipline of "fix only this bug + add a test that catches it"? Against "code is a liability" — in your last review, did you consider that **the best change might be a deletion**?
