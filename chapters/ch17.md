# 第 17 章｜Code Search（程式碼搜尋）

> *"Understanding code is key to developing and maintaining it, and this means that investing in understanding code will yield dividends."*
> 「理解程式碼，是開發與維護它的關鍵；這意味著，投資於『讓人理解程式碼』會帶來回報。」

## 摘要

**Code Search** 是 Google 用來瀏覽與搜尋程式碼的工具（前端 UI ＋多個後端）。它和許多 Google 開發工具一樣，**直接源於「擴展到程式碼庫規模」的需求**——起初是「內部 grep 工具」＋「外部 Code Search 的排序與 UI」，再因整合 **Kythe**（提供 cross-reference 交叉引用、跳到符號定義）而從「搜尋」轉向「瀏覽」，並以一條原則導引後續開發：**「用一次點擊回答關於程式碼的下一個問題」**。如今「這符號在哪定義？哪裡用到？怎麼 include？何時加入？甚至全機隊耗多少 CPU cycle？」都能一兩下點到。與 IDE 不同，**Code Search 為「大規模地讀、理解、探索程式碼」而最佳化**，重度依賴雲端後端。

研究顯示工程師用它在回答關於程式碼的問題，意圖可歸為五類：**How（約 1/3，看別人怎麼做的範例）、What（約 1/4，瀏覽某部分在做什麼）、Where（16%，某資訊在哪）、Why（16%，為何這樣行為，常用於除錯）、Who/When（8%，誰何時加的）**。

**為什麼要另做一個 web 工具而非用 IDE？** ①**Scale（規模）**：整份程式碼庫塞不進單機；集中式索引「**做一次、人人受惠**」、隨每次提交**增量更新（線性成本）**，而「每個開發者各自在 IDE 建索引」是**二次方**擴展。②**零設定的全域程式視野**。③**特化（不是 IDE 反而是優點）**：UX 為瀏覽而非編輯最佳化，每次點擊都有意義。④**與其他工具整合**：Code Search 連結是程式的 canonical「位置」（log viewer、stack frame、文件都連回它）。實作上有兩個索引：**即時搜尋索引（增量、中位 < 10 秒可見）** 與 **每日的 cross-reference 索引（Kythe，需近乎全建置、無法增量）**。

關鍵數字與取捨：每天 **逾百萬次查詢**；**每次多 1 秒 ≈ 每天 35 個全職工程師閒置**，故鎖定 **sub-200ms**；索引 1.5TB 內容、200 qps、中位搜尋 < 50ms（暴力解需 30 萬核）；索引從 trigram → suffix array → **sparse n-gram（比暴力解快 500 倍）**。取捨圍繞 **completeness（完整性）vs 速度/資源/信任**：可丟掉二進位/產生的 JS，但**寧可索引太多以維持工程師的信任**；用 priority 分片同時服務「最相關」與「全部結果」；索引 Piper 線性歷史以支援「回到某時間點」與 workspace；token vs substring vs regex 的表達力取捨。

## 為什麼要讀這一篇

- **你以為「程式碼搜尋」不就是 grep ＋ IDE 索引？** 本章說明在 Google 規模下為何需要專用工具：整份程式碼庫**塞不進單機**、IDE 各自建索引是**二次方**擴展。看懂「集中式索引做一次、人人受惠」，你會重新評估你團隊的搜尋/索引基礎設施。
- **你想知道工程師到底在搜尋什麼。** 本章用實證把意圖拆成五類（**How/What/Where/Why/Who-When**）——最大宗是「**看別人怎麼做的範例（約 1/3）**」。這提醒你：理解程式（而非只是找到它）才是生產力的核心，值得投資。
- **你在為互動式工具設效能目標。** 本章給你一把可量化的尺：**每次查詢多 1 秒 ≈ 每天 35 個工程師閒置**、**< 200ms 才算 responsive、1 秒注意力開始飄、10 秒就 context switch**。這把「延遲」直接換算成「工程時間」，幫你向上爭取後端投資。
- **你在做一個會被全公司用的開發工具。** Code Search 的成功心法：**成為其他工具的平台與 canonical「位置」**（log/stack frame/文件都連回它）、**讓人知道它存在（納入 Noogler 訓練）**、以及**先有「夠用但有限」的版本也能被信任使用**——只要限制被理解。

## 細節重點

### The Code Search UI（UI）
核心主張：搜尋框是核心，像 web search 有「建議（suggestions）」可快速導航到檔案/符號/目錄；複雜情況回傳含程式片段的結果頁。搜尋本身像即時的「find in files（grep）」，但加上**相關性排序**與程式特有的增強（語法高亮、scope 感知、認得註解與字串字面量）。也提供 **CLI 與 RPC API**（需後處理或結果集太大時很有用）。

工程意涵：看單一檔案時，**多數 token 可點擊**（由 Kythe 等編譯器型索引驅動）——函式呼叫連到定義、import 連到原始檔、註解裡的 bug ID 連到 bug 報告；點符號名打開「所有使用處」面板；hover 區域變數高亮所有出現。透過與 Piper 整合（[[ch16]]）還能看檔案歷史、各版本 diff、跳到 Critique（[[ch19]]）、經典的 blame，甚至從目錄視圖看被刪除的檔案。（一個良性循環：無所不在的 code browser **鼓勵人寫出易瀏覽的程式**——別把階層巢得太深、用具名型別而非 string/int。）

### How Do Googlers Use Code Search?（工程師怎麼用）
核心主張：用它回答關於程式碼的問題，意圖可歸五類（依 Sadowski 等人的研究）：
- **How（約 1/3，最大宗）：** 看別人怎麼做的**範例**（已找到某 API，想看它該怎麼套用、怎麼穩健處理錯誤；或先找對的函式庫再挑最合適的實作）。典型用「搜尋＋交叉引用瀏覽」組合。
- **What（約 1/4）：** 經典檔案瀏覽——某部分在**做什麼**（較探索性，為了在改動前/讀懂別人改動前先理解程式）。靠 call hierarchy 瀏覽與相關檔案間快速導航（header/實作/測試/build 檔）。
- **Where（16%）：** 某資訊（函式定義、設定、API 所有用處、檔案位置）**在哪**——很精準，可用查詢或語意連結（jump to definition）回答；常見於重構/清理或協作。靠**排序**處理常見情況、**豐富查詢語言**處理罕見情況。UI 也讓**分享結果連結**很容易（code review、文件、bug、postmortem 引用程式的 canonical 方式；連舊版本都可引用，連結隨程式演進仍有效）。
- **Why（16%）：** 為何某段程式被加、或為何如此行為——常見於除錯。關鍵能力是**搜尋與探索「某時間點的確切程式狀態」**（除生產問題可能要看數週/數月前的狀態，除測試失敗可能只是幾分鐘前的變更）。
- **Who/When（8%）：** 誰、何時引入（與版本控制互動，像 Git blame、跳到對應 review；也用來找「最該問的人」）。

### Why a Separate Web Tool?（為何另做 web 工具，而非 IDE）
核心主張：IDE 外多數人在本地做這些事，為何還要另一個工具？
- **Scale（規模）：** Google 程式碼庫**塞不進單機**（IDE 的前提）；即使沒到這障礙，「每個開發者各自建本地搜尋與交叉引用索引」也有成本（常在 IDE 啟動時付、拖慢速度）。**集中式索引＝做一次、投資人人受惠**，且**隨每次提交增量更新（線性成本）**——相對地「每人各自在 IDE 建索引」大致**二次方**擴展（程式碼線性成長 × 線性個 IDE 各自做線性的工）。但 **cross-reference 索引無法增量**（任何變更都可能影響整個程式碼庫、實務上常影響數千檔），需近乎全建置，故 Kythe **每天**才建一次（Chromium 約 6 小時）——即時搜尋索引與每日交叉引用索引的落差是少數但反覆出現的困擾來源。
- **Zero Setup Global Code View（零設定的全域程式視野）：** 能即時瀏覽整個程式碼庫 →易找可重用的函式庫與好範例；IDE 因啟動建索引而有「縮小專案範圍」的壓力，Code Search 無需設定、學任何程式都快，也不會漏掉程式相依。
- **Specialization（特化——不是 IDE 反而是優點）：** UX 可為**瀏覽與理解**最佳化而非編輯——因為沒有編輯游標，**每次點擊都能有意義**（顯示所有使用處、跳到定義），而非只是移動游標。故開發者常同時開多個 Code Search 分頁與編輯器。
- **Integration（與其他工具整合）：** 它是看原始碼的主要方式，故是「曝光程式碼資訊」的合理平台（許多全庫分析的結果都浮現在它上面，如標記 dead code）。反向地，**Code Search 連結是程式的 canonical「位置」**——log viewer、crash 的 stack frame、編譯錯誤/測試、codelab/文件都連回它（連未提交的 workspace 程式都可），連結因可綁定特定 revision 而長期有效。也**對外暴露 search/cross-ref/syntax-highlight 的 API**，並有 vim/emacs/IntelliJ 外掛。

### Impact of Scale on Design（規模對設計的衝擊）
核心主張：搜尋程式碼最大的擴展挑戰是 **corpus size（語料庫大小）**——幾 MB 用 grep 暴力解就行，幾百 MB 用本地索引快一個數量級，到 GB/TB 級就得用多機雲端方案。
- **Search Query Latency（查詢延遲）：** 快是好的，但低延遲不免費——可用「省下的全體工程時間」來證成投入。Google 每天**逾百萬次查詢**：每次多 1 秒 ≈ **每天 35 個全職工程師閒置**；而後端用約其 1/10 的人就能建與維護——故約**每天 10 萬次查詢（< 5000 開發者）就是「1 秒延遲」論點的損益兩平點**。且生產力損失非線性：**< 200ms 才算 responsive、1 秒後注意力開始飄、再 10 秒就完全 context switch**——故對所有頻繁操作鎖定 **sub-200ms 端到端**。
- **Index Latency（索引延遲）：** 多數時候開發者不在意索引過時；但對「自己剛寫/審的變更」，不同步會造成大量困惑——寫程式時**期待即時索引**（新檔/函式/類別找不到很挫折）；搜尋取代式重構也需要被移除的程式立即從結果消失。除錯生產事故時，索引與執行中程式的落差尤其麻煩（可能藏住真因）。但建交叉引用索引在 Google 規模需數小時、且只保留一個「版本」。

### Google's Implementation（Google 的實作）
核心主張：早期用 trigram（Russ Cox 後來開源了簡化版）；目前索引約 **1.5 TB 內容、200 qps、中位伺服端搜尋 < 50ms、中位索引延遲（提交到可見）< 10 秒**。粗估暴力解：RE2（regex 庫）在 RAM 約 100MB/秒，50ms 內處理 1.5TB 需 **30 萬核**；換 substring（約 1GB/秒）省 10 倍，但 200 qps 下 50ms 窗內同時有 10 個查詢，又回到 30 萬核——這說明了為何持續投資索引。索引歷經 **trigram → 自製 suffix array → 目前的 sparse n-gram（比暴力解快逾 500 倍、且能高速答 regex）**；改用 token-based n-gram 是為了**沾光 Google 主索引/搜尋技術堆疊**（reverse index 建構、編碼、serving、instant indexing 都是現成的；以實作簡單性換效能的取捨）。索引從記憶體搬到 **flash**（便宜一個數量級、但延遲高兩個數量級，故記憶體裡好用的索引未必適合 flash）。索引完整歷史只增 **2.5 倍**資源。本地 workspace 用多機暴力搜尋。

- **Ranking（排序）：** 程式碼庫愈大、排序愈重要（任何短子字串都出現上千上百萬次）。分兩類訊號——**query independent（與查詢無關，可離線算）：file views（檔案被看次數，反映重要性，但有 exploitation vs exploration 的回饋迴路問題，新檔吃虧）、references（類似 PageRank，用 include/import 取代網頁連結，靠 Kythe 圖提供可靠引用）→「priority（優先度）」**；**query dependent（與查詢相關，須便宜可算）：token 乾淨匹配的 boost、大小寫、把 symbol/filename 匹配排在內容匹配前**。
- **Retrieval（檢索）：** 評分前要先找出候選；**supplemental retrieval（補充檢索）**——把原查詢改寫成更特化的（如限定定義與檔名），把新檢到的加進輸出，避免「定義那一筆在固定數量結果前就被截掉」。
- **Result diversity（結果多樣性）：** 在意圖不明時，跨類別給最佳結果（如同時給 Java 與 Python 的匹配，而非整頁同一種）。

### Selected Trade-Offs（選定的取捨）
核心主張：在 Google 規模下實作且保持 responsive 牽涉許多取捨：
- **Completeness：Repository at Head（完整性：以 head 為準）：** 可丟掉不給人讀的二進位/圖片/影片（只留檔名）、和混淆過的產生 JS（不可讀），省資源也降噪。但**丟檔有大缺點**：開發者要能**信任** Code Search，而對「沒被索引的檔案」無法給出「結果不完整」的回饋——故**寧可錯在索引太多**，上限主要為防濫用與系統穩定而非省資源。
- **Completeness：All vs Most-Relevant（全部 vs 最相關）：** 一般搜尋為速度犧牲完整性（賭排序讓 top 結果涵蓋所需）；但有時開發者要**全部結果**（重構/全域取代替換、工具分析）——這是與 web search 的根本差異。解法：把程式碼庫按 priority 分片，通常只看各片高優先檔，但**需要時可取全部**（約 1/3 查詢結果 < 20 筆）；以更複雜的實作/API 換取「兼顧平均速度與可選的完整性」。
- **Completeness：Head vs Branches vs History vs Workspaces：** 索引超過「當前快照」會大幅增加複雜度與成本（沒有 IDE 索引非當前版本）。Google 索引 Piper 的**線性歷史**——可在任意快照搜尋、找被刪的程式、依作者搜；好處之一是**過時程式可直接刪除**（以前要搬到 obsolete 目錄才找得回）；也為 workspace（未提交變更）搜尋奠基。
- **Expressiveness：Token vs Substring vs Regex（表達力）：** Code Search 支援 **regex**（強大、可用於沒有深層語意工具的文件/語言、開發者熟悉），代價是難建高效索引。**token-based 索引**擴展好（只存原始碼一小部分）但對程式碼很多有意義的字元（`function()` vs `function(x)`、`(x ^ y)`、`=== myClass`）難或不可能、且 tokenize 程式識別字（CamelCase/snake_case/全黏一起）定義不清、會模糊大小寫與詞幹、無法搜空白與分隔符。再上一階是 **substring search（trigram 索引）**——索引小但召回精度較低（需從結果濾掉非匹配）。有了 substring 索引就能把 regex automaton 轉成一組 substring 搜尋（trigram 直接、可推廣）；沒有完美的 regex 索引，但因複雜 regex 查詢佔比小，substring 近似實務上很好用。

工程意涵（結論）：對小到能塞進 IDE 的專案，這些大概用不上；但若你負責**較大程式碼庫上工程師的生產力**，核心洞見是——**理解程式碼是開發與維護的關鍵，投資「讓人理解程式碼」會帶來（難量化但真實的）回報**。Code Search 最重要的兩個功能 **Kythe 整合（語意理解）與找可用範例**，正是最直接連到「理解」的。而**沒人會用一個他不知道存在的工具**，故 Google 把它納入 **Noogler 訓練**。

## 關鍵字補充

- **Code Search** — Google 瀏覽與搜尋程式碼的工具（前端 UI＋雲端後端）；為「大規模地讀、理解、探索」而最佳化，非 IDE。
- **Kythe** — 編譯器型索引服務，提供 cross-reference（某符號的所有使用處），用完整 build 資訊消歧義；其整合讓 Code Search 從搜尋轉向瀏覽。
- **"Answer the next question in a single click"** — 整合 Kythe 後的導引原則；定義在哪、用在哪、何時加入、全機隊 CPU cycle 都一兩下點到。
- **五類搜尋意圖（How/What/Where/Why/Who-When）** — How 約 1/3（範例）、What 約 1/4（瀏覽）、Where 16%、Why 16%（除錯）、Who/When 8%。
- **集中式索引 vs 每人 IDE 索引** — 集中式做一次、增量更新（線性成本）、人人受惠；每人各自在 IDE 建索引大致二次方擴展。
- **Search Index vs Cross-reference Index** — 搜尋索引可增量、中位 < 10 秒可見；交叉引用索引（Kythe）無法增量、需近乎全建置、每日才建一次。
- **Sub-200ms / 1-second / 10-second** — < 200ms 才算 responsive、1 秒後注意力飄、10 秒就 context switch；每次多 1 秒 ≈ 每天 35 個工程師閒置。
- **Sparse n-gram Index** — 目前 Code Search 的索引方案，比暴力解快逾 500 倍、且能高速答 regex；歷經 trigram → suffix array → sparse n-gram。
- **Query Independent vs Dependent Signals（排序訊號）** — 與查詢無關（file views、references/priority，可離線算）vs 與查詢相關（token 匹配 boost、symbol/filename boost，須便宜）。
- **Supplemental Retrieval（補充檢索）** — 把原查詢改寫成更特化的（如限定定義/檔名），避免關鍵結果在固定數量內被截掉。
- **Completeness Trade-offs（完整性取捨）** — 丟二進位/產生 JS 省資源但傷信任，故寧可索引太多；priority 分片同時服務「最相關」與「全部結果」。
- **Token vs Substring vs Regex** — token 擴展好但對程式碼許多有意義字元失效；substring（trigram）索引小但召回較低；regex 最強但難建高效索引（可由 substring 索引近似）。

## 延伸提問

1. 你團隊找程式碼是靠 grep ＋ 各自的 IDE 索引嗎？算算「每人各自建索引」的二次方成本：隨程式碼庫與人數成長，這還撐得住嗎？一個**集中式、做一次、人人受惠**的索引（即使只是共用的 ctags/egrep profile）值不值得投資？
2. 把你最近用搜尋解決的問題分類成 **How/What/Where/Why/Who-When**：最大宗是不是「**看別人怎麼做的範例**」？這對你「該投資哪種工具/文件」有什麼啟示？對照 [[ch10]]「測試/程式即可執行的範例」。
3. 你能不能把你工具的延遲**換算成工程時間**（像「每次多 1 秒 ≈ 每天 N 個工程師閒置」）？你目前的搜尋/建置/開啟延遲，落在 **200ms／1 秒／10 秒** 的哪一檔？這把尺能不能幫你向上爭取後端投資？
4. 你的開發工具有沒有成為「**canonical 的位置**」——log、stack frame、文件、bug 都連回同一個地方？還是各工具各做各的 UI、彼此不知道對方存在？把資訊集中在一個「主要瀏覽介面」上，會帶來什麼複利？
5. 套用 Code Search 的**完整性取捨**到你的工具：你有沒有為了省資源而悄悄丟掉某些結果、卻沒讓使用者知道？這對「信任」的傷害值不值得？哪些該「寧可做太多」、哪些該明確標示限制？對照 [[ch14]]「無聲的上限」。

## 跨章連結

- [[ch16]] Version Control and Branch Management — Code Search 與 Piper 整合（看歷史、blame、各版本 diff）；monorepo 的單一程式碼庫正是「集中式索引」得以成立的前提。
- [[ch19]] Critique — Code Search 可跳到 Critique 的對應 review；兩者都是「以瀏覽/理解為核心」的開發者介面，互相連結。
- [[ch15]] Deprecation — Code Search/Kythe 是棄用「discovery（誰在用舊系統）」的主力；「可搜尋歷史」讓過時程式可直接刪除、也減輕情感依附。
- [[ch10]] Documentation — Google 因 Code Search 夠強而幾乎不另產獨立 reference；註解與程式並列、可直接搜尋瀏覽，是「文件貼著程式碼」的基礎設施。
- [[ch07]] Measuring Engineering Productivity — 「延遲換算成工程時間」「理解程式碼帶來難量化但真實的回報」正是衡量生產力的具體案例。
- [[ch23]] Continuous Integration — Kythe 也用於 CI；cross-reference 索引需近乎全建置，與建置系統（[[ch18]]）緊密相關。
- [[ch01]] What Is Software Engineering? — 「索引完整歷史」「回到某時間點的確切狀態」直接體現「整合時間維度」；集中式索引的線性 vs 二次方擴展呼應 Scale 主軸。
