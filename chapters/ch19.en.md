# Chapter 19 — Critique: Google's Code Review Tool

> *"Trust and communication are core to the code review process. A tool can enhance the experience, but it can't replace them."*
> A tool can make review smoother, but trust and communication — between humans — are what make it work.

## Summary

[[ch09]] covered the *process* of code review; this chapter covers the *tool* that supports it — Google's beloved in-house **Critique** (by far the most popular review tool internally). A good review tool doesn't just implement the process; its design principles are shaped by culture. Critique's four guiding principles: **① Simplicity** — a fast UI, easy navigation, clear visual state markers, "no unnecessary choices" (the principle with the most impact); **② Foundation of trust** — review is for empowering, not slowing others; trust authors to address minor comments rather than adding another review round; **③ Generic communication** — communication problems are rarely solved by tooling, so encourage plain comments over a complex data model; **④ Workflow integration**. Simplicity and integration are in tension — Google considered a "Code Central" (edit + review + search in one) but **deliberately declined** it, keeping code review the primary focus (features are linked, not embedded).

Critique supports six stages of **precommit review**: **① create a change (upload a snapshot → triggers automatic analyzers), ② request review (mail reviewers), ③ comment (unresolved by default = the author must address; resolved = optional; published atomically; anyone can drive-by review), ④ modify and reply (diff any two snapshots; ③④ repeat), ⑤ approval (LGTM; UI turns green), ⑥ commit (after presubmits pass)**. Other essentials: powerful **diffing** (LCS + syntax highlighting + Kythe cross-references + intraline diff + ignore-whitespace + move detection + side-by-side), analysis results as chips (red/yellow/gray, binary actionability, with fix suggestions), **tight tool integration** (Cider/Code Search/Tricorder/Rapid/Zapfhahn, preferring links over embedding), **GwsQ** auto-assigning reviewers, the **attention set ("whose turn")**, and a **Changelist Search**–powered dashboard. The scoring model is simplified to three things: **LGTM + Approval + unresolved comments** — every change needs at least one LGTM (two pairs of eyes), and LGTM/Approval are always positive (no unmotivated thumbs-down). Conclusion: **trust and communication are core, a tool can enhance but not replace them, and tight integration is key to Critique's success.**

## Why read this

- **You're choosing or building a review tool, agonizing over how many features to add.** Critique's top principle is **Simplicity** — "no unnecessary choices." It even **deliberately declined** an all-in-one "Code Central." Understand "why less is more," and you'll more readily reject tempting but complexity-adding features.
- **Your reviews stall on "whose turn is it now?"** Critique's **attention set ("whose turn")** is a tiny feature with outsized effect — it bolds "who the change is blocked on," and once built, no one could imagine working without it. Small workflow optimizations cut friction substantially.
- **Your scoring/approval mechanism is complex (many scores, vetoes, re-review rounds).** Critique simplifies to **LGTM + Approval + unresolved comments** and makes **LGTM/Approval always positive** (negative feedback must tie to a specific unresolved comment) — which not only simplifies the flow but **positively shapes review culture.**
- **You want to plug a review tool into your whole dev ecosystem.** One key to Critique's success is **tight tool integration** (jump to Code Search, edit in Cider, view Tricorder analysis, check whether it's in a Rapid release) — but via **links, not embedding**, to avoid sacrificing the core review experience. Learn that balance.

## Key details

### Code Review Tooling Principles
The core claim: why is Critique successful? Because it's shaped by Google's culture of treating code review as a core workflow, into four principles:
- **Simplicity:** a fast-loading UI, easy navigation (hotkey-supported), clear visual state markers; "no unnecessary choices." **This principle had the most impact** — many interesting features were considered but dropped to avoid complicating the model for a small set of users.
- **Foundation of trust:** review is for **empowering**, not slowing others; trust colleagues as much as possible — e.g., trust authors to address minor comments rather than adding a re-review round; changes are openly viewable/reviewable across Google.
- **Generic communication:** **communication problems are rarely solved by tooling**; Critique prefers plain comments (even suggested edits) over complicating the data model and process — "even with the best tool, communication can go wrong because users are human."
- **Workflow integration:** multiple integration points with other core tools (jump to Code Search, edit in Cider, view test results).

Takeaway: **Simplicity and integration are in tension** — Google considered a "Code Central" (edit + review + search in one) but **deliberately declined** it to keep code review the primary focus; features are **linked out** from Critique and implemented in other subsystems.

### Code Review Flow (six stages)
The core claim: Critique supports **precommit review**, in six typical steps:
1. **Create a change:** the author writes a change in their workspace, uploads a **snapshot** to Critique, triggering automatic analyzers ([[ch20]]).
2. **Request review:** once happy with the diff and analysis results, mail it to one or more reviewers.
3. **Comment:** reviewers draft comments; **unresolved by default (the author must address)**, plus optional resolved (optional/informational) ones; comments are **published atomically** (a complete thought after reviewing the whole change); **anyone can do a drive-by review.**
4. **Modify & reply:** the author revises, uploads new snapshots, and replies; addresses (at least) all unresolved comments; can diff **any two snapshots**; ③④ may repeat.
5. **Change approval:** when happy, reviewers **LGTM** (optionally with comments); when committable, the UI turns prominently **green.**
6. **Commit:** after approval, the author triggers the commit; if presubmits find no problems, it commits.

Takeaway: the system offers much flexibility to deviate (reviewers can unassign/reassign, authors can postpone); in emergencies the author can **force-commit and review afterward.** A **notifications** model keeps Critique focused as a primary review tool: it emits events, and other systems (a Chrome extension, email) build on them — some love instant notifications, others find them too disruptive.

### Stage 1: Diffing & Analysis Results
The core claim: understanding the change itself is the core of review, so **optimizing the diff is a core requirement.** Critique's diffing starts from an optimized **LCS (longest common subsequence)** algorithm, plus: **syntax highlighting, Kythe cross-references ([[ch17]]), intraline diff (character-level, word-boundary-aware), adjustable ignore-whitespace, and move detection (moved code marked "moved," not "removed here/added there").** Views can be side-by-side or overlay — for side-by-side, they simplified the diff structure (no borders, no padding) and tuned fonts to fit Java's 100-character line limit. A snapshot-version widget lets you drag to compare versions (auto-collapsing similar snapshots); **prefetching everything** makes loading fast.

Uploading a snapshot triggers **analyzers ([[ch20]])**, summarized as **chips** (red = highlight, yellow = in progress, gray = otherwise; **actionability is binary** — no more options); findings can show in the diff (styled differently from comments) and often include **fix suggestions** to preview and apply in one click (e.g., a linter catching trailing whitespace → two clicks to the problem, one to apply the fix).

**Tight tool integration:** Critique is a "melting pot" for one-click/hover access to **Cider (cloud IDE), Code Search, Tricorder (static analysis), Rapid (release), Zapfhahn (test coverage).** But it **prefers links over embedding** to avoid distracting from the core review experience (the one exception is test coverage — shown via gutter background colors). Tight integration is possible because workspaces live in a **FUSE-based filesystem**, with the Source of Truth in the cloud, accessible to all these tools.

### Stage 2: Request Review
The core claim: the author picks reviewers — even small teams benefit from distributing reviews evenly and accounting for vacations. A team can provide an **email alias**, and **GwsQ** (named after the team that pioneered it) auto-assigns a specific reviewer per the alias's config. At Google's scale, "who's best to review a change outside your project" is hard — Critique's reviewer suggestions consider: **who owns the changed code, who's most familiar (changed it recently), who's available (not OOO, ideally same time zone), and the GwsQ setup.**

Takeaway: assigning a reviewer triggers **presubmits**: auto-adding email lists for transparency, running the project's test suites, and enforcing project-specific invariants on code and descriptions (e.g., to generate release notes). Since tests are resource-intensive, they're part of presubmits (run at review request and commit), not per-snapshot like Tricorder. A failed presubmit **blocks** the change from review or commit and emails the author.

### Stages 3 & 4: Commenting & Understanding State
The core claim: **commenting is the second-most-common action in Critique** (after viewing), and **anyone can comment** (not just author and assigned reviewers). Per-person state lets reviewers check off individual files as reviewed (cleared for everyone when the author modifies that file). Seeing a finding, a reviewer can click "**Please fix**" to create an unresolved comment; they can also inline-edit a suggested fix (Critique turns it into a comment with an attached fix); common responses have **Done** (addressed) / **Ack** (read — for informational comments) shortcuts. Comments are drafted as-you-go but **published atomically.**

**Two mechanisms for understanding state:**
- **"Whose turn" / attention set:** with multiple reviewers, knowing it's your turn accelerates review. The attention set is "who the change is currently blocked on"; those in it should respond promptly; Critique renders their names in **bold.** After it shipped, users "couldn't imagine the previous state" — it also emphasizes review's **turn-based** nature: it's always at least one person's turn.
- **Dashboard & Changelist Search:** Critique's landing page is the user's dashboard, powered by **Changelist Search** (indexing the latest state of all changes across Google, regex queries, fast enough for interactive use). The default first section shows "changes needing your attention" (the attention set), but it's customizable (LSC global reviewers customize it to avoid being flooded).

### Stage 5: Change Approvals (Scoring a Change)
The core claim: beyond specific suggestions via comments, you need a high-level "OK." Google simplifies scoring to **three things**: **LGTM (looks good to me) + Approval + unresolved comments.**
- **LGTM** = "I've reviewed this, believe it meets our standards, and it's okay to commit after addressing unresolved comments."
- **Approval** = "as a gatekeeper, I allow this into this part of the codebase."
- **Every change needs at least one LGTM** (regardless of approval), ensuring **at least two pairs of eyes.**

Takeaway: there used to be "Needs More Work" and "LGTM++" ratings, later **simplified** so LGTM/Approval are always positive — if a second review is truly needed, primary reviewers add comments without LGTM. This also **positively shaped culture**: reviewers **can't thumbs-down without specific feedback** — all negative feedback must tie to something concrete to fix (an unresolved comment, a phrase chosen to sound relatively nice). The distinction between **hard requirements (LGTM/Approval — only reviewers can grant/revoke) and soft requirements (unresolved comments — the author can mark resolved)** promotes and relies on **trust** — a reviewer can LGTM with unresolved comments without later verifying they were addressed, saving time especially across time zones.

### Stage 6, After Commit & Gerrit
The core claim:
- **Stage 6 (commit):** Critique has a commit button to avoid context-switching to the CLI.
- **After commit (tracking history):** Critique is also used for **change archaeology** — anyone can browse a change's history (comments and evolution) for later auditing, understanding "why a change was made" or "how a bug was introduced"; aggregated review data even produces training. It also supports **post-commit comments** (when a problem is found later) and **rollbacks** (and seeing whether a change has been rolled back).
- **Case Study | Gerrit:** Critique isn't externally available due to tight coupling with the monorepo and internal tools; teams on open-source projects (Chrome/Android) or non-monorepo work use **Gerrit** — standalone, open source, tightly integrated with Git, with a fine-grained permission model and a more sophisticated (configurable) scoring system (a reviewer can **veto with −2**). Both share the same model (each commit reviewed separately).

Takeaway (conclusion): a review tool carries implicit trade-offs. **Time in review is time not coding**, so any process optimization is a productivity gain; most cases need only two people (author + reviewer) to agree, keeping velocity high. Critique must handle **scale** in several ways (no degradation under many review requests, fast loading on the critical path, usable for unusually large changes, helping find reviewers without figuring out the ownership landscape). It **favors an opinionated process and a simple interface** but allows some customization (custom analyzers/presubmits, team policies like requiring multiple LGTMs). **Trust and communication are core; a tool can enhance but not replace them; tight integration is key to success.**

## Key terms

- **Critique** — Google's in-house, most-popular code review tool; supports the six-stage precommit review flow, tightly integrated with other dev tools.
- **The four principles (Simplicity / Trust / Generic communication / Workflow integration)** — Critique's design guides; Simplicity had the most impact (Google declined an all-in-one "Code Central" for it).
- **Precommit review** — review before code enters the codebase; six stages: create → request → comment → modify → approve → commit.
- **Snapshot** — a change's patch at a point in time; uploading one triggers analyzers, and you can diff any two snapshots.
- **Diffing** — LCS + syntax highlighting + Kythe cross-references + intraline diff + ignore-whitespace + move detection + side-by-side; optimizing the diff is a core requirement of a good tool.
- **Analyzer chips / Tricorder** — analysis results as chips (red/yellow/gray, binary actionability, with fix suggestions); Tricorder is the static-analysis tool (see [[ch20]]).
- **GwsQ** — auto-assigns reviewers via an email alias; suggestions consider owner/familiarity/availability/GwsQ setup.
- **Presubmits** — run at review request and commit (add email lists, run test suites, enforce invariants); a failure blocks the change.
- **Attention set / "whose turn"** — bolds "who the change is currently blocked on"; emphasizes review's turn-based nature; a small optimization that cuts friction substantially.
- **Changelist Search** — the search system powering the dashboard, indexing the latest state of all changes across Google, regex-queryable, fast enough for interactive use.
- **LGTM / Approval / unresolved comments (three-part scoring)** — LGTM = meets standards, okay to commit; Approval = gatekeeper allows it in; unresolved = the author must address; every change needs at least one LGTM (two pairs of eyes).
- **Gerrit** — a standalone, open-source review tool tightly integrated with Git; used for open-source/non-monorepo projects; supports −2 vetoes and more configurable scoring.

## Questions to push your thinking

1. Is your review tool oriented toward **Simplicity (no unnecessary choices)** or ever-more features? Are there features that "complicate the model for a small set of users" you should cut? Could you, like Google declining "Code Central," **resist cramming everything into one tool**?
2. Do your reviews stall on "whose turn is it now?" Could you add an **attention set**-like mechanism (clearly marking "who it's blocked on")? Recall a change idling between author and reviewer because no one knew whose turn it was — how much time would this save?
3. Is your approval/scoring mechanism too complex (many scores, vetoes, re-review rounds)? Could you simplify to **LGTM + Approval + unresolved comments** and require **negative feedback to tie to something specific to fix**? What would that do to your review **culture**?
4. How tightly is your review tool integrated with others (search, IDE, CI, release, coverage)? Which should be **linked**, which worth **embedding**? Recall Critique's "links by default, embed only coverage" balance — have you sacrificed core-experience simplicity for integration? Per [[ch17]], [[ch20]].
5. Does "**trust the author to address minor comments without re-checking**" work on your team? Is your review **empowering** or **slowing**? Across time zones, how much round-tripping would that trust save? Per [[ch09]], [[ch02]]'s HRT.
