CHAPTER 01

What Is Software Engineering?

什麼是軟體工程？
作者 Titus Winters PDF p.31–52 Time 時間 Scale 規模 Trade-offs 取捨
"Software engineering is programming integrated over time." 軟體工程，是把時間維度整合進來的程式設計。

摘要

本章是全書的論題（thesis），回答一個看似基本卻長期被混淆的問題：軟體工程和寫程式（programming）差在哪？作者提出三條判準——時間、規模、取捨。程式設計是「當下產生程式碼」；軟體工程則是在「程式碼預期被使用、維護多久」的時間尺度上，加上「多少人協作、組織多大」的規模，再疊上「在不完美資訊下做高風險決策」的取捨後，形成的一整套政策、實務與工具。

核心是可持續性（sustainability）：只要在程式碼的預期壽命內，你「有能力」回應任何有價值的變化，專案就是可持續的——重點是能力，而非一定要改。短命程式碼幾乎不受時間影響，可以「夠用就好」；但壽命跨越數年到數十年的系統，幾乎所有依賴都會變，於是「能跑」與「可維護」的差距會被時間無限放大。這章奠定了後面 24 章共用的思考框架。

為什麼要讀這一篇

把「會寫程式」升級成「會做工程」。 L4 最常見的盲點是把「在我機器上能跑」當完成；這章給你一把尺——程式碼的預期壽命——重新定義什麼叫「做完」。
替團隊做決策，卻說不清理由。 本章把成本拆成六類，並給鐵律：決策只能因「非做不可」或「現有證據下最佳」，絕不能因為「我說了算」。這是向上溝通與帶人的共同語言。
預判哪些做法「現在沒事、長大就爆」。 Hyrum's Law、不可擴展的棄用政策、分支氾濫——用「組織放大 10×、100×」的思想實驗，提早辨識超線性成長的成本。
它是全書地圖。 Time／Scale／Trade-offs 三主軸貫穿後面每一章；先讀懂這章，後面所有章節都有了掛鉤點。

細節重點

TIME Time and Change 時間與變化

關鍵提問：「你的程式碼預期壽命有多長？」合理答案跨度約 10 萬倍——從「跑一次、活幾分鐘」到「活幾十年」。

短命程式碼幾乎不受時間影響；隨壽命拉長，變化會從可忽略變成主宰因素。從一次性腳本到跨十年專案之間，存在一個「必須開始回應外部變化」的轉折點，沒有從一開始規劃升級的專案，第一次跨過去通常極度痛苦（隱藏假設多、缺經驗、一次補做好幾年升級）。

工程意涵：可持續性不是「永不改變」，而是「有能力且划算地改變」。熬過第一次大升級、並建立「能持續跟上」的能力，才是長期可持續的本質。

TIME Hyrum's Law 海勒姆定律

當一個 API 的使用者夠多，你在契約裡承諾什麼都不重要——系統所有可被觀察到的行為，都會有人依賴。

作者類比為熵：無法根除，只能緩解。即使最無害的改動，只要時間夠久、用戶夠多，都會弄壞某個東西。

工程意涵：把它從哲學變成日常風險評估——評估變更價值時，必須把「調查、定位、修復破壞」的難度算進去。

TIME Example: Hash Ordering 雜湊排序範例

雜湊容器迭代順序是 Hyrum's Law 活教材：短命且環境不變時可假設順序固定；不知壽命或不能保證依賴不變時，這假設就是錯的。

即使你不依賴順序，序列化進 RPC 回應後，呼叫端可能反而依賴它；甚至有人把雜湊順序當廉價亂數產生器，移除隨機性反而弄壞他們。這是「能跑」與「正確」的最小範例。

金句總結：「如果『clever』是稱讚，那是 programming；如果『clever』是指控，那是 software engineering。」

TIME Why Not Just Aim for "Nothing Changes"? 為何不追求「什麼都不變」

「能不能不要改？」——看情況。純 C、零外部依賴或只依賴 POSIX 這類極穩定的東西，或許能避開重構；但多數專案暴露在會變動的底層技術下。

Heartbleed、Meltdown、Spectre 等安全問題，加上硬體演進帶來的效能機會（舊演算法在新硬體上反而低效），都讓「假設永不變」變成高風險豪賭。

工程意涵：「為改而改」不對，但「沒有能力改」更危險。變化本身不是好事，但「能否變化」是必須具備、且值得投資讓它變便宜的能力。

SCALE Scale and Efficiency 規模與效率

組織賴以生產與維護程式碼的一切都必須在成本與資源上可擴展；尤其每件「重複做」的事，都必須在人力投入上可擴展。

不只程式本身要對 CPU／記憶體／儲存／頻寬擴展良好，連開發流程、測試叢集、版控與建置系統也要。許多問題像「溫水煮青蛙」，緩慢惡化卻從不爆成單一危機。

SCALE Policies That Don't / Do Scale 不可擴展 vs 可擴展的政策

辨識壞政策的方法：想像「組織放大 10×、100×」時單一工程師的負擔會不會等比增加。

不可擴展：傳統棄用法（「8/15 砍舊 Widget，請大家自己遷移」）把成本推給下游；開發分支政策（「合併太亂就少合併」）導致分支爆炸。
可擴展對策：Churn Rule——基礎設施團隊自己負責遷移內部用戶，由少數專家深入學一次、套到每個子問題；The Beyoncé Rule——「喜歡它就該幫它寫 CI 測試」，沒進共用 CI 的測試不算數。

SCALE Example: Compiler Upgrade 編譯器升級範例

Google 2006 年傳奇升級——五年沒升級、數千工程師多數沒經驗——極度痛苦。但關鍵不是難度，而是意識到任務必要後，用自動化、一致性、專家化讓升級能以「固定人數」完成，即使程式碼持續長大。
工程意涵：停滯（stagnation）是一個選項，但通常不明智——會多付約 25% 運算成本並承擔安全風險。影響升級難易的因素：專家、穩定度、一致性、熟悉度、政策。

SCALE Shifting Left 左移

在開發時間軸上（構想→設計→實作→審查→測試→提交→金絲雀→上線），愈早（往「左」）發現問題，修復愈便宜。

源自資安界「shift left on security」。靜態分析與 code review 在提交前抓到的 bug 遠比上線後便宜。沒有單一工具完美，所以採縱深防禦，盡量在左側多攔截缺陷。

TRADE Trade-offs and Costs 取捨與成本

「成本」不只是錢，包含財務、資源、人力、交易、機會、社會六類。在高創造、高報酬的軟體工程裡，限制因素通常是人力而非錢——專注度差異輕易就有 10–20%。
鐵律：決策只能因「非做不可」或「現有證據下的最佳選擇」，不能是「我說了算」。追求共識（consensus）而非全體一致（unanimity）。

TRADE Examples: Markers / Distributed Builds 白板筆 / 分散式建置

白板筆：Google 選擇開放補給，明確權衡「讓腦力激盪零阻礙，遠比防止有人順手拿走幾支筆重要」。

分散式建置：自建系統整體效益遠大於成本，但故事沒有完美結局——當工程師不再直接感受建置變慢的痛，膨脹的依賴失去約束（呼應 Jevons 悖論：效率提升反而刺激消耗）。即使簡單取捨也有未預見的下游效應。

TRADE Revisiting Decisions & SWE vs Programming 重新檢視決策／兩者之別

資料驅動文化的隱藏好處，是有能力也有必要承認錯誤。決策者必須有「認錯的權利」——勇於認錯的領導者更受尊敬。

最後澄清：軟體工程並非「優於」程式設計，而是兩個不同問題領域，各有約束與最佳實務。短命專案不需整合測試、CD、SemVer；長壽專案則離不開它們。用對工具才是重點。

關鍵字補充

Software Engineering vs Programming（軟體工程 vs 程式設計）
程式設計是「當下產生程式碼」；軟體工程是「在程式碼有用壽命內維護它、支援團隊協作」所需的政策、實務與工具總和。差別在維度：time、scale、trade-offs。
Sustainability（可持續性）
在程式碼預期壽命內，你「有能力」安全地回應一切該回應的變化。重點是能力，不是強制每次都升級。
Hyrum's Law（海勒姆定律 / 隱含依賴定律）
用戶夠多時，API 所有可觀察行為都會被某人依賴。是評估變更難度的核心風險因子，類比於熵：可緩解不可根除。
Expected Life Span（預期壽命）
程式碼預期被建置、執行、維護多久。合理跨度約 10 萬倍；壽命決定該用「夠用就好」還是「乾淨可維護」。
Sublinear Scaling（次線性擴展）
組織與程式碼長大時，重複性工作的人力成本必須次線性成長；超線性成長的政策遲早撐不住。
The Beyoncé Rule（碧昂絲法則）
「喜歡它就該幫它寫 CI 測試」。沒進共用 CI 的測試不算數，破壞責任歸於沒寫測試者。
Churn Rule（流失/翻騰規則）
基礎設施團隊自己負責把內部用戶遷移到新版，而非把成本推給每個下游；由專家集中處理，擴展性更好。
Shifting Left（左移）
在開發時間軸上愈早發現問題，修復成本愈低。配合縱深防禦，盡量在提交前攔截缺陷。
Cost（成本，六類）
財務、資源、人力、交易、機會、社會。軟體工程的限制因素通常是人力成本而非財務成本。
Boiled Frog Problem（溫水煮青蛙問題）
低效與成本緩慢惡化、從不爆成單一危機，因而被忽視；需靠全組織的擴展意識持續監看。

延伸提問

  1. 拿你正在維護的一個系統，誠實估它的預期壽命。你的做法（測試、依賴管理、升級策略）和這壽命相稱嗎？哪裡用「夠用就好」反而埋了長期地雷？
  2. 套用 Hyrum's Law：你的服務有哪些「未承諾、卻已被下游依賴」的可觀察行為（輸出順序、時序、錯誤訊息格式、效能特性）？哪一個一旦改動最危險？
  3. 用「組織放大 10 倍」思想實驗檢視你團隊某條流程（發版、權限審批、依賴升級）。它是次線性還是超線性成長？能否用「專家集中處理」或「自動化」改造？
  4. 回想最近一個重大技術決策。它屬於「非做不可」「現有證據下最佳」哪一種？還是混入了「大家都這樣做／我說了算」？你能列出六類成本嗎？
  5. 你的團隊文化允許公開承認決策錯誤嗎？若不允許，是什麼制度或誘因在阻止它？對照下一章的無究責事後檢討。

Part I — Thesis
CHAPTER 01

What Is Software Engineering?

The book's thesis: programming + time
by Titus Winters PDF p.31–52 Time Scale Trade-offs
"Software engineering is programming integrated over time." In plain terms: software engineering = programming + the dimension of time.

Summary

This chapter is the book's thesis. It answers a question that sounds basic but is constantly muddled: how does software engineering differ from programming? The author gives three discriminators — time, scale, and trade-offs. Programming is producing code right now; software engineering is what programming becomes once you add the time the code must live, the scale of people producing it, and the pressure of high-stakes decisions under imperfect information — yielding a whole apparatus of policies, practices, and tools.

The central idea is sustainability: your project is sustainable if, for the expected life span of your code, you are able to react to any valuable change. The key word is capability, not obligation. Short-lived code is essentially untouched by time and can be "good enough"; but for code that lives years or decades, almost every dependency eventually changes, and the gap between "it happens to work" and "it is maintainable" is magnified without bound. This installs the mental framework reused across the next 24 chapters.

Why read this

You're upgrading from "can code" to "can engineer." The classic L4 blind spot is treating "it runs on my machine" as done. This chapter hands you a ruler — the expected life span of the code — that redefines what done even means.
You decide for your team but can't articulate why. The chapter splits "cost" into six kinds and gives one hard rule: a decision is justified only because it's mandatory or the best option on current evidence — never "because I said so."
You want to predict what's "fine now, explodes at scale." Hyrum's Law, non-scaling deprecation, branch proliferation — use the "imagine the org 10× / 100× bigger" thought experiment to spot super-linear costs early.
It is the map for the whole book. Time / scale / trade-offs run through every later chapter; grasp this one and everything else has a hook.

Key details

TIME Time and Change

The clarifying question: "What is the expected life span of your code?" Reasonable answers span roughly 100,000× — from "runs once, lives minutes" to "lives for decades."

Short-lived code is largely immune to time; as life span grows, change shifts from negligible to dominant. Between a one-off script and a decade-long project lies a transition point where the project must start reacting to external change — and projects that didn't plan for upgrades find that first crossing brutal (baked-in assumptions, inexperience, years of upgrades at once).

Takeaway: sustainability isn't "never change" — it's "able to change, affordably." Getting through that first upgrade and staying current is the essence of long-term sustainability.

TIME Hyrum's Law

With a sufficient number of users of an API, it does not matter what you promise in the contract — all observable behaviors of your system will be depended on by somebody.

Likened to entropy: mitigable, never eradicable. Given enough time and users, even the most innocuous change breaks something.

Takeaway: turn it into routine risk assessment — a change's value analysis must include the cost of investigating, locating, and fixing the breakages it causes.

TIME Example: Hash Ordering

Hash-container iteration order is the textbook Hyrum's Law case: fine to assume a fixed order if your code is short-lived and the environment is frozen; wrong if you don't know the life span or can't promise dependencies won't change.

Even if you don't depend on order, serializing into an RPC response can make the caller depend on it; some even use hash order as a cheap RNG, so removing the randomness breaks them. The smallest example of "it works" vs "it is correct."

Mnemonic: "It's programming if 'clever' is a compliment, but it's software engineering if 'clever' is an accusation."

TIME Why Not Just Aim for "Nothing Changes"?

Can you simply avoid change? — it depends. Pure C with no external deps (or only ultra-stable ones like POSIX) might dodge refactors; most projects are exposed to shifting underlying technology.

Security disclosures (Heartbleed, Meltdown, Spectre) plus hardware evolution (old algorithms get inefficient on new CPUs) make "assume nothing ever changes" a high-risk bet.

Takeaway: changing for its own sake is wrong, but being unable to change is worse. Change isn't inherently good, but the capability to change is mandatory — and worth investing to make cheap.

SCALE Scale and Efficiency

Everything the org relies on to produce and maintain code must scale in cost and resources; in particular, everything done repeatedly must scale in human effort.

Not just the software (CPU/memory/storage/bandwidth) but the dev workflow, test clusters, version control, and build systems. Many such problems are "boiled-frog": they worsen slowly and never spike into a single crisis.

SCALE Policies That Don't / Do Scale

Spot a bad policy by imagining the org 10× / 100× bigger and asking whether one engineer's burden grows proportionally.

Doesn't scale: the classic deprecation move ("we delete the old Widget on Aug 15th, migrate yourselves") pushes cost downstream; the dev-branch reflex ("merge less") causes branch explosion.
Scales: Churn Rule — infra teams migrate their own internal users; a few experts learn it once and apply to every sub-problem. Beyoncé Rule — "put a CI test on it"; tests outside shared CI don't count.

SCALE Example: Compiler Upgrade

Google's legendary 2006 upgrade — five years without upgrading, thousands of engineers, most inexperienced — was extremely painful. The lesson isn't the difficulty but that, once recognized as necessary, automation/consistency/expertise made upgrades doable with a constant headcount even as the codebase grew.
Takeaway: stagnation is an option, usually unwise (~25% extra compute cost plus security exposure). Ease factors: expertise, stability, conformity, familiarity, policy.

SCALE Shifting Left

On the workflow timeline (conception → design → implementation → review → testing → commit → canary → production), the earlier ("more to the left") you catch a problem, the cheaper to fix.

From security ("shift left on security"). Bugs caught by static analysis and code review before commit are far cheaper than those reaching production. No single tool is perfect, so use defense in depth.

TRADE Trade-offs and Costs

"Cost" is more than money: financial, resource, personnel, transaction, opportunity, societal. In a high-creativity field the limiting factor is usually personnel, not money — a 10–20% focus swing is easy.
Hard rule: a decision is legitimate only when mandatory or the best option on current evidence — not "because I said so." Aim for consensus, not unanimity.

TRADE Examples: Markers / Distributed Builds

Markers: Google stocks open supply closets — an explicit trade-off: frictionless brainstorming matters far more than guarding a few markers.

Distributed builds: net savings vastly outweighed cost, but no fairy-tale ending — once engineers stopped feeling slow builds, dependency bloat lost its constraint (an echo of Jevons' paradox: efficiency can increase consumption). Even simple trades have unforeseen downstream effects.

TRADE Revisiting Decisions & SWE vs Programming

A hidden benefit of a data-driven culture is the ability — and necessity — of admitting mistakes. Deciders must have the right to be wrong: leaders who admit mistakes are respected more, not less.

Finally: software engineering isn't "better" than programming — they're two different problem domains with distinct best practices. A few-day project needs no integration tests, CD, or SemVer; a decades-long one can't live without them. Using the right tool is the point.

Key terms

Software Engineering vs Programming
Programming = producing code now. Software engineering = the policies, practices, and tools to keep code useful as long as needed and to enable team collaboration. Difference = dimensionality: time, scale, trade-offs.
Sustainability
Being able, for the code's expected life span, to safely respond to every change you ought to. About capability, not forcing upgrades.
Hyrum's Law (Law of Implicit Dependencies)
With enough users, all observable API behaviors get depended on. The core risk factor in assessing change difficulty; entropy-like.
Expected Life Span
How long code is expected to be built, run, maintained. Spans ~100,000×; decides "good enough" vs "clean and maintainable."
Sublinear Scaling
As org and codebase grow, the human cost of repeated work must grow sub-linearly; super-linear policies eventually fail.
The Beyoncé Rule
"Put a CI test on it." Tests outside shared CI don't count; breakage blame falls on whoever didn't write the test.
Churn Rule
Infra teams migrate their own internal users instead of pushing cost downstream; centralized expertise scales better.
Shifting Left
Catch problems earlier in the workflow to lower fix cost; pair with defense in depth.
Cost (six kinds)
Financial, resource, personnel, transaction, opportunity, societal. The limiting factor is usually personnel, not money.
Boiled Frog Problem
Inefficiency worsens slowly and never spikes into a crisis, so it's ignored; only org-wide scaling awareness keeps it in check.

Questions to push your thinking

  1. Take a system you maintain and honestly estimate its expected life span. Do your testing, dependency-management, and upgrade strategies match it? Where has "good enough" planted a long-term landmine?
  2. Apply Hyrum's Law: which "unpromised but already-depended-on" behaviors does your service have (output order, timing, error format, performance)? Which is most dangerous to change?
  3. Run the "org 10× bigger" thought experiment on a process (releases, approvals, dependency upgrades). Does it grow sub- or super-linearly? Can centralized expertise or automation fix it?
  4. Recall a recent major decision. Was it "mandatory" or "best option on current evidence" — or did "everyone does it / because I said so" sneak in? Can you list the six costs?
  5. Does your culture allow publicly admitting a wrong decision? If not, what incentive blocks it? Compare with the blameless post-mortem in the next chapter.
