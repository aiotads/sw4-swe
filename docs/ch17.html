CHAPTER 17

Code Search

程式碼搜尋
作者 Alexander Neubeck、Ben St. John PDF p.379–398 Scale 規模 Time 時間 Trade-offs 取捨
"Understanding code is key to developing and maintaining it … investing in understanding code will yield dividends." 理解程式碼，是開發與維護它的關鍵；投資於「讓人理解程式碼」會帶來回報。

摘要

Code Search 是 Google 瀏覽與搜尋程式碼的工具（前端 UI＋雲端後端），直接源於「擴展到程式碼庫規模」的需求。整合 Kythe（cross-reference、跳到定義）後從「搜尋」轉向「瀏覽」，並以 「用一次點擊回答關於程式碼的下一個問題」 為原則。與 IDE 不同，它為「大規模地讀、理解、探索」而最佳化。

工程師用它回答五類意圖：How（約 1/3，看範例）、What（約 1/4，瀏覽）、Where（16%）、Why（16%，除錯）、Who/When（8%）。為何另做 web 工具？ ①Scale：整份程式碼庫塞不進單機；集中式索引做一次、人人受惠、增量更新（線性成本），IDE 各自建索引是二次方。②零設定全域視野。③特化（不是 IDE 反而是優點）。④與其他工具整合（Code Search 連結＝程式的 canonical「位置」）。

實作有兩個索引：即時搜尋索引（增量、中位 < 10 秒） 與 每日 cross-reference 索引（Kythe、無法增量）。關鍵數字：每天逾百萬查詢，每次多 1 秒 ≈ 每天 35 個工程師閒置 →鎖定 sub-200ms；索引 1.5TB、200 qps、中位搜尋 < 50ms；trigram → suffix array → sparse n-gram（比暴力解快 500 倍）。取捨圍繞 completeness vs 速度/資源/信任。

為什麼要讀這一篇

你以為「程式碼搜尋」不就是 grep ＋ IDE 索引？ 在 Google 規模下整份程式碼庫塞不進單機、IDE 各自建索引是二次方擴展。「集中式索引做一次、人人受惠」會讓你重評你的搜尋/索引基礎設施。
你想知道工程師在搜尋什麼。 五類意圖中最大宗是「看別人怎麼做的範例（約 1/3）」。提醒你：理解程式（而非只找到它）才是生產力核心，值得投資。
你在為互動式工具設效能目標。 一把可量化的尺：每次多 1 秒 ≈ 每天 35 個工程師閒置、<200ms 才 responsive、1 秒注意力飄、10 秒 context switch。把延遲換算成工程時間。
你在做會被全公司用的工具。 成功心法：成為其他工具的平台與 canonical「位置」、讓人知道它存在（Noogler 訓練）、先有「夠用但有限」的版本也能被信任——只要限制被理解。

為何 Why a Separate Web Tool? 為何不用 IDE

  • Scale：整份程式碼庫塞不進單機；集中式索引做一次、人人受惠、隨提交增量更新（線性成本），而「每人各自在 IDE 建索引」大致二次方。但 cross-reference 索引無法增量（任何變更可能影響整庫），故 Kythe 每天才建一次（Chromium 約 6 小時）。
  • 零設定全域視野：即時瀏覽整庫 →易找可重用庫與好範例；無需專案描述/build 環境。
  • 特化（不是 IDE 反而是優點）：沒有編輯游標，每次點擊都能有意義（跳定義、看用處）；UX 純為瀏覽與理解最佳化。
  • 與其他工具整合：Code Search 連結＝程式的 canonical「位置」——log viewer、stack frame、編譯錯誤、codelab/文件都連回它（連未提交 workspace 程式都可）；也暴露 API、有 vim/emacs/IntelliJ 外掛。

規模 Impact of Scale、Implementation 延遲換算工程時間、sparse n-gram

  • 查詢延遲：每天逾百萬查詢；每次多 1 秒 ≈ 每天 35 個全職工程師閒置，後端只需約 1/10 的人 →約每天 10 萬查詢就是損益兩平。<200ms 才 responsive、1 秒注意力飄、10 秒 context switch →鎖定 sub-200ms。
  • 索引延遲：寫程式時期待即時索引；中位提交到可見 < 10 秒。除錯生產事故時索引與執行中程式的落差尤其麻煩。
  • 實作：1.5TB、200 qps、中位搜尋 < 50ms；暴力解需 30 萬核；trigram → suffix array → sparse n-gram（快 500 倍、能答 regex）；沾光 Google 主搜尋堆疊；索引從記憶體搬到 flash；完整歷史只增 2.5 倍。
  • 排序：query independent（file views、references/priority，離線算）vs query dependent（token/symbol/filename boost，須便宜）；supplemental retrieval；result diversity。

取捨 Selected Trade-Offs 完整性 vs 速度/資源/信任

  • Repository at Head：可丟二進位/產生 JS 省資源降噪；但丟檔傷信任（無法對沒索引的檔給「結果不完整」回饋）→寧可索引太多。
  • All vs Most-Relevant：一般搜尋為速度犧牲完整；但重構/工具需全部結果。解法：按 priority 分片、需要時可取全部（約 1/3 查詢 < 20 筆）。
  • Head vs History vs Workspaces：索引 Piper 線性歷史 →可搜任意快照、找被刪程式、依作者搜；過時程式可直接刪除。
  • Token vs Substring vs Regex：token 擴展好但對程式碼有意義字元失效；substring（trigram）索引小但召回較低；regex 最強但難建高效索引（可由 substring 近似）。

延伸提問

  1. 你團隊找程式碼靠 grep ＋各自的 IDE 索引嗎？算算「每人各自建索引」的二次方成本：隨程式碼庫與人數成長還撐得住嗎？一個集中式、做一次、人人受惠的索引值不值得投資？
  2. 把你最近用搜尋解決的問題分類成 How/What/Where/Why/Who-When：最大宗是不是「看別人怎麼做的範例」？這對你「該投資哪種工具/文件」有什麼啟示？對照 ch10。
  3. 你能不能把工具延遲換算成工程時間（「每次多 1 秒 ≈ 每天 N 個工程師閒置」）？你的搜尋/建置/開啟延遲落在 200ms／1 秒／10 秒 哪一檔？
  4. 你的工具有沒有成為「canonical 的位置」——log、stack frame、文件、bug 都連回同一處？還是各做各的 UI、彼此不知道對方存在？
  5. 套用 Code Search 的完整性取捨：你有沒有為省資源悄悄丟掉某些結果卻沒讓使用者知道？這對「信任」的傷害值不值得？對照 ch14「無聲的上限」。

Part IV — Tools
CHAPTER 17

Code Search

Investing in helping people understand code pays off
by Alexander Neubeck & Ben St. John PDF p.379–398 Scale Time Trade-offs
"Understanding code is key to developing and maintaining it … investing in understanding code will yield dividends." Investing in helping people understand code pays off — the real point of Code Search.

Summary

Code Search is Google's tool for browsing and searching code (frontend UI + cloud backends), which arose directly from the need to scale to the codebase's size. Integrating Kythe (cross-references, jump-to-definition) shifted it from searching to browsing, guided by "answering the next question about code in a single click." Unlike an IDE, it's optimized for reading, understanding, and exploring at scale.

Engineers use it for five intents: How (~1/3, examples), What (~1/4, browsing), Where (16%), Why (16%, debugging), Who/When (8%). Why a separate web tool? ① Scale: the codebase won't fit on one machine; a centralized index is built once, benefits everyone, incrementally updated (linear cost), while per-IDE indexing scales quadratically. ② Zero-setup global view. ③ Specialization (not being an IDE is an advantage). ④ Integration (a Code Search link is code's canonical "location").

Two indexes: an instant search index (incremental, median < 10s) and a daily cross-reference index (Kythe, not incremental). Key numbers: over a million queries a day, one extra second ≈ 35 idle engineers daily → target sub-200ms; 1.5TB index, 200 qps, median search < 50ms; trigram → suffix array → sparse n-gram (500× faster than brute force). Trade-offs center on completeness vs speed/resources/trust.

Why read this

You think "code search" is just grep + IDE indexing? At Google's scale the codebase won't fit on one machine and per-IDE indexing scales quadratically. "A centralized index built once, benefiting everyone" will make you rethink your search/index infrastructure.
You want to know what engineers search for. The biggest of five intents is "examples of how others did it (~1/3)." The lesson: understanding code (not just finding it) is the productivity core worth investing in.
You're setting performance targets for an interactive tool. A yardstick: one extra second ≈ 35 idle engineers daily, and <200ms is responsive, 1s drifts attention, 10s context-switches. It converts latency into engineering time.
You're building a company-wide tool. The playbook: be a platform and canonical "location" for other tools, make people aware it exists (Noogler training), and an early "useful but limited" version can still be trusted — if its limits are understood.

WHY Why a Separate Web Tool? why not an IDE

  • Scale: the codebase won't fit on one machine; a centralized index is built once, benefits everyone, incrementally updated per commit (linear cost), while per-IDE indexing scales roughly quadratically. But the cross-reference index can't be incremental (any change can affect the whole codebase), so Kythe builds it daily (~6 hours for Chromium).
  • Zero-setup global view: instantly browse the whole codebase → easy to find reusable libraries and good examples; no project description/build environment needed.
  • Specialization (not being an IDE is an advantage): with no editor cursor, every click can be meaningful (jump to definition, show usages); the UX is purely for browsing and understanding.
  • Integration: a Code Search link is code's canonical "location" — log viewers, stack frames, compile errors, codelabs/docs link back (even unsubmitted workspace code); it also exposes APIs and has vim/emacs/IntelliJ plugins.

SCALE Impact of Scale & Implementation latency → engineer-time, sparse n-gram

  • Query latency: over a million queries a day; one extra second ≈ 35 idle full-time engineers daily, with the backend taking ~1/10 of those engineers → ~100,000 queries/day is break-even. <200ms responsive, 1s drifts, 10s context-switch → target sub-200ms.
  • Index latency: instant indexing is expected while writing code; median commit-to-visible < 10s. The index/running-code gap is especially troublesome during incident debugging.
  • Implementation: 1.5TB, 200 qps, median search < 50ms; brute force needs 300,000 cores; trigram → suffix array → sparse n-gram (500× faster, answers regex); rides Google's main search stack; index moved memory → flash; full history adds only 2.5×.
  • Ranking: query-independent (file views, references/priority, offline) vs query-dependent (token/symbol/filename boosts, cheap); supplemental retrieval; result diversity.

TRADE-OFFS Selected Trade-Offs completeness vs speed/resources/trust

  • Repository at head: drop binaries/generated JS to save resources and noise; but dropping files hurts trust (you can't signal "incomplete" for unindexed files) → err toward indexing too much.
  • All vs most-relevant: normal search trades completeness for speed; but refactoring/tooling needs all results. Fix: shard by priority, fetch all when requested (~1/3 of queries have < 20 results).
  • Head vs history vs workspaces: indexing Piper's linear history → search any snapshot, find deleted code, by author; obsolete code can simply be deleted.
  • Token vs substring vs regex: token scales well but fails for code's meaningful characters; substring (trigram) is small but lower-recall; regex is most powerful but hard to index efficiently (approximated via substring).

Key terms

Code Search
Google's tool for browsing and searching code (frontend UI + cloud backends); optimized for reading, understanding, exploring at scale, not an IDE.
Kythe
A compiler-based indexing service providing cross-references, using full build info to disambiguate; its integration shifted Code Search from searching to browsing.
"Answer the next question in a single click"
The guiding principle post-Kythe; where defined/used, when added, fleet-wide CPU cycles, all a click or two away.
The five search intents
How ~1/3 (examples), What ~1/4 (browsing), Where 16%, Why 16% (debugging), Who/When 8%.
Centralized vs per-IDE index
Centralized is built once, incrementally updated (linear cost), benefiting everyone; per-IDE indexing scales roughly quadratically.
Search vs cross-reference index
The search index is incremental, median < 10s to appear; the cross-reference index (Kythe) can't be incremental, needs a near-full build, built daily.
Sub-200ms / 1s / 10s
<200ms is responsive, 1s drifts attention, 10s context-switches; one extra second ≈ 35 idle engineers daily.
Sparse n-gram index
The current scheme, 500×+ faster than brute force and fast for regex; evolved trigram → suffix array → sparse n-gram.
Query-independent vs dependent signals
Query-independent (file views, references/priority — offline) vs query-dependent (token/symbol/filename boosts — must be cheap).
Supplemental retrieval
Rewriting the query into more specialized ones (e.g., restricted to definitions/filenames) so key results aren't cut off within a fixed count.
Completeness trade-offs
Dropping binaries/generated JS saves resources but hurts trust, so err toward indexing too much; priority shards serve both "most-relevant" and "all results."
Token vs substring vs regex
Token scales well but fails for code's meaningful characters; substring (trigram) is small but lower-recall; regex is most powerful but hard to index efficiently.

Questions to push your thinking

  1. Does your team find code via grep + per-developer IDE indexing? Tally the quadratic cost of "everyone indexes their own": does it hold as the codebase and team grow? Is a centralized, built-once, everyone-benefits index worth investing in?
  2. Classify your recent search-solved problems into How/What/Where/Why/Who-When: is the biggest "examples of how others did it"? What does that imply for which tools/docs to invest in? Per ch10.
  3. Can you convert your tool's latency into engineering time ("one extra second ≈ N idle engineers daily")? Where does it fall — 200ms / 1s / 10s?
  4. Has your tooling become a canonical "location" — logs, stack frames, docs, bugs all linking to one place? Or does each build its own UI, unaware of the others?
  5. Apply Code Search's completeness trade-offs: do you quietly drop results to save resources without telling users? Is the trust cost worth it? Per ch14's "no silent caps."
