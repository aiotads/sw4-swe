Compute as a Service

運算即服務
作者 Onufry Wojtaszczyk PDF p.545–574 Scale 規模 Time 時間 Trade-offs 取捨
"I don't try to understand computers. I try to understand the programs." —Barbara Liskov 我不試著去理解電腦，我試著去理解程式。——Barbara Liskov

摘要

寫完程式後你需要硬體來跑它——這本質上就是 Compute as a Service（CaaS，運算即服務）。本章（全書最後一章）追蹤 Google 內部 Borg（Kubernetes/Mesos 的先驅）的演化，講「給我硬體跑我的東西」如何長成能隨組織擴展的系統，分四部：馴服運算環境／為受管運算寫軟體／CaaS 隨時間與規模／選擇運算服務。

貫穿全章的是「cattle, not pets（把機器當牛、不當寵物）」——壞了能全自動 stamp out 替換、不需人工，由此自癒；這被視為 Google 15 年成功的根本工具。最大敵人是狀態：本機狀態當短暫、真儲存外置到複本同步的持久系統（GFS/Bigtable/Spanner），靠 service discovery 與 idempotency 補位。容器初為隔離/多租戶（cgroups 2007 由 Google 貢獻進 kernel），更重要的是成了應用與機器間的抽象邊界。

Hyrum's Law 在容器抽象上特別猛（PID 32,000 案例修了八年仍未完）。Borg 一統 batch 與 serving：serving 變牛省線性管理成本、batch 填 serving slack 幾乎免費跑。選運算服務是高 lock-in 的取捨：集中 vs 客製、抽象層級（serverless 要真正無狀態、犧牲環境控制；Google 不重押）、公有 vs 私有（外包 overhead、怕 lock-in，用開源/多雲/混合雲緩解）。結論：規模需要統一的共用運算基礎設施，容器是關鍵，用好它要把應用設計成「牛」。

為什麼要讀這一篇

你還在 SSH 進機器、手動部署與重啟行程。 本章從 Jeff Dean 2002 的「50+ 台手動啟動/監控/遷移」惡夢講起，給你清楚的演化路徑：簡單自動化 → 自動排程 → 容器化多租戶 → rightsizing/autoscaling。你會知道 naive 解法在什麼規模、為什麼會崩。
你的服務是「寵物」，壞了得有人 panic 地救。 「cattle, not pets」是核心心法：能全自動 stamp out 替換的牛，維護負擔才不會隨 fleet 線性/超線性成長。但本章誠實提醒：當牛還不夠，狀態才是真正的難關。
你在為團隊/組織選運算服務（VM？容器？serverless？公有雲？）。 最後一節講清楚高 lock-in 的本質、集中 vs 客製的張力、serverless 的真正比較對象（持久容器而非 VM 寵物）、以及公有/多雲/混合雲緩解 lock-in 的手法。
你想理解 Hyrum's Law 在基礎設施層有多可怕。 PID 32,000 案例：一個沒人承諾的實作細節被無數系統當保證依賴，修起來跨八年、分三階段、還要 backport 資料。永久改變你對「可觀察行為＝API」的警覺。

細節重點

演化 管理運算的光譜：寵物 → 牛 → serverless pets → cattle → serverless

naive 的 SSH/SFTP 手動解法在規模下崩潰；馴服運算環境是一條「抽象漸增、管理 overhead 漸減」的演化路徑——但抽象越多越好並不成立，每一步都是取捨。
馴服運算環境：簡單自動化（部署腳本、自癒重啟）→ 自動排程（中央服務管機器清單、故障重排）→ 容器化多租戶（按資源需求 bin-pack、需隔離；cgroups 2007 進 kernel）→ rightsizing/autoscaling（自動定資源/複本數，免人工填數字，「簡單的事該簡單」）。為失敗而設計：工作切成可動態分配的小 chunk；狀態當短暫、真儲存外置（state replication，GFS/Bigtable/Spanner）；service discovery 取代寫死 hostname；mutating 請求要 idempotency。

效率 一統天下：Borg 把 batch 填進 serving slack one service to rule them all

把 batch 與 serving 併成單一池有兩大效率：serving 變牛（免每隊各管一池的線性成本）＋batch 填 serving 為抗尖峰保留的閒置，使 batch 幾乎免費跑。
容器作為抽象／Hyrum's Law：容器隔開應用與機器，機器變了只有容器層要適配。但 PID 32,000：Borg 把 PID 限在 0–32,000 成了被依賴的隱式「API」（log 假設五位數），修法分三階段（暫時上限→拆執行緒/行程 PID 空間→PID namespace），八年後仍在進行——故用廣社群開發的容器系統有價值。選運算服務高 lock-in：集中 vs 客製（Borg 為 Cloud VM/Search 磁碟做 bifurcation 讓 API 龐雜）、公有 vs 私有（怕 Zimki 式倒閉，用開源/多雲/混合雲＋破繭路徑緩解）。

關鍵字補充

Compute as a Service（CaaS，運算即服務）
「給我硬體跑我的東西」長成的受管運算系統；Borg 是 Kubernetes/Mesos 的先驅。
Cattle, not Pets（把機器當牛、不當寵物）
把伺服器設計成能全自動 stamp out 替換的牛而非要人搶救的寵物；規模化與自癒的根本。
Bin-packing（裝箱）／Multitenancy（多租戶）
按程式資源需求把複本裝箱到機器池共享機器；需隔離以免互相干擾。
Isolation（隔離）／Container（容器）
多租戶必備；容器是比 VM 輕量的隔離（cgroups 2007 由 Google 貢獻進 kernel）＋應用與機器間的抽象邊界。
Rightsizing / Autoscaling
自動決定資源需求與複本數，免人工填數字；「簡單的事該簡單、複雜的事該可能」。
Architecting for Failure（為失敗而設計）
工作切成可動態分配的小 chunk、serving job 收下線預告後優雅排空；當牛還不夠。
Batch vs Serving
batch 關心吞吐、短命、可無預警殺；serving 關心延遲、長命、重排要節流且需下線預告。
State Replication／Transient Local Storage
本機狀態當短暫、真儲存外置到複本同步的持久系統（GFS/Bigtable/Spanner）；快取按延遲、核心按總負載配置。
Service Discovery／Idempotency（冪等性）
一層間接定址取代寫死 hostname；retry 與重複請求的解藥是冪等（client 指派 ID）。
Hyrum's Law（在容器抽象上）
PID 32,000 案例：沒人承諾的實作細節被當 API 依賴，修正跨八年三階段；故用廣社群開發的容器系統有價值。
One Service to Rule Them All
Borg 把 batch 與 serving 併成單一池：serving 變牛省線性管理成本＋batch 填 serving slack 幾乎免費跑。
Serverless / Lock-in
只寫 Handler、框架 server 多租戶、能縮到零；要求真正無狀態、犧牲環境控制權；比較對象是持久容器。運算服務選擇高 lock-in，用開源/多雲/混合雲緩解。

延伸提問

  1. 你的服務是「寵物」還是「牛」？能不能做到「一台壞了，自動化全自動 stamp out 一台新的、不需任何人 SSH 進去設定」？若不行，卡在哪——部署不自動，還是藏了狀態（行程內/本機磁碟/寫死 hostname）讓它無法替換？對照 ch22。
  2. 你把本機狀態當「短暫」了嗎？哪些其實該外置到持久儲存？你的快取是「按延遲目標配置、核心按總負載配置」，還是把快取當不能失去的依賴（一掛就 outage）？你的 mutating API 有 idempotency 嗎？
  3. 你的基礎設施有沒有「PID 32,000」式的隱式依賴——某個沒人承諾的實作細節（埠範圍、ID 格式、檔案佈局）被無數地方當保證依賴？要改估得出代價嗎？這正是 ch01 Hyrum's Law 在基礎設施層的體現。
  4. 你在選運算服務嗎？評估過 lock-in 的真實代價（Hyrum's Law＋周邊工具生態難遷移）嗎？你的需求是真正無狀態（適合 serverless）還是有狀態（需持久容器）？小團隊用公有雲 serverless、大組織用統一容器——你在哪一格？
  5. 你的組織是 batch/serving 各管一池，還是有機會像 Borg 那樣統一以吃到「serving 變牛省管理成本＋batch 填 slack 幾乎免費」？你算過各隊各管一池的線性管理成本嗎？對照 ch07。

Part IV — Tools
CHAPTER 25

Compute as a Service

Just give me hardware to run my stuff — at scale, over time
by Onufry Wojtaszczyk PDF p.545–574 Scale Time Trade-offs
"I don't try to understand computers. I try to understand the programs." —Barbara Liskov CaaS abstracts away the toil of managing hardware so engineers can focus on the programs.

Summary

After writing code you need hardware to run it — that, in essence, is Compute as a Service (CaaS). This final chapter traces Google's Borg (precursor to Kubernetes/Mesos), showing how "just give me hardware to run my stuff" grows into a system that scales with the org, in four parts: taming the compute environment / writing software for managed compute / CaaS over time and scale / choosing a compute service.

Running through it all is "cattle, not pets" — a new instance stamped out fully automatically, no manual setup, yielding self-healing; held to be fundamental to Google's 15 years. The biggest enemy is state: treat local state as transient and externalize real storage to a replicated persistent system (GFS/Bigtable/Spanner), backed by service discovery and idempotency. Containers began for isolation/multitenancy (cgroups, into the kernel in 2007) but mattered more as an abstraction boundary between application and machine.

Hyrum's Law hits the container abstraction especially hard (the PID-32,000 case, still being fixed eight years on). Borg unifying batch and serving: serving becomes cattle (saving linear management cost), batch fills serving's slack (running effectively for free). Choosing a compute service is a high-lock-in trade-off: centralization vs customization, level of abstraction (serverless demands true statelessness and cedes control; Google didn't bet heavily on it), and public vs private (outsource overhead, fear lock-in, mitigate with open source / multicloud / hybrid). Conclusion: scale requires a common shared compute infrastructure; containers are key; use it well by designing applications as cattle.

Why read this

You're still SSH-ing into machines, manually deploying and restarting. The chapter starts from Jeff Dean's 2002 nightmare ("start/monitor/migrate by hand on 50+ machines") and gives a clear evolution: simple automation → automated scheduling → containerized multitenancy → rightsizing/autoscaling. You'll learn at what scale, and why, the naive solution collapses.
Your service is a "pet" that needs human rescue. "Cattle, not pets" is the core: cattle that can be stamped out fully automatically, so maintenance doesn't grow linearly/superlinearly with the fleet. But it's honest: cattle isn't enough — state is the real challenge.
You're choosing a compute service (VM? containers? serverless? public cloud?). The final section spells out the high-lock-in nature, the centralization-vs-customization tension, serverless's real comparison (persistent containers, not VM pets), and public/multi/hybrid-cloud mitigations.
You want to grasp how terrifying Hyrum's Law is at the infrastructure layer. The PID-32,000 case: an unpromised detail depended on as a guarantee everywhere, whose fix spans eight years, three phases, and data backfill. It permanently sharpens "observable behavior = API."

Key details

EVOLUTION The spectrum of managing compute: pets → cattle → serverless

The naive SSH/SFTP manual solution collapses at scale; taming the compute environment is a path of "increasing abstraction, decreasing management overhead" — but "more abstraction is always better" is false; each step is a trade-off.
Taming the compute environment: simple automation (deploy scripts, self-healing restart) → automated scheduling (a central service owns the machine list, reschedules on failure) → containerized multitenancy (bin-pack by resource needs, requires isolation; cgroups into the kernel in 2007) → rightsizing/autoscaling (auto-determine resources/replicas, sparing manual sizing). Architect for failure: chunk work for dynamic assignment; treat state as transient and externalize real storage (state replication, GFS/Bigtable/Spanner); service discovery instead of hardcoded hostnames; mutating requests need idempotency.

EFFICIENCY One service to rule them all: batch fills serving's slack

Unifying batch and serving into one pool brings two efficiencies: serving becomes cattle (no per-team linear cost) + batch fills the slack serving reserves for spikes, so batch runs effectively for free.
Containers as abstraction / Hyrum's Law: containers separate application from machine, so only the container layer adapts when the machine changes. But PID 32,000: Borg limiting PIDs to 0–32,000 became a depended-on implicit "API" (logs assumed five digits), fixed in three phases (temporary cap → split thread/process PID space → PID namespaces), still ongoing eight years on — hence the value of a broadly-used container system. Choosing is high-lock-in: centralization vs customization (Borg's Cloud-VM/Search-disk bifurcations made the API unwieldy), public vs private (fearing a Zimki-style shutdown, mitigated by open source / multicloud / hybrid + a break-out path).

Key terms

Compute as a Service (CaaS)
The managed compute system "just give me hardware to run my stuff" grows into; Borg is the precursor to Kubernetes/Mesos.
Cattle, not pets
Design servers as cattle stamped out fully automatically, not pets a human rescues; the root of scaling and self-healing.
Bin-packing / multitenancy
Pack replicas onto a machine pool by resource needs to share machines; requires isolation to prevent interference.
Isolation / container
What multitenancy needs; lighter isolation than VMs (cgroups, into the kernel in 2007) plus an abstraction boundary between application and machine.
Rightsizing / autoscaling
Auto-determine resource needs and replica counts, sparing manual sizing; "easy things easy, complex things possible."
Architecting for failure
Chunk work for dynamic assignment; serving jobs drain gracefully on a shutdown notice; being cattle isn't enough.
Batch vs serving
batch cares about throughput, short-lived, killable without warning; serving cares about latency, long-lived, needs throttled rescheduling and notice.
State replication / transient local storage
Treat local state as transient, externalize real storage to a replicated persistent system; provision cache for latency, core for total load.
Service discovery / idempotency
A layer of indirection instead of hardcoded hostnames; the antidote to retries/duplicates is idempotency (client-assigned IDs).
Hyrum's Law (on the container abstraction)
The PID-32,000 case: an unpromised detail depended on as API, fixed over eight years and three phases; hence the value of a broadly-used container system.
One service to rule them all
Borg unifying batch and serving: serving becomes cattle (saving linear management cost) + batch fills serving slack (effectively free).
Serverless / lock-in
Write only the Handler, multitenant servers, scales to zero; demands true statelessness, cedes control; compared to persistent containers. Compute choices are high-lock-in, mitigated by open source / multicloud / hybrid.

Questions to push your thinking

  1. Is your service a "pet" or "cattle"? Can you do "one dies, automation stamps out a new one, no SSH required"? If not, where are you stuck — non-automated deploy, or hidden state (in-process / local disk / hardcoded hostnames)? Per ch22.
  2. Do you treat local state as transient? What should be externalized? Is your cache "provisioned for latency, core for total load," or a dependency you can't lose (an outage when it dies)? Does your mutating API have idempotency?
  3. Does your infrastructure have a "PID-32,000" implicit dependency — an unpromised detail (port range, ID format, file layout) depended on as a guarantee everywhere? Can you estimate the cost of changing it? This is ch01's Hyrum's Law at the infrastructure layer.
  4. Choosing a compute service? Have you assessed the real cost of lock-in (Hyrum's Law + the tooling ecosystem)? Are your needs truly stateless (serverless) or stateful (persistent containers)? Small team on serverless, big org on unified containers — which box?
  5. Does your org run batch and serving in separate pools, or could you unify like Borg for "serving becomes cattle + batch fills the slack (free)"? Have you measured the linear management cost of every team running its own pool? Per ch07.
