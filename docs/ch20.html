Static Analysis

靜態分析
作者 Caitlin Sadowski PDF p.445–456 Scale 規模 Trade-offs 取捨 Time 時間
"Focus on developer happiness." 以開發者的幸福感為念——建立反饋迴路、贏得信任，靜態分析才會真正被使用。

摘要

靜態分析是分析原始碼（而非執行中程式＝dynamic）以找 bug、反模式與可在不執行下診斷的問題。它能及早抓 bug，但用途遠不只如此——還用來編碼最佳實務、跟上現代 API、預防技術債，更是 API 棄用防「回潮」的關鍵（ch15、ch22）。

兩大特性：Scalability（shardable/incremental，只分析待審變更所影響的檔案/行；擴展分析的數量與種類）與 Usability（權衡成本效益、聚焦新引入的警告、能自動修就自動修、平順整合工作流）。三條關鍵教訓：①以開發者幸福為念（低誤報、反饋迴路、信任；effective false positive＝開發者沒採取正面行動就算誤報）②成為核心工作流的一部分（code review 是甜蜜點）③賦能使用者貢獻（簡單 API 如 Refaster）。

平台是 Tricorder（與 Critique 整合、每天 5 萬+ 變更）；新 check 四準則（可理解/可行動易修/<10% 有效誤報/顯著影響潛力，整體 <5%）。其餘：整合工具（Error Prone/clang-tidy）、反饋管道（Not useful/Please fix）、suggested fixes、per-project（非 per-user）客製、presubmits、compiler 整合（從不發 warning——要嘛 error 要嘛不顯示）。

為什麼要讀這一篇

你導入過靜態分析，結果大家嫌吵然後忽略。 關鍵是低誤報率與信任，衡量標準是 effective false positive（開發者沒採取正面行動就算誤報）。建反饋迴路、壓到 10% 以下，工具才會被用。
你不知道該在哪個環節跑。 完整地圖：IDE（<100ms）→ code review（甜蜜點）→ presubmit（擋提交）→ compiler（break build）——愈左愈早、愈右愈強制。
你想規模化又不想養龐大工具團隊。 答案是 empower users to contribute：建易 plug-in 的生態、用簡單 API（Refaster：給前後片段）讓領域專家自己寫——100+ 分析器多來自團隊外。
你在猶豫要不要讓使用者各自關掉檢查。 血淚教訓：做 per-project、不要 per-user——移除 user 客製反而幫忙找出工具真 bug；user 客製只會藏 bug、壓反饋。

細節重點

教訓 三條關鍵教訓 Key Lessons

讓靜態分析有效的可擴展性與可用性，源於三條核心教訓：
  • Scalability：shardable/incremental，只對待審變更所影響的檔案/行顯示結果；擴展分析的數量與種類、直接秀給相關工程師避免瓶頸。
  • Usability：聚焦新引入的警告（既有運作正常的程式，「修」可能反引入 bug）；能自動修就自動修。

整合 Tricorder 與工作流整合點 愈左愈早、愈右愈強制

Tricorder 與 Critique 整合（灰色框）、微服務、每天 5 萬+ 變更；新 check 四準則（可理解/可行動易修/<10% 有效誤報/顯著影響潛力）。靜態分析可在工作流的多個點浮現：
  • 整合工具：Error Prone（Java）/clang-tidy（C++）找 AST 反模式（int 右移 32 是 no-op，修了 31 處）；IfThisThenThat、binary size checker；幾乎全是 intraprocedural。
  • 反饋管道：「Not useful」（對作者開 bug）/「Please fix」；有時修好「訊息」就解決問題（printf %s 例）。
  • Suggested fixes：從 Critique 或 CLI 一鍵套用；風格問題該自動修。每天 Please fix 數千次、自動修 ~3,000 次、Not useful ~250 次。
  • Per-project 客製：專案層級確保全隊一致視圖；移除 user 客製反而找出真 bug（C++ linter 跑 Obj-C、HTML linter 高誤報）。
  • Presubmit：擋提交（DO NOT SUBMIT、測試、格式）；`CLEANUP=` 跳過部分以利 LSC。

原文引用

"An issue is an 'effective false positive' if developers did not take some positive action after seeing the issue."一個問題是「有效誤報」，只要開發者在看到它之後沒有採取任何正面行動。
Focus on Developer Happiness——把誤報從技術定義改為以使用者行動定義。
"Who wants to wade through hundreds of false reports in search of a few true ones?"誰想為了找出少數幾個真的問題，去涉水翻過數百個假的報告？
低 false-positive 率為何比減少漏報更重要——工具被真正採用的關鍵。
"Code review is a sweet spot for analysis results."程式碼審查，是呈現分析結果的甜蜜點。
改動心態、被阻塞時有時間跑分析、同儕壓力、替審查者省時。
"Anything that can be fixed automatically should be fixed automatically."任何能被自動修正的，都應該被自動修正。
Usability／Suggested fixes——降低成本效益裡的「成本」；風格問題不該浪費人工審查時間。
"We have found repeatedly that developers ignore compiler warnings."我們一再發現：開發者會忽略編譯器警告。
Compiler Integration——故力求從不發 warning：要嘛當 error（break build），要嘛不顯示。

延伸提問

  1. 你導入的靜態分析工具，false-positive 率是多少？你用「技術正確性」還是 effective false positive（開發者有沒有採取正面行動）來衡量？有沒有一條反饋迴路讓使用者回報「沒用」並真的據以調整？
  2. 你的靜態分析在哪個環節跑？沿 IDE→code review→presubmit→compiler，你落在哪、該往哪推？哪些檢查機械到可 break the build、哪些該留在 code review？對照 ch19、ch23。
  3. 你的團隊是不是讓使用者各自關掉煩人的檢查？能不能改成專案層級客製、並把「煩人」當成找 bug 的線索而非壓住反饋？
  4. 你有沒有把領域專家的知識規模化？哪些「修過一次卻可能再現」的 bug 值得寫成 check 防回潮？有沒有像 Refaster 那樣夠簡單的 API 讓非專家貢獻？
  5. 你的工具有自動修正嗎？哪些目前靠人工審查者一一指出的問題（格式、風格、deprecated 用法）其實該被 formatter/auto-fix 取代？對照 ch09。

Static Analysis

Build the feedback loop and earn the trust
by Caitlin Sadowski PDF p.445–456 Scale Trade-offs Time
"Focus on developer happiness." Build the feedback loop and earn the trust — that's what makes static analysis actually get used.

Summary

Static analysis analyzes source code (not a running program = dynamic) to find bugs, antipatterns, and issues diagnosable without execution. It catches bugs early, but does far more — codifying best practices, keeping code current with APIs, preventing tech debt, and is key to preventing API-deprecation backsliding (ch15, ch22).

Two properties: Scalability (shardable/incremental, analyzing only the files/lines affected by a pending change; scaling the number and variety of analyses) and Usability (weigh cost-benefit, focus on newly introduced warnings, fix automatically when possible, integrate smoothly). Three key lessons: ① focus on developer happiness (low false positives, feedback loop, trust; the effective false positive = no positive action taken) ② make it part of the core workflow (code review is the sweet spot) ③ empower users to contribute (simple APIs like Refaster).

The platform is Tricorder (integrated with Critique; 50,000+ changes a day); new checks meet four criteria (understandable / actionable & easy to fix / < 10% effective false positives / significant impact potential; overall < 5%). Plus: integrated tools (Error Prone/clang-tidy), feedback channels (Not useful/Please fix), suggested fixes, per-project (not per-user) customization, presubmits, and compiler integration (never issue warnings — make it an error or don't show it).

Why read this

You rolled out static analysis and everyone found it noisy and ignored it. The key is low false positives and trust, measured by the effective false positive (no positive action taken). Build a feedback loop, drive it under 10%, and it gets used.
You don't know where to run it. The full map: IDE (<100ms) → code review (sweet spot) → presubmit (block commit) → compiler (break build) — earlier left, more forcing right.
You want to scale without a huge tools team. The answer is empower users to contribute: an easy-to-plug-into ecosystem with simple APIs (Refaster: before/after snippets) so domain experts write checks — most of 100+ analyzers come from outside.
You're tempted to let users turn off annoying checks. The hard lesson: do per-project, not per-user — removing user customization helped find real tool bugs; user customization only hides bugs and suppresses feedback.

Key details

LESSONS The Three Key Lessons

The scalability and usability that make static analysis effective arise from three core lessons:
  • Scalability: shardable/incremental, showing results only for the files/lines affected by a pending change; scale the number and variety of analyses, show directly to relevant engineers.
  • Usability: focus on newly introduced warnings (for working code, "fixing" may introduce a bug); fix automatically when possible.

INTEGRATION Tricorder & workflow integration points earlier left, more forcing right

Tricorder is integrated with Critique (gray boxes), microservices, 50,000+ changes a day; four criteria for new checks. Static analysis can surface at several workflow points:
  • Integrated tools: Error Prone (Java)/clang-tidy (C++) find AST antipatterns (an int right-shift by 32 is a no-op — fixed 31 occurrences); IfThisThenThat, a binary-size checker; almost all intraprocedural.
  • Feedback channels: "Not useful" (file a bug against the writer) / "Please fix"; sometimes fixing the message fixes the problem (the printf-%s example).
  • Suggested fixes: one-click apply from Critique or a CLI; style should be auto-fixed. ~thousands of "Please fix," ~3,000 auto-fixes, ~250 "Not useful" a day.
  • Per-project customization: a consistent team view; removing user customization revealed real bugs (a C++ linter on Obj-C, a high-false-positive HTML linter).
  • Presubmit: blocks commits (DO NOT SUBMIT, tests, formatting); `CLEANUP=` skips some to ease LSCs.

Key terms

Static analysis
Analyzing source code (not a running program) to find bugs/antipatterns; catches bugs early, codifies best practices, keeps code current, prevents tech debt and deprecation backsliding.
Scalability
Shardable/incremental, analyzing only the files/lines affected by a pending change; scale the number and variety, show directly to relevant engineers.
Usability
Weigh the cost-benefit; focus on newly introduced warnings, fix automatically when possible, integrate smoothly.
Effective false positive
A false positive defined by user action: any issue where the developer took no positive action; perception and message clarity matter as much as correctness.
The three key lessons
Developer happiness (trust, low false positives, feedback loop), part of the core workflow (code review sweet spot), empower users to contribute (simple APIs, domain experts).
Tricorder
Google's static-analysis platform, integrated with Critique, microservices, 50,000+ changes a day; four criteria for new checks; overall effective-false-positive rate < 5%.
Error Prone / clang-tidy
Extend the compiler to find Java/C++ AST antipatterns; Error Prone "ERROR" checks are enabled in the compiler to prevent recurrence.
Refaster
Write an analyzer from before/after code snippets; lets non-analysis/language experts contribute.
Suggested fixes
One-click apply from Critique or a CLI; style issues should be auto-fixed (formatters).
Per-project customization
Customize at the project, not user, level, ensuring a consistent team view; user customization only hides bugs and suppresses feedback.
Presubmit
A commit-blocking check (DO NOT SUBMIT, tests, formatting, specific Tricorder categories); `CLEANUP=` can skip some to ease LSCs.

Questions to push your thinking

  1. What's your tool's false-positive rate? Do you measure by "technical correctness" or the effective false positive (did the developer take positive action)? Do you have a feedback loop for users to report "not useful" and actually retune?
  2. Where does your static analysis run? Along IDE → code review → presubmit → compiler, where are you and where should you push? Which checks are mechanical enough to break the build, which belong in code review? Per ch19, ch23.
  3. Does your team let users turn off annoying checks individually? Could you switch to project-level customization and treat "annoying" as a bug-finding clue rather than suppressing feedback?
  4. Have you scaled domain-expert knowledge? Which "fixed once but could recur" bugs deserve a check? Is there an API simple enough (like Refaster) for non-experts to contribute?
  5. Does your tooling auto-fix? Which problems your reviewers point out one by one (formatting, style, deprecated usage) should be replaced by a formatter/auto-fix? Per ch09.
