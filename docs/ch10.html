CHAPTER 10

Documentation

文件
作者 Tom Manshreck PDF p.213–234 Time 時間 Scale 規模 Trade-offs 取捨
"Documentation is hugely important over time and scale. … Write for your audience, not yourself." 文件在時間與規模兩個維度上都極其重要。——為你的讀者而寫，別只為自己寫。

摘要

幾乎每位工程師都抱怨過文件「品質差、數量少、根本沒有」。本章的核心診斷：2010 年代末工程文件的處境，像極了 1980 年代末的軟體測試——大家都知道該投入更多，但組織還沒真正認識到它的關鍵價值。Google 最成功的做法，是把文件當成程式碼來對待（treat documentation like code），納入既有工程工作流。

「文件」泛指工程師為做好工作而寫的一切補充文字——不只獨立文件，程式碼註解也算。文件效益都在下游：你只寫一次，卻會被讀數百、數千次，初始成本被攤提到所有未來讀者身上——這就是它在時間與規模上的價值。最重要的心法是 C++ 風格指南的格律：「為讀者最佳化」。

全章主張：文件要有 owner、進版本控制、走 code review、用 bug 系統追蹤、定期檢查（freshness date）；寫之前先認清受眾（seekers 要一致性、stumblers 要清晰；customer 與 provider 分開）；一份文件只服務單一目的；五種文件類型各有寫法；好文件在 completeness／accuracy／clarity 三者間取捨。結論坦承：文件在 Google 還不是「一等公民」，但工程師「既是問題、也是解方」。

為什麼要讀這一篇

你覺得「寫文件」是浪費時間的雜事。 用「攤提」翻轉它：文件寫一次、被讀上千次，成本分攤到所有讀者；它和測試一樣是對未來的投資。看懂這點，你會願意付那筆前期成本。
你的團隊 wiki 早已腐爛。 GooWiki 的崩壞（約 90% 文件數月無人瀏覽或更新、一度成為 Google 開發者年度調查頭號抱怨）是前車之鑑。解方很具體：搬進版本控制、給 owner、走 code review、用 bug 追蹤。
你其實只寫給「現在的自己」。 這是本章最常被引用的錯誤。學會先問「我的讀者是誰」（seekers／stumblers、customer／provider），並用 WHO/WHAT/WHEN/WHERE/WHY 框住開頭兩段，文件立刻好用很多。
什麼時候該找 technical writer？ 違反直覺：團隊幾乎都能寫好「給自己」的文件，真正需要協助的是寫給「另一個受眾」、跨 API 邊界的文件——writer 的關鍵價值是「以局外人身分挑戰你團隊的假設」。

細節重點

為何 Why Is Documentation Needed? 為什麼需要文件

效益都在下游、不立即回饋作者（與測試相反）；但會隨時間自行回本——寫一次、被讀千次，成本攤提給所有未來讀者。它回答「當初為什麼這樣設計／實作」（包括兩年後回看的你）。
即使對作者也有四個即時好處：①幫你把 API 想清楚（「解釋不了、定義不了，大概就是沒設計好」）；②維護路線圖與歷史紀錄；③讓程式更專業（「文件好」常是「維護得好」的指標）；④減少別人來問。但絕大多數效益歸讀者——故一切回到為讀者最佳化。

核心 Documentation Is Like Code 把文件當程式碼對待

文件是一種工具，用另一種語言（通常英文）寫成，目的和程式碼裡的規範相同：強制一致、提升清晰、避免理解錯誤。所以盡量比照程式碼處理。
Case Study｜GooWiki 崩壞：沒有 owner →過時；無新增流程 →重複文件激增；扁平命名（光設定 Borg 就 7~10 份）；能修的人不是用的人。解方＝搬進版本控制，品質戲劇性改善；引進 Markdown 與 g3doc 後，能在同一個變更裡同時改程式與文件。

受眾 Know Your Audience 認清你的受眾

最重要的錯誤就是只寫給自己。動筆前先辨識主要受眾並對其書寫。好文件不必完美——「如果你能讀，你就能寫」；讀者站在你曾站過的地方，只是還沒有你現在的領域知識。
  • seekers（搜尋者）知道要什麼→關鍵是一致性（reference 格式統一好掃讀）；stumblers（誤撞者）還不確定→關鍵是清晰（檔頭給概觀、TL;DR 標明「不感興趣可停讀」）。
  • customer（API 使用者）與 provider（專案成員）的文件要分開——實作細節對維護者重要，終端使用者不該需要讀到。
  • 訣竅是讓文件保持精短——寫短常需先寫長再刪修。Pascal：「要是我有更多時間，這封信會寫得更短。」

類型 Documentation Types 五種類型，別混用

一份文件該有單一目的並堅守它（像 API「只做一件事、做好」）。早期「巨型 wiki 頁」把連結、概念、reference 全攪在一起而失敗。
  • Reference（參考，含註解）：最常見。API 註解 vs 實作註解；盡量單一來源、從註解生成。function comment 用祈使動詞開頭（Merges, Deletes, Creates）；Google 不用 Returns:／Throws: 樣板，一段散文把參數/回傳/例外一起講更清楚。
  • Design Docs（設計文件）：重大專案前須核准；模板強制考慮 security／i18n／storage／privacy；涵蓋目標、策略、關鍵決策與取捨、替代方案；是「寫程式前的 code review」與日後衡量成敗的標尺。
  • Tutorials（教學）：「Hello World」；最佳時機是你剛加入團隊時；明確編號每一步、合併原子操作、別編號系統回應。
  • Conceptual（概念）：補充而非取代 reference；最難寫、最常被忽略；犧牲完整與準確換清晰。「註解是文件的 unit test，概念文件就是 integration test」。
  • Landing Pages（著陸頁）：只當交通警察——標示用途、只放連結；別一頁服侍兩個主人（顧客入口 vs 團隊首頁）。

審查 Documentation Reviews 三種審查

想「測試」文件有沒有用，就找別人讀讀看。三種審查：technical review（求準確，領域專家、常就是 code review 一部分）、audience review（求清晰，不熟領域者）、writing review（求一致，technical writer）。文件一旦 tied 進工程工作流，就會隨時間改善。

哲學 Documentation Philosophy WHO/WHAT/WHY、三參數取捨

多數技術文件只答 HOW，工程師常忽略其餘五問（設計文件例外——WHY 與 HOW 同等重要）；在開頭兩段補足 WHO/WHAT/WHEN/WHERE/WHY 可避免混淆。文件至少該有開頭、中段、結尾；在文件裡適度冗餘是有用的。
Deprecating（棄用）：文件不再有用就移除或標為過時；freshness date（逾期三個月寄提醒）＋「Last reviewed by…」署名是低成本維護法。何時需 technical writer：團隊都能寫好「給自己」的文件，外援用在跨 API 邊界、寫給另一受眾的文件——以局外人挑戰假設。結論：文件在 Google 還不是一等公民（本質主觀、由讀者非同步衡量）；工程師「既是問題、也是解方」。

關鍵字補充

本章核心方法論：文件要有 owner、進版本控制、走 code review、用 bug 系統追蹤、定期評估；沿用既有開發者工作流，而非另建新流程。

取自 C++ 風格指南；文件效益絕大多數歸讀者，寫一次被讀千次，故以讀者視角為準。
Canonical Documentation（權威文件）
多份文件衝突時指定的唯一主要來源；用 go/ 連結與「放進原始碼並列」扶植，其他重複者併入或棄用。
g3doc
Google 把文件嵌在程式碼旁的框架（Markdown、可在 Code Search 瀏覽）；讓工程師在同一變更裡同時改程式與文件。
Seekers vs Stumblers（搜尋者 vs 誤撞者）
seeker 知道要什麼、關鍵是一致性；stumbler 還不確定、關鍵是清晰（檔頭概觀、TL;DR）。
Customer vs Provider（顧客 vs 提供者）
API 使用者 vs 專案成員；兩者文件要分開，實作細節與設計決策不該塞進 API 文件。
API vs Implementation Comments（API vs 實作註解）
前者不談實作/設計、不假設讀者熟 API；後者可假設較多領域知識（但別假設太多，人會離開專案）。
五種文件類型（Reference/Design/Tutorial/Conceptual/Landing）
各有單一目的、不可混用；conceptual 最難寫、補充而非取代 reference，犧牲完整與準確換清晰。
Completeness／Accuracy／Clarity（完整／準確／清晰）
好文件三參數，難三者兼得；「好文件」＝「正在做它該做的那件事」，一份文件別做超過一件事。
Freshness Date（新鮮度日期）
記錄上次審查時間、逾期（如三個月）自動寄提醒；配「Last reviewed by…」署名，低成本確保定期檢視。
WHO/WHAT/WHEN/WHERE/WHY
多數技術文件只答 HOW；在開頭兩段補足這五問避免混淆（設計文件的 WHY 與 HOW 同等重要）。

延伸提問

  1. 把你最近寫的一份文件攤開：它其實是寫給「現在的你」還是寫給「讀者」？主要受眾是 seeker 還是 stumbler、customer 還是 provider？在開頭兩段補上 WHO/WHAT/WHY，它會不會立刻好用很多？
  2. 盤點你團隊的文件：有多少像 GooWiki 那樣沒有 owner、過時、重複？哪幾份可以搬進版本控制、給 owner、走 code review、加 freshness date，讓它「像程式碼一樣」被維護？（對照 ch15 棄用）
  3. 你的 API 文件裡有沒有混進設計決策或實作細節？把它們挪到 design doc 會不會更清晰？有沒有試圖「一頁服侍兩個主人」（landing page 同時當團隊首頁與顧客入口）？
  4. 用 completeness／accuracy／clarity 三角檢視你最不滿意的一份文件：它想三者兼得卻什麼都沒做好嗎？明確選定它的單一目的後，你會犧牲哪一項來成全另一項？
  5. 你的團隊是不是把「找 technical writer」當萬靈丹？對照本章：你們真正需要外援的，是跨 API 邊界、寫給另一受眾的文件嗎？還是「給自己的文件」其實團隊自己就能寫好，只缺把它當程式碼維護的紀律？

Part III — Processes
CHAPTER 10

Documentation

Treat docs like code — and write for the reader, not yourself
by Tom Manshreck PDF p.213–234 Time Scale Trade-offs
"Documentation is hugely important over time and scale. … Write for your audience, not yourself." Docs pay off across time and scale — and the reader, not you, is who they're for.

Summary

Nearly every engineer has complained about documentation — its quality, quantity, or sheer absence. The core diagnosis: the state of engineering docs in the late 2010s resembles software testing in the late 1980s — everyone knows more effort is needed, but the org hasn't recognized its critical benefits. Google's most successful approach: treat documentation like code and fold it into the existing engineering workflow.

"Documentation" means every supplemental text an engineer writes — not just standalone docs but code comments too. The benefits are all downstream: written once, read hundreds or thousands of times, with cost amortized across all future readers — that's its value over time and scale. The guiding maxim, from the C++ Style Guide: "optimize for the reader."

The claims: docs should have owners, live in source control, go through code review, be bug-tracked, and be periodically checked (freshness dates); know your audience first (seekers want consistency, stumblers want clarity; keep customer and provider apart); a doc serves a single purpose; the five types each have their craft; good docs trade off among completeness, accuracy, clarity. The candid conclusion: docs aren't yet a first-class citizen at Google, but engineers are "both the problem and the solution."

Why read this

You think writing docs is wasted time. Amortization flips that: a doc is written once, read thousands of times, its cost spread across all readers. Like testing, an investment in the future. See that and you'll pay the upfront cost.
Your team wiki has rotted. GooWiki's collapse (~90% of docs unviewed/unupdated for months; once Google's #1 developer-survey complaint) is the cautionary tale. The fix is concrete: source control, owners, code review, bug tracking.
You write only for present-you. The chapter's most-cited mistake. Ask "who is my audience" first (seekers/stumblers, customer/provider), and frame the first two paragraphs with WHO/WHAT/WHEN/WHERE/WHY — instantly more useful.
When do you need a technical writer? Counterintuitively: teams can write their own docs fine; help is needed for docs aimed at another audience, crossing API boundaries — the writer's value is "challenging your team's assumptions as an outsider."

Key details

WHY Why Is Documentation Needed?

Benefits are all downstream and don't immediately reward the author (the opposite of testing); but they pay off over time — written once, read thousands of times, amortized across all readers. Docs answer "why was this designed/implemented this way" (including for the you reading it two years later).
Even for the author, four immediate wins: ① it formulates the API ("can't explain or define it → not designed well enough"); ② a maintenance road map and historical record; ③ more professional code ("good docs" indicates "well maintained"); ④ fewer questions from others. But most benefit accrues to readers — so it returns to optimize for the reader.

CORE Documentation Is Like Code

Docs are a tool, written in another language (usually English) for the same goals as code conventions: enforce consistency, improve clarity, avoid comprehension errors. So treat them like code wherever possible.
Case Study | GooWiki's collapse: no owners → obsolescence; no add-process → duplicate proliferation; flat namespace (7–10 docs just for Borg setup); those who could fix weren't those who used. Fix = move into source control, quality improved dramatically; Markdown + g3doc let engineers change code and docs in the same change.

AUDIENCE Know Your Audience

The biggest mistake is writing only for yourself. Identify your primary audience first and write to it. Good docs needn't be perfect — "if you can read, you can write"; your reader stands where you once stood, minus your domain knowledge.
  • Seekers know what they want → key is consistency (scannable reference); stumblers don't → key is clarity (overviews at the top, a TL;DR like "not interested? stop reading").
  • Keep customer docs (API users) apart from provider docs (project members) — implementation details matter to maintainers, not end users.
  • The trick is to keep docs short — writing short usually means writing long, then editing down. Pascal: "If I had more time, I would have written you a shorter letter."

TYPES Documentation Types five types, don't mix

A doc should have a single purpose and stick to it (like an API that "does one thing well"). Early monolithic wiki pages — links, concepts, reference jumbled — failed for serving no single purpose.
  • Reference (incl. comments): most common. API vs implementation comments; single-sourced, generated from comments. Function comments start with an indicative verb (Merges, Deletes, Creates); Google avoids Returns:/Throws: boilerplate — a single prose sentence is clearer.
  • Design docs: approved before any major project; the template forces security/i18n/storage/privacy; covers goals, strategy, key decisions with trade-offs, alternatives; "a form of code review before any code" and a yardstick for success.
  • Tutorials: "Hello World"; best written when you first join; number every step, combine atomic operations, don't number system responses.
  • Conceptual: augment, not replace reference; the hardest and most neglected; sacrifice completeness/accuracy for clarity. "Comments are the unit tests; conceptual docs are the integration tests."
  • Landing pages: just a traffic cop — state the purpose, link out only; don't serve two masters (customer entry vs team home).

REVIEWS Documentation Reviews three kinds

To "test" whether a doc works, have someone else review it. Three kinds: technical review (accuracy, an SME — often part of a code review), audience review (clarity, someone unfamiliar with the domain), writing review (consistency, a technical writer). Once docs are tied into the workflow, they improve over time.

PHILOSOPHY Documentation Philosophy WHO/WHAT/WHY, the trade-off triangle

Most technical docs answer only HOW; engineers ignore the other questions (design docs are the exception — WHY matters as much as HOW); cover WHO/WHAT/WHEN/WHERE/WHY in the first two paragraphs. A doc should have a beginning, middle, and end; in docs, some redundancy is useful.
Deprecating: remove or mark obsolete when no longer useful; a freshness date (reminder after three months) + a "Last reviewed by…" byline is low-cost upkeep. When you need a technical writer: teams write "docs for themselves" fine; use writers for cross-API-boundary docs for another audience — outsiders who challenge assumptions. Conclusion: docs aren't yet a first-class citizen at Google (inherently subjective, judged by the reader asynchronously); engineers are "both the problem and the solution."

Most docs answer only HOW; covering these in the first two paragraphs avoids confusion (a design doc's WHY matters as much as HOW).

Questions to push your thinking

  1. Open a doc you recently wrote: was it written for present-you or the reader? Is its audience a seeker or stumbler, customer or provider? Would adding WHO/WHAT/WHY to the first two paragraphs make it instantly more useful?
  2. Audit your team's docs: how many are like GooWiki — ownerless, stale, duplicated? Which could be moved into source control, given owners, code-reviewed, and given a freshness date to be maintained "like code"? (Per ch15 deprecation.)
  3. Do your API docs leak design decisions or implementation details? Move them to a design doc — clearer? Does any doc "serve two masters" (a landing page that's both team home and customer entry)?
  4. Apply the completeness/accuracy/clarity triangle to your least-liked doc: is it failing all three by trying for all three? Once you pick its single purpose, which would you sacrifice for another?
  5. Does your team treat "get a technical writer" as the cure-all? Per this chapter: do you really need help only for cross-API-boundary docs for another audience? Or could the team write "docs for itself" fine, lacking only the discipline to maintain them like code?
