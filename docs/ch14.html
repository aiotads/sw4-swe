CHAPTER 14

Larger Testing

大型測試
作者 Joseph Graves PDF p.309–338 Scale 規模 Trade-offs 取捨 Time 時間
"A good design includes a test strategy that identifies risks and larger tests that mitigate them." 好的設計包含一份測試策略：辨識風險，並用對應的大型測試去緩解它們。

摘要

單元測試是地基，但只靠它無法確認整個系統真的能運作。大型測試是風險緩解策略的重要一環——不受小測試限制，故可能慢（預設逾時 15 分或 1 小時，有的跑數天）、非 hermetic、非確定性。存在的首要理由是 fidelity（保真度）：測試多大程度反映 SUT（被測系統） 的真實行為——保真度愈高，成本與風險也愈高（production 最高保真、也最危險）。

大型測試補單元測試的五個系統性缺口：不忠實的替身、設定問題（config 是 Google 重大故障頭號原因；2013 一次未測的網路設定推送造成全球當機）、負載下的問題、未預期行為（Hyrum's Law）、湧現/真空效應。但大型測試違反「可靠/快/可擴展」，還有擁有權與標準化兩大挑戰。

結構四階段：取得 SUT → 植入資料 → 執行動作 → 驗證。SUT 由常衝突的 hermeticity 與 fidelity 衡量；驗證三法：manual／assertions／A/B diff。十種大型測試中，A/B diff 回歸測試是 Google 最常見的（始於 2001）。納入流程的守則：做「可能的最小測試」、輪詢取代 sleep、驅除 flakiness、讓失敗可被理解、每個測試都有明確 owner。

為什麼要讀這一篇

你寫了一堆單元測試卻仍不敢說「系統能動」。 本章點名單元測試的五個系統性盲區——尤其 configuration 是 Google 重大故障頭號原因。看懂才知道哪些風險非得用大型測試補。
你的金字塔長成了冰淇淋甜筒。 成因：從手動測試起家，手動 e2e 累積成主流、又難補單元測試——「幾天內就造出 legacy code」。解方：頭幾天就往金字塔靠。
你想知道大型測試有哪幾種、緩解什麼風險。 本章給十種類型的「SUT／資料／驗證」配方表——這正是 Test Engineer 要會的「測試策略」。
你的大型測試又慢又愛掛、沒人維護。 守則：輪詢取代 sleep、降內部 timeout、用 Dapper 追蹤、失敗訊息帶上下文、每個測試都有 OWNERS——否則大型測試會 rot。

細節重點

為何 Fidelity 與單元測試的缺口 大型測試存在的首要理由

大型測試存在的首要理由是 fidelity——反映 SUT 真實行為的程度。可從環境看（單元測試 →production 是一條光譜）也從測試內容看（手刻資料窄且帶偏誤 vs 從 production 複製）。保真度↑ ⇒ 成本與風險↑。
  • 不忠實的替身：mock 多由「寫測試者」定義、可能誤解真實行為、且會 stale（真實實作改了卻無訊號）。
  • 設定問題：config 是 Google 重大故障頭號原因（2013 未測的網路設定推送造成全球當機）——設定也要進版本控制。
  • 負載／未預期行為（Hyrum's Law）／湧現與真空效應：單元測試像「真空中的理論物理」，乾淨卻錯失某些缺陷。

代價 Why Not Have Larger Tests? 三違反＋兩挑戰

大型測試常違反「可靠/快/可擴展」三美德（更 flaky、更慢、彼此碰撞）；還有擁有權（跨多 unit/owner，無 owner 就 rot）與標準化缺乏（各團隊用不同框架，使其在 LSC 時常被跳過、也教不了新人）兩大挑戰。

結構 Structure of a Large Test SUT・資料・動作・驗證

大型測試的四階段工作流：
  • 縮小 SUT：在 UI/API 邊界拆開（UI 測試脆又貴）；第三方依賴用 record/replay（Record Mode post-submit 錄流量、Replay Mode 開發/presubmit 重放；Google 因 protobuf 不用 Pact 等 contract test）；資料庫換記憶體版以縮到單機。
  • Webdriver Torso：在 public YouTube 頻道測 production 轉檔被媒體解謎——要預期使用者會發現 production 裡的測試資料。
  • 規模：節點數 N 增加 →路徑乘法/指數成長；低保真替身的乘法效應（兩個 10% 替身 →出 bug 機率 99%）；用 chained tests 把輸出串成下一測的輸入。

分類 Types of Larger Tests 十種大型測試

把 SUT／資料／驗證組合成緩解不同風險的測試（這就是 Test Engineer 的「測試策略」）：functional（1+ binary）、browser/device、performance/load/stress、deployment config（不崩潰即過）、exploratory（含 bug bash）、A/B diff 回歸（Google 最常見、始於 2001；A-A/A-B-C）、UAT（Google 做得少）、prober/canary（對 production 跑唯讀 smoke）、DiRT/chaos（行星級故障注入；Netflix chaos；Google 自製 Catzilla 每週數千次）、user evaluation（dogfooding／experimentation——AdWords 背景陰影色實驗／rater evaluation）。

流程 Developer Workflow 納入、加速、去 flaky、可理解、擁有

  • 納入：多數大型測試不屬於 TAP，故有獨立 post-submit 持續建置、也鼓勵 presubmit；A/B diff 的人工 bless 可納入 code review。
  • 加速＋去 flaky：縮小範圍或拆成可平行的小測試；用輪詢/事件取代 time-based sleep；調低內部 timeout、預建依賴。
  • 可理解：失敗訊息要帶上下文（「預期 10 筆卻得 1 筆」）、用 Dapper 追蹤跨行程呼叫鏈、提供聯絡人。
  • 擁有：每個大型測試都必須有 owner（用 OWNERS 檔或逐測試標註），否則更難修、更久解、然後 rot。

原文引用

"Unit tests can give you confidence about individual functions, objects, and modules, but large tests provide more confidence that the overall system works as intended."單元測試能給你對個別函式、物件、模組的信心；但大型測試給你的，是「整體系統如預期運作」的更高信心。
開章對「為何要大型測試」的核心回答；自動化測試能以手動做不到的方式擴展。
"At Google, configuration changes are the number one reason for our major outages."在 Google，設定變更是我們重大故障的頭號原因。
Configuration issues——2013 未測的網路設定推送造成全球當機；設定也要進版本控制、納入大型測試。
"A unit test is like a problem in theoretical physics: ensconced in a vacuum, neatly hidden from the mess of the real world."單元測試就像理論物理裡的問題：安置在真空中，與現實世界的混亂俐落地隔開。
Emergent behaviors / vacuum effect——對速度與可靠很好，卻錯失只在真實依賴/網路/資料下現形的缺陷。
"Often these two factors are in direct conflict."這兩個因素（封閉性與保真度）往往直接衝突。
The System Under Test——hermeticity 與 fidelity 難兼得，SUT 形態的選擇就是在兩者間取捨。
"Manual regression testing does not scale sublinearly: the larger a system grows … the more human time is needed."手動回歸測試無法次線性擴展：系統長得愈大、路徑愈多，需要的人力時間就愈多。
Verification（manual）——故把探索式/手動找到的 bug 補成自動測試、偏好 A/B diff 等自動化驗證。

關鍵字補充

Larger Test（大型測試）
不受小測試（一執行緒/行程/機器）限制的測試，可能慢、非 hermetic、非確定性；補單元測試的系統性盲區，是風險緩解策略的一環。
Fidelity（保真度）
測試（環境與內容）反映 SUT 真實行為的程度；大型測試存在的首要理由。保真度↑ ⇒ 成本與風險↑。
SUT（System Under Test）
大型測試的核心；由 hermeticity 與 fidelity 兩個常衝突的因素衡量；形態 single-process/single-machine/multimachine/shared/hybrid。
Hermeticity（封閉性）
SUT 與其他元件/流量的隔離程度；高封閉性最不受並行與基礎設施 flakiness 影響，但常犧牲保真。
冰淇淋甜筒（Ice Cream Cone）
大量 e2e、少單元的反模式；常源於「從手動測試起家」，幾天內就造出 legacy code。解方：頭幾天往金字塔靠。
A/B Diff 回歸測試
跑兩版 SUT、送相同流量、比輸出；Google 最常見的大型測試（始於 2001）；變體 A-A/A-B-C。挑戰：approval/noise/coverage/setup。
Record/Replay
跑大型測試時錄外部服務流量（Record Mode，post-submit），開發/presubmit 時重放（Replay Mode）；Google 因 protobuf 採此。
Seeded Data vs Test Traffic
植入資料（SUT 起始狀態：domain data、realistic baseline）vs 測試流量（執行期送入）；產生 handcrafted/copied/sampled。
Prober / Canary Analysis
prober 對 production 跑唯讀確定性 smoke 斷言；canary analysis 在灰度推送時比較 canary 與 baseline 的健康指標。
DiRT / Chaos / Catzilla
DiRT 是年度災難演練（行星級故障注入）；chaos engineering（Netflix）持續注入背景故障；Catzilla 是 Google 自製、每週數千次。
Dapper
以單一 request ID 串起 RPC 呼叫鏈所有日誌的追蹤框架；讓跨行程的大型測試失敗可被追根究柢。
Test Engineer（測試工程師）
Google 的專門角色；核心能力之一是為產品擬定「測試策略」（辨識風險向量、選對應大型測試）。

延伸提問

  1. 盤點你系統的「風險向量」：哪些是單元測試結構上補不到的（不忠實替身？設定？負載？未預期輸入？湧現行為）？你有沒有為每個主要風險配一個對應的大型測試（一份明確的測試策略）？
  2. 你的設定（config/部署腳本/feature flag）有沒有進版本控制、有沒有被測試？回想 Google「設定是重大故障頭號原因」——你上次出包是程式還是設定造成的？
  3. 你的金字塔像金字塔還是冰淇淋甜筒？是不是「從手動測試起家、幾天內就造出 legacy code」？哪些手動 e2e 該被自動整合測試取代？對照 ch11。
  4. 挑你最大、最痛的 e2e 測試：它的 SUT 能不能縮小（UI/API 邊界拆開？第三方換 record/replay？資料庫換記憶體版）？在 hermeticity vs fidelity 上你落在哪、該往哪移？
  5. 你的大型測試壞掉時別人看得懂為什麼嗎（訊息有上下文？有 Dapper 追蹤？知道找誰）？它有明確 owner 嗎？presubmit/post-submit/只在發布跑——這取捨對你的團隊對嗎？

Part III — Processes
CHAPTER 14

Larger Testing

Match each risk with a larger test that mitigates it
by Joseph Graves PDF p.309–338 Scale Trade-offs Time
"A good design includes a test strategy that identifies risks and larger tests that mitigate them." Match each risk with a larger test that mitigates it — that's a test strategy.

Summary

Unit tests are the foundation, but they alone can't confirm the whole system works. Larger tests are an essential part of risk mitigation — free of small-test constraints, they may be slow (default timeout 15 min or 1 hr; some run days), nonhermetic, nondeterministic. Their primary reason is fidelity: how well a test reflects the real behavior of the SUT — higher fidelity, higher cost and risk (production is highest-fidelity and riskiest).

They fill five gaps in unit tests: unfaithful doubles, configuration issues (config is Google's #1 cause of major outages; a 2013 untested network-config push caused a global outage), issues under load, unanticipated behaviors (Hyrum's Law), emergent/vacuum effects. But they violate "reliable/fast/scalable" and face two challenges: ownership and lack of standardization.

Four phases: obtain a SUT → seed data → perform actions → verify. A SUT is judged on the often-conflicting hermeticity and fidelity; verification comes in three forms: manual / assertions / A/B diff. Of ten kinds, the A/B diff regression test is Google's most common (since 2001). Workflow rules: the smallest possible test, poll instead of sleep, drive out flakiness, make failures understandable, give every test a clear owner.

Why read this

You've written many unit tests but can't say "the system works." The chapter names unit testing's five blind spots — especially that configuration is Google's #1 cause of major outages. Now you know which risks need larger tests.
Your pyramid has become an ice cream cone. Cause: starting with manual testing, letting manual e2e dominate while units lag — "legacy code within days." Cure: move toward the pyramid early.
You want to know the kinds of larger test and what each mitigates. The chapter gives a "SUT / data / verification" recipe for ten types — exactly the test strategy a Test Engineer designs.
Your large tests are slow, flaky, unowned. Rules: poll instead of sleep, lower internal timeouts, trace with Dapper, give failures context, give every test an OWNERS — or they rot.

Key details

WHY Fidelity & the gaps in unit tests

The primary reason larger tests exist is fidelity — how well a test reflects the SUT's real behavior. Seen by environment (unit → production is a spectrum) and by content (handcrafted data is narrow/biased vs copied from production). Higher fidelity ⇒ higher cost and risk.
  • Unfaithful doubles: mocks defined by the test author may misunderstand real behavior and go stale.
  • Configuration: config is Google's #1 cause of major outages (a 2013 untested network-config push caused a global outage) — version-control config too.
  • Load / unanticipated behavior (Hyrum's Law) / emergent & vacuum effects: unit tests are like "theoretical physics in a vacuum" — clean, but missing some defects.

COST Why Not Have Larger Tests? three violations + two challenges

Larger tests often violate "reliable/fast/scalable" (flakier, slower, collide); plus two challenges: ownership (multiple units/owners; without one, the test rots) and lack of standardization (different frameworks per team, getting them skipped during LSCs and unteachable to newcomers).

STRUCTURE Structure of a Large Test SUT · Data · Action · Verify

The four-phase workflow of a large test:
  • Shrink the SUT: split at the UI/API boundary (UI tests are brittle and costly); use record/replay for third parties (Record Mode records traffic post-submit, Replay Mode replays in dev/presubmit; Google's protobuf use means no Pact-style contract tests); in-memory databases to fit on one machine.
  • Webdriver Torso: testing production rendering on a public YouTube channel got a media mystery — expect users to discover test data in production.
  • Scale: N nodes → multiplicative/exponential paths; low-fidelity doubles' multiplicative effect (two 10% doubles → 99% bug chance); use chained tests to thread one test's output into the next's input.

TYPES Types of Larger Tests ten kinds

Combine SUT/data/verification into tests mitigating different risks (the Test Engineer's "test strategy"): functional (1+ binaries), browser/device, performance/load/stress, deployment config (passes if it doesn't crash), exploratory (incl. bug bash), A/B diff regression (Google's most common, since 2001; A-A/A-B-C), UAT (Google does little), prober/canary (read-only smoke against production), DiRT/chaos (planetary-scale fault injection; Netflix chaos; Google's homegrown Catzilla thousands/week), user evaluation (dogfooding / experimentation — the AdWords ad-shading experiment / rater evaluation).

WORKFLOW Developer Workflow integrate, speed up, de-flake, understand, own

  • Integrate: most don't belong in TAP, so they get a separate post-submit build, with presubmit encouraged; A/B diff's manual blessing folds into code review.
  • Speed up + de-flake: reduce scope or split into parallel smaller tests; poll/use events instead of time-based sleeps; lower internal timeouts, prebuild dependencies.
  • Understandable: failures need context ("expected 10 but got 1"), cross-process tracing via Dapper, and a contact.
  • Own: every large test must have an owner (OWNERS file or per-test annotation), or it gets harder to fix, slower to resolve, and rots.

Key terms

Larger test
A test free of small-test constraints; may be slow, nonhermetic, nondeterministic; fills unit tests' systematic gaps as part of risk mitigation.
Fidelity
How well a test reflects the SUT's real behavior; the primary reason larger tests exist. More fidelity ⇒ more cost and risk.
SUT (system under test)
The core of a large test; judged on hermeticity and fidelity (often conflicting); forms: single-process/single-machine/multimachine/shared/hybrid.
Hermeticity
The SUT's isolation from other components/traffic; high hermeticity minimizes concurrency and infrastructure flakiness, often at the cost of fidelity.
Ice cream cone
The many-e2e, few-unit antipattern; often from "starting with manual testing," creating legacy code within days. Cure: move toward the pyramid early.
A/B diff regression test
Run two SUT versions, send the same traffic, compare outputs; Google's most common large test (since 2001); variants A-A/A-B-C. Challenges: approval/noise/coverage/setup.
Record/replay
Record traffic to external services during a large test (Record Mode, post-submit), replay in dev/presubmit (Replay Mode); Google's choice over contract tests.
Seeded data vs test traffic
The SUT's starting state (domain data, realistic baseline) vs data sent during execution; generated handcrafted/copied/sampled.
Prober / canary analysis
A prober runs read-only deterministic smoke assertions against production; canary analysis compares canary vs baseline health metrics during a staged rollout.
DiRT / chaos / Catzilla
DiRT is Google's annual disaster drill (planetary-scale fault injection); chaos engineering (Netflix) continuously injects faults; Catzilla is homegrown, thousands a week.
Dapper
Google's tracing framework correlating all logs in an RPC chain by one request ID; lets cross-process large-test failures be traced to a root cause.
Test Engineer
Google's specialized role; a core skill is designing a product's "test strategy" (risk vectors, matching larger tests).

Questions to push your thinking

  1. Inventory your "risk vectors": which are structurally beyond unit tests (unfaithful doubles? configuration? load? unanticipated inputs? emergent behavior)? Do you have a test strategy matching each major risk to a larger test?
  2. Is your configuration under version control and tested? Recall "config is the #1 cause of major outages" — was your last incident code or config?
  3. Does your pyramid look like a pyramid or an ice cream cone? Did you "start with manual testing and create legacy code within days"? Which manual e2e should become automated integration tests? Per ch11.
  4. Take your biggest, most painful e2e test: can its SUT shrink (split at the UI/API boundary? record/replay for third parties? in-memory databases)? On hermeticity vs fidelity, where are you and where should you move?
  5. When your large tests fail, can others understand why (context? Dapper tracing? a contact)? Does each have a clear owner? Presubmit/post-submit/release-only — is that trade-off right?
