CHAPTER 13

Test Doubles

測試替身
作者 Andrew Trenk、Dillon Bly PDF p.285–308 Trade-offs 取捨 Time 時間 Scale 規模
"A real implementation should be preferred over a test double. Interaction testing should be avoided when possible." 應該優先用真實實作，而非測試替身；能避免就避免互動測試。

摘要

Test double（測試替身）是在測試中頂替真實實作的物件/函式（像替身演員）。它讓你寫大量「小、快、不 flaky」的單元測試，去測那些本來會慢又不穩的程式（打外部伺服器、寫資料庫）。但用替身有三項取捨：testability（可測試性）、applicability（適用性，誤用造成脆弱）、fidelity（保真度）。

本章最重要的立場（Google 血淚教訓）：優先用真實實作（classical testing），而非以 mock 框架為主的 mockist testing（難以規模化）。Google 曾把 mock 當「萬用鐵鎚」，數年後才發現這些測試好寫卻要不斷維護、又很少抓到 bug。偏好真實勝於隔離：跑 production 會跑的程式更有信心；真實實作有 bug 時測試失敗——是好事。

三種技術：faking（輕量假實作）、stubbing（硬寫回傳值）、interaction testing（驗證有沒有被呼叫）。優先序：真實 ＞ fake ＞（節制的）stub／interaction。決定要不要用真實，看執行時間／確定性／依賴建構三點。fake 該由真實實作的 owner 維護、且自己有 contract test。過度 stubbing 使測試不清晰、脆弱、低效；interaction testing 應盡量避免（淪為 change-detector test），只對狀態改變函式做、避免 overspecification。

為什麼要讀這一篇

你一遇依賴難搞就 @Mock 一切。 代價：mock 測試好寫卻要不斷維護又很少抓到 bug，還擋住 API owner 改實作（被 mock 成千上萬次）。優先序：真實 ＞ fake ＞ stub／interaction。
你分不清 fake／stub／mock。 三種技術：fake（輕量真實作）、stubbing（硬寫回傳值）、interaction testing（驗證有沒有被呼叫）。搞懂才知道面對某依賴該用哪招。
你的測試是 change-detector——改 production 一定跟著紅、即使行為沒變。 多半是 interaction testing 過度使用。改成偏好 state testing（驗證結果/狀態，而非「呼叫了什麼」）。
你想讓測試又快又真。 關鍵是為依賴寫一個高保真 fake——一個好 fake 大幅提升全組織速度；它該由真實實作 owner 維護、且有 contract test 防走鐘。

細節重點

衝擊 The Impact of Test Doubles 三項取捨

用替身引入三件取捨：testability（程式得設計成能抽換依賴，事後加測試常得大改）、applicability（用得差會脆弱複雜低效，很多情況該用真實實作）、fidelity（替身行為與真實的接近程度，太低就沒價值，但完美保真常不可行）。
Google 的教訓：過度依賴 mock 框架很危險——初看像「萬用鐵鎚」，數年後發現這些測試好寫卻要不斷維護、又很少抓到 bug，鐘擺已盪回「寫更真實的測試」。

基本 Seam、Mocking Framework 接縫與依賴注入

Seam（接縫）是讓程式可測試的手段——能為被測系統替換不同依賴；最常透過 dependency injection（依賴注入）：依賴被傳進來而非自己 new，於是測試能換成替身。

取捨 優先序：真實 ＞ fake ＞ stub／interaction Prefer Realism Over Isolation

測試第一選擇是依賴的真實實作（與 production 同一份）——更高保真、更有信心。偏好真實＝classical testing；偏好 mock 框架＝mockist testing（Google 認為難以規模化）。好測試應獨立於實作、以被測 API 表述。
真實實作有 bug 時測試失敗——這是好事（代表 production 也會壞）；好的 CI 工具通常容易追出肇因變更。

技術一 Faking 假實作：理想的替身

真實實作不可行時，fake 通常是最佳選擇——行為類似真實，被測系統甚至分不出。
  • 誰寫：需領域知識、隨真實行為更新 →應由真實實作的 owner 寫與維護；只建在「不適合測試的程式的根部」（為 DB API 寫 fake，而非每個呼叫者）。
  • 保真度：對真實的 API 契約保真（同輸入同輸出/狀態變化），但只需從測試的視角看（hashing fake 不必產生相同 hash 值）；不支援的路徑要 fail fast。
  • fake 自己要被測：否則隨真實實作演進走鐘——用 contract test（同一組測試跑在真實與 fake 上）。
  • 沒 fake：先請 owner 寫；不行就自己包一層；再不行回去用真實實作。fake 可視為一種最佳化。

技術二 Stubbing：過度使用的危險 硬寫回傳值

stubbing 快又簡單，但過度使用嚴重損及生產力：①不清晰（得在腦中走過系統才懂為何 stub）②脆弱（洩漏實作細節，實作一改就得改測試）③低效（無法保證 stub 行為與真實一致、無法存狀態）。
何時恰當：需函式回傳特定值把系統帶入某狀態，或模擬真實/fake 難觸發的回傳/錯誤；且每個 stub 都該與斷言直接相關、數量要少（要 stub 很多＝過度使用或系統太複雜該重構）。

技術三 Interaction Testing：偏好 State Testing 驗證呼叫 vs 驗證結果

偏好 state testing：驗證回傳值/狀態改變（能確認系統真的正常）；interaction testing 只能驗證「函式被呼叫」（排序例子裡無從得知數字真的排好沒）。它洩漏實作細節 →脆弱的 change-detector test。
  • 何時恰當：①無法 state testing（無真實也無 fake）時當 fallback；②呼叫的次數或順序會造成不良行為（如驗證快取減少 DB 呼叫，atMostOnce()）。
  • 最佳實務：①只對狀態改變函式做（sendEmail/saveRecord，而非 getUser/readFile）；②避免 overspecification（用 eq()/any() 只驗證最小相關集合）。呼應 ch12「測行為而非方法」。

關鍵字補充

Test Double（測試替身）
在測試中頂替真實實作的物件/函式（如替身演員）；讓你寫大量小而快、不 flaky 的單元測試。「mocking」本章刻意避用。
Classical vs Mockist Testing
classical 偏好真實實作（Google 採用）；mockist 偏好 mock 框架取代真實實作（Google 認為難以規模化）。
Fidelity（保真度）
替身行為與真實實作的接近程度；fake 應對真實的 API 契約保真，但只需從測試的視角看，不必完美。
Seam（接縫）
讓程式可測試的手段：可為被測系統替換不同依賴。最常透過 dependency injection 引入。
Dependency Injection（依賴注入，DI）
依賴被傳進來而非自己 new，使測試能抽換替身；Java 常用 Guice／Dagger，動態語言依賴較低。
Fake（假實作）
輕量、行為類似真實但不適合 production 的實作（記憶體 DB）；需替身時的理想選擇；由 owner 維護、自己要有測試。
Stubbing（樁）
給本無行為的函式硬寫回傳值；過度使用使測試不清晰、脆弱、低效、無法存狀態。
Interaction Testing（互動測試）
驗證函式有沒有被正確呼叫而不真執行它；應盡量避免，只對狀態改變函式做、避免 overspecification。
Change-Detector Test（變更偵測器測試）
過度互動測試的後果：production 一改就失敗、即使行為沒變；洩漏實作細節、脆弱。
@DoNotMock
ErrorProne 的標註，讓 API owner 宣告「此型別不該被 mock，有更好替代」；保護 owner 改實作的自由。
Contract Test（契約測試）
對 API 公開介面寫測試、同時跑在真實實作與 fake 上，確保 fake 不走鐘。
State vs Interaction Testing
state＝驗證回傳值/狀態改變；interaction＝驗證函式被呼叫。偏好 state（更能確認系統真的正常）。

延伸提問

  1. 翻你測試裡的 @Mock／when().thenReturn()：多少可換成真實實作或 fake？哪些依賴是「快、確定、依賴簡單」的 value object（金額、日期），根本不該被 mock（試試 @DoNotMock 的精神）？
  2. 你的某個測試是不是「change-detector」——改 production 一定跟著紅、即使行為沒變？它是不是在做 interaction testing？能不能改成 state testing？
  3. 你最難在測試裡用的依賴（資料庫、外部服務）有沒有 fake？若沒有，該請 owner 寫、還是自己包一層？這個 fake 有沒有 contract test 防走鐘？
  4. 用「執行時間／確定性／依賴建構」三準則檢視一個你猶豫「真實還是替身」的測試：真實慢在哪？非確定來自哪（時鐘？多執行緒？非 hermetic？）？哪一點才是你真正該用替身的理由？
  5. 你做 interaction testing 時有沒有 overspecify？哪些其實是非狀態改變函式（getUser、readFile）根本不必驗證？收斂到最小集合後測試會不會更耐改？對照 ch12。

Part III — Processes
CHAPTER 13

Test Doubles

Reach for the real thing first; a double is a trade-off
by Andrew Trenk & Dillon Bly PDF p.285–308 Trade-offs Time Scale
"A real implementation should be preferred over a test double. Interaction testing should be avoided when possible." Reach for the real thing first; a double is a trade-off, not a default.

Summary

A test double stands in for a real implementation in a test (like a stunt double). It lets you write many small, fast, non-flaky unit tests of code that would otherwise be slow and unstable (external servers, databases). But doubles bring three trade-offs: testability, applicability (misuse → brittleness), and fidelity.

The central stance (Google's hard-won lesson): prefer real implementations (classical testing) over mocking-heavy mockist testing (which doesn't scale). Google once treated mocks as "a hammer fit for every nail," only to find those tests easy to write but constantly needing maintenance while rarely finding bugs. Prefer realism over isolation: real dependencies run what production runs; a real bug failing your test is good.

Three techniques: faking (a lightweight working impl), stubbing (hardcode a return), interaction testing (verify a call). Order: real > fake > (sparing) stub/interaction. Deciding on real turns on execution time / determinism / dependency construction. A fake should be maintained by the real implementation's owner and have its own contract tests. Over-stubbing makes tests unclear, brittle, ineffective; interaction testing should be avoided (it becomes a change-detector test) — do it only on state-changing functions, without overspecification.

Why read this

You `@Mock` everything when a dependency is awkward. The cost: mocked tests are easy to write but need constant maintenance while rarely finding bugs, and block API owners (mocked thousands of times). Order: real > fake > stub/interaction.
You can't tell fake/stub/mock apart. Three techniques: fake (lightweight working impl), stubbing (hardcode return), interaction testing (verify a call). Knowing them tells you which to reach for.
Your tests are change-detectors — production changes turn them red even with unchanged behavior. Usually over-used interaction testing. Prefer state testing (validate result/state, not "what was called").
You want tests fast and realistic. The key is a high-fidelity fake — one good fake boosts a whole org; maintained by the real implementation's owner, guarded by contract tests against drift.

Key details

IMPACT The Impact of Test Doubles

Three trade-offs: testability (the code must be designed to swap dependencies; retrofitting tests often means a major refactor), applicability (misuse → brittle/complex/ineffective; often a real implementation is better), fidelity (too low and the double is worthless; perfect fidelity is often infeasible).
Google's lesson: over-using mocking frameworks is dangerous — they first look like "a hammer fit for every nail," but years later prove easy to write yet in constant need of maintenance while rarely finding bugs; the pendulum has swung back to "more realistic tests."

BASICS Seams, Mocking Frameworks

A seam makes code testable — letting the system under test use different dependencies; most often via dependency injection: dependencies are passed in rather than constructed, so tests can substitute doubles.

TRADE-OFF Order: Real > Fake > Stub/Interaction Prefer Realism Over Isolation

The first choice is real implementations (same as production) — higher fidelity, more confidence. Preferring real = classical testing; preferring mocks = mockist testing (Google finds it hard to scale). A good test is independent of implementation, written to the API.
When a real implementation has a bug, the test fails — good (it means production would break too); with good CI it's usually easy to find the offending change.

TECHNIQUE 1 Faking the ideal double

When a real implementation is infeasible, a fake is often best — it behaves like the real one, so the system can't tell.
  • Who writes it: needs domain knowledge and updates as real behavior changes → the real implementation's owner; build it only at the root of the un-testable code (a fake for the DB API, not each caller).
  • Fidelity: faithful to the real API contracts (same output/state for an input), but only from the test's perspective (a hashing fake needn't match real hashes); unsupported paths should fail fast.
  • Fakes need tests: else they drift as the real impl evolves — use contract tests (one set run against both real and fake).
  • No fake: ask the owner; else wrap a layer; else fall back to the real implementation. A fake can be seen as an optimization.

TECHNIQUE 2 Stubbing: The Dangers of Overusing

Stubbing is quick and easy, but over-use severely hurts productivity: ① unclear (you must step through the system to see why functions are stubbed), ② brittle (leaks implementation details; impl changes force test changes), ③ ineffective (no guarantee the stub matches real behavior; can't store state).
When appropriate: when you need a function to return a specific value to get the system into a state, or to simulate returns/errors hard to trigger; and each stub should relate directly to an assertion, kept few (many stubs → over-use or an over-complex system to refactor).

TECHNIQUE 3 Interaction Testing: Prefer State Testing

Prefer state testing: validate the returned value/changed state (confirms the system actually works); interaction testing only validates "a function was called" (in the sorting example, no way to know the numbers are really sorted). It leaks implementation details → brittle change-detector tests.
  • When appropriate: ① a fallback when state testing is impossible (no real, no fake); ② when the number or order of calls matters (verify a cache reduces DB calls — atMostOnce()).
  • Best practices: ① only on state-changing functions (sendEmail/saveRecord, not getUser/readFile); ② avoid overspecification (use eq()/any() for the minimal relevant set). Echoes ch12's "test behaviors, not methods."

Key terms

Test double
An object/function standing in for a real implementation (like a stunt double); enables many small, fast, non-flaky unit tests. The chapter avoids "mocking."
Classical vs mockist testing
Classical prefers real implementations (Google's choice); mockist prefers mocking frameworks (hard to scale).
Fidelity
How closely a double behaves like the real implementation; a fake honors the real API contracts, but only from the test's perspective.
Seam
A way to make code testable by allowing different dependencies; most often via dependency injection.
Dependency injection (DI)
Dependencies are passed in rather than constructed, so tests substitute doubles; Java uses Guice/Dagger, dynamic languages rely on it less.
Fake
A lightweight working implementation that behaves like the real one but isn't production-suitable; the ideal double; owner-maintained and itself tested.
Stubbing
Hardcoding a no-behavior function's return; over-use makes tests unclear, brittle, ineffective, and stateless.
Interaction testing
Verifying a function was called correctly without executing it; avoid where possible — only state-changing functions, no overspecification.
Change-detector test
The result of over-used interaction testing: fails on any production change even with unchanged behavior; leaky and brittle.
@DoNotMock
An ErrorProne annotation letting an API owner declare "don't mock this type"; protects freedom to change the implementation.
Contract test
Tests against an API's public interface run against both the real implementation and the fake, ensuring the fake doesn't drift.
State vs interaction testing
State = validate the returned value/changed state; interaction = validate a function was called. Prefer state.

Questions to push your thinking

  1. Scan your @Mock / when().thenReturn(): how many could be real implementations or fakes? Which are "fast, deterministic, simple" value objects (money, date) that shouldn't be mocked (try the @DoNotMock spirit)?
  2. Is one test a "change-detector" — red on any production change even when behavior is unchanged? Is it interaction testing? Could it become state testing?
  3. Does your hardest-to-test dependency have a fake? If not, should the owner write it, or should you wrap a layer? Does it have a contract test against drift?
  4. Apply "execution time / determinism / dependency construction" to a test you're torn over: where is the real impl slow? Where does nondeterminism come from (clock? threads? non-hermetic?)? Which is the real reason for a double?
  5. In your interaction tests, are you overspecifying? Which are non-state-changing (getUser, readFile) and needn't be verified? After narrowing with any()/eq(), are they more change-resilient? Per ch12.
