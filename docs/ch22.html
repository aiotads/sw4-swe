CHAPTER 22

Large-Scale Changes

大規模變更
作者 Hyrum Wright PDF p.487–506 Scale 規模 Time 時間 Trade-offs 取捨
"Making LSCs means making a habit of making LSCs." 要會做大規模變更，靠的是養成「常做大規模變更」的習慣。

摘要

Large-Scale Change（LSC，大規模變更）＝任何邏輯相關、但實務上無法當成單一原子提交一次送出的變更集合（檔案太多、改動必有 merge conflict、或 repo 拓樸讓跨 repo 原子變更不可能）。本章最反直覺的觀察：隨程式碼庫與工程師人數成長，「可能的最大原子變更」反而縮小。

既然原子變更撐不住，Google 改以分而治之：自動工具產生一份巨大全域變更，再用 Rosie 依專案邊界與 OWNERS 切成可獨立提交的碎片（shard），各自跑 test→mail→submit。測試靠 TAP train（測試列車）併車攤平成本。文化上把 commit 當「牛（cattle）而非寵物（pets）」——無名、可隨時回退、丟幾頭不心疼。

誰來做？基礎設施團隊（有領域知識、攤平成本、避免「無預算強制令」）。流程四階段：授權 → 變更產生 → 碎片管理 → 清理。案例：scoped_ptr→std::unique_ptr（>50 萬處引用、高峰每天 >700 變更觸及 >15,000 檔）、填坑（單字元檔名修正）、Operation RoseHub（>50 名 Googler 手發 >2,600 patch 修補 Mad Gadget 漏洞）。關鍵原則：變更產生工具的人力必須次線性於 repo 大小。

為什麼要讀這一篇

你的「最大可原子提交」其實比你想的小，而且會越來越小。 repo 與團隊越大，原子變更上限越低。本章給你「放棄原子性、改用分片」的完整心法與工具觀，讓程式碼庫對基礎設施變更保持靈活。
你被一個沒人敢碰的老系統卡住（鬧鬼墓園）。 把 SRE 的「No Haunted Graveyards」搬到程式碼層次；唯一解藥是扎實的測試——有測試，再老再複雜的系統都能放心改。
你想推更好的新系統，卻指望「使用者自己遷移」。 為何有機遷移幾乎不會完成（照抄舊範例、好處分散），以及為何該由擁有舊系統的基礎設施團隊負責——把無預算強制令的外部性內部化。
你把每個 commit 都當寶貝。 「cattle vs pets」改變你對自動化變更的心態：機器大量產生時，個別 commit 該是可丟棄的牛、不是寵物；被回退別往心裡去。

細節重點

反直覺 原子上限縮小 → 分而治之 Atomic ceiling shrinks

repo 與人數越大，可能的最大原子變更反而縮小；解法不是「更努力地做大原子提交」，而是把全域變更切成可獨立提交的碎片。
原子變更的障礙：VCS 操作隨變更大小線性擴展、大提交卡住其他人；merge conflict 機率被人數放大；鬧鬼墓園（老到沒人敢碰、解藥是測試）；異質性（多種 VCS/CI/格式讓自動掃改難）；測試（Google 跑直接＋傳遞依賴，刻意接受小碎片重跑測試的成本）。

流程 TAP train 與四階段流程 Authorization → Creation → Shard → Cleanup

LSC 鮮少互動、多數測試對多數 LSC 都通過——故能併車同測、攤平成本。整體流程四階段、由 Rosie 與委員會支撐。
四階段：授權（填文件＋FAQ、領域審查、委員會把多數審查導向單一「全域核准者」）→ 變更產生（盡量自動化、像人寫的）→ 碎片管理（Rosie 切片、限流、OWNERS 偵測挑審查者、全域核准者用模式工具自動核准）→ 清理（用 Tricorder 在審查時擋住舊符號再被引入，防倒退）。scoped_ptr→unique_ptr 靠「先變型別別名→文字替換→移除舊別名」完成。

關鍵字補充

Large-Scale Change（LSC，大規模變更）
邏輯相關但無法當單一原子單元提交的變更集合；幾乎都由自動工具產生（清理反模式、替換棄用特性、基礎設施升級、遷移使用者）。
Atomic Change（原子變更）
能一次測試/審查/提交的單一不可分割變更；隨 repo 與人數成長，可能的最大原子變更反而縮小。
Haunted Graveyard（鬧鬼墓園）
SRE 箴言「No Haunted Graveyards」；老到沒人敢碰的關鍵系統，阻擋遷移與升級；解藥是扎實測試。
Unfunded Mandate（無預算強制令）
外部強加要求卻不給補償；解釋為何該由基礎設施團隊（而非各消費團隊）負責 LSC、把外部性內部化。
TAP Train（測試列車）
把多個 LSC 併車共享測試以攤平成本；五步、每三小時一班，產出「可安全提交」的證據。
Rosie
把 master change 切成可原子提交碎片、並管理各片 test-mail-submit 的平台；LSC 規模化的核心。
Shard（碎片）
由 Rosie 依專案邊界/OWNERS 切出的可獨立提交小變更；必須能獨立提交、各自過 presubmit。
Cattle vs Pets（牛 vs 寵物）
對待變更的心態對比；LSC 的個別 commit 是可丟棄的牛，不是親手打磨的寵物。
Global Approver（全域核准者）
有全庫核准權、用基於模式的工具自動核准 LSC 碎片的專家，取代逐一找本地擁有者。
Kythe / ClangMR / Refaster
語意索引與 AST 層的分析/轉換工具，提供 LSC 工具所需的程式碼庫洞察。
Type Alias / Forwarding Function
讓舊用戶持續運作、同時非原子遷移的語言特性（scoped_ptr→unique_ptr 即靠型別別名過渡）。
Tricorder
在審查時標記「引入已棄用物件新用法」的框架，防止 LSC 清理後的倒退。

延伸提問

  1. 回到開篇：你能可靠地在單一原子提交裡更新幾個檔案？限制是什麼（VCS 算力？merge conflict？presubmit？）？緊急時做得到嗎？若這數字遠小於你的程式碼庫，你的「分而治之」工具（誰是你的 Rosie？）準備好了嗎？
  2. 你的程式碼庫裡有沒有鬧鬼墓園——老到沒人敢碰、卻在關鍵路徑上的系統？擋住你升級或移除舊系統的是不是它？你願不願意先補測試把它解放出來？對照 ch11、ch14。
  3. 你最近推新系統，是指望使用者有機遷移，還是由你（系統擁有者）負責 LSC？回想「無預算強制令」與「照抄舊範例」——有機遷移在你的組織真的會完成嗎？對照 ch15。
  4. 你對 LSC 的個別 commit 抱「牛」還是「寵物」心態？若你的自動化還做不到「廉價重生變更」，你卡在哪——工具（產生）、測試（TAP train 等價物），還是文化（本地擁有者的否決權與信任）？
  5. 你的變更產生工具，人力成本是否次線性於 repo 大小？repo 長大十倍，產生「全部所需變更」的人時會跟著暴增嗎？那條「>500 處編輯就該學工具」的經驗法則，在你的團隊該設在哪？對照 ch17、ch20。

Part IV — Tools
CHAPTER 22

Large-Scale Changes

Give up atomicity; shard, test, and submit instead
by Hyrum Wright PDF p.487–506 Scale Time Trade-offs
"Making LSCs means making a habit of making LSCs." The capability is a muscle you build by exercising it — not a one-off heroic feat.

Summary

A Large-Scale Change (LSC) is any set of changes logically related but not practically submittable as one atomic commit (too many files, always merge-conflicts, or a repo topology that makes cross-repo atomic changes impossible). The chapter's most counterintuitive observation: as a codebase and its engineer count grow, the largest possible atomic change shrinks.

Since atomic changes don't scale, Google does divide and conquer: tooling generates one giant global change, then Rosie shards it — by project boundaries and OWNERS — into independently-submittable shards, each running test→mail→submit. Testing rides the TAP train to amortize cost. Culturally, commits are cattle, not pets — nameless, rollback-able anytime, cheap to lose.

Who does it? Infrastructure teams (domain knowledge, amortized cost, avoiding the unfunded mandate). Four phases: Authorization → Change creation → Shard management → Cleanup. Cases: scoped_ptr→std::unique_ptr (>500,000 references; >700 changes touching >15,000 files per day at peak), filling potholes (a single-character filename fix), and Operation RoseHub (>50 Googlers sending >2,600 patches to fix the Mad Gadget vulnerability). Key principle: the tooling's human cost must be sublinear in repo size.

Why read this

Your "largest atomic commit" is smaller than you think — and shrinking. The bigger the repo and team, the lower the atomic ceiling. The chapter hands you the mindset and tooling view of "give up atomicity, shard instead," keeping your codebase flexible to infrastructure change.
You're stuck behind an old system no one dares touch (a haunted graveyard). It lifts the SRE mantra "No Haunted Graveyards" to the code level; the one antidote is solid testing — with tests, you can change a system of any age confidently.
You push a better new system but expect users to migrate themselves. Why organic migrations almost never finish (copying old examples, diffuse benefits) and why the infra team that owns the old system should own the migration — internalizing the unfunded mandate's externality.
You treat every commit as precious. "Cattle vs pets" reframes automated change: when machine-generated en masse, an individual commit is disposable cattle, not a pet; don't take rollbacks personally.

Key details

COUNTERINTUITIVE Atomic ceiling shrinks → divide and conquer

The bigger the repo and team, the lower the atomic-change ceiling; the fix isn't "try harder at big atomic commits" but to shard the global change into independently-submittable pieces.
Barriers to atomic changes: VCS operations scale linearly with change size, and big commits block others; merge-conflict probability is compounded by engineer count; haunted graveyards (too old to touch — antidote is testing); heterogeneity (many VCS/CI/format styles make auto-sweeps hard); testing (Google runs direct + transitive deps, deliberately accepting that small shards rerun tests).

PROCESS The TAP train & the four-phase process

LSCs rarely interact and most tests pass for most LSCs — so you can share a train and amortize cost. The overall process has four phases, supported by Rosie and a committee.
Four phases: Authorization (a doc + FAQ, domain review, committee often routes reviews to a single global approver) → Change creation (as automated and human-looking as possible) → Shard management (Rosie shards, throttles, uses OWNERS detection to pick reviewers, global approver auto-approves with pattern tooling) → Cleanup (use Tricorder at review time to block reintroduction and prevent backsliding). scoped_ptr→unique_ptr finished via "make it a type alias → textual substitution → remove the old alias."

Key terms

Large-Scale Change (LSC)
A set of changes logically related but not submittable as one atomic unit; almost always tool-generated (clean antipatterns, replace deprecated features, upgrade infra, migrate users).
Atomic change
A single indivisible change you can test/review/commit at once; the largest possible one shrinks as repo and engineer count grow.
Haunted graveyard
From "No Haunted Graveyards"; a critical system so old no one dares touch it, blocking migrations and upgrades; the antidote is solid testing.
Unfunded mandate
Externally imposed requirements without compensating resources; why infra teams (not consuming teams) should own LSCs, internalizing the externality.
TAP train
Batching multiple LSCs onto one train to share test results; five steps, every 3 hours, producing "safe to submit" evidence.
Rosie
The platform that shards a master change into atomically-submittable pieces and manages each one's test-mail-submit; the core of LSC scaling.
Shard
An independently-submittable small change carved by Rosie along project/OWNERS boundaries; must be independently committable and pass its own presubmits.
Cattle vs pets
A mindset contrast; an LSC's individual commit is disposable cattle, not a hand-raised pet.
Global approver
An expert with repo-wide approval rights who auto-approves LSC shards with pattern-based tooling, replacing per-shard local-owner review.
Kythe / ClangMR / Refaster
Semantic-index and AST-level analysis/transform tools providing the codebase insight LSC tooling needs.
Type alias / forwarding function
Language features that let old users keep working while migrating non-atomically (scoped_ptr→unique_ptr transitioned via a type alias).
Tricorder
A framework that flags new uses of deprecated objects at review time, preventing backsliding after an LSC cleanup.

Questions to push your thinking

  1. Back to the opening: how many files can you reliably update in one atomic commit? What constrains it (VCS capacity? merge conflicts? presubmits?)? Could you in an emergency? If that's far smaller than your codebase, is your "divide and conquer" tooling (who's your Rosie?) ready?
  2. Does your codebase have a haunted graveyard — a system on a critical path too old to touch? Is it what blocks upgrading or removing old systems? Would you invest in tests first to free it? Per ch11, ch14.
  3. When you recently pushed a new system, did you expect organic migration or own the LSC yourself? Recall "unfunded mandate" and "copying old examples" — will organic migration actually finish in your org? Per ch15.
  4. Do you hold a "cattle" or "pet" mindset toward individual LSC commits? If your automation can't yet cheaply regenerate changes, where are you stuck — tooling (generation), testing (a TAP-train equivalent), or culture (owners' veto and trust)?
  5. Is your change-generation tooling's human cost sublinear in repo size? If the repo grows tenfold, does the person-time balloon? Where should the ">500 edits → learn the tools" rule sit for your team? Per ch17, ch20.
