Critique: Google's Code Review Tool

Critique：Google 的程式碼審查工具
作者 Caitlin Sadowski 等 PDF p.427–444 Scale 規模 Trade-offs 取捨 Time 時間
"Trust and communication are core to the code review process. A tool can enhance the experience, but it can't replace them." 信任與溝通是程式碼審查流程的核心。工具能增強這份體驗，卻無法取代它們。

摘要

ch09 談 code review 的流程，本章談支撐它的工具——Google 自家、最受歡迎的 Critique。四條指導原則：Simplicity（簡單，影響最大）、Foundation of trust（信任的地基）、Generic communication（通用的溝通）、Workflow integration（與工作流整合）。Simplicity 與 integration 有張力——Google 曾考慮做「Code Central」三合一但刻意放棄，讓 code review 保持為主要焦點。

支撐提交前審查的六階段：建立變更（上傳快照→觸發分析器）→請求審查→留言（unresolved 預設）→修改回覆→核准（LGTM、標綠）→提交。其餘：強大 diffing（LCS＋Kythe 交叉引用＋intraline＋move detection＋並排）、分析 chip（紅/黃/灰、附 fix suggestion）、緊密工具整合（偏好連結而非嵌入）、GwsQ 自動指派、attention set、Changelist Search dashboard。

評分簡化成三件事：LGTM＋Approval＋unresolved comments——每變更至少一個 LGTM（兩雙眼睛），LGTM/Approval 永遠正面（不能無理由 thumbs-down）。結論：信任與溝通是核心、工具能增強卻無法取代、緊密整合是成功關鍵。

為什麼要讀這一篇

你在挑/做審查工具，糾結加多少功能。 頭號原則是 Simplicity（沒有不必要的選擇），甚至刻意放棄「Code Central」三合一。看懂「少即是多」，你更敢拒絕增複雜度的功能。
你的審查常卡在「現在輪到誰？」 attention set 是個小到不行卻效果驚人的功能——粗體標出「卡在誰身上」，實作後大家無法想像沒有它的日子。
你的評分機制很複雜。 Critique 簡化成 LGTM＋Approval＋unresolved comments，並讓 LGTM/Approval 永遠正面（負面回饋必須綁定具體 unresolved comment）——還正面影響了審查文化。
你想把審查工具接進整個生態。 成功關鍵之一是緊密工具整合（跳 Code Search、在 Cider 編輯、看 Tricorder）——但用連結而非嵌入，以免犧牲核心審查體驗。

細節重點

原則 Code Review Tooling Principles 四條指導原則

  • Simplicity（簡單）：UI 快、導航易、清楚的視覺狀態；「沒有不必要的選擇」。影響最大——為一小撮使用者讓模型變複雜的功能都被放棄。
  • Foundation of trust（信任的地基）：審查是賦能而非拖慢；信任作者會處理小意見、不必再加一輪複查。
  • Generic communication（通用的溝通）：溝通問題鮮少靠工具解決；優先用一般留言而非複雜協定。
  • Workflow integration：與其他核心工具多個整合點。張力：曾考慮「Code Central」三合一但刻意放棄、功能用連結而非嵌入。

理解 Diffing、分析、整合、Attention Set stage 1–4

  • Diffing：LCS＋語法高亮＋Kythe 交叉引用（ch17）＋intraline diff（字元級）＋忽略空白＋move detection＋並排；prefetch 一切。最佳化 diff 是好工具的核心要求。
  • 分析結果：上傳快照觸發 Tricorder（ch20）；chip 紅/黃/灰、binary 可行動性、常附 fix suggestion 一鍵套用。
  • 緊密工具整合：Cider/Code Search/Tricorder/Rapid/Zapfhahn；偏好連結而非嵌入（唯覆蓋率用 gutter 色嵌入）；靠 FUSE workspace＋雲端 Source of Truth。
  • 請求審查：GwsQ 依 email alias 自動指派；審查者建議考慮 owner/熟悉/可用/GwsQ。presubmits（加 email list、跑測試、強制不變式）失敗會擋住。
  • 留言：人人皆可、per-person 已審勾選、Please fix/Done/Ack 快捷、原子式 publish。
  • Attention set（「輪到誰」）：粗體標出「變更卡在誰身上」；凸顯 code review 的回合制本質；實作後「無法想像以前怎麼活的」。dashboard 由 Changelist Search（regex、夠快可互動）驅動。

評分 Change Approvals 三件式評分：LGTM＋Approval＋unresolved

除了用留言給具體建議，還需要高層次的「OK」機制。Google 把評分簡化成三件事。
Hard（LGTM/Approval，只審查者能給/撤）vs Soft（unresolved comment，作者可標 resolved） 的區分促進並仰賴信任——審查者可附 unresolved 留言就 LGTM、不必事後逐一檢查，跨時區特別省時。

之後 After Commit、Gerrit 程式考古與開源替代

提交後：用於程式考古（瀏覽變更歷史、留言與演進，用於稽核/理解「為何這樣改」「bug 怎麼引入」、製作教材）；支援提交後留言與回滾。Gerrit：Critique 因與 monorepo 緊密耦合而不對外；開源（Chrome/Android）或非 monorepo 專案用 Gerrit——獨立、與 Git 整合、更可設定的評分（−2 否決）。

  1. 你用/做的審查工具，偏向 Simplicity 還是不斷加功能？有沒有「為一小撮使用者讓模型變複雜」的功能該砍？你敢不敢像 Google 放棄「Code Central」那樣克制把所有東西塞進一個工具？

Part IV — Tools
CHAPTER 19

Critique: Google's Code Review Tool

A tool can make review smoother — but trust and communication make it work
by Caitlin Sadowski et al. PDF p.427–444 Scale Trade-offs Time
"Trust and communication are core to the code review process. A tool can enhance the experience, but it can't replace them." A tool can enhance review, but trust and communication — between humans — are what make it work.

Summary

Ch.9 covered the process of code review; this chapter covers the tool — Google's beloved in-house Critique. Four guiding principles: Simplicity (most impact), Foundation of trust, Generic communication, Workflow integration. Simplicity and integration are in tension — Google considered an all-in-one "Code Central" but deliberately declined it, keeping code review the primary focus.

It supports six stages of precommit review: create a change (upload a snapshot → triggers analyzers) → request review → comment (unresolved by default) → modify and reply → approval (LGTM, turns green) → commit. Plus: powerful diffing (LCS + Kythe cross-references + intraline + move detection + side-by-side), analyzer chips (red/yellow/gray, with fix suggestions), tight integration (links over embedding), GwsQ auto-assignment, the attention set, and a Changelist Search dashboard.

Scoring is simplified to three things: LGTM + Approval + unresolved comments — every change needs at least one LGTM (two pairs of eyes), and LGTM/Approval are always positive (no unmotivated thumbs-down). Conclusion: trust and communication are core, a tool can enhance but not replace them, tight integration is key.

Why read this

You're choosing/building a review tool, agonizing over features. The top principle is Simplicity (no unnecessary choices), even declining an all-in-one "Code Central." Understand "less is more" and reject complexity-adding features.
Your reviews stall on "whose turn?" The attention set is a tiny feature with outsized effect — bolding "who it's blocked on," and once built no one could imagine working without it.
Your scoring is complex. Critique simplifies to LGTM + Approval + unresolved comments and makes LGTM/Approval always positive (negative feedback ties to a specific unresolved comment) — positively shaping culture.
You want to plug a review tool into your ecosystem. A key to success is tight integration (jump to Code Search, edit in Cider, view Tricorder) — but via links, not embedding, to avoid sacrificing the core experience.

Key details

PRINCIPLES Code Review Tooling Principles

  • Simplicity: fast UI, easy navigation, clear visual state; "no unnecessary choices." The most impactful — features that complicate the model for a few users were dropped.
  • Foundation of trust: review is for empowering, not slowing; trust authors to address minor comments rather than adding a re-review round.
  • Generic communication: communication problems are rarely solved by tooling; prefer plain comments over complex protocols.
  • Workflow integration: multiple integration points. Tension: considered "Code Central" but declined it; features are linked, not embedded.

UNDERSTAND Diffing, analysis, integration, attention set stages 1–4

  • Diffing: LCS + syntax highlighting + Kythe cross-references (ch17) + intraline diff (character-level) + ignore-whitespace + move detection + side-by-side; prefetch everything. Optimizing the diff is a core requirement.
  • Analysis results: uploading a snapshot triggers Tricorder (ch20); chips red/yellow/gray, binary actionability, often with a fix suggestion to apply in one click.
  • Tight integration: Cider/Code Search/Tricorder/Rapid/Zapfhahn; links over embedding (only coverage embedded via gutter colors); enabled by FUSE workspaces + a cloud Source of Truth.
  • Request review: GwsQ auto-assigns via an email alias; suggestions consider owner/familiarity/availability/GwsQ. Failed presubmits (email lists, test suites, invariants) block the change.
  • Commenting: anyone can; per-person reviewed checkboxes; Please fix/Done/Ack shortcuts; published atomically.
  • Attention set ("whose turn"): bolds "who the change is blocked on"; emphasizes review's turn-based nature; once built, "no one could imagine the previous state." The dashboard is powered by Changelist Search (regex, fast enough for interactive use).

SCORING Change Approvals three-part: LGTM + Approval + unresolved

Beyond specific suggestions via comments, you need a high-level "OK." Google simplifies scoring to three things.
The distinction between hard (LGTM/Approval — only reviewers grant/revoke) and soft (unresolved comments — the author can resolve) promotes and relies on trust — a reviewer can LGTM with unresolved comments without later verifying, saving time especially across time zones.

AFTER After Commit & Gerrit archaeology and the open-source alternative

After commit: used for change archaeology (browse a change's history, comments and evolution, for auditing/understanding "why" and "how a bug was introduced," producing training); supports post-commit comments and rollbacks. Gerrit: Critique isn't external (tightly coupled to the monorepo); open-source (Chrome/Android) or non-monorepo teams use Gerrit — standalone, Git-integrated, with more configurable scoring (−2 vetoes).

Key terms

Critique
Google's in-house, most-popular code review tool; supports the six-stage precommit review flow, tightly integrated with other tools.
The four principles
Simplicity / Trust / Generic communication / Workflow integration; Simplicity had the most impact (Google declined an all-in-one "Code Central").
Precommit review
Review before code enters the codebase; six stages: create → request → comment → modify → approve → commit.
Snapshot
A change's patch at a point in time; uploading triggers analyzers, and you can diff any two snapshots.
Diffing
LCS + syntax highlighting + Kythe cross-references + intraline + ignore-whitespace + move detection + side-by-side; optimizing the diff is a core requirement.
Analyzer chips / Tricorder
Analysis results as chips (red/yellow/gray, binary actionability, with fix suggestions); Tricorder is the static-analysis tool (see ch20).
GwsQ
Auto-assigns reviewers via an email alias; suggestions consider owner/familiarity/availability/GwsQ setup.
Presubmits
Run at review request and commit (email lists, test suites, invariants); a failure blocks the change.
Attention set / "whose turn"
Bolds "who the change is blocked on"; emphasizes review's turn-based nature; a small optimization that cuts friction.
Changelist Search
The search system powering the dashboard, indexing the latest state of all changes across Google, regex-queryable, fast for interactive use.
LGTM / Approval / unresolved (three-part scoring)

Questions to push your thinking

  1. Is your review tool oriented toward Simplicity or ever-more features? Are there features that "complicate the model for a few users" to cut? Could you, like Google declining "Code Central," resist cramming everything into one tool?
  2. Do your reviews stall on "whose turn?" Could you add an attention set-like mechanism? Recall a change idling because no one knew whose turn it was — how much time would this save?
  3. Is your scoring too complex? Could you simplify to LGTM + Approval + unresolved comments and require negative feedback to tie to something specific? What would that do to your review culture?
  4. How tightly is your review tool integrated with others? Which should be linked, which worth embedding? Recall Critique's "links by default, embed only coverage" — have you sacrificed simplicity for integration? Per ch17, ch20.
  5. Does "trust the author to address minor comments without re-checking" work on your team? Is your review empowering or slowing? Across time zones, how much round-tripping would that trust save? Per ch09, ch02's HRT.
