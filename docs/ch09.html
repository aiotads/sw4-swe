CHAPTER 09

Code Review

程式碼審查
作者 Tom Manshreck、Caitlin Sadowski PDF p.193–212 Scale 規模 Time 時間 Trade-offs 取捨
"Code review acts as the glue connecting engineers with one another." 程式碼審查是把工程師彼此黏在一起的膠水——測試、靜態分析、CI 全都掛在它上面。

摘要

程式碼審查（code review）是「由作者以外的人來審視程式碼、通常在程式進入程式碼庫之前」的流程。Google 的版本極端徹底：幾乎每一次變更、不論多小，都要在提交前被審（precommit review）；而每位工程師既發起審查、也審查別人。本章談「流程」而非工具（工具 Critique 另闢 ch19）。

審查的終點是另一位工程師的 LGTM（looks good to me）。但 Google 把「批准」拆成三個獨立的權限位元：①另一位工程師的 LGTM（正確且可理解）；②owner 核准（適不適合放這塊程式碼庫）；③具readability 者核准（符合語言風格）。常由一人扮演全部，但拆開來才能靈活又能擴展——「為何不讓一人包辦？答案是規模」。

審查帶來六項效益：正確性、可理解性、一致性、集體擁有感、知識分享、歷史紀錄。書中明言：正確性並非首要效益——「程式長期是否可被理解與維護」更重要，因為程式被讀遠多於被寫。最後是讓流程擴展的最佳實務：有禮且專業、寫小變更（約 200 行）、寫好說明、審查者降到最低（多半剛好一人）、能自動化就自動化。一句根本心法：程式碼是負債。

為什麼要讀這一篇

你把 review 當成「抓 bug 的關卡」。 這章翻轉認知：抓正確性不是 Google 最看重的效益。真正的長期價值是可理解性、一致性、知識分享、歷史紀錄——全為時間與規模服務。看懂這點，你寫留言的角度會完全不同。
你的 review 又慢又痛。 本章把「慢」歸因到實作方式而非審查本身：寫小變更（約 200 行、35% 只動一檔）、24 個工作小時內回饋、別擠牙膏式給意見、審查者降到一人（「最重要的是第一個 LGTM」）。
「擁有權」「readability」到底在解什麼？ 答案是 scale：把 LGTM／owner／readability 三個位元拆開，讓 intern 與 tech lead 走同一套流程卻有不同路徑；OWNERS 檔讓「誰負責」沿目錄樹一查就到，無需中央登記。
不同類型的變更該用不同力道審。 greenfield 最嚴、bug fix 最聚焦（別順手改別的）、機器產生的 LSC 只挑「你這塊特有」的問題（不能對流程行使否決權）。把力道用對地方，是讓 review 規模化的關鍵。

細節重點

流程 Code Review Flow 審查的六步流程

Google 採 precommit review（提交前審查）。終點是拿到 LGTM——一個「必要的權限位元」，但要和 owner／readability 位元組合才放行提交。

心法 Code Is a Liability 程式碼是負債

程式碼本身是負債，而非資產。 就算必要，它對某時某地的某人也終究是維護工作——像飛機的燃油，有重量。動筆前先確認功能值得；重複碼甚至比沒有更糟。「如果你在從頭造輪子，你大概做錯了。」
工程意涵：審查不是重開設計辯論的場合。設計該在設計審查／API 審查就定案；code review 既不該為「全新程式」感到意外，也不該被當成翻舊帳的機會。

核心 三個批准位元 The three approval bits

任何變更要被「批准」，牽涉三種審查——三個位元可任意組合，由一人或多人扮演。
  • 兩步常態：多數需多重批准者先拿 peer 的 LGTM（聚焦正確性），再找 owner／readability 核准（owner 更關心「會不會難維護？增加技術債？我們有能力維護嗎？」）。
  • 彈性：既是 owner 又有 readability 的 tech lead，只要再一個 LGTM 就能提交；intern 送同一變更則需「具 readability 的 owner」核准。

擁有權 Ownership OWNERS 檔・stewardship（by Hyrum Wright）

小團隊把全 repo 開給所有人；團隊一大就撐不住。Google 用 ownership 記錄「誰對哪一塊有知識與責任」——不是「佔有」程式碼，而是 stewardship（受託管理）（作者自承「stewards 管家」是更好的詞）。
  • 機制：OWNERS 檔列出某目錄及子目錄負責人；階層式相加（一檔由其目錄樹上方所有 OWNERS 成員的聯集擁有），最終解析到個人；鼓勵小而聚焦。
  • 支撐其他實務：root OWNERS 可當 LSC（ch22）的全域核准者；OWNERS 本身是一種文件（沿樹往上找負責人）；新專案放一個新 OWNERS 就成立，無需中央登記。已穩定擴展二十年。

效益 Code Review Benefits 六項效益

  • 正確性：多一雙眼睛確認做到所宣稱的事；IBM 研究指出越早發現缺陷越省時——但前提是流程必須輕量。作者享有 deference，審查者只在能改善可理解性/功能時才提替代。令人意外：正確性並非首要效益。
  • 可理解性：第一次有「作者以外的人」給不受作者偏誤影響的回饋；程式被讀遠多於被寫。對可理解性提問用「顧客永遠是對的」——你現在的疑問未來會放大許多倍。正確性＋可理解性＝LGTM 兩大判準。
  • 一致性：規模化下你的程式會被別人依賴、重構（甚至工具）；一致＋簡單更易被工具更新。這就是 LGTM 與 readability 分離的原因（Java 程式需「Java readability」核准）。
  • 心理與文化：強化「程式不是『你的』」；流程當「壞人」、審查者當「好人」（工具就叫 Critique）；提供 validation 對抗冒牌者症候群；發起審查逼作者「把 ducks 排好隊」。
  • 知識分享：最被低估的效益。資訊雙向流動；「工程師未必讀每封 email，但傾向回應每個寄來的 code review」——及時、可行動、跨時區跨專案。
  • 歷史紀錄：每次變更都進庫，審查本身成為可考古的歷史——調出某模式何時被引入、當時那次審查。

實務 Code Review Best Practices 讓流程能擴展

審查的摩擦與延遲多半是實作方式的問題，而非審查本身。保持敏捷、快速才能擴展。
  • 有禮且專業：批評嚴守專業範疇；尊重作者做法；24 個工作小時內回饋；別擠牙膏式給意見。作者也要專業——「你不是你的程式碼」，每個留言當 TODO，不同意用 PTAL（請再看一下）。
  • 寫小變更：保持敏捷最重要的一條，約 200 行為界；多數約一天內獲初步回饋、約 35% 只動一個檔案；讓「一人審查」與「易回滾」成為可能。當成最佳化，允許偶爾的大變更。
  • 寫好變更說明：第一行是黃金地段（工具摘要、email 主旨、Code Search 歷史那行）；說清「改什麼、為什麼」——「Bug fix」毫無幫助。描述是歷史紀錄。
  • 審查者降到最低：多數剛好一人。「最重要的是第一個 LGTM，後續邊際效益遞減，成本很快超過價值」；確需多人時各自聚焦不同面向。
  • 能自動化就自動化：用 presubmit（提交前檢查）在寄給審查者前攔下問題、退回作者先修，讓人專注在比格式更重要的事（見 ch20）。

分類 Types of Code Reviews 四種類型，不同力道

  • Greenfield（全新程式）：最不常見、卻最該嚴審——評估「能否經得起時間與規模」最重要的時機。確認 API 符合議定設計、完整測試、有 owner（常先審一個 OWNERS 檔）、足夠註解、納入 CI（ch23）。
  • 行為變更／改良／最佳化：多數變更屬此。判準：必要嗎？改善了程式碼庫嗎？（最棒的修改有些其實是刪除！）行為變更要連帶修測試；最佳化可能需附 benchmark。
  • 修錯與回滾：抗拒順手改別的——bug fix 只修這個 bug＋補一個能攔住它的測試。Rollback 數秒可生成但仍需審查；故所有變更都該小而原子化，以免回滾連鎖弄壞下游。
  • 重構與 LSC：許多變更機器自動產生（作者是機器）但仍要審。審機器變更時只標「你這塊特有」的疑慮，別質疑底層工具／流程（流程已被審過、個別團隊不能否決，否則無法擴展）；也別擴大範圍。

關鍵字補充

Precommit Review（提交前審查）
程式進庫前必須先被審。Google 對幾乎所有變更（不論大小）都強制執行，是少數的全面性流程之一。
LGTM（looks good to me）
審查者對「正確性＋可理解性」的同意，是一個必要的權限位元；預設只需一個。
三個批准位元（The three approval bits）
LGTM、owner approval、readability。可由一人或多人扮演、任意組合，拆開是為了彈性與規模。
OWNERS 檔（OWNERS file）
列出某目錄及子目錄負責人；階層式相加、解析到個人。讓「誰負責」沿目錄樹一查就到，新專案放一個就成立。
Ownership / Stewardship（擁有權／受託管理）
不是「佔有」程式碼，而是以公司最大利益看顧某段程式碼的責任（作者自承「stewards」是更好的詞）。
Code Is a Liability（程式碼是負債）
程式本身是維護成本而非資產；動筆前先確認功能值得，重複碼甚至比沒有更糟。
Readability（可讀性認證）
經該語言可讀性訓練取得、能核准「符合語言風格與最佳實務」的資格；與 LGTM 分離，專責一致性與可維護性。
Small Changes（小變更）
保持審查敏捷最重要的實務，約以 200 行為界；多數一天內獲初步回饋、約 35% 只動一檔；讓「一人審查」與「易回滾」成為可能。
PTAL（please take another look）
作者不同意審查者時，提出替代方案並請對方再看一次的禮貌用語，用來保持討論文明。
Presubmit（提交前檢查）
變更送審時自動跑的靜態分析／格式檢查，在寄給審查者前先攔問題、退回作者，讓人專注在更重要的事。
Rollback（回滾）
撤銷前一變更、回到已知狀態的變更；數秒可生成但仍需審查。故所有變更都應小而原子化。

延伸提問

  1. 你團隊把 code review 的價值主要押在「抓 bug」上嗎？若 Google 說正確性不是首要效益，那你們從審查得到的可理解性、一致性、知識分享、歷史紀錄有被認真對待嗎？下次寫留言時，你會不會多問一句「半年後的人讀得懂嗎」？
  2. 把你最近送審的變更量一量行數：有多少超過約 200 行？哪些其實可以切成數個小而原子化的變更？回想一次「為了等大 review 而卡住後續工作」——小變更能不能根治它？
  3. 你們的審查需要幾個人點頭才放行？對照「最重要的是第一個 LGTM」：你們是不是在用「找一群人取得一致同意」換安心，卻付出遞延與遞減效益？哪些變更其實一人審就夠？
  4. 套用「三個批准位元」想像你的團隊：你們有沒有把「正確性」「這塊程式碼庫的擁有權」「語言風格一致性」混成一團、由同一種人一次審完？拆開會不會讓 intern 與資深工程師都更好走？你們的 OWNERS 清楚嗎？
  5. 你修 bug 時會不會「順手」改其他東西？回想一次因此難以回滾或回歸測試的痛。你能不能養成「bug fix 只修這個 bug＋補一個能攔住它的測試」的紀律？對照「程式碼是負債」——你上次審查時，有沒有想過「最好的修改其實是刪除」？

Part III — Processes
CHAPTER 09

Code Review

The glue that connects engineers — and every other process
by Tom Manshreck & Caitlin Sadowski PDF p.193–212 Scale Time Trade-offs
"Code review acts as the glue connecting engineers with one another." Code review is the connective tissue — testing, static analysis, and CI all hang off it.

Summary

Code review is "a process in which code is reviewed by someone other than the author, often before the introduction of that code into a codebase." Google's version is unusually thorough: essentially every change, no matter how small, is reviewed before being committed (a precommit review), and every engineer both initiates and performs reviews. This chapter is about the process, not the tool — Critique gets its own chapter (ch19).

The end goal is another engineer's LGTM ("looks good to me"). But Google splits "approval" into three independent permission bits: ① an LGTM (correct and comprehensible); ② code-owner approval (appropriate for this part of the codebase); ③ readability approval (conforms to the language's style). One person often plays all three, but keeping them separable is what makes review flexible *and* able to scale — "why not one all-powerful reviewer? The short answer is scale."

Review yields six benefits: correctness, comprehension, consistency, collective ownership, knowledge sharing, and a historical record. Strikingly, correctness is not the primary benefit — whether code stays understandable and maintainable matters more, because code is read far more than written. Then the practices that let it scale: be polite and professional, write small changes (~200 lines), write good descriptions, keep reviewers to a minimum (usually exactly one), and automate where possible. One root idea: code is a liability.

Why read this

You treat review as "the bug-catching gate." This flips that: catching correctness is not the benefit Google values most. The real payoff is comprehension, consistency, knowledge sharing, and a historical record — all serving time and scale. Internalize it and your comments change.
Your reviews are slow and painful. The chapter pins "slow" on the implementation, not review itself: write small changes (~200 lines; ~35% touch one file), feedback within 24 working hours, don't review piecemeal, keep reviewers to one ("the most important LGTM is the first one").
What do "ownership" and "readability" actually solve? The answer is scale: splitting LGTM / owner / readability lets an intern and a tech lead run the same process by different paths; OWNERS files answer "who's responsible" by walking the tree — no central registry.
Different changes need different review effort. Greenfield gets the strictest look; a bug fix stays focused (don't fix unrelated things); a machine-generated LSC gets only comments specific to your code (you can't veto the process). Putting effort where it belongs is how review scales.

Key details

FLOW Code Review Flow

Google does precommit review. The end goal is an LGTM — a *necessary* permission bit, but it combines with the owner and readability bits before a commit is allowed.

MINDSET Code Is a Liability

Code itself is a liability, not an asset. Even a necessary one is a maintenance task for someone down the line — like an airplane's fuel: it has weight. Confirm a feature is warranted before writing it; duplicated code can be worse than none. "If you're writing it from scratch, you're doing it wrong!"
Takeaway: a review is not the place to rehash design. Design belongs in design/API reviews; a review of brand-new code shouldn't surprise, and review isn't a chance to relitigate the past.

CORE The Three Approval Bits

Three aspects require "approval" for any change — three bits that combine freely, played by one or more people.
  • Two-step norm: most reviews needing more than one approval get a peer's LGTM first (correctness), then owner/readability approval (the owner cares about "hard to maintain? technical debt? do we have the expertise?").
  • Flexibility: a tech lead who owns the project and has readability needs only one more LGTM; an intern sending the same change needs approval from an owner with readability.

OWNERSHIP Ownership OWNERS files · stewardship (by Hyrum Wright)

Small teams grant everyone everything; as the team grows it fails to scale. Google records "who has what knowledge and responsibility" via ownership — not *possession* of code but *stewardship* (the author admits "stewards" was the better word).
  • Mechanism: OWNERS files list the people responsible for a directory and its children; hierarchically additive (a file is owned by the union of all OWNERS above it), resolving to individuals; keep lists small and focused.
  • Underpins other practices: the root OWNERS set acts as global approvers for LSCs (ch22); OWNERS double as documentation (walk up the tree); a new project just adds a new OWNERS file — no central registry. Scaled well for two decades.

BENEFITS Code Review Benefits the six

  • Correctness: a second set of eyes; finding defects earlier costs less — but only if the process stays lightweight. Authors get deference; reviewers propose alternatives only to improve comprehension/functionality. Surprisingly, correctness is not the primary benefit.
  • Comprehension: the first feedback unbiased by the author's perspective; code is read far more than written. Treat comprehension questions by "the customer is always right" — every question now multiplies over time. Correctness + comprehension = the two LGTM criteria.
  • Consistency: at scale your code is depended on and refactored by others (and tools); consistent + simple is easier for tools to update. Hence LGTM is separated from readability (Java code needs a "Java readability" approver).
  • Psychological & cultural: reinforces "code is not 'yours'"; the process is the "bad cop" so the reviewer stays the "good cop" (the tool is called Critique); offers validation against imposter syndrome; initiating a review makes you "get your ducks in a row."
  • Knowledge sharing: the most underrated benefit. Information flows both ways; "an engineer may not read every email, but they tend to respond to every code review sent" — timely, actionable, cross-team.
  • Historical record: each change joins the codebase, so review is a record you can do "archeology" on — when a pattern was introduced and the original review.

PRACTICES Code Review Best Practices keep it scalable

Friction and delay are usually the chosen implementation, not review itself. Keep it nimble and quick to scale.
  • Be polite and professional: keep criticism strictly professional; defer to authors' approaches; feedback within 24 working hours; don't review piecemeal. Authors too — "you are not your code," treat each comment as a TODO, use PTAL ("please take another look") when you disagree.
  • Write small changes: the most important practice for nimble review, bounded at ~200 lines; most get initial feedback within a day, ~35% touch one file; what makes one-reviewer review and easy rollback possible. Treat "prefer small" as an optimization, allowing occasional larger changes.
  • Write good descriptions: the first line is prime real estate (tool summary, email subject, Code Search history line); say what changed and why — "Bug fix" helps no one. The description is the historical record.
  • Keep reviewers to a minimum: most are exactly one. "The most important LGTM is the first one; subsequent ones hit diminishing returns; cost soon outweighs value." When multiple are needed, each focuses on a different aspect.
  • Automate where possible: use presubmits to catch problems before a change reaches a reviewer (avoiding an awkward email), freeing reviewers for things more important than formatting (see ch20).

TYPES Types of Code Reviews four buckets, different effort

  • Greenfield (brand-new code): least common, most important to review hard — the best time to judge resilience to time and scale. Ensure the API matches an agreed design, is fully tested, has owners (often a single OWNERS file first), is well-commented, and enters CI (ch23).
  • Behavioral changes / improvements / optimizations: most changes. Criteria: is it necessary, does it improve the codebase? (Some of the best changes are deletions!) Behavioral changes need revised tests; optimizations may need benchmarks.
  • Bug fixes and rollbacks: resist fixing unrelated things — fix only this bug + a test that catches it. A rollback is created in seconds but still requires review; so all changes should be small and atomic to avoid cascading breakage.
  • Refactorings and LSCs: many changes are machine-generated (the author is a machine) but still require review. Review them by flagging only concerns specific to your code, not the underlying tool/process (already reviewed; teams can't veto it, or it couldn't scale); and don't expand scope.

Key terms

Precommit review
Code must be reviewed before entering the codebase. Mandated for almost every change regardless of size — one of Google's few blanket processes.
LGTM ("looks good to me")
A reviewer's agreement on correctness + comprehension; a necessary permission bit. Only one is required by default.
The three approval bits
LGTM, owner approval, readability. Played by one or more people, combined freely; split for flexibility and scale.
OWNERS file
Lists the people responsible for a directory and its children; hierarchically additive, resolving to individuals. Answers "who's responsible" by walking the tree; a new project just adds one.
Ownership / stewardship
Not *possession* of code but responsibility to steward a section in the company's interest (the author admits "stewards" was the better word).
Code is a liability
Code is a maintenance cost, not an asset; confirm a feature is warranted first, and duplicated code can be worse than none.
Readability
A certification, earned via that language's readability training, to approve that code follows the language's style; separated from LGTM, dedicated to consistency and maintainability.
Small changes
The most important practice for nimble review, bounded at ~200 lines; most get initial feedback within a day, ~35% touch one file; enables one-reviewer review and easy rollback.
PTAL ("please take another look")
A polite cue from a disagreeing author offering an alternative and asking the reviewer to look again — keeps debates civil.
Presubmit
Automated static-analysis/format checks run when a change is sent, catching problems before they reach a reviewer.
Rollback
A change that undoes a prior one to a known state; created in seconds but still requires review. Hence all changes should be small and atomic.

Questions to push your thinking

  1. Does your team pin review's value mostly on "catching bugs"? If correctness is not the primary benefit, are you taking comprehension, consistency, knowledge sharing, and the historical record seriously? Next comment, will you also ask "can someone read this in six months"?
  2. Measure your recent changes' line counts: how many exceed ~200 lines? Which could be split into small, atomic changes? Recall follow-on work stalling on a big review — would small changes have cured it?
  3. How many approvals do your changes need to land? Against "the most important LGTM is the first one": are you buying reassurance with a group's unanimous consent at the cost of delay and diminishing returns? Which changes really need only one reviewer?
  4. Apply the three approval bits: do you lump "correctness," "ownership of this codebase," and "language-style consistency" into one review by one kind of person? Would splitting them ease the path for interns and seniors alike? Are your OWNERS clear?
  5. Do you "while I'm here" fix unrelated things in a bug fix? Recall the pain of a change then hard to roll back or regression-test. Can you build the discipline of "fix only this bug + add a test that catches it"? Against "code is a liability" — did your last review consider that the best change might be a deletion?
