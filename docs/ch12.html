CHAPTER 12

Unit Testing

單元測試
作者 Erik Kuefler PDF p.259–284 Time 時間 Scale 規模 Trade-offs 取捨
"Test behaviors, not methods. Follow DAMP over DRY when sharing code for tests." 測試「行為」而非「方法」；共用程式碼時奉行 DAMP（描述性而有意義）而非 DRY。

摘要

接續 ch11 的 size／scope 兩軸，本章聚焦 unit test（範圍窄、通常 size 小）——Google 最常見的測試（建議約 80% 單元／20% 較廣，一天常跑數千個）。整章核心只有一個詞：maintainability（可維護性）。可維護的測試「就是會動」——寫完不必再想，直到失敗，而失敗指向有清楚成因的真實 bug。Mary 的故事是反例：改幾行卻被噴一整頁錯誤、修一整天，這些測試沒抓到真 bug，只是壞在對程式內部結構的假設上。

可維護性的兩大敵人是 brittle（脆弱） 與 unclear（不清晰）。防脆弱三招：①追求「不會變」的測試（四種變更只有「改變既有行為」該動測試）；②透過 public API 測（走正門，而非戳實作細節）；③測『狀態』而非『互動』（過度依賴 mock 是元兇）。寫清晰測試：完整且精簡、測行為而非方法（given/when/then、以行為命名）、別放邏輯、寫清楚的失敗訊息。共用程式碼：DAMP, not DRY——一點重複可以，只要更具描述性與意義。

為什麼要讀這一篇

你每次小改動都被一堆測試噴錯，卻沒introduce 任何 bug。 那是 brittle tests。三帖藥：追求不會變的測試、透過 public API 測、測狀態而非互動。讀完你會寫「改 production 不必跟著改」的測試。
你習慣「一個 method 配一個 testXxx」。 那會隨 method 變複雜而失控。改成測行為——given… when… then…——每個測試只管一個行為，讀來像自然語言、失敗時一眼定位。
你以為測試也該 DRY。 反直覺：測試該 DAMP。因為測試本身沒有測試，複雜度代價更高；把資訊藏進 helper 反而讓測試「不完整」、藏住 bug。一點重複換來「讀完就懂」很划算。
你被難懂的測試拖垮、想直接刪掉。 刪掉看不懂的測試＝開個你不知道的覆蓋率洞，且代表它可能多年來價值為零。不如一開始就寫清晰、不放邏輯、失敗訊息明確的測試。

細節重點

核心 The Importance of Maintainability 可維護性為何最重要

抓 bug 之後，測試最重要的目的是提升生產力。但 Mary 的故事顯示反面：壞測試讓生產力不增反減——改動沒引入真 bug，卻因綁死內部結構被迫逐一修補。壞測試必須在 check-in 前修好。問題分兩類：brittle 與 unclear。

防脆弱① Strive for Unchanging Tests 追求不會變的測試

理想的測試是不會變的——除非被測系統的需求改變，否則永不需動。對照工程師對 production 做的四種變更：

防脆弱②③ Public APIs、State not Interactions 走正門、測狀態

  • 透過 public API 測：用使用者的方式呼叫系統（「先走正門」），而非戳實作細節（拿掉 private、戳 isValid/saveToDatabase）。能弄壞測試的改動按定義也會弄壞使用者 →形成明示契約、更不脆弱、還能當文件。
  • 測狀態，而非互動：state＝觀察呼叫後系統變成什麼；interaction＝檢查對協作者做了哪些呼叫。互動測試更脆弱（在乎「如何抵達」而非「結果」）——verify(database).put(...) 兩種出錯方式。元兇是過度依賴 mocking 框架；能用真實物件就用（見 ch13）。

清晰 Writing Clear Tests 測行為、別放邏輯、好失敗訊息

測試遲早會失敗（好事）。失敗只有兩因：①系統有問題（測試正做該做的事）②測試本身有瑕疵（brittle）。診斷速度取決於 clarity——而測試常比寫它的人活得更久，看不懂就刪＝開覆蓋率洞、且它可能多年價值為零。
  • 完整且精簡：本體含理解「如何得到結果」的一切（完整）、無分散注意的雜訊（精簡）。
  • 別在測試裡放邏輯：清晰的測試一看就對；URL 例子裡僅一個字串串接就藏住「兩條斜線」的 bug——攤平即現形。寧用直線程式碼。
  • 寫清楚的失敗訊息：清楚區分預期 vs 實際＋脈絡。Google 的 Truth 庫比 JUnit assertTrue 訊息更有用。

共用 DAMP, Not DRY 描述性而有意義

DRY（每概念只一處）在 production 划算；但測試裡利弊不同——好測試本就該穩定、你希望它在系統變時壞掉，且測試本身沒有測試、複雜度代價更高。故測試該 DAMP（Descriptive And Meaningful Phrases）：一點重複可以，只要更清晰。DAMP 與 DRY 互補。
  • Shared Values：用帶預設值的 helper（具名參數／Builder）勝過語意含糊的共用常數；可輕微隨機化未設的預設。
  • Shared Setup：@Before 適合建被測物件與協作者；但測試在乎的特定值該在本體直接覆寫（別藏在 setup）。
  • Shared Helpers / Validation：避免每測結尾的固定 validate()（不以行為驅動）；好的 validation helper 只斷言單一概念事實。
  • Test Infrastructure：跨多 suite 共用＝更像 production，須有自己的測試、儘早標準化（Google 強制 Mockito 為唯一 Java mock 框架）。

關鍵字補充

Unit Test（單元測試）
範圍窄（單一 class/method）、通常 size 小；Google 最常見、建議約 80%；首要價值是可維護地提升生產力。

Unchanging Tests（不會變的測試）
除非系統需求改變否則永不需動；四種變更中只有「改變既有行為」該更新既有測試。
Test via Public APIs（透過 public API 測）
用使用者的方式呼叫系統（走正門）而非戳實作細節；弄壞測試＝弄壞使用者，形成明示契約。
State vs Interaction Testing（測狀態 vs 測互動）
state＝觀察呼叫後系統變成什麼；interaction＝檢查對協作者做了哪些呼叫。互動測試較脆弱。
Behaviors, Not Methods（行為而非方法）
為每個行為寫一個測試；behavior 是「某狀態下對輸入的回應保證」，用 given/when/then 表達，與 method 多對多。
Given / When / Then
行為的三部分（arrange/act/assert）：設定／動作／驗證；讓測試可在名稱、註解、程式三種粒度閱讀。
DAMP（Descriptive And Meaningful Phrases）
測試的程式共用哲學：容忍一點重複以換取描述性與意義；與 DRY 互補（因測試本身無測試）。
Truth
Google 的 Java 斷言庫，能產生比 JUnit assertTrue 更有用的失敗訊息（明確點出 subject 與預期）。
Test Infrastructure（測試基礎設施）
跨多 suite 共用的程式碼；更像 production，須有自己的測試、儘早標準化（如強制 Mockito）。

延伸提問

  1. 拿你「改 production 卻被迫跟著改一堆測試」的經驗：那些測試是不是綁死了實作細節（戳 private、verify 互動）？若改成透過 public API 測狀態，這次改動還會弄壞它們嗎？你動到的是「四種變更」的哪一種？
  2. 翻一個你寫的 testXxx（對應某 method）：它是不是其實在測多個行為？用 given/when/then 拆成數個 behavior-driven 測試、用「should…」重新命名——失敗報告會不會更快告訴你哪個行為壞了？
  3. 你的測試裡有沒有邏輯（迴圈、條件、字串拼接）？挑一段攤平成直線程式碼——會不會像 URL 例子一樣，攤平後看見一個原本藏住的 bug？
  4. 你的套件「太 DRY」還是「DAMP」？有沒有把關鍵資訊藏進 createX／validateX helper，導致讀單一測試得跳出去好幾處？哪些 helper 該內聯回本體？
  5. 你團隊的測試基礎設施有沒有自己的測試？mocking／斷言庫有沒有標準化成一套？值不值得像 Google 強制 Mockito 那樣收斂？對照 ch13。

Part III — Processes
CHAPTER 12

Unit Testing

Test what the system promises, not how it's built
by Erik Kuefler PDF p.259–284 Time Scale Trade-offs
"Test behaviors, not methods. Follow DAMP over DRY when sharing code for tests." Test what the system promises, not how it's built — and let tests repeat a little if it makes them clear.

Summary

Continuing ch11's size/scope axes, this chapter focuses on the unit test (narrow scope, usually small) — Google's most common (mix ≈ 80% unit / 20% broader, thousands run daily). The whole chapter turns on one word: maintainability. Maintainable tests "just work" — forgotten after writing until they fail, with failures pointing to real bugs with clear causes. Mary's story is the anti-example: a few-line change yields a screenful of errors that catch no real bug — they merely broke on assumptions about internal structure.

Maintainability's two enemies are brittleness and unclarity. Three cures for brittleness: ① strive for unchanging tests (of four kinds of change, only "changing existing behavior" should touch existing tests); ② test via public APIs (front door, not implementation details); ③ test state, not interactions (mock over-use is the chief cause). For clarity: complete and concise, test behaviors not methods (given/when/then, named after the behavior), don't put logic in tests, write clear failure messages. For code sharing: DAMP, not DRY — a little duplication is fine if it's more descriptive and meaningful.

Why read this

Every small change gives you a wall of failures, with no bug introduced. Those are brittle tests. Three cures: strive for unchanging tests, test via public APIs, test state not interactions. You'll write tests that don't change when production does.
You write one `testXxx` per method. That spirals as methods grow. Switch to testing behaviors — given… when… then… — one behavior per test, reading like natural language, pointing at the broken behavior.
You assume tests should be DRY too. Counterintuitively, tests should be DAMP: since tests aren't themselves tested, complexity costs more; hiding info in helpers makes tests incomplete and hides bugs. Duplication buys "read it top to bottom."
Unintelligible tests drag you down, tempting you to delete them. Deleting one opens an unknown coverage hole and signals it gave zero value for years. Better to write clear, logic-free tests with explicit failure messages.

Key details

CORE The Importance of Maintainability

After catching bugs, a test's top purpose is improving productivity. But Mary's story shows the flip side: bad tests drain it — her change introduced no bug, yet tests coupled to internal structure forced patching. Bad tests must be fixed before check-in. Problems split into brittle and unclear.

CURE 1 Strive for Unchanging Tests

The ideal test is unchanging — never needs changing unless the system's requirements change. Map against the four kinds of production change:

CURES 2 & 3 Public APIs, State not Interactions

  • Test via public APIs: call the system as a user would ("use the front door"), not via implementation details (dropping private, poking isValid/saveToDatabase). A change that breaks the test by definition breaks a user → an explicit contract, less brittle, doubles as docs.
  • Test state, not interactions: state = observe what the system becomes; interaction = check calls on collaborators. Interaction tests are more brittle (care about "how," not "result") — verify(database).put(...) can go wrong two ways. The cause is mock over-use; prefer real objects (see ch13).

CLARITY Writing Clear Tests

Tests will eventually fail (good). Failures have two causes: ① the system has a problem (what tests are for) ② the test is flawed (brittle). Diagnosis speed depends on clarity — and tests outlive their authors, so deleting what you can't understand opens a coverage hole and may have given zero value for years.
  • Complete & concise: the body holds all info to understand "how it reaches its result" (complete) and nothing distracting (concise).
  • Don't put logic in tests: clear tests are obviously correct on inspection; in the URL example a single concatenation hid a double-slash bug — flattening reveals it. Prefer straight-line code.
  • Write clear failure messages: distinguish expected vs actual with context. Google's Truth gives more useful messages than JUnit's assertTrue.

SHARING DAMP, Not DRY

DRY (one canonical place) pays off in production; but the calculus differs for tests — good tests should be stable, you want them to break when the system changes, and tests aren't themselves tested, so complexity costs more. So tests should be DAMP (Descriptive And Meaningful Phrases): a little duplication is fine if clearer. DAMP complements DRY.
  • Shared values: prefer helper methods with defaults (named params / Builder) over ambiguous shared constants; even slightly randomize unset defaults.
  • Shared setup: @Before suits the object under test and collaborators; values a test cares about should be overridden in the body (not hidden in setup).
  • Shared helpers / validation: avoid a fixed validate() at every test's end (not behavior-driven); good validation helpers assert a single conceptual fact.
  • Test infrastructure: code shared across suites is more like production, needs its own tests, standardized early (Google mandated Mockito as the only Java mocking framework).

Key terms

Unit test
Narrow-scope (one class/method), usually small; Google's most common, ~80% recommended; its prime value is improving productivity maintainably.
Maintainability
Maintainable tests "just work": forgotten after writing until they fail, with failures pointing to real bugs with clear causes. The chapter's core.
Brittle test
Fails on an unrelated change introducing no real bug; usually coupled to implementation details. Distinct from flaky (nondeterministic).
Unchanging tests
Never need changing unless requirements change; only "changing existing behavior" should update existing tests.
Test via public APIs
Call the system as a user would (front door), not via implementation details; breaking the test = breaking a user, an explicit contract.
State vs interaction testing
State = observe what the system becomes; interaction = check calls on collaborators. Interaction tests are more brittle.
Behaviors, not methods
A test per behavior; a behavior is a "guarantee about response to inputs in a state," expressed as given/when/then; method↔behavior is many-to-many.
Given / when / then
The three parts of a behavior (arrange/act/assert): set up / act / validate; readable at name, comment, and code granularities.
DAMP (Descriptive And Meaningful Phrases)
The test code-sharing philosophy: tolerate some duplication for descriptiveness; complementary to DRY (tests aren't themselves tested).
Truth
Google's Java assertion library, producing more useful failure messages than JUnit's assertTrue (names the subject and expectation).
Test infrastructure
Code shared across suites; more like production, needs its own tests and early standardization (e.g., mandating Mockito).

Questions to push your thinking

  1. Take a "changed production, forced to change many tests" experience: were those tests coupled to implementation details (poking privates, verifying interactions)? Testing state via the public API, would this change still break them? Which of the "four kinds" was yours?
  2. Pick a testXxx (mapped to a method): is it testing multiple behaviors? Split it into behavior-driven tests with given/when/then, renamed "should…" — would failures now tell you faster which behavior broke?
  3. Is there logic in your tests (loops, conditionals, concat)? Flatten one into straight-line code — like the URL example, does it reveal a hidden bug?
  4. Is your suite "too DRY" or "DAMP"? Have you hidden key info in createX/validateX helpers, so reading one test means jumping around? Which should be inlined back?
  5. Does your test infrastructure have its own tests? Are your mocking/assertion libraries standardized to one? Worth converging like Google's Mockito mandate? Per ch13.
