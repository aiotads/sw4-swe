CHAPTER 18

Build Systems and Build Philosophy

建置系統與建置哲學
作者 Erik Kuefler PDF p.399–426 Scale 規模 Time 時間 Trade-offs 取捨
"Limiting engineers' power and flexibility can improve their productivity." 限制工程師的權力與彈性，反而能提升他們的生產力。

摘要

Google 從零打造了 Blaze（2015 開源版 Bazel），目標是讓建置快（單一指令、幾秒）且正確（任何人任何機器、相同輸入永遠相同結果）。絕大多數的 build 是自動觸發的，幾乎所有開發工具都掛在它上面。一個反覆出現的主題：管理依賴是建置系統最根本的工作。

核心對比兩種哲學：Task-based（任務式，Ant/Maven/Gradle）——基本單元是任意腳本，彈性高但給工程師太多權力、給系統太少，故難平行化/增量/維護；Artifact-based（產物式，Blaze/Bazel）——BUILD 檔是宣告式 manifest，工程師說「建什麼」、系統決定「怎麼建」，故能保證正確並大幅平行化與重用。

由此推展出分散式建置（遠端快取 ObjFS＋遠端執行 Forge，自 2008）。取捨是時間/規模/彈性：DIY 腳本→task-based→artifact-based。Google 相信幾乎每個新專案從一開始就該用 artifact-based。模組與依賴：細粒度模組（1:1:1）、最小 visibility、嚴格傳遞依賴、手動釘版本、One-Version Rule。

為什麼要讀這一篇

你的 build 又慢又愛因「換台機器就壞」抓狂。 你用的可能是 task-based，系統不知道腳本在幹嘛、無法安全平行化或可靠增量。看懂 artifact-based 才懂「限制彈性反而提升生產力」。
你以為「有 compiler 就夠」或「寫個 build 腳本就好」。 跨語言、外部依賴、新人 bootstrap、自動建置——每步都讓 DIY 腳本崩潰。本章幫你判斷「何時該投資真正的建置系統」。
你被鑽石依賴、版本衝突、latest 害慘過。 三帖藥：手動釘版本、One-Version Rule、嚴格傳遞依賴。理解「依賴是 target 的公開契約」。
你在決定模組要粗還是細。 細粒度模組擴展更好（1:1:1）——更快的分散式 build、更少重建、測試只跑受影響子集；配合工具自動管理 BUILD 檔。

細節重點

目的 Purpose、沒有建置系統會怎樣 Fast & Correct

所有建置系統的目的：把原始碼轉成可執行檔。好系統最佳化兩性質——Fast（單一指令、幾秒）與 Correct（任何人任何機器、相同輸入永遠相同結果）。Bazel 的目標是不必在快與正確間二選一。
沒有建置系統：javac *.java 在單一目錄夠用；但一旦跨語言/編譯單元/外部依賴，shell 腳本最終淪為冗長、慢、不可重現、新人 bootstrap 痛苦的經典規模問題。管理自己的程式碼很直接，管理依賴才是難題。

對比 Task-based vs Artifact-based 本章核心哲學

解方是把權力從工程師手中拿回給系統：別讓工程師定義任意任務，而是工程師說 what、系統決定 how。
Bazel 巧招：tools as dependencies（toolchain 達平台獨立）、custom rule→action、sandboxing（LXC）讓 action 只見宣告的 I/O、外部依賴用密碼學雜湊確定化。

規模 Distributed Builds 遠端快取＋遠端執行

20 億行、依賴鏈極深（簡單 binary 常依賴數萬 target），單機不可能在合理時間建完——唯一解是分散式建置。

取捨 模組與依賴 1:1:1、One-Version、手動釘版本

  • 細粒度模組（1:1:1）：每目錄一個 package/target/BUILD（Pants 命名）；好處在規模下顯現——更快分散式 build、更少重建、測試只跑受影響子集。最小化 visibility。
  • 內部依賴：從源碼建、無「版本」；strict transitive deps——引用沒直接依賴的符號就失敗（避免「依賴變成公開契約」、讓 build 更快）。
  • 外部依賴：手動釘版本（不要 latest，可發現/回滾）；One-Version Rule 避免鑽石依賴；Bazel 不自動下載傳遞依賴。
  • 安全：鏡像到自控伺服器、要求 hash、或 vendoring（check-in 到 third_party，把外部變內部）——Google 用之（仰賴超大 monorepo）。
結論：限制工程師的權力與彈性反而提升生產力；「能信任 build 是強大的——增量 build 就是會動，幾乎不需清快取或跑 clean」。

CHAPTER 18

Build Systems and Build Philosophy

Take power from engineers, give it to the system — everyone gets faster
by Erik Kuefler PDF p.399–426 Scale Time Trade-offs
"Limiting engineers' power and flexibility can improve their productivity." Counterintuitively, giving power to the system makes everyone faster.

Summary

Google built Blaze from scratch (open-sourced 2015 as Bazel), aiming for builds that are fast (one command, seconds) and correct (any developer/machine, same inputs → same result). The vast majority of builds are triggered automatically, and nearly every dev tool hangs off it. A recurring theme: managing dependencies is a build system's most fundamental job.

The core contrast of two philosophies: Task-based (Ant/Maven/Gradle) — the unit is an arbitrary script; flexible but gives engineers too much power and the system too little, so it's hard to parallelize/incrementalize/maintain; Artifact-based (Blaze/Bazel) — BUILD files are a declarative manifest where engineers say what to build and the system decides how, so it can guarantee correctness and heavily parallelize and reuse.

From there comes distributed builds (remote caching ObjFS + remote execution Forge, since 2008). The trade-offs are time/scale/flexibility: DIY scripts → task-based → artifact-based. Google believes almost every new project should use artifact-based from the start. Modules and dependencies: fine-grained modules (1:1:1), minimal visibility, strict transitive deps, manual version pinning, the One-Version Rule.

Why read this

Your build is slow and maddening because "it breaks on another machine." You're probably task-based; the system doesn't know what scripts do, so it can't safely parallelize or do reliable incrementals. Artifact-based explains why "limiting flexibility improves productivity."
You think "a compiler is all I need" or "a build script will do." Multiple languages, external deps, newcomer bootstrapping, automated builds — each breaks DIY scripts. The chapter helps you judge "when to invest in a real build system."
You've been burned by diamond dependencies, version conflicts, latest. Three cures: pin versions manually, the One-Version Rule, strict transitive deps. Understand "a dependency is a target's public contract."
You're deciding coarse vs fine modules. Fine-grained scales better (1:1:1) — faster distributed builds, fewer rebuilds, running only the affected test subset; tools auto-manage BUILD files.

Key details

PURPOSE Purpose & life without a build system Fast & Correct

Every build system turns source into executables. A good one optimizes Fast (one command, seconds) and Correct (any developer/machine, same inputs → same result). Bazel's goal: avoid choosing between speed and correctness.
Without a build system: javac *.java suffices for one directory; but across languages/units/external deps, shell scripts become verbose, slow, unreproducible, a bootstrapping nightmare — the classic problem of scale. Managing your own code is straightforward; managing dependencies is the hard part.

CONTRAST Task-based vs Artifact-based the core philosophy

The fix is to take power back from engineers and give it to the system: not arbitrary tasks, but engineers say what, the system decides how.
Bazel's tricks: tools as dependencies (toolchains for platform independence), custom rules → actions, sandboxing (LXC) so an action sees only its declared I/O, and external deps made deterministic via a cryptographic hash.

SCALE Distributed Builds remote caching + remote execution

2 billion lines, deep dependency chains (simple binaries depend on tens of thousands of targets) — a single machine can't build in reasonable time. The only answer is a distributed build.

TRADE-OFFS Modules & dependencies 1:1:1, One-Version, pin manually

  • Fine-grained modules (1:1:1): one package/target/BUILD per directory (Pants's term); pays off at scale — faster distributed builds, fewer rebuilds, running only the affected test subset. Minimize visibility.
  • Internal deps: built from source, no "version"; strict transitive deps — fail when referencing a symbol without depending on it (avoids "dependencies as a public contract," speeds builds).
  • External deps: pin versions manually (no latest; discoverable/rollback-able); the One-Version Rule avoids diamond dependencies; Bazel doesn't auto-download transitive deps.
  • Security: mirror to servers you control, require hashes, or vendoring (check into third_party, converting external to internal) — used by Google (relying on a huge monorepo).
Conclusion: limiting engineers' power and flexibility improves productivity; "being able to trust the build is powerful — incremental builds just work, and there's almost never a need to clear caches or run clean."

Key terms

Blaze / Bazel
Google's internal Blaze and its 2015 open-source Bazel; artifact-based, aiming for both speed and correctness.
Fast & correct
A good build system's two goals: one command, seconds; any developer/machine, same inputs → same output.
Task-based build system
The unit is an arbitrary script (task), like Ant/Maven/Gradle; flexible but gives the system too little info.
Artifact-based build system
BUILD files are a declarative manifest; engineers say what, the system decides how (Blaze/Bazel/Pants/Buck).
BUILD / target / workspace
BUILD files define targets (java_binary/java_library with name/srcs/deps); a WORKSPACE file marks the root.
Tools as dependencies / toolchains
Treat tools like the compiler as dependencies (changing them rebuilds); toolchains give platform independence.
Sandboxing
LXC (Docker's tech) lets each action see only its declared I/O, making conflicts impossible and restricting the network.
Remote caching / remote execution
A shared cache avoids rebuilds (ObjFS); actions spread across a worker pool (Forge); prerequisite is reproducible builds.
1:1:1 rule
One package/target/BUILD per directory (Pants's term); prefer fine-grained modules for distributed builds, incrementality, precise testing.
Strict transitive dependencies
Fail when a symbol is referenced without a direct dependency; avoids "dependencies as a public contract," speeds builds.
One-Version Rule / diamond dependency
Only one version of each third-party dependency, avoiding A depending on both v1 and v2 (see ch21).
Vendoring / third_party
Checking external deps into the source tree (third_party), converting external to internal; used by Google (huge monorepo).

Questions to push your thinking

  1. Is your build task-based or artifact-based? Can it safely parallelize and do reliable incremental builds, or are you forced to clean and maddened by "breaks on another machine"? How would you restructure it?
  2. Recall your "no real build system" pain: where are you — DIY scripts / task-based / artifact-based? "Changing the build system gets more expensive as the project grows" — invest early?
  3. Are your external deps pinned manually or latest/range? Burned by diamond dependencies? Apply One-Version + manual + hash: which are a time bomb? Per ch16, ch21.
  4. Are your modules coarse or fine? Tally what "run only the affected test subset" saves your CI. Move toward 1:1:1 and auto-manage dependency lists? Per ch11, ch23.
  5. The chapter says "limiting flexibility improves productivity." Where did you "give people too much power while the system can't guarantee correctness or efficiency"? Which decision, given to automated tools, eases everyone's life?
