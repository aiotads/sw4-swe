CHAPTER 23

Continuous Integration

持續整合
作者 Rachel Tannenbaum PDF p.507–532 Time 時間 Scale 規模 Trade-offs 取捨
"CI is the continuous assembling and testing of our entire complex and rapidly evolving ecosystem." 持續整合，是對我們整個複雜且快速演化的生態系，做持續的組裝與測試。

摘要

CI（持續整合）經典定義＝頻繁整合工作、每次整合都由自動建置（含測試）驗證以盡早抓出問題。但在微服務、RPC、演化的 OS/雲端、資料/ML 模型的現代世界，弄壞你應用的變更越來越不在自家程式碼裡——故新定義：對整個複雜且快速演化的生態系，做持續的組裝與測試。

CI 本質是回答：何時跑哪些測試、每個時點如何組裝 SUT（保真度 vs 建置成本）。四大概念：①快速回饋迴圈（bug 越晚抓成本近乎指數→左移 shift left；回饋要可取得＋可行動）②自動化（CB：在 head 跑建置＋測試，分 true head／green head；CD：持續組裝 RC）③持續測試（隨右移施以更大範圍測試；presubmit 只放快又可靠的）④CI 即告警（flaky 測試＝亂叫的假警報；不可行動就不該告警）。

Google 實作：全球持續建置 TAP（每天 >50,000 變更、>40 億測試案例，monorepo 幾乎所有變更的閘門；通過 presubmit 有 95%+ 機率通過其餘、平均等 ~11 分鐘）＋批次/二分搜尋 culprit finding＋Build Cop＋兩下點擊/高信心自動回退＋依賴圖只跑必要測試。以 Google Takeout 四幕轉型作結，並回應「我負擔不起 CI」：CI 不是新成本，而是把成本左移到更便宜的階段。

為什麼要讀這一篇

你還在問「為什麼不把所有測試都放 presubmit？」 答案是太貴、被無關 flaky 卡住、還有 mid-air collision（兩個碰不同檔的變更合起來弄壞測試）。原則：presubmit 只放快又可靠的，接受覆蓋率損失與一定回退率。
你的端到端套件「幾乎總是紅的」，大家已不看了。 用 hermetic testing 把不穩定打下來（Assistant 改全封閉後 runtime 降 14 倍、近乎零 flaky），用「bug tag 負責任停用＋自動清理」保持綠，而非淪為一堆死測試。
你不知道 CI 回饋該長怎樣才有用。 「accessible & actionable」是關鍵：Takeout 把難讀 log 改成「紅/綠一目了然＋可點 log 連結」，介入除錯比例就降 35%。回饋要好用到能直接修。
你把「CI 全綠」當鐵律、紅了就禁止提交。 「CI 即告警」會鬆動教條：不是每個失敗都等於生產問題；根因清楚且不影響 production 時，硬卡提交不合理。

細節重點

左移 一段程式變更的一生與快速回饋 Life of a code change · shift left

bug 抓得越晚，成本近乎指數上升；CI 鼓勵把驗證盡量左移，用一連串由快到慢的回饋迴圈盡早抓問題。
CB／CD：CB 在 head 跑建置＋測試，「破壞 build」含弄壞編譯與測試；通過＝綠，repo 因而有 true head（最新提交）與 green head（CB 已驗證）兩版。CD 持續把 head 組裝成 RC（多在 green head 切）＝程式＋設定＋依賴的可部署單元；設定要一起測（大量 production bug 來自蠢設定）。RC 跨環境晉級不重編（Docker/Borg 確保一致）。

實作 TAP 持續測試管線與失敗管理 presubmit → post-submit → RC → prod

為何 presubmit 不夠？太貴、被 flaky 卡、會有 mid-air collision。原則：presubmit 只放快又可靠的，其餘左移右接；TAP 容許可能弄壞的變更先落地，再非同步補測。
CI 即告警／hermetic：flaky 測試＝每幾分鐘亂叫的假警報；不可行動就不該告警、沒違反不變量就不該算失敗，追求 100% 綠像追 100% uptime 一樣昂貴。對抗不穩定靠 hermetic testing（完全自包含、有決定性與隔離；fake 是一種封閉後端）與 record/replay（錄/重放，便宜但脆弱，要在 false positive/negative 間拿捏）。Takeout 四幕：沙箱 presubmit（擋 95% 壞設定、夜間失敗減半）／友善 UI＋可點連結（介入除錯 −35%）／對 prod 跑同套做失敗隔離／bug tag 負責任停用＋自動清理（MTTCU）。

回應「我負擔不起 CI」——先算你已經在為 production 救火付的代價。

關鍵字補充

Continuous Integration（CI，持續整合）
頻繁整合＋自動建置驗證以盡早抓問題；現代定義擴及「對整個複雜演化生態系的持續組裝與測試」。
Shift Left（左移）／Fast Feedback Loop
bug 越晚抓成本近乎指數上升，故把驗證往早端移、縮短回饋迴圈。
Continuous Build（CB）／true head vs green head
在 head 跑建置＋測試；true head＝最新提交，green head＝CB 已驗證的最新（開發常 sync green head）。
Continuous Delivery（CD）／Release Candidate（RC）
持續組裝 RC 並晉級各環境；RC＝程式＋設定＋依賴、通過 CB 的可部署單元（設定必須一起測）。
Version Skew（版本偏移）
分散式系統同時存在多個不相容的程式/資料/設定版本；canary 與多版本並存的典型風險。
Mid-air Collision（空中相撞）
兩個碰不同檔的變更各自通過卻合起來弄壞測試；Google 規模幾乎天天發生，是 presubmit 不夠的原因之一。
Presubmit vs Post-submit
presubmit 只放快又可靠的（接受覆蓋率損失與回退率）；post-submit 容忍較長時間與不穩定，補抓漏網。
CI Is Alerting（CI 即告警）
把 SRE 監控/告警原則左移到 CI；不可行動不該告警、沒違反不變量不該算失敗、追 100% 綠像追 100% uptime 一樣昂貴。
Hermetic Testing（封閉測試）
對完全自包含、無外部依賴的環境跑測試；具決定性與隔離性，是降低大型測試不穩定的利器。
Record/Replay（錄製/重放）
錄真後端回應、快取、在封閉環境重放；比啟整個沙箱 stack 便宜，但要在 false positive/negative 間拿捏。
TAP（Test Automation Platform）
Google 全球持續建置；每天 >50,000 變更、>40 億測試案例，是 monorepo 幾乎所有變更的閘門。
Build Cop（建置警長）／Culprit Finding
專人讓專案測試保持綠、壞了就回退；批次＋二分搜尋找出弄壞測試的元兇。

延伸提問

  1. 你的團隊把哪些測試放在 presubmit？它們真的都「快又可靠」嗎，還是有 flaky/慢測試天天卡大家、害人除無關的錯？哪些該下放到 post-submit？你承受過 mid-air collision 嗎？對照 ch11。
  2. 你的端到端套件現在是綠的還是「幾乎總是紅的」？你是用 hermetic testing 把不穩定打下來、用「bug tag 負責任停用＋自動清理」保持綠，還是任由它爛成一堆被註解掉的死測試？對照 ch13、ch14。
  3. 接受「CI 即告警」後：你的「最新 CI 不綠就禁止提交」政策合理嗎？你能區分「真會變成 production 問題的失敗」與「根因清楚、不影響 production 的失敗」嗎？你追的「100% 綠」值那代價嗎？
  4. 你的 CI 回饋夠 actionable 嗎？一個失敗訊息能不能直接帶你到 log、到出錯那一行？回想 Takeout 改可點連結就讓介入除錯降 35%——你的回饋還停在「看得到但不好用」嗎？
  5. 你的設定有沒有跟程式一起進版本控制、一起被測（納入 RC）？回想「大量 production bug 來自蠢設定」——你是把設定當二等公民，還是讓它走同樣的 review 與測試？對照 ch24。

Part IV — Tools
CHAPTER 23

Continuous Integration

Catch problematic changes as early as possible — shift left
by Rachel Tannenbaum PDF p.507–532 Time Scale Trade-offs
"CI is the continuous assembling and testing of our entire complex and rapidly evolving ecosystem." The change that breaks you increasingly lives outside your own code — integrate the whole ecosystem.

Summary

CI classically = integrate work frequently, each integration verified by an automated build (including test) to catch problems early. But in a microservices/RPC world with evolving OS/cloud and data/ML models, the change that breaks you increasingly lives outside your own code — hence the new definition: continuous assembling and testing of the whole complex, rapidly evolving ecosystem.

CI essentially answers: which tests to run when, and how to compose the SUT at each point (fidelity vs setup cost). Four concepts: ① fast feedback loops (bug cost grows almost exponentially → shift left; feedback must be accessible + actionable) ② automation (CB: build + test at head, splitting true head / green head; CD: continuously assemble RCs) ③ continuous testing (larger-scoped as it shifts right; presubmit holds only fast, reliable tests) ④ CI is alerting (a flaky test = a spurious alert; if not actionable, don't alert).

Google's implementation: the global continuous build TAP (>50,000 changes and >4 billion test cases/day, the gateway for almost all monorepo changes; passing presubmit → 95%+ pass the rest; ~11-min average wait) + batching/binary-search culprit finding + a Build Cop + two-click/auto rollback + dependency-graph minimal test sets. It closes with Google Takeout's four-act transformation and answers "I can't afford CI": CI isn't a new cost — it's a cost shifted left to a cheaper stage.

Why read this

You still ask "why not run all tests on presubmit?" Too expensive, blocked by unrelated flakiness, and mid-air collisions (two changes touching different files combine to fail a test). Rule: presubmit holds only fast, reliable tests — accept coverage loss and some rollbacks.
Your end-to-end suite is "almost always red" and nobody looks. Use hermetic testing to crush instability (Assistant cut presubmit runtime 14× with virtually no flakiness) and "tag-based responsible disablement + auto-cleanup" to keep it green instead of a pile of dead tests.
You don't know what useful CI feedback looks like. "Accessible & actionable" is key: Takeout turned indecipherable logs into "red/green at a glance + clickable log links," cutting debugging involvement 35%. Feedback must be usable enough to fix directly.
You treat "CI all green" as law and block commits when red. The "CI is alerting" lens loosens that: not every failure equals a production problem; when the root cause is understood and won't affect production, blocking commits is unreasonable.

Key details

SHIFT LEFT Life of a code change & fast feedback cost rises rightward

The later a bug is caught, the more it costs — almost exponentially; CI moves verification left, using a chain of fastest-to-slowest feedback loops to catch problems early.
CB / CD: CB builds + tests at head — "breaking the build" covers compilation and tests; passing = green, so the repo has true head (latest commit) and green head (CB-verified). CD continuously assembles head into an RC (usually cut at green) = a deployable unit of code + config + deps; test config with it (many production bugs are silly config). RCs aren't recompiled across environments (Docker/Borg enforce consistency).

IN PRACTICE TAP's continuous-testing pipeline & failure management

Why presubmit isn't enough: too expensive, blocked by flakiness, mid-air collisions. Rule: presubmit holds only fast, reliable tests; the rest shifts left/right. TAP lets potentially-breaking changes land first, then tests asynchronously.
CI is alerting / hermetic: a flaky test = a spurious alert firing every few minutes; if not actionable, don't alert; if no invariant violated, it's not a failure, and 100% green is as costly as 100% uptime. Fight instability with hermetic testing (self-contained, deterministic, isolated; a fake is one kind of hermetic backend) and record/replay (cheap but brittle, balancing false positives/negatives). Takeout's four acts: sandboxed presubmit (blocked 95% bad-config, halved nightly failures) / friendly UI + clickable links (−35% debugging involvement) / same suite vs prod for failure isolation / tag-based responsible disablement + auto-cleanup (MTTCU).

Key terms

Continuous Integration (CI)
Frequent integration + automated build verification to catch problems early; modern definition extends to "continuous assembling and testing of the whole evolving ecosystem."
Shift left / fast feedback loop
A bug's cost grows almost exponentially the later it's caught, so move verification to the early end and shorten feedback loops.
Continuous Build (CB) / true head vs green head
Build + test at head; true head = latest commit, green head = latest CB-verified change (dev often syncs green head).
Continuous Delivery (CD) / Release Candidate (RC)
Continuously assemble RCs and promote through environments; an RC = a deployable unit of code + config + deps that passed CB (test config with it).
Version skew
A distributed system simultaneously holding multiple incompatible versions of code/data/config; the classic risk of canarying.
Mid-air collision
Two changes touching different files each pass but combine to break a test; happens most days at Google's scale.
Presubmit vs post-submit
Presubmit holds only fast, reliable tests (accepting coverage loss and rollbacks); post-submit tolerates longer times and instability.
CI is alerting
Shift SRE monitoring/alerting principles to CI; if not actionable don't alert, if no invariant violated it isn't a failure, 100% green is as costly as 100% uptime.
Hermetic testing
Testing against an entirely self-contained environment with no external dependencies; deterministic and isolated.
Record/replay
Record live backend responses, cache, and replay in a hermetic environment; cheaper than a full sandboxed stack but brittle.
TAP (Test Automation Platform)
Google's global continuous build; >50,000 changes and >4 billion test cases/day, the gateway for almost all monorepo changes.
Build Cop / culprit finding
A dedicated person keeping a project's tests green and rolling back breakages; batching + binary search to find the offending change.

Questions to push your thinking

  1. Which tests run on presubmit? Are they all truly "fast and reliable," or are flaky/slow ones blocking everyone and forcing debugging of unrelated failures? Which should move to post-submit? Ever hit a mid-air collision? Per ch11.
  2. Is your end-to-end suite green or "almost always red"? Do you crush instability with hermetic testing, keep it green with "tag-based disablement + auto-cleanup," or let it rot into commented-out dead tests? Per ch13, ch14.
  3. Adopting "CI is alerting": is "no commits unless CI is green" reasonable? Can you distinguish "failures that become production problems" from "failures with a clear root cause that won't"? Is your "100% green" worth its cost?
  4. Is your CI feedback actionable? Can a failure message take you straight to the log, the offending line? Recall Takeout cutting debugging involvement 35% with clickable links — is yours "visible but not usable"?
  5. Is your config in version control and tested with the code (in the RC)? Recall "many production bugs come from silly config" — second-class citizen, or same review and testing? Per ch24.
