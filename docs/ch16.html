CHAPTER 16

Version Control and Branch Management

版本控制與分支管理
作者 Titus Winters PDF p.355–378 Scale 規模 Time 時間 Trade-offs 取捨
"Developers must not have a choice where to commit, or which version of an existing component to depend upon." 工程師不該有「要把程式提交到哪」或「要依賴某元件的哪個版本」的選擇。

摘要

「工具」篇開場。版本控制（VCS）大概是軟體工程界最普遍採用的工具。它把全書主軸具象化——軟體工程＝整合了時間維度的程式設計，而 VCS 正是管理「原始碼 × 時間」的主要工具，可想成擴充檔案系統：VCS(filename, time, branch) => 內容。它真正的價值是讓團隊與組織能擴展。

兩大架構：centralized（集中式，Subversion/Perforce/Piper） 與 distributed（DVCS，Git/Mercurial）。Git 是主流。但核心功能相同、差別在 UX；真正重要的是你放什麼進去、怎麼用——而「怎麼用」幾乎全是政策與慣例。最關鍵的是 Source of Truth（單一事實來源）：集中式內建、DVCS 需靠政策明訂。

核心立場：trunk-based development（主幹開發：一個 repo、沒有開發分支）特別能擴展。長壽 dev branch 是「以為換來穩定、實為生產力拖累」的安全錯覺；正解是重度依賴測試與 CI、保持 build 綠燈、用 runtime flag 關閉未完成功能。release branch 相對良性。Google 在 Piper monorepo 上奉行 One-Version Rule：加依賴時絕不能有「選哪個版本」的選擇，否則 fork 造成鑽石依賴。monorepo 讓 One Version 想違反都難，但不是唯一正解。

為什麼要讀這一篇

你還在開「這週合併策略怎麼喬」的會、有「合併協調員」。 那是長壽 dev branch 的純開銷——分支不換穩定，只把症狀放大。正解：trunk-based＋CI＋feature flag，「branches are a drag on productivity」。
你的團隊為「該用哪個版本」吵過、或被鑽石依賴坑過。 終結它的政策：One-Version Rule——加依賴時不該有選擇。看懂 fork 為何危險（shading 騙得了函式騙不了型別）。
你在猶豫 monorepo 還是 manyrepo、Git 還是別的。 本章把這降為次要：真正重要的是堅守 One-Version 與 Source of Truth。並介紹兩全其美的 VMR（虛擬 monorepo）。
你的新人覺得「回退舊版本」是高成本奢侈品。 那是把「程式設計」誤當「軟體工程」。「VCS＝管理原始碼 × 時間的主要工具」一句話幫你說清為何它不可或缺。

細節重點

本質 What Is Version Control? 原始碼 × 時間

VCS 追蹤檔案隨時間的版本；早期靠鎖檔、進階靠原子提交（atomicity）。可想成 VCS(filename, time, branch) => 內容——把時間變成明確的一部分（程式設計裡多餘，軟體工程裡關鍵）。
抗拒它，幾乎來自混淆「程式設計」與「軟體工程」——新人把「整個專案的 undo」看成多餘奢侈品；但 VCS 真正價值是橋接單人與多人流程、讓組織擴展，並順帶滿足稽核、觸發提交前的反思（跑 static analysis、測試）。

架構 Centralized vs Distributed、Source of Truth 集中 vs 分散、事實來源

  • 核心功能相同、差別在 UX：集中式（Subversion/Perforce/Piper）有單一中央 repo；DVCS（Git/Mercurial）任一 clone 都是可提交的 repo。Git 是主流，沒把握就用它。
  • Source of Truth：集中式內建（trunk 最新提交＝當前版本）；DVCS 沒有內建，需明訂「某 repo 的某 branch」為事實來源，否則退回「Presentation v5 - final - redlines - Josh's version v2」的混亂。沒有它，總會有人去維護一份清單——那就是在重造一個集中式事實來源。
  • Google 為何用集中式：DVCS 要傳整個歷史與 metadata，在 5 萬工程師/數千萬提交的規模幾乎全是浪費；集中＋雲端儲存對擴展至關重要。

分支 Trunk-based vs Dev Branch 分支管理的核心

WIP ≈ 一個分支。長壽 dev branch 被認為「以為換來穩定、實為生產力拖累」；正解是 trunk-based。

規則 One-Version Rule、Fork 的危險 加依賴時不該有選擇

在 Single Source of Truth 之上：「每個依賴只能有一個版本可選」。根本規則：工程師加新依賴時，絕不該有「該依賴哪個版本」的選擇。對個人像任意阻礙，對組織卻是高效擴展的關鍵。
(幾乎)沒有長壽分支：新 Widget 該提交到 trunk、runtime 關閉、用 visibility 隱藏，而非活在平行分支讓「選擇」偷溜回來。Google 約 1000 個團隊中只有不到 10 個有長壽 dev branch；build horizon 每 6 個月封頂版本偏移。

未來 Monorepo 與未來 VMR・版本號是時間戳

monorepo 讓 One Version 想違反都難，但不是唯一正解（開源用 manyrepo 也好）；重點不在 monorepo 而在盡可能堅守 One-Version。工具正提供「混合細粒度 repo 與 monorepo」的 VMR（虛擬 monorepo）（Git submodules/Bazel/CMake）。一句收束：版本號其實是時間戳；允許版本偏移＝加進「時間」這個維度複雜度，而 One-Version 能砍掉它。

關鍵字補充

Version Control System（VCS）
追蹤檔案隨時間的版本；可視為 VCS(filename, time, branch) => 內容 的擴充檔案系統。管理「原始碼 × 時間」的主要工具。
Centralized vs Distributed
集中式（Subversion/Perforce/Piper）有單一中央 repo；DVCS（Git/Mercurial）任一 clone 都是可提交的 repo。核心功能相同，差別在 UX。
Source of Truth（單一事實來源）
哪個 repo 的哪條 branch 是權威版本；集中式內建、DVCS 需靠政策明訂；有相對性。
Trunk-based Development（主幹開發）
一個 repo、沒有（長壽）開發分支；小增量直接提交 trunk＋CI＋feature flag。DORA 證實與高效組織強正相關。
Dev Branch / Release Branch
dev branch 長壽功能分支（誤導，實為拖累）；release branch 代表發布 build 的確切程式，相對良性、預期被拋棄。
One-Version Rule（單一版本規則）
加依賴時不該有「選哪個版本」的選擇；每個依賴只能有一個版本可選。對組織高效擴展至關重要。
Fork without renaming
違反 One Version 的典型；造成鑽石依賴、build 失敗或執行期 bug（見 ch21）。
Shading（遮蔽）
Java 把內部依賴改名以隱藏；對函式可行，對跨套件傳遞的型別失敗——只是 patch「需要多版本」的底層問題。
Monorepo / VMR
monorepo 讓 One Version 想違反都難但非唯一正解；VMR 用 Git submodules/Bazel/CMake 縫合細粒度 repo，逼近 monorepo 功能。
Piper
Google 自製集中式 VCS，作為分散式微服務、80TB+、約 5 萬工程師、每工作日 6–7 萬次提交。
Build Horizon（建置地平線）
每個 production job 最多每 6 個月重建重部署；封頂了潛在的版本偏移。

CHAPTER 16

Version Control and Branch Management

No choice of where to commit or which version to depend on
by Titus Winters PDF p.355–378 Scale Time Trade-offs
"Developers must not have a choice where to commit, or which version of an existing component to depend upon." Give developers no choice of where to commit or which version to depend on.

Summary

The "Tools" part opens. Version control (VCS) is perhaps the most universally adopted tool in software engineering. It makes the book's thesis concrete — software engineering is programming integrated over time — and VCS is the primary tool for managing "source × time," an extended filesystem: VCS(filename, time, branch) => contents. Its real value is letting teams and orgs scale.

Two architectures: centralized (Subversion/Perforce/Piper) and distributed (DVCS) (Git/Mercurial). Git dominates. But the core function is the same, differing in UX; what matters is what you put in and how you use it — and "how" is almost entirely policy and convention. The key is the Source of Truth: baked in for centralized, policy-defined for DVCS.

Central stance: trunk-based development (one repo, no dev branches) is particularly scalable. Long-lived dev branches are a "perceived safety crutch that's actually a productivity drag"; the fix is heavy tests and CI, keep the build green, disable incomplete features at runtime. Release branches are relatively benign. On the Piper monorepo, Google enforces the One-Version Rule: a developer adding a dependency must never have a choice of "which version", or a fork causes diamond dependencies. A monorepo makes One Version hard to violate, but it's not the one true way.

Why read this

You still hold "what's this week's merge strategy" meetings with a "merge coordinator." That's pure overhead from long-lived dev branches — branches don't buy stability, they amplify the symptom. The fix: trunk-based + CI + feature flags, "branches are a drag on productivity."
Your team has argued over "which version" or been burned by diamond dependencies. The policy that ends it: the One-Version Rule — no choice when adding a dependency. See why forks are dangerous (shading fools functions, not types).
You're torn between monorepo and manyrepo, Git or other. The chapter demotes this to secondary: what matters is adhering to One Version and a Source of Truth. It introduces the best-of-both-worlds VMR (virtual monorepo).
Your newcomers see "rolling back" as a high-overhead luxury. That's mistaking "programming" for "software engineering." "VCS = the primary tool for managing source × time" helps you explain why it isn't optional.

Key details

NATURE What Is Version Control? source × time

A VCS tracks revisions over time; early systems used locking, advanced ones atomicity. Think VCS(filename, time, branch) => contents — making time explicit (unnecessary in programming, critical in software engineering).
Resistance comes from conflating "programming" with "software engineering" — newcomers see a "project-wide undo" as a luxury; but VCS's real value is bridging single- and multi-developer processes, letting orgs scale, plus audit needs and a commit-time moment of reflection (run static analysis, tests).

ARCHITECTURE Centralized vs Distributed, Source of Truth

  • Same core function, different UX: centralized (Subversion/Perforce/Piper) has one central repo; DVCS (Git/Mercurial) makes any clone committable. Git dominates; when in doubt, use it.
  • Source of Truth: baked in for centralized (latest at trunk = current); DVCS has none inherently, so declare "one branch in one repo" or regress to "Presentation v5 - final - redlines - Josh's version v2." Without it, someone keeps a list — reproducing a centralized source of truth.
  • Why Google is centralized: DVCS transmits full history/metadata — almost all waste at 50,000 engineers / tens of millions of commits; centrality and in-cloud storage are critical to scaling.

BRANCHES Trunk-based vs Dev Branch

WIP ≈ a branch. Long-lived dev branches are "presumed stability but actually a productivity drag"; the fix is trunk-based.

RULE One-Version Rule, the danger of forks no choice when adding a dependency

On top of Single Source of Truth: "only one version of each dependency to choose." The core rule: a developer adding a dependency must never have a choice of "which version." An arbitrary impediment to one developer, but critical for efficient organizational scaling.
(Nearly) no long-lived branches: a new Widget should be committed to trunk, disabled at runtime, hidden by visibility — not in a parallel branch where "choice" sneaks back. Of ~1,000 teams, fewer than 10 have a long-lived dev branch; a 6-month build horizon caps version skew.

FUTURE Monorepo & the future VMR · version numbers are timestamps

A monorepo makes One Version hard to violate, but isn't the one true way (open source thrives on manyrepo); what matters is adhering to One Version. Tools offer a VMR (virtual monorepo) blending fine-grained repos with monorepo behavior (Git submodules/Bazel/CMake). The closer: version numbers are really timestamps; allowing version skew adds the dimensional complexity of time, which One Version cuts down.

Key terms

Version control system (VCS)
Tracks revisions over time; an extended filesystem mapping VCS(filename, time, branch) => contents. The primary tool for managing source × time.
Centralized vs distributed
Centralized (Subversion/Perforce/Piper) has one central repo; DVCS (Git/Mercurial) makes any clone committable. Same core function, different UX.
Source of Truth
Which branch in which repo is authoritative; baked in for centralized, policy-defined for DVCS; relative.
Trunk-based development
One repo, no (long-lived) dev branches; small increments straight to trunk + CI + feature flags. DORA-correlated with high performance.
Dev branch / release branch
A dev branch is a long-lived feature branch (misguided, a drag); a release branch represents the release build's exact code, relatively benign, eventually abandoned.
One-Version Rule
No choice of "which version" when adding a dependency; only one version of each to choose. Critical for efficient organizational scaling.
Fork without renaming
The classic One-Version violation; causes diamond dependencies, build failures, or runtime bugs (see ch21).
Shading
Java renaming internal dependencies to hide them; works for functions but fails for cross-package types — a patch over "multiple versions needed."
Monorepo / VMR
A monorepo makes One Version hard to violate but isn't the only answer; a VMR stitches fine-grained repos (Git submodules/Bazel/CMake) to approximate it.
Piper
Google's custom centralized VCS, a distributed microservice, 80TB+, ~50,000 engineers, 60,000–70,000 commits per workday.
Build horizon
Every production job is rebuilt and redeployed at most every 6 months; caps potential version skew.

Questions to push your thinking

  1. Does your team hold "merge strategy" meetings, have a "merge coordinator," or maintain long-lived dev branches? Tally the overhead — what disappears under trunk-based + CI + feature flags? Can you solve "trunk instability" with tests rather than branches? Per ch23.
  2. In your codebase, does adding a dependency offer a choice of "which version"? Burned by an un-renamed fork or diamond dependency? Which packages are secretly a time bomb? Per ch21.
  3. Do you have a clear Source of Truth? Or "after pulling a teammate's repo, I'm unsure I have all changes"? Where does a newcomer get a "known-good" copy? Does it scale sublinearly?
  4. Choosing monorepo vs manyrepo — the chapter says that's secondary. Refocus on "can I adhere to One Version and a Source of Truth?": does your repo structure help or hurt? Is a VMR your best of both worlds?
  5. Does your team treat version control as a luxury "project-wide undo," or "the primary tool for managing source × time"? Is anyone mistaking "software development" for "software engineering"? Ties to ch01.
